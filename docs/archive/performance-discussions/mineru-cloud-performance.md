# MinerU 云端解析性能瓶颈与优化方向

## 背景

当前离线入库链路中，教材 PDF 解析依赖云端 MinerU。项目已完成大文件分片解析优化：本地先对 PDF 做体检，满足阈值后按页拆分为多个 shard，并发提交云端解析，最后合并为统一的原始文档块与全局页码。

默认配置口径：

```yaml
parser:
  cloud:
    sharding:
      enabled: true
      min_pages: 120
      min_file_mb: 80
      pages_per_shard: 20
      max_concurrency: 4
      text_sample_pages: 5
```

## 当前主要瓶颈

1. 云端 OCR 与版面分析耗时不可控

   对扫描版教材、图片密集型 PDF、表格密集型 PDF，MinerU 云端处理时间主要消耗在 OCR、版面分析、图片/表格识别和结果打包阶段。OSS 上传通常不是主要瓶颈。

2. 单个大 PDF 容易放大尾部等待

   如果整本教材作为一个云端任务提交，某几页复杂页面会拖慢整个任务完成时间，也更容易遇到超时、连接断开或结果 ZIP 下载失败。

3. 云端并发与限流边界不透明

   当前本地可以把大文件拆成多个 shard 并发提交，但云端是否限流、限流阈值是多少、失败后是否适合立即重试，都需要用真实教材压测确认。

4. 失败恢复粒度仍偏粗

   当前分片只是内部执行单元，最终仍合并成一个文档。后续如果某个 shard 失败，需要进一步支持 shard 级断点续跑、失败 shard 重试和前端 shard 进度展示。

## 已有优化

- 大文件自动分片，避免单个 MinerU 云任务过大。
- 多 worker 并发解析 shard，降低总等待时间。
- 合并时修正全局页码，保持用户看到的页码仍对应原 PDF。
- 每个 shard 使用独立输出目录，避免图片文件覆盖。
- OSS object 名称包含 shard 标识，避免多 shard 上传覆盖。
- MinerU 提交和轮询阶段已有网络瞬断重试。

## 可讨论的优化方向

### 方向一：调优 shard 粒度与并发

需要用真实教材分别测试：

- `pages_per_shard = 10 / 20 / 30`
- `max_concurrency = 1 / 2 / 4`
- 扫描件、文字型 PDF、图表密集 PDF 的耗时差异

初步判断：

- shard 太大：单任务仍慢，失败代价高。
- shard 太小：任务数量变多，提交/轮询/合并开销变大，也更可能触发限流。
- 并发太高：本地等待缩短，但云端限流或网络错误概率可能上升。

### 方向二：引入 shard 级断点续跑

把每个 shard 的状态持久化，至少记录：

- shard 页码范围
- OSS object
- MinerU task id
- parse status
- result url
- 本地输出路径
- error message

这样失败后只重试失败 shard，不需要重跑整本教材。

### 方向三：解析 Provider 可切换

保留 302AI MinerU 作为默认 provider，同时引入阿里 Document Mind 或其他解析服务做 POC。短期建议只做显式切换和 A/B 对比，不默认自动 fallback。

对比维度：

- 解析速度
- 表格还原质量
- 图片定位质量
- 章节结构质量
- 失败率
- 成本
- 输出格式适配成本

### 方向四：按 PDF 类型选择策略

入库前先做轻量体检：

- 页数
- 文件大小
- 是否扫描件
- 文本层密度
- 图片占比
- 表格/公式密集程度

然后选择不同策略：

- 文字型 PDF：优先轻量解析，减少 OCR。
- 扫描件：走 OCR 强解析，并降低并发避免云端失败。
- 图表密集教材：保守 shard 粒度，优先保证图片路径和页码稳定。
- 超大教材：强制分片，并启用 shard 级恢复。

## 初步建议

第一阶段先不要急着替换 MinerU，而是补齐可观测性和压测数据：

1. 记录每个 shard 的上传、提交、轮询、下载、解析合并耗时。
2. 用 2-3 本真实教材跑不同 shard 参数。
3. 找出云端限流和失败概率的拐点。
4. 再决定是继续调 MinerU，还是推进 Provider A/B。

## 如果云端 MinerU 继续阻塞，推荐路线

如果实际使用中确认云端 MinerU 已经成为入库链路的主要阻塞，不建议只靠继续调大并发硬扛。更稳妥的路线是把“解析”从单一 Provider 改成可切换、可分流、可降级的解析层。

### 短期：继续保留 MinerU，但降低阻塞感

1. 解析任务后台化

   大文件上传后立即创建任务，前端显示分阶段进度，不要求用户停在当前页面等待。任务状态至少拆成：

   - 文件准备
   - PDF 体检
   - shard 上传
   - 云端解析
   - 结果下载
   - 合并
   - 清洗 / 切片 / 向量化

2. shard 级状态持久化

   当前已支持内部 shard 并发解析，但失败恢复粒度还不够。下一步应把 shard 状态落库或落 Redis，失败后只重跑失败 shard。

3. 允许部分结果先预览

   对超大教材，可以考虑 shard 解析完成一批就先进入草稿预览。这样用户能先检查前几章，不必等整本书完成。

4. 解析质量分档

   增加“快速解析 / 标准解析 / 高精度解析”三档：

   - 快速解析：优先文本层和基本版面，少做重 OCR。
   - 标准解析：当前默认策略。
   - 高精度解析：图表、公式、扫描件优先，允许更慢。

### 中期：做混合解析，不让所有 PDF 都走重 MinerU

1. 文字型 PDF 走轻量本地解析

   如果 PDF 自带文本层且文本密度足够，优先用本地 PyMuPDF / pdfplumber 提取文本、页码和基础块结构。只有图片页、表格页或低置信页再交给云端 OCR。

2. 扫描件 / 图表密集页才走云端重解析

   先按页判断：

   - 文本层为空或很少：走 OCR。
   - 图片占比高：走 OCR / 视觉解析。
   - 普通正文页：走本地文本抽取。

   这会把“整本书都慢”变成“只有困难页面慢”。

3. 表格单独 Provider

   教材里的表格如果对问答很关键，可以把表格页送到更擅长表格结构化的 Provider，而正文仍走轻量解析。

4. 图片说明异步补全

   图像理解、图注生成、图表摘要可以后台异步做，不阻塞文档进入基础检索。

### 长期：解析 Provider 可切换

建议实现统一接口：

```text
PDFParserProvider
  -> parse(file, options) -> ParsedDocument
```

候选 Provider：

| 方案 | 适合场景 | 优点 | 风险 |
| --- | --- | --- | --- |
| 302AI MinerU 云端 | 当前默认链路、教材结构解析 | 已接入，输出适配成本最低 | 大文件慢，云端排队和失败不可控 |
| 本地 MinerU / 自建 MinerU | 数据安全、批量离线处理 | 可控，不受第三方排队影响 | GPU/环境成本高，部署维护重 |
| 阿里 Document Mind | 国内云服务、文档结构化解析 A/B | 国内访问稳定性可能更好，适合做 POC | 输出格式需要适配，质量需实测 |
| Azure Document Intelligence | 版面、表格、段落结构 | 成熟云服务，结构化输出强 | 国内访问、成本、合规和账号体系需评估 |
| AWS Textract | 表格、表单、OCR | 异步分析能力成熟 | 教材/中文/复杂版面效果需实测 |
| Mistral OCR | 快速转 Markdown / 结构化文本 | 面向 RAG 的文档 OCR 能力明确 | 国内网络、中文教材质量和页码/图片定位需实测 |
| 本地轻量解析 PyMuPDF/pdfplumber | 有文本层 PDF | 快、便宜、可控 | 扫描件、复杂表格和阅读顺序弱 |

## 推荐取舍

我倾向于把下一步拆成两条并行但轻量的线：

1. 先做“混合解析快速通道”

   入库前检测 PDF 是否有可用文本层。能本地解析的页面不送 MinerU，只把疑难页送云端。这条线最可能直接降低等待时间。

2. 同时做 Provider POC

   用同一批真实教材对比 MinerU、阿里 Document Mind、Mistral OCR 或其他候选服务。不要先大规模替换，先看：

   - 端到端耗时
   - 页码稳定性
   - 标题层级
   - 表格还原
   - 图片路径 / 图注
   - chunk 后召回效果

如果只选一个最优先动作，建议先做 **文字型 PDF 本地快速解析 + 扫描页云端兜底**。它不推翻现有 MinerU 链路，但能把很多不需要 OCR 的页面从云端阻塞里拿出来。

## 待确认问题

- 真实教材中，最常见文件大小和页数范围是多少？
- 用户更不能接受的是“慢”，还是“失败后要重来”？
- 是否允许同一本教材后台解析数十分钟，但前端持续显示精确进度？
- 是否希望大文件解析完成前先允许部分章节进入预览或问答？
