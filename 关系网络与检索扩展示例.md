# 关系网络与检索扩展示例（大白话版）

## 一、先搞清楚几个基本概念

### 1.1 什么是 chunk？

**大白话**：chunk 就是把一篇长文章切成的小段。

**为什么切？**
- 文章太长，不好检索
- 切成小段，每段好理解、好检索
- 就像把一本书切成一页一页的笔记

**例子**：

```
原始文章（5000字）:
┌─────────────────────────────────────────────────────────────┐
│ 3.2 神经元的结构与功能                                       │
│ 神经元是神经系统结构和功能的基本单位，主要由细胞体、树突和    │
│ 轴突组成。如图3-1所示，树突负责接收信息，轴突负责传导神经    │
│ 冲动。神经元可分为感觉神经元、运动神经元和中间神经元...      │
│ （后面还有4000多字）                                        │
└─────────────────────────────────────────────────────────────┘

切成 chunks:
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│ chunk-001     │  │ chunk-002     │  │ chunk-003     │
│ "神经元是..."  │  │ "树突负责..." │  │ "如图3-1..."  │
└───────────────┘  └───────────────┘  └───────────────┘
```

---

### 1.2 什么是 parent、child、enhanced 三层？

**大白话**：这是三种不同"身份"的 chunk。

| 身份 | 大白话解释 | 例子 |
|------|-----------|------|
| **parent** | 章节标题，充当"文件夹" | "3.2 神经元的结构与功能" |
| **child** | 正文内容，真正检索用的 | "神经元由细胞体、树突和轴突组成" |
| **enhanced** | child 的"精简版摘要"，更容易被搜到 | "神经元由三部分构成：细胞体、树突、轴突" |

**为什么要有 enhanced？**

假设用户问："神经元的组成部分是什么？"

```
原始 child 内容：
"神经元是神经系统结构和功能的基本单位，主要由细胞体、树突和轴突组成，
其中树突呈树枝状，可以接收其他神经元传来的信息..."

问题：内容太多，用户问法和原文不一样，可能搜不到。
```

```
enhanced 内容：
"神经元由细胞体、树突和轴突三部分组成。
可能的问题：神经元有哪些组成部分？神经元由什么构成？"

好处：更简洁，还预判了用户可能的问法，更容易被搜到。
```

---

### 1.3 enhanced 和 child 的关系是什么？

**大白话**：enhanced 是 child 的"孩子"，child 是 enhanced 的"爹"。

**数据结构上**：

```python
# child chunk
{
    "id": "child-001",
    "content": "神经元是神经系统结构和功能的基本单位...",
    "layer": "child"
}

# enhanced chunk（child-001 的孩子）
{
    "id": "enhanced-001",
    "content": "神经元由细胞体、树突和轴突三部分组成...",
    "layer": "enhanced",
    "parent_id": "child-001"  # 这就是关系！指向它爹 child-001
}
```

**为什么要这样关联？**

想象这个场景：

```
用户问："神经元由什么组成？"

系统检索到 enhanced-001：
"神经元由细胞体、树突和轴突三部分组成"

但用户可能还想看原文，怎么办？
→ 通过 parent_id = "child-001" 找到原文
→ 把原文也一起返回给用户

最终返回给用户：
1. 摘要（enhanced）：神经元由细胞体、树突和轴突三部分组成
2. 原文（child）：神经元是神经系统结构和功能的基本单位...
```

## 二、为什么要建立关系？

### 2.1 没有关系的痛点

**场景1**：用户问"如图3-1所示的内容是什么？"

```
如果没有关系：
- 搜到正文 chunk："如图3-1所示，树突负责接收信息..."
- 但图片在另一个 chunk 里
- 用户看不到图片，只能看到"如图3-1所示"这几个字
- 用户懵了：图呢？？？
```

**场景2**：用户问"神经元的分类表格内容是什么？"

```
如果没有关系：
- 搜到正文 chunk："具体分类见表3-1。"
- 但表格在另一个 chunk 里
- 用户看不到表格内容
- 用户懵了：表呢？？？
```

**场景3**：用户问"染色后下一步做什么？"

```
如果没有关系：
- 搜到"第三步：滴加染液染色"
- 但不知道下一步是什么
- 用户得自己翻书找
```

---

### 2.2 建立关系后的好处

**大白话**：关系就像是"线索"，帮你找到相关的东西。

```
命中一个 chunk → 顺着关系 → 带出其他相关 chunk
```

**回到上面的场景**：

**场景1（图片引用）**：

```
用户问："如图3-1所示的内容是什么？"

1. 搜到正文 chunk：
   "如图3-1所示，树突负责接收信息..."
   
2. 发现它有 refers_to 关系 → 指向图片 chunk

3. 把图片 chunk 也带出来：
   "图3-1 神经元结构示意图" + 图片路径
   
4. 最终返回：
   - 正文内容
   - 图片内容
   - 图片文件（可以展示）
```

**场景2（表格引用）**：

```
用户问："神经元的分类表格内容是什么？"

1. 搜到正文 chunk：
   "具体分类见表3-1。"
   
2. 发现它有 refers_to 关系 → 指向表格 chunk

3. 把表格 chunk 也带出来：
   表3-1 神经元分类
   类型     功能     位置
   感觉神经元  接收刺激   外周
   运动神经元  传导冲动   中枢
   
4. 最终返回：
   - 正文内容
   - 表格完整内容
```

**场景3（步骤关系）**：

```
用户问："染色后下一步做什么？"

1. 搜到 chunk：
   "第三步：滴加亚甲基蓝染液，染色3-5分钟。"
   
2. 发现它有 next_step 关系 → 指向下一步

3. 把下一步也带出来：
   "第四步：用吸水纸吸去多余染液，加盖玻片。"
   
4. 最终返回：
   - 当前步骤
   - 下一步骤（甚至可以返回整个步骤链）
```

## 三、关系是怎么建立的？（逐个举例）

### 3.1 refers_to（引用关系）

**大白话**：正文里提到"如图X"、"如表X"，就和那个图/表建立关系。

**怎么判断的？**

```python
# 代码做的事（简化版）：

1. 扫描所有图片 chunk，提取编号：
   图片 chunk-002 的内容："图3-1 神经元结构示意图"
   → 提取出编号 "3-1"
   → 记录：图编号 "3-1" 对应 chunk-002

2. 扫描所有表格 chunk，提取编号：
   表格 chunk-004 的内容："表3-1 神经元分类..."
   → 提取出编号 "3-1"
   → 记录：表编号 "3-1" 对应 chunk-004

3. 扫描所有正文 chunk，找引用：
   正文 chunk-001 的内容："...如图3-1所示..."
   → 发现提到了 "图3-1"
   → 查到 "图3-1" 对应 chunk-002
   → 建立关系：chunk-001 ↔ chunk-002
   
   正文 chunk-003 的内容："...见表3-1。"
   → 发现提到了 "表3-1"
   → 查到 "表3-1" 对应 chunk-004
   → 建立关系：chunk-003 ↔ chunk-004
```

**建立后的数据结构**：

```python
# 正文 chunk-001
{
    "id": "child-001",
    "content": "如图3-1所示，树突负责接收信息...",
    "relations": [
        {"target_id": "child-002", "rel_type": "refers_to", "evidence": "如图3-1所示"}
    ]
}

# 图片 chunk-002
{
    "id": "child-002",
    "content": "图3-1 神经元结构示意图",
    "is_image_chunk": True,
    "image_path": "data/output/images/page12_img1.png",
    "relations": [
        {"target_id": "child-001", "rel_type": "refers_to", "evidence": "如图3-1所示"}
    ]
}
```

**检索时怎么用？**

```python
# 伪代码
def search_with_relations(query):
    # 1. 先搜最相关的 chunk
    main_chunk = vector_search(query)  # 搜到 child-001
    
    # 2. 看它有什么关系
    related_chunks = []
    for relation in main_chunk.relations:
        if relation.rel_type == "refers_to":
            # 找到它引用的图片/表格
            target = find_chunk_by_id(relation.target_id)
            related_chunks.append(target)
    
    # 3. 返回主内容 + 相关内容
    return {
        "main": main_chunk,
        "related": related_chunks  # 图片/表格
    }
```

### 3.2 sibling（兄弟关系）

**大白话**：同一个章节下的内容，互称"兄弟"。

**为什么要建立这个关系？**

```
场景：
用户问："神经元的结构是什么？"

搜到正文 chunk："神经元由细胞体、树突和轴突组成。"

但同一章节里还有：
- 图片 chunk：神经元结构示意图
- 表格 chunk：神经元分类表

如果建立了 sibling 关系，就能把这些"兄弟"都带出来，
给用户更完整的答案。
```

**怎么判断的？**

```python
# 代码做的事（简化版）：

1. 按 parent_id（章节）分组：
   parent-001 下的 child：
   - child-001（正文）
   - child-002（图片）
   - child-003（正文）
   - child-004（表格）

2. 同一组的正文和媒体块互建关系：
   child-002（图片）和 child-001、child-003 建立 sibling
   child-004（表格）和 child-001、child-003 建立 sibling
```

**建立后的数据结构**：

```python
# 图片 chunk-002
{
    "id": "child-002",
    "content": "图3-1 神经元结构示意图",
    "parent_id": "parent-001",
    "relations": [
        # 既有 refers_to（被引用）
        {"target_id": "child-001", "rel_type": "refers_to"},
        # 又有 sibling（同章节）
        {"target_id": "child-001", "rel_type": "sibling"},
        {"target_id": "child-003", "rel_type": "sibling"},
    ]
}
```

### 3.3 next_step / prev_step（步骤关系）

**大白话**：像链条一样，把步骤串起来。

**为什么要建立这个关系？**

```
场景：
用户问："怎么做神经细胞染色观察？"

搜到："第三步：滴加亚甲基蓝染液染色"

如果建立了 next_step 关系：
→ 第四步：用吸水纸吸去多余染液
→ 第五步：加盖玻片
→ 然后观察

用户就能看到完整的步骤链！
```

**怎么判断的？**

```python
# 代码做的事（简化版）：

1. 识别"步骤型"chunk：
   开头是 "第一步"、"第1步"、"1."、"1)"、"首先"、"然后" 等

2. 同一章节内，按顺序连接：
   第一步 → next_step → 第二步 → next_step → 第三步...
```

**建立后的数据结构**：

```python
# 第一步
{
    "id": "child-001",
    "content": "第一步：取洁净载玻片...",
    "is_procedure_chunk": True,
    "procedure_order": 1,
    "relations": [
        {"target_id": "child-002", "rel_type": "next_step"}
    ]
}

# 第二步
{
    "id": "child-002",
    "content": "第二步：用牙签刮取细胞...",
    "is_procedure_chunk": True,
    "procedure_order": 2,
    "relations": [
        {"target_id": "child-001", "rel_type": "prev_step"},
        {"target_id": "child-003", "rel_type": "next_step"}
    ]
}

# 第三步
{
    "id": "child-003",
    "content": "第三步：滴加染液染色...",
    "is_procedure_chunk": True,
    "procedure_order": 3,
    "relations": [
        {"target_id": "child-002", "rel_type": "prev_step"},
        {"target_id": "child-004", "rel_type": "next_step"}
    ]
}
```

**检索时怎么用？**

```python
def search_procedure(query):
    # 1. 搜到某个步骤
    step = vector_search(query)  # 搜到第二步
    
    # 2. 顺着 next_step 找后面的步骤
    forward = []
    current = step
    while has_relation(current, "next_step"):
        next_step = get_related_chunk(current, "next_step")
        forward.append(next_step)
        current = next_step
    
    # 3. 顺着 prev_step 找前面的步骤
    backward = []
    current = step
    while has_relation(current, "prev_step"):
        prev_step = get_related_chunk(current, "prev_step")
        backward.append(prev_step)
        current = prev_step
    
    # 4. 返回完整步骤链
    return {
        "before": backward,      # 前面的步骤
        "current": step,         # 当前步骤
        "after": forward         # 后面的步骤
    }
```

### 3.4 semantic_similar / duplicate_of（语义相似）

**大白话**：意思相近的内容，即使在不同章节，也连起来。

**为什么要建立这个关系？**

```
场景：
不同章节都提到了"神经元"：

章节3.2："神经元是神经系统结构和功能的基本单位..."
章节5.1："神经细胞即神经元，是构成神经系统的主要成分..."
章节7.3："神经元由细胞体、树突和轴突三部分组成..."

如果建立了语义相似关系，当用户问"神经元是什么"时，
即使只搜到其中一个，也能把其他章节的相关内容带出来！
```

**怎么判断的？**

```python
# 代码做的事（简化版）：

1. 计算所有 chunk 的向量（embeddings）

2. 计算两两之间的相似度（余弦相似度）

3. 相似度 ≥ 0.85 的，建立 semantic_similar 关系
   相似度 ≥ 0.95 的，建立 duplicate_of 关系（近似重复）
```

**例子**：

```python
# chunk-001（章节3.2）
{
    "id": "child-001",
    "content": "神经元是神经系统结构和功能的基本单位...",
    "parent_id": "parent-001"
}

# chunk-005（章节5.1，相似度 0.87）
{
    "id": "child-005",
    "content": "神经细胞即神经元，是构成神经系统的主要成分...",
    "parent_id": "parent-002",
    "relations": [
        {"target_id": "child-001", "rel_type": "semantic_similar", "weight": 0.87}
    ]
}

# chunk-008（章节7.3，相似度 0.91）
{
    "id": "child-008",
    "content": "神经元由细胞体、树突和轴突三部分组成...",
    "parent_id": "parent-003",
    "relations": [
        {"target_id": "child-001", "rel_type": "semantic_similar", "weight": 0.91}
    ]
}
```

**检索时怎么用？**

```python
def search_with_semantic(query):
    # 1. 搜到最相关的内容
    main = vector_search(query)  # 搜到 child-001
    
    # 2. 找语义相似的内容
    similar = []
    for relation in main.relations:
        if relation.rel_type in ["semantic_similar", "duplicate_of"]:
            similar.append(find_chunk_by_id(relation.target_id))
    
    # 3. 返回：主内容 + 相似内容（可能跨章节）
    return {
        "main": main,
        "similar": similar  # 其他章节的相似内容
    }
```

### 3.5 cause_of / effect_of（因果关系）

**大白话**：A 导致了 B，A 是原因，B 是结果。

**为什么要建立这个关系？**

```
场景：
用户问："为什么膜电位会上升？"

如果没有因果关系：
- 搜到"膜电位迅速上升"，但不知道原因

如果有因果关系：
- 搜到"膜电位迅速上升"
- 顺着 effect_of 关系找到原因：
  "钠离子通道大量开放，钠离子快速内流"
- 答案更完整！
```

**怎么判断的？**

```python
# 代码做的事（简化版）：

1. 扫描每个 chunk，找因果标记词：
   "因为"、"由于"、"导致"、"使得"、"引起"、"造成"、"所以"、"因此"

2. 如果当前 chunk 有因果标记词：
   - 它可能是"结果"
   - 前一个 chunk 可能是"原因"
   - 建立关系：前一个 → cause_of → 当前
```

**例子**：

```python
# 原因 chunk
{
    "id": "child-001",
    "content": "钠离子通道大量开放，钠离子快速内流",
    "relations": [
        {"target_id": "child-002", "rel_type": "cause_of"}  # 导致了 child-002
    ]
}

# 结果 chunk
{
    "id": "child-002",
    "content": "由于钠离子快速内流，导致膜电位迅速上升...",
    "relations": [
        {"target_id": "child-001", "rel_type": "effect_of"}  # 是 child-001 的结果
    ]
}
```

### 3.6 enhanced 继承 child 的关系

**大白话**：child 有什么关系，它的孩子 enhanced 也就有什么关系。

**为什么要这样？**

```
场景：
child-001 有关系：
- refers_to → 图片 chunk-002
- sibling → 表格 chunk-004

enhanced-001 是 child-001 的孩子

如果 enhanced-001 不继承关系：
- 搜到 enhanced-001 时，不知道相关图片/表格在哪

如果继承关系：
- 搜到 enhanced-001 时，也能找到图片/表格
- 就像"儿子继承父亲的遗产"
```

**怎么实现的？**

```python
# 代码做的事（简化版）：

for chunk in chunks:
    if chunk.layer != "enhanced":
        continue
    
    # 找它爹（parent_id 指向的 child）
    parent = find_chunk_by_id(chunk.parent_id)
    
    # 继承爹的所有关系
    for relation in parent.relations:
        chunk.relations.append(relation)  # 复制过来
```

**例子**：

```python
# child-001 的关系
{
    "id": "child-001",
    "relations": [
        {"target_id": "child-002", "rel_type": "refers_to"},
        {"target_id": "child-004", "rel_type": "sibling"}
    ]
}

# enhanced-001 继承后
{
    "id": "enhanced-001",
    "parent_id": "child-001",
    "relations": [
        {"target_id": "child-002", "rel_type": "refers_to"},  # 继承
        {"target_id": "child-004", "rel_type": "sibling"}     # 继承
    ]
}
```

## 四、关系建立后的数据结构长什么样？

### 4.1 一个完整的例子

假设有这篇教材内容：

```
【章节】3.2 神经元的结构与功能

【正文】神经元是神经系统结构和功能的基本单位，主要由细胞体、树突和轴突组成。
如图3-1所示，树突负责接收信息，轴突负责传导神经冲动。

【图片】图3-1 神经元结构示意图

【正文】神经元的分类方式多种多样，具体分类见表3-1。

【表格】表3-1 神经元分类...

【正文】突触是神经元之间传递信息的关键结构。
```

### 4.2 切成 chunks 后的结构

```python
chunks = [
    # ─────────────────────────────────────────
    # 索引 0：章节标题（parent 层）
    # ─────────────────────────────────────────
    {
        "id": "parent-001",
        "content": "3.2 神经元的结构与功能",
        "layer": "parent",
        "relations": []  # 标题通常没有关系
    },
    
    # ─────────────────────────────────────────
    # 索引 1：正文第一段（child 层）
    # ─────────────────────────────────────────
```

## 五、命中一个 chunk，怎么带出其他 chunk？

### 5.1 核心逻辑（伪代码）

```python
def search_and_expand(query):
    """检索 + 关系扩展"""
    
    # 第1步：向量检索，找到最相关的 chunk
    main_chunks = vector_search(query, top_k=5)
    
    # 第2步：对每个命中的 chunk，找它的关系
    all_chunks = []
    all_relations = []
    
    for main in main_chunks:
        all_chunks.append(main)
        
        # 遍历它的所有关系
        for relation in main.relations:
            target = find_chunk_by_id(relation.target_id)
            
            # 根据关系类型决定是否扩展
            if should_expand(relation.rel_type):
                all_chunks.append(target)
                all_relations.append({
                    "from": main.id,
                    "to": target.id,
                    "type": relation.rel_type,
                    "evidence": relation.evidence
                })
    
    # 第3步：返回主内容 + 扩展内容 + 关系链
    return {
        "main_chunks": main_chunks,
        "expanded_chunks": [c for c in all_chunks if c not in main_chunks],
        "relations": all_relations
    }


def should_expand(rel_type):
    """哪些关系类型需要扩展"""
    expand_types = [
        "refers_to",      # 引用 → 带出图片/表格
        "next_step",      # 下一步
        "prev_step",      # 上一步
        "cause_of",       # 原因
        "effect_of",      # 结果
        # "sibling",      # 兄弟 → 可选，可能太多
        # "semantic_similar",  # 语义相似 → 可选
    ]
    return rel_type in expand_types
```

## 六、关系网络的意义总结

| 关系类型 | 解决的问题 | 实际效果 |
|----------|-----------|----------|
| **refers_to** | 正文提到图/表，但图/表在别处 | 自动带出相关图片、表格 |
| **sibling** | 同章节内容分散，用户想看全 | 自动带出同章节的其他内容 |
| **next_step/prev_step** | 步骤分散，用户想知道前后 | 自动串成完整的步骤链 |
| **cause_of/effect_of** | 用户想知道原因或结果 | 自动带出因果相关的内容 |
| **semantic_similar** | 不同章节有相似内容 | 自动带出跨章节的相关内容 |
| **enhanced 继承** | 搜到 enhanced 但想看原文 | 通过 parent_id 找回原文 |

---

## 七、一句话总结

**关系的本质就是"线索"**：

```
命中一个 chunk
    ↓
看它的 relations 字段
    ↓
找到相关 chunk 的 ID
    ↓
把相关 chunk 也带出来
    ↓
给用户更完整的答案
```

**enhanced 和 child 的关系**：

```
enhanced 是 child 的"精简版"
通过 parent_id 关联
搜到 enhanced 时，可以找到原文 child
```

**为什么要建这么多关系？**

```
因为用户的问题千奇百怪：
- 问步骤 → 需要 next_step
- 问原因 → 需要 cause_of
- 问图表 → 需要 refers_to
- 问相似 → 需要 semantic_similar

关系建得越丰富，检索越能"举一反三"！
```
