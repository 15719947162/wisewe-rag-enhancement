## 一、什么是 Embedding？为什么需要它？

### 1.1 Embedding 的本质

Embedding（嵌入/向量化）是将文本转换为**高维向量**的过程。

```
文本： "针灸学是研究经络的学科"
           │
           ▼ Embedding API
向量： [0.12, -0.34, 0.56, 0.78, ..., 0.23]  (1024 个浮点数)
```

这个向量是文本的**语义指纹**——语义相似的文本，其向量在空间中的距离更近。

### 1.2 为什么 Embedding 是 RAG 的核心？

RAG（检索增强生成）的核心流程：

```
┌─────────────────────────────────────────────────────────────────┐
│                        离线阶段（文档入库）                        │
├─────────────────────────────────────────────────────────────────┤
│  文档 → 切片 → Embedding → 存入向量数据库                          │
│                     ↓                                            │
│            ["针灸学..."] → [0.1, 0.2, ...] → pgvector            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                        在线阶段（用户查询）                        │
├─────────────────────────────────────────────────────────────────┤
│  用户问题 → Embedding → 向量相似度检索 → 返回相关文档              │
│      ↓                                                           │
│  "针灸能治什么病？" → [0.15, 0.18, ...]                           │
│                              ↓                                   │
│                    与数据库中所有向量计算距离                       │
│                              ↓                                   │
│                    返回距离最近的 Top-K 个切片                     │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 向量相似度计算

最常用的是**余弦相似度**：

```
相似度 = cos(θ) = (A · B) / (|A| × |B|)

其中：
- A · B 是向量点积
- |A| 是向量模长
- θ 是向量夹角

相似度范围：[-1, 1]
- 1 表示完全相同
- 0 表示正交（无关）
- -1 表示完全相反
```

**直观理解**：

```
向量空间中的位置关系：

        "机器学习"
            ●
           /
          / 距离近 = 语义相似
         /
    "深度学习"●────────●"量子力学"
                    距离远 = 语义无关
```

### 3.2 配置选择的设计考量

| 配置项 | 默认值 | 选择原因 |
|--------|--------|----------|
| `batch_size = 10` | 10 | 平衡 API 调用次数和单次请求大小，太小效率低，太大可能超时 |
| `max_concurrency = 10` | 10 | 避免过度并发导致限流，同时保证足够的吞吐量 |
| `dimensions = 1024` | 1024 | 适合语义检索，更高的维度（如 1536）会增加存储和计算成本 |
| `key_cooldown = 30s` | 30 | 给 API 服务恢复时间，避免反复触发限流 |
| `query_cache_ttl = 1800s` | 30分钟 | 用户可能重复提问相似问题，缓存可显著减少 API 调用 |

```
批大小过小（如 1）：

┌─────────────────────────────────────┐

    问题：单次请求时间过长，失败影响面大

批大小适中（如 10）：
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 文本1-10  │ │ 文本11-20 │ │ 文本21-30 │  ← 3 次 API 调用
└──────────┘ └──────────┘ └──────────┘
    优势：平衡效率和容错性
```

### 4.2 _QueryEmbeddingCache - 查询缓存（第 123-191 行）

#### 4.2.1 为什么需要查询缓存？

```
场景：用户对话式 RAG

用户：什么是机器学习？
系统：[调用 Embedding API] → 返回答案

用户：机器学习是什么？  ← 语义几乎相同
系统：[再次调用 Embedding API] → 浪费资源！

使用缓存后：
用户：什么是机器学习？
系统：[调用 API，缓存结果]

用户：机器学习是什么？
系统：[缓存命中，直接返回] → 零延迟，零成本
```

#### 4.2.2 缓存实现详解

```python
class _QueryEmbeddingCache:
    """
    查询向量缓存（LRU + TTL）。
    
    设计要点：
    1. LRU（Least Recently Used）：缓存满时移除最久未使用的条目
    2. TTL（Time To Live）：条目在指定时间后自动失效
    3. 线程安全：使用锁保护并发访问
    
    为什么用 OrderedDict？
    - Python 的 OrderedDict 会记住插入顺序
    - move_to_end() 可以 O(1) 时间实现 LRU
    - popitem(last=False) 可以 O(1) 时间移除最旧条目
    """
    
    def __init__(self) -> None:
        # key → (创建时间, 向量)
        self._items: OrderedDict[str, tuple[float, list[float]]] = OrderedDict()
        self._lock = threading.Lock()
    
    def get(self, key: str, ttl_seconds: int) -> list[float] | None:
        """
        从缓存获取向量。
        
        流程：
        1. 检查缓存是否存在
        2. 检查是否过期（TTL）
        3. 命中则移动到链表尾部（LRU 更新）
        """
        if ttl_seconds <= 0:
            return None  # TTL=0 表示禁用缓存
        
        now = time.monotonic()  # 单调时钟，不受系统时间调整影响
        
        with self._lock:
            item = self._items.get(key)
            if item is None:
                return None  # 缓存未命中
            
            created_at, embedding = item
            
            # 检查是否过期
            if now - created_at > ttl_seconds:
                self._items.pop(key, None)  # 过期，移除
                return None
            
            # 命中缓存，移动到链表尾部（最近使用）
            self._items.move_to_end(key)
            return list(embedding)
    
    def set(self, key: str, embedding: list[float], max_size: int) -> None:
        """
        将向量存入缓存。
        
        流程：
        1. 存入新条目
        2. 移动到链表尾部
        3. 如果超过容量，移除链表头部（最久未使用）
        """
        if max_size <= 0:
            return  # max_size=0 表示禁用缓存
        
        with self._lock:
            self._items[key] = (time.monotonic(), list(embedding))
            self._items.move_to_end(key)  # 移动到尾部
            
            # LRU 淘汰：超过容量时移除最旧的（链表头部）
            while len(self._items) > max_size:
                self._items.popitem(last=False)  # last=False 表示移除头部
```

## 六、API 密钥池管理

### 6.1 为什么需要密钥池？

**问题场景：单密钥限流**

```
用户有 1000 条文本需要向量化
批大小 = 10，需要 100 次 API 调用

单密钥场景：
┌────────┐              ┌──────────────────┐
│ Key 1  │ ──请求过多──→ │ 触发 429 限流    │
└────────┘              └──────────────────┘
                              │
                              ▼
                        服务暂停等待
                        （冷却 30-60 秒）
                              │
                              ▼
                        总耗时 = 2-3 分钟
```

**解决方案：多密钥轮换**

```
用户配置了 3 个密钥：
- Key 1: sk-aaa
- Key 2: sk-bbb
- Key 3: sk-ccc

密钥池轮换：
┌────────┐              ┌──────────────────┐
│ Key 1  │ ──被限流───→ │ 冷却 30 秒       │
└────────┘              └──────────────────┘
     │ 自动切换
     ▼
┌────────┐              ┌──────────────────┐
│ Key 2  │ ──继续服务──→ │ 正常处理         │
└────────┘              └──────────────────┘
     │
     ▼
Key 1 冷却结束，恢复使用

总耗时 = 30-40 秒（节省 70%+）
```

### 6.3 密钥池核心实现（第 373-545 行）

```python
class _EmbeddingKeyPool:
    """
    API 密钥池管理器。
    
    核心职责：
    1. 轮询分配可用密钥（负载均衡）
    2. 管理限流冷却期
    3. 记录每个密钥的使用统计
    
    数据结构：
    - _keys: 密钥列表
    - _inflight: 每个密钥的"在途"请求数
    - _cooldown_until: 每个密钥的冷却结束时间
    - _usage_by_key: 每个密钥的使用统计
    """
    
    def __init__(self, keys: list[str], cooldown_seconds: int) -> None:
        self._keys = list(keys)
        
        # 为每个密钥分配别名（便于日志追踪）
        self._aliases = {
            key: f"embedding-key-{index + 1}" 
            for index, key in enumerate(self._keys)
        }
        
        # 索引映射（快速定位密钥位置）
        self._index_by_key = {key: index for index, key in enumerate(self._keys)}
        
        # 在途请求数（当前正在使用的请求数）
        self._inflight: Counter[str] = Counter()
        
        # 冷却结束时间（被限流的密钥在此时间后才可用）
        self._cooldown_until: dict[str, float] = {}
        
        # 每个密钥的使用统计
        self._usage_by_key = {
            key: {
                "calls": 0,       # 总调用次数
                "successes": 0,   # 成功次数
                "failures": 0,    # 失败次数
                "throttles": 0,   # 被限流次数
                "totalMs": 0,     # 总耗时（毫秒）
            }
            for key in self._keys
        }
        
        self._cursor = 0                  # 轮询游标
        self._cooldown_seconds = cooldown_seconds
        self._throttle_count = 0          # 总限流次数
        self._retry_count = 0             # 总重试次数
        self._cooldown_count = 0          # 总冷却次数
        self._lock = threading.Lock()     # 线程锁
```

### 6.4 密钥选择策略（第 428-491 行）

```python
    def acquire(self, exclude_keys: set[str] | None = None) -> _EmbeddingKeyLease | None:
        """
        获取一个可用的密钥租约。
        
        选择策略：
        1. 优先选择不在冷却期的密钥
        2. 选择在途请求数最少的密钥（负载均衡）
        3. 如果所有密钥都在冷却，忽略冷却限制选择一个
        
        参数：
        - exclude_keys: 要排除的密钥集合（通常是被限流后暂时排除）
        """
        exclude = exclude_keys or set()
        
        with self._lock:
            if not self._keys:
                return None
            
            now = time.monotonic()
            
            # 第一次尝试：尊重冷却期
            key = self._select_key(now, exclude, respect_cooldown=True)
            
            if key is None:
                # 所有密钥都在冷却，第二次尝试：忽略冷却期
                key = self._select_key(now, exclude, respect_cooldown=False)
            
            if key is None:
                return None
            
            # 增加在途计数
            self._inflight[key] += 1
            
            # 移动游标到下一个位置（轮询）
            self._cursor = (self._index_by_key[key] + 1) % len(self._keys)
            
            return _EmbeddingKeyLease(key=key, alias=self._aliases[key])
    
    def _select_key(self, now: float, exclude: set[str], *, respect_cooldown: bool) -> str | None:
        """
        选择最佳密钥。
        
        选择标准：
        1. 不在排除列表中
        2. （可选）不在冷却期
        3. 在途请求数最少
        4. 位置最靠前（当在途数相同时）
        
        评分公式：(在途请求数, 位置偏移) - 元组比较，越小越好
        """
        best_key: str | None = None
        best_score: tuple[int, int] | None = None
        count = len(self._keys)
        
        for offset in range(count):
            key = self._keys[(self._cursor + offset) % count]
            
            if key in exclude:
                continue
            
            # 检查冷却期
            if respect_cooldown and self._cooldown_until.get(key, 0.0) > now:
                continue
            
            # 评分：(在途请求数, 位置偏移)
            score = (self._inflight[key], offset)
            
            if best_score is None or score < best_score:
                best_key = key
                best_score = score
        
        return best_key
```

### 6.6 密钥释放与统计（第 493-518 行）

```python
    def release(self, lease: _EmbeddingKeyLease, elapsed_ms: int, *, success: bool, throttle: bool = False) -> None:
        """
        释放密钥租约并记录结果。
        
        参数：
        - lease: 密钥租约
        - elapsed_ms: 调用耗时（毫秒）
        - success: 是否成功
        - throttle: 是否被限流
        
        行为：
        1. 减少在途计数
        2. 更新统计指标
        3. 如果被限流，将密钥加入冷却期
        """
        with self._lock:
            # 减少在途计数
            self._inflight[lease.key] = max(0, self._inflight[lease.key] - 1)
            
            # 更新统计
            usage = self._usage_by_key[lease.key]
            usage["calls"] += 1
            usage["totalMs"] += max(0, elapsed_ms)
            
            if success:
                usage["successes"] += 1
                return
            
            usage["failures"] += 1
            
            if throttle:
                usage["throttles"] += 1
                self._throttle_count += 1
                
                # 如果配置了冷却时间，将被限流的密钥加入冷却
                if self._cooldown_seconds > 0:
                    self._cooldown_until[lease.key] = time.monotonic() + self._cooldown_seconds
                    self._cooldown_count += 1
```

### 7.5 为什么使用 as_completed 而不是 map？

```python
# 方式一：executor.map() - 按提交顺序返回
results = list(executor.map(process, batches))
# 问题：必须等待最慢的批次完成才能处理后续批次

# 方式二：as_completed() - 按完成顺序返回
futures = [executor.submit(process, batch) for batch in batches]
for future in as_completed(futures):
    result = future.result()
    # 优势：哪个批次先完成就先处理哪个
```

**时间对比**：

```
假设 3 个批次，处理时间分别为 2s、1s、3s

executor.map()：
时间 0: 提交批次0、1、2
时间 2: 批次0完成，等待
时间 3: 批次2完成，等待
时间 3: 全部完成（实际上被最慢的批次3s阻塞）

as_completed()：
时间 0: 提交批次0、1、2
时间 1: 批次1完成 → 立即处理结果
时间 2: 批次0完成 → 立即处理结果
时间 3: 批次2完成 → 立即处理结果
优势：更早开始处理已完成的结果
```

### 8.1 限流错误检测（第 331-356 行）

```python
def _is_throttle_error(error: str) -> bool:
    """
    检测是否为限流错误。
    
    限流错误的特征：
    - HTTP 429 状态码
    - 包含 "rate limit"、"throttle"、"quota" 等关键词
    - 中英文都支持
    
    为什么需要检测？
    - 普通错误：直接重试即可
    - 限流错误：需要切换密钥或等待冷却
    """
    text = (error or "").lower()
    if not text:
        return False
    
    markers = (
        "429",                    # HTTP 状态码
        "rate limit",             # 速率限制
        "ratelimit",              # 无空格版本
        "too many requests",      # 请求过多
        "throttl",                # 节流（throttle/throttled）
        "quota",                  # 配额
        "qps",                    # 每秒查询数
        "限流",                    # 中文
        "请求过多",                 # 中文
        "配额",                    # 中文
    )
    return any(marker in text for marker in markers)
```

### 8.2 指数退避重试（第 689-733 行）

```python
def _embed_batch_with_retry(
    client: openai.OpenAI,
    batch_index: int,
    offset: int,
    batch: list[str],
    model: str,
    dimensions: int,
    max_retries: int,
    token_usage: ThreadSafeTokenUsage | None = None,
) -> tuple[int, list[list[float]], int]:
    """
    带重试机制的批次向量化。
    
    重试策略：指数退避
    - 第 1 次重试：等待 0.25 秒
    - 第 2 次重试：等待 0.50 秒
    - 第 3 次重试：等待 0.75 秒
    - 最大等待 1.0 秒
    
    为什么用指数退避？
    - 避免立即重试导致雪崩
    - 给服务恢复时间
    - 逐步增加等待时间，提高成功率
    """
    attempts = 0
    while True:
        try:
            return offset, _embed_batch(client, batch, model, dimensions, token_usage), attempts
        except Exception as exc:
            if attempts >= max_retries:
                # 超过最大重试次数，抛出异常
                end = offset + len(batch) - 1
                raise RuntimeError(
                    f"Embedding batch {batch_index} failed after {attempts + 1} attempts "
                    f"(offset {offset}-{end}): {exc}"
                ) from exc
            
            attempts += 1
            
            # 指数退避：0.25s, 0.5s, 0.75s, ... 最大 1s
            time.sleep(min(0.25 * attempts, 1.0))
```

**重试时间线**：

```

尝试 1 ──失败──→ 等待 0.25s
     │
     ▼
尝试 2 ──失败──→ 等待 0.50s
     │
     ▼
尝试 3 ──失败──→ 抛出异常（不再重试）

如果 max_retries = 5：

尝试 2 ──失败──→ 等待 0.50s
尝试 3 ──失败──→ 等待 0.75s
尝试 4 ──失败──→ 等待 1.00s（达到上限）
尝试 5 ──失败──→ 等待 1.00s
尝试 6 ──失败──→ 抛出异常
```

### 8.3 密钥池模式下的重试（第 735-832 行）

```python
def _embed_batch_with_key_pool(
    pool: _EmbeddingKeyPool,
    clients_by_key: dict[str, openai.OpenAI],
    client_lock: threading.Lock,
    # ... 其他参数
) -> tuple[int, list[list[float]], int, int]:
    """
    使用密钥池的批次向量化。
    
    与普通重试的区别：
    - 普通重试：等待后使用同一密钥重试
    - 密钥池重试：切换到其他密钥重试
    
    策略：
    1. 从密钥池获取密钥
    2. 调用 API
    3. 如果被限流，将当前密钥加入排除列表，尝试其他密钥
    4. 如果普通错误，等待后使用同一密钥重试
    """
    attempts = 0
    total_key_retries = 0
    excluded_keys: set[str] = set()  # 暂时排除的密钥
    
    while True:
        # 从密钥池获取一个可用的密钥
        lease = pool.acquire(excluded_keys)
        if lease is None:
            raise RuntimeError("Embedding key pool has no available key")
        
        # 获取或创建对应的客户端（线程安全）
        with client_lock:
            client = clients_by_key.get(lease.key)
            if client is None:
                client = get_embedding_client(api_key=lease.key, base_url=base_url)
                clients_by_key[lease.key] = client
        
        call_started = time.perf_counter()
        try:
            embeddings = _embed_batch(client, batch, model, dimensions, token_usage)
            
            # 成功：释放密钥
            pool.release(
                lease,
                int((time.perf_counter() - call_started) * 1000),
                success=True,
            )
            return offset, embeddings, attempts, total_key_retries
        
        except Exception as exc:
            elapsed_ms = int((time.perf_counter() - call_started) * 1000)
            throttle = _is_throttle_error(str(exc))
            
            # 释放密钥并记录结果
            pool.release(lease, elapsed_ms, success=False, throttle=throttle)
            
            # 如果是限流错误且还有密钥切换机会，尝试其他密钥
            if throttle and total_key_retries < key_retries:
                total_key_retries += 1
                pool.record_retry()
                excluded_keys.add(lease.key)  # 暂时排除这个密钥
                continue  # 立即尝试其他密钥，不等待
            
            # 检查是否超过重试次数
            if attempts >= max_retries:
                end = offset + len(batch) - 1
                raise RuntimeError(
                    f"Embedding batch {batch_index} failed after {attempts + 1} attempts "
                    f"(offset {offset}-{end}): {exc}"
                ) from exc
            
            attempts += 1
            excluded_keys.clear()  # 普通重试时清空排除列表
            time.sleep(min(0.25 * attempts, 1.0))
```

```python
def embed_query_cached(
    query: str,                       # 查询文本
    model: Optional[str] = None,      # 模型名称
    api_key: Optional[str] = None,    # API 密钥
    base_url: Optional[str] = None,   # API 基础 URL
    dimensions: int = 1024,           # 向量维度
) -> tuple[list[float], bool]:        # 返回 (向量, 是否命中缓存)
    """
    向量化查询（带缓存）。
    
    返回值：
    - list[float]: 查询的向量
    - bool: True 表示命中缓存，False 表示未命中
    
    缓存配置（通过环境变量）：
    - RAG_QUERY_EMBEDDING_CACHE_TTL_SECONDS: 缓存有效期（默认 1800 秒）
    - RAG_QUERY_EMBEDDING_CACHE_MAX_SIZE: 最大缓存条目数（默认 512）
    """
    resolved_model = _resolve_model(model)
    
    # 读取缓存配置
    ttl_seconds = _int_env("RAG_QUERY_EMBEDDING_CACHE_TTL_SECONDS", 1800, minimum=0)
    max_size = _int_env("RAG_QUERY_EMBEDDING_CACHE_MAX_SIZE", 512, minimum=0)
    
    # 生成缓存键
    key = _query_cache_key(query, resolved_model, base_url, dimensions)
    
    # 尝试从缓存获取
    cached = _QUERY_EMBEDDING_CACHE.get(key, ttl_seconds)
    if cached is not None:
        return cached, True  # 命中缓存
    
    # 缓存未命中，调用 API
    embedding = embed_texts(
        [query],
        model=resolved_model,
        api_key=api_key,
        base_url=base_url,
        batch_size=1,
        dimensions=dimensions,
    )[0]
    
    # 存入缓存
    _QUERY_EMBEDDING_CACHE.set(key, embedding, max_size)
    
    return embedding, False  # 未命中缓存
```

### 10.3 缓存键生成（第 1085-1112 行）

```python
def _query_cache_key(
    query: str,
    model: str,
    base_url: str | None,
    dimensions: int,
) -> str:
    """
    生成查询缓存键。
    
    缓存键组成：
    - Provider 标识（从 base_url 推断）
    - 模型名称
    - 向量维度
    - 归一化的查询文本
    
    为什么这样设计？
    - 不同服务商的向量可能不同，需要区分
    - 不同模型的向量不同，需要区分
    - 不同维度的向量不同，需要区分
    - 查询文本需要归一化（去除多余空格、小写），提高命中率
    """
    # 归一化查询文本
    normalized_query = " ".join((query or "").strip().split()).lower()
    
    # 推断 Provider
    provider_marker = (
        base_url 
        or os.environ.get("LLM_BASE_URL") 
        or ("dashscope" if os.environ.get("DASHSCOPE_API_KEY") else "default")
    )
    
    return f"{provider_marker}|{model}|{dimensions}|{normalized_query}"
```

## 十一、性能优化策略总结

### 11.1 优化策略一览

| 策略 | 实现位置 | 优势 | 适用场景 |
|------|----------|------|----------|
| **批处理** | `_embed_batch()` | 减少 HTTP 开销 | 大量文本向量化 |
| **并发处理** | `ThreadPoolExecutor` | 提高吞吐量 | 批次数 > 1 |
| **多密钥轮换** | `_EmbeddingKeyPool` | 应对限流，提高可用性 | 有多个 API Key |
| **指数退避重试** | `_embed_batch_with_retry()` | 避免雪崩，提高成功率 | 网络不稳定 |
| **查询缓存** | `embed_query_cached()` | 零延迟，零成本 | 重复查询 |
| **客户端复用** | `clients_by_key` | 减少连接建立开销 | 多密钥模式 |

### 11.2 性能对比

```
场景：1000 条文本向量化

┌─────────────────────────────────────────────────────────────────┐
│ 无优化（单线程，无批处理）                                        │
├─────────────────────────────────────────────────────────────────┤
│ • 1000 次 HTTP 请求                                              │
│ • 单密钥，第 100 次后被限流，等待 60 秒                           │
│ • 总耗时：约 5-10 分钟                                           │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 批处理 + 并发（batch_size=10, max_concurrency=10）               │
├─────────────────────────────────────────────────────────────────┤
│ • 100 次 HTTP 请求（减少 90%）                                    │
│ • 10 个并发，单密钥，可能被限流                                   │
│ • 总耗时：约 1-2 分钟                                            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 批处理 + 并发 + 多密钥池（3 个 Key）                              │
├─────────────────────────────────────────────────────────────────┤
│ • 100 次 HTTP 请求                                               │
│ • 10 个并发，3 个密钥轮换，限流时自动切换                         │
│ • 总耗时：约 30-60 秒                                            │
└─────────────────────────────────────────────────────────────────┘

```
