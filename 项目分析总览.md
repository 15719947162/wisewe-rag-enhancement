## 1. 先用一句大白话说清楚：这个项目到底是什么

这个项目不是“一个 PDF 上传后调大模型回答问题的小工具”。

它实际上是一整套围绕知识库构建和 RAG 问答的完整系统，里面至少包含这几层：

1. 文档解析层
2. 文档清洗层
3. 文档切片层
4. 向量化层
5. 关系构建层
6. 图谱构建层
7. 在线检索层
8. 在线回答层
9. 前端控制台与治理层

如果再说得直白一点，这个项目在做两件大事：

1. 先把 PDF / 文档处理成“可检索的知识库”
2. 再基于这个知识库做“带证据的问答”

所以它的核心路径不是：

`PDF -> 大模型 -> 答案`

而是：

`PDF -> 解析 -> 清洗 -> 切片 -> 增强 -> 质量门控 -> 向量化 -> 关系补全 -> 入库 -> 检索 -> 生成答案`

如果你第一次看这个仓库，建议不要一上来先看前端页面，也不要一上来先看某个单点优化。

更好的方式是：

1. 先看离线入库链路
2. 再看在线问答链路
3. 最后回头看前端和控制台

因为这个项目最核心的业务事实是：

1. 没有离线入库，就没有后面的知识库
2. 没有知识库，在线问答也只是空壳

所以正确的阅读顺序应该是先“造库”，再“用库”。

## 5. 先看离线入库主链路

离线入库是整个项目最重要的基础链路。

因为只有这条链路真的把文档变成知识库，后面的问答才有东西可查。

### 5.1 总入口在哪

总入口在 [backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py)。

这个文件不要把它只当“上传服务”看。它实际上是整个入库编排器。

先抓住两个关键函数：

1. `run_pipeline_real(...)`
2. `confirm_pipeline(...)`

这两个函数一起构成了离线入库的两段式执行模型：

#### 第一段：先生成 chunk 草稿

这一段主要做：

1. 解析
2. 清洗
3. 切片
4. 增强
5. 基础关系补链
6. 草稿保存

#### 第二段：再完成正式入库

这一段主要做：

1. 质量门控
2. 向量化
3. 语义 / 步骤 / 因果关系补充
4. 实体物化
5. 正式写入 pgvector

这个设计很重要。

它说明项目不是“切完就直接写正式库”，而是中间有一个 `chunk_drafts` 的草稿阶段。

## 6. 离线入库第 1 步：解析文档

### 6.1 解析做的到底是什么

解析阶段做的不是“把文档切片”，而是先把原始文档变成结构化内容块。

输入是：

1. PDF
2. 图片文档
3. Office 文档
4. 某些其他受支持来源

输出是：

`ContentBlock[]`

也就是一组结构化内容块。

这些块通常包含：

1. 标题块
2. 正文块
3. 表格块
4. 图片块

所以解析阶段的目标不是直接服务检索，而是先把“原始文档”变成“结构化中间结果”。

### 6.2 解析入口代码在哪

解析分发入口在 [core/parser/provider.py](/E:/all_project/AI/rag-enhancement/core/parser/provider.py)。

这个文件很重要，因为它决定当前到底走哪条 parser 通道。

你可以直接抓这里几个点：

1. `PDF_PARSER_PROVIDER_ENV = "PDF_PARSER_PROVIDER"`
2. `DEFAULT_PDF_PARSER_PROVIDER = "mineru"`
3. `PDF_PARSER_CHANNELS`
4. `get_pdf_parser_provider()`
5. `parse_pdf(...)`

它的执行逻辑非常直接：

1. 先从环境变量 `PDF_PARSER_PROVIDER` 读 provider
2. 如果没配，就用默认值 `mineru`
3. 如果配了不支持的值，就报错
4. 再根据 provider 去动态导入对应模块
5. 调那个模块里的 `parse_pdf(...)`

这说明一件事：

这个项目当前不是“系统先分析 PDF 再自动选择最佳解析器”。

而是：

`配置决定 parser provider`

### 6.3 当前主要的 parser provider 有哪些

从 `provider.py` 里看，当前主要有三条通道：

1. `mineru`
2. `mineru_official`
3. `ali_document_mind`

它们的共同点是：

1. 最终都要产出 `ContentBlock[]`
2. 后面都进入同一套清洗、切片、向量化、入库流程

它们的差异主要在：

1. 上游服务是谁
2. 鉴权怎么做
3. 任务怎么提交
4. 结果怎么回收和映射

但从“主链路”角度看，你只要记住一件事：

不管前面选了哪条 provider，最后都要把结果统一收敛成 `ContentBlock[]`。

### 6.4 为什么解析阶段和切片阶段要分开

这是很多人第一次看代码时最容易混的地方。

这里一定要分清：

#### 解析阶段

目标是：

把原始文档读懂，变成结构化内容块。

#### 切片阶段

目标是：

把这些结构化内容块重新组织成适合检索和问答的知识块。

也就是说：

1. parser 负责“识别文档结构”
2. chunker 负责“组织知识单元”

这两层不是一回事。

## 7. 离线入库第 2 步：清洗 `ContentBlock[]`

### 7.1 清洗的真实目标是什么

清洗不是“美化文字”。

清洗的真实目标是：

1. 去掉明显没价值的块
2. 降低噪声
3. 减少后面切片和检索被污染

比如这类内容通常就是清洗对象：

1. 空块
2. 很短、几乎没有语义的块
3. 标点或符号占比过高的块
4. 版权、广告类噪声

### 7.2 清洗入口在哪

入口在 [core/cleaner/__init__.py](/E:/all_project/AI/rag-enhancement/core/cleaner/__init__.py)。

核心函数是：

`clean_blocks(...)`

它的输入是：

`list[ContentBlock]`

它的输出是：

`CleanResult`

里面至少包含：

1. 清洗后的 `blocks`
2. 删除数量
3. 修改数量
4. 删除原因
5. metrics

### 7.3 清洗怎么执行

`clean_blocks(...)` 的执行顺序也很清楚：

#### 先跑规则清洗

默认规则包括：

1. `RemoveEmptyBlocks()`
2. `RemoveShortBlocks(min_chars=2)`
3. `RemovePunctuation(threshold=0.8)`
4. `RemoveCopyrightAds()`

也就是说，规则清洗是主链路的基础清洗层。

#### 再决定是否跑 LLM 清洗

如果 `use_llm=True`，还会再调用 `LLMCleaner(...)`。

所以这里是一个“规则优先、LLM 可选”的结构，不是全靠 LLM。

### 7.4 为什么这个设计合理

因为规则清洗有几个现实优势：

1. 快
2. 稳
3. 成本低
4. 对明显垃圾内容很有效

而 LLM 清洗更适合处理：

1. 规则不容易判定的边界内容
2. 需要更细语义判断的内容

所以这套设计并不是“全规则”或“全 LLM”，而是：

`规则打底 + LLM 补充`

### 7.5 清洗后到底得到什么

清洗完并不会得到 `Chunk[]`。

清洗完得到的仍然是：

`更干净的 ContentBlock[]`

这一点非常重要。

因为从执行顺序上看，此时项目还没进入知识库切片阶段。

## 8. 离线入库第 3 步：切片，把结构化块变成知识块

### 8.1 切片入口在哪

切片策略入口在 [core/chunker/__init__.py](/E:/all_project/AI/rag-enhancement/core/chunker/__init__.py)。

这里最重要的是：

1. `get_strategy(...)`
2. `link_related_chunks`

在主链路里，`ingestion_service.py` 会先通过 `get_strategy(...)` 拿到策略对象，再执行：

`strategy_obj.chunk(cleaned)`

所以切片本质上就是：

`cleaned ContentBlock[] -> strategy -> Chunk[]`

### 8.2 为什么说现在最重要的是 `hierarchical`

因为当前项目里最值得重点理解的切片策略就是：

[core/chunker/hierarchical.py](/E:/all_project/AI/rag-enhancement/core/chunker/hierarchical.py)

这个文件不是简单的“按长度切分”。

它做的是一套三层结构切片：

1. `parent`
2. `child`
3. `enhanced`

而且它内部还带：

1. 文本分组
2. 表格处理
3. 图片处理
4. 增强任务调度
5. 最终 chunk 合并渲染

所以这个文件本身已经像一个“小型流水线”。

### 8.3 `hierarchical.chunk(...)` 开始时先做什么

这个函数一开始不是马上增强，也不是马上写库。

它先做三件基础准备：

1. 初始化本轮的计时器和 token 计数
2. 准备文本缓冲区 `buffer`
3. 准备切片计划表 `entries` 和增强任务表 `tasks`

你可以把它理解成：

它先给整次切片建立一个“施工图”。

### 8.4 它怎么扫描 `ContentBlock[]`

`hierarchical.chunk(...)` 的主循环是按 `blocks` 逐个扫描的。

扫描时，最重要的判断是当前块是什么类型：

1. 标题
2. 表格
3. 图片
4. 普通正文

然后每种类型走不同逻辑。

#### 遇到标题块

它会：

1. 先把之前缓冲的正文刷出来
2. 生成一个 `parent` 层 chunk
3. 更新当前标题上下文

这说明标题在这里不是拿来直接检索的正文块，而是上层结构容器。

#### 遇到普通正文块

它不会立刻直接生成 chunk。

而是先放进 `buffer` 里累计。

为什么这么做？

因为普通正文通常需要合并后再按句子或长度切，不能一块一块原样照抄成 chunk。

#### 遇到表格块

它会：

1. 先刷正文缓冲区
2. 直接生成一个 `child` 层表格 chunk
3. 如果配置允许，再登记一个表格增强任务

#### 遇到图片块

它会：

1. 先刷正文缓冲区
2. 直接生成一个 `child` 层图片 chunk
3. 如果配置允许，再登记一个图片增强任务

### 8.5 文本缓冲区刷出时发生了什么

这是 `hierarchical` 里很关键的一步。

当正文缓冲区被刷出时，会调用 `_make_children(...)`。

这一步的本质是：

1. 先把多段正文合并成候选文本
2. 再按句子边界和长度切成多个 `child` chunk
3. 给每个 `child` 记录标题、页码、来源、父级信息

所以从“对象变化”的角度看：

`多个 ContentBlock -> 一组 child chunks`

这一步才是正文真正变成知识单元的地方。

### 8.6 增强任务是什么时候登记的

在 `hierarchical` 里，增强不是等全部切完以后再重新遍历 chunk 去判断。

而是在生成 `child` 时顺手登记。

也就是说，逻辑更接近：

1. 先生成一个 `child`
2. 立刻判断它是否需要增强
3. 如果需要，就给它加一个增强任务占位

这点很重要，因为它说明增强不是独立于切片的后处理，而是切片流程内部的一部分。

### 8.7 “增强”到底是在做什么

这里的“增强”不是把原始 `child` 改写掉。

它真正做的是：

`基于 child 再额外生成一个 enhanced chunk`

所以正确理解应该是：

1. 原始 `child` 保留
2. 增强结果作为 `layer="enhanced"` 的新块附加上去
3. 两者最后一起进入后续入库流程

这也是为什么项目后面既能用原文证据，又能利用增强表达做召回。

### 8.8 增强分几种

在 `hierarchical` 里，增强大致分四种类型：

1. `text`
2. `fragment`
3. `table`
4. `image`

#### `text`

普通正文增强，重点是让内容更适合检索。

#### `fragment`

针对碎片化内容，重点是补背景、补上下文。

#### `table`

针对表格，重点是把结构化表格转成更好召回的说明文本。

#### `image`

针对图片，重点是让图像内容也能进入检索语义空间。

### 8.9 增强任务怎么真正执行

切片规划结束后，`hierarchical` 会进入真正的增强执行阶段：

1. 先统计一共多少增强任务
2. 再根据配置决定串行还是并行
3. 对每个任务调用对应的 LLM / VL 分支
4. 回收结果
5. 记录耗时、token、失败数、key pool 使用情况

这说明增强不只是“有或没有”，它本身还是一个独立调度器。

### 8.10 最终 chunk 列表是怎么合并出来的

增强结果不是一直独立保存成另一个数组往后传。

在 `hierarchical` 里，最终会通过 `_render_plan(...)` 把这些东西统一渲染成一个最终 `Chunk[]`：

1. 基础 `parent`
2. 基础 `child`
3. 增强 `enhanced`

这里有一个很关键的事实：

`enhanced` 通常是紧跟其来源 `child` 排列的，不是全部堆到数组末尾。

这意味着：

1. 数据结构上更容易看出来源关系
2. 后续预览和证据展示也更自然
3. 同时保留原文与增强层不会互相覆盖

### 8.11 切片阶段的输出到底是什么

到这里，`hierarchical.chunk(...)` 返回的已经不是中间块，而是完整的切片结果：

`Chunk[]`

这个数组里可能同时包含：

1. `parent` chunk
2. `child` chunk
3. `enhanced` chunk

但注意：这还不是正式入库完成。

因为后面还有很多事没做。

## 9. 离线入库第 4 步：切片完成后先补基础关系

### 9.1 切片返回后第一件事不是 embedding

这是一个非常关键的执行顺序点。

在 [backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py) 里，切片返回 `chunks` 之后，第一件事不是向量化，而是：

`link_related_chunks(chunks)`

也就是说：

`Chunk[] -> 基础关系补链`

### 9.2 这一步在补什么关系

这一步补的是“切片阶段已经能明确看出来”的结构关系。

常见包括：

1. 图文引用
2. 表格引用
3. 相邻关系
4. 同父级关系
5. enhanced 继承来源 child 的相关关系

这类关系的特点是：

1. 不需要等 embedding
2. 不需要做复杂语义计算
3. 只靠结构和文本提示就能先连起来

所以把它放在 embedding 前做是合理的。

## 10. 离线入库第 5 步：先保存到 `chunk_drafts`

### 10.1 为什么不是直接写正式库

补完基础关系后，主链路会先把 `chunks` 存到 `chunk_drafts`。

这一步的意义很大：

1. 它把切片结果先保存成中间态
2. 它把“切片成功”和“正式入库完成”分开了
3. 它让后半段流程有一个稳定接力点

所以你可以把 `chunk_drafts` 理解成：

`正式知识库之前的中间草稿仓`

### 10.2 这个设计解决了什么问题

它至少解决了三个现实问题：

1. 切片结果可预览、可确认
2. 后半段失败时，不会把前半段结果彻底丢掉
3. 流程可以被拆成更可控的阶段

这比“一次性跑完直接入正式库”的设计更稳。

## 11. 离线入库第 6 步：进入 `confirm_pipeline(...)`

### 11.1 为什么这个函数很关键

很多人看项目时会以为“切片结束就差不多了”。

其实不是。

真正把数据变成正式知识库的关键函数是：

[backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py:1506)

也就是：

`confirm_pipeline(task_id)`

它做的事情比名字听起来重得多。

你可以把它理解成：

`草稿收口 -> 正式入库总入口`

## 12. 离线入库第 7 步：质量门控

### 12.1 为什么质量门控放在增强后

这也是一个很值得注意的顺序。

项目不是：

`先质量门控，再增强`

而是：

`先切片和增强，再做质量门控`

这样设计的原因是：

1. 增强后内容可能更完整
2. 某些碎片原文本身看起来差，但增强后有价值
3. 质量门控应该针对“最终准备入库的 chunk”做判断

所以从工程逻辑上看，质量门控放在增强后是合理的。

### 12.2 质量门控入口在哪

入口在 [core/cleaner/quality_gate.py](/E:/all_project/AI/rag-enhancement/core/cleaner/quality_gate.py)。

核心函数是：

`apply_quality_gate(...)`

### 12.3 它先怎么筛

先看一个很具体的执行细节：

表格块和图片块在质量门控里是先直接豁免的。

对于普通文本块，第一道规则是看标点 / 符号占比。

如果某段内容绝大多数都是标点、噪声符号，那就直接丢掉。

### 12.4 什么时候会调用 LLM 评分

如果 `min_score > 0`，就会进入 LLM 评分阶段。

也就是说，LLM 评分不是永远执行，而是一个“按配置启用”的质量层。

执行逻辑可以理解成：

1. 把候选 chunk 按批次送给模型
2. 要求模型打 1 到 5 分
3. 如果分数低于阈值，就丢弃
4. 如果只是 `score_only=True`，就只打分不拦截

### 12.5 这一阶段输出什么

输出是 `QualityGateResult`，里面至少包含：

1. 通过的 `chunks`
2. 丢弃数量
3. 丢弃原因
4. LLM 分数
5. metrics

所以从数据流角度看，这一步是：

`chunk_drafts -> passed chunks`

也就是：并不是所有切片结果都能进入正式知识库。

## 13. 离线入库第 8 步：向量化

### 13.1 向量化什么时候开始

质量门控通过后，主链路才进入 embedding 阶段。

这意味着向量化的对象不是所有草稿 chunk，而是：

`通过质量门控后的 passed`

### 13.2 向量化入口在哪

主要看 [core/embedding/client.py](/E:/all_project/AI/rag-enhancement/core/embedding/client.py)。

在 `ingestion_service.py` 里，对应调用是：

`embed_texts_with_metrics(texts)`

### 13.3 这个 embedding 客户端不只是“调一次接口”

这个文件其实做了很多工程化事情，不只是简单调 embedding API。

你可以重点记住它有这些能力：

1. 模型与 base_url 解析
2. batch size 控制
3. 最大并发控制
4. 重试控制
5. API key pool
6. 限流后冷却
7. query embedding cache
8. metrics 统计

也就是说，这里的 embedding 层已经不是一个薄封装，而是一个带并发调度和可靠性处理的客户端。

### 13.4 为什么这很重要

因为大规模入库时，embedding 通常是主耗时之一。

如果没有：

1. 批处理
2. 并发
3. 重试
4. key pool

那入库吞吐和稳定性都会差很多。

所以这部分代码不是“锦上添花的优化”，而是这个项目能跑大任务的重要基础设施。

## 14. 离线入库第 9 步：embedding 后继续补关系

这一步很多人容易漏掉。

不是 embedding 做完就直接写库。

而是还要继续补关系。

### 14.1 语义关系

先执行：

`link_semantic(passed, embeddings)`

它的作用是基于 embedding 相似度补出 chunk 之间的语义边。

常见结果包括：

1. `semantic_similar`
2. `duplicate_of`

也就是说，项目的知识网络不只是基于显式引用，也会基于语义相似再补一层。

### 14.2 步骤关系

然后执行：

1. `detect_procedure_chunks(...)`
2. `link_procedure(...)`

这一步主要是识别某些具有步骤性质的内容，再补：

1. `next_step`
2. `prev_step`

这说明项目不仅在做“文档分块”，还在尝试识别内容内部的流程结构。

### 14.3 因果关系

接着执行：

`link_causal(...)`

这一步主要补：

1. `cause_of`
2. `effect_of`

所以从这里你就能看出，这个项目的关系层不是单一的。

它至少有：

1. 显式结构关系
2. 语义关系
3. 步骤关系
4. 因果关系

这也是它后面能做 Graph RAG 的一个基础原因。

## 15. 离线入库第 10 步：实体物化

### 15.1 实体物化在什么时候做

实体物化在关系补全之后、正式写 chunk 之前。

这一步的入口是：

[core/kg/extraction_pipeline.py](/E:/all_project/AI/rag-enhancement/core/kg/extraction_pipeline.py)

核心函数：

`materialize_entities(conn, chunks, kb_id)`

### 15.2 它怎么做

从代码看，它的顺序大致是：

1. 只遍历 `enhanced` 层 chunk
2. 收集 `extracted_entities`
3. 通过 `EntityMerger` 做合并
4. 为实体生成定义
5. 再对实体本身做 embedding
6. 把实体与 chunk 之间补 `mentions` 关系
7. 最后写实体表

这说明两件事：

#### 第一，实体层主要承接增强层输出

也就是说，增强不仅是为了提高召回，它还是实体和图谱信息的重要来源。

#### 第二，实体也有自己的 embedding

所以项目不是只给 chunk 向量化，实体本身也会向量化。

这进一步说明项目的图谱层不是装饰性的，而是认真落了一层实体表示。

## 16. 离线入库第 11 步：正式写入 pgvector

### 16.1 真正的最终落库入口

最终落库核心函数在：

[core/output/pgvector_writer.py](/E:/all_project/AI/rag-enhancement/core/output/pgvector_writer.py)

核心函数：

`write_to_pgvector(...)`

它不是只写一个 `chunks` 表，而是完整接管正式落库。

### 16.2 `write_to_pgvector(...)` 的执行顺序

这个函数内部顺序很清楚：

1. 如果是默认知识库，先确保默认库存在
2. 计算文件 hash
3. 建立数据库连接
4. 确保 schema 存在
5. 如果同 hash 文档已存在，先判断是否跳过或更新元数据
6. upsert `documents`
7. 批量写 `chunks`
8. 批量写 `chunk_relations`
9. 批量写 `kg_triples`
10. commit

### 16.3 这里写入了哪些核心表

至少要记住这四张：

1. `documents`
2. `chunks`
3. `chunk_relations`
4. `kg_triples`

#### `documents`

保存文档级信息，比如：

1. 文件名
2. file_hash
3. chunk_count
4. 来源存储信息
5. parser provider

#### `chunks`

这是最核心的检索层表。

它保存的不只是正文，还保存：

1. `content`
2. `source`
3. `page`
4. `chunk_index`
5. `strategy`
6. `title`
7. `char_count`
8. `is_table_chunk`
9. `is_image_chunk`
10. `image_path`
11. `layer`
12. `parent_id`
13. `related_ids`
14. `search_text`
15. `search_vector`
16. `embedding`

这一点非常关键。

因为它说明一个 chunk 同时服务三类需求：

1. 文本展示
2. 全文检索
3. 向量检索

#### `chunk_relations`

这里保存的是 chunk 之间的边。

字段含义可以简单记成：

1. 谁指向谁
2. 关系类型是什么
3. 权重多少
4. 来源是什么
5. 证据是什么

#### `kg_triples`

这里主要保存三元组。

特别要注意一点：

代码里明确只从 `enhanced` 层 chunk 的 `extracted_triples` 中抽三元组。

这说明增强层不仅为召回服务，也为图谱层服务。

### 16.4 为什么 writer 里有 `file_hash`

因为它要处理“重复文档”问题。

如果文件 hash 已经存在，它可以：

1. 识别文档是否已经入过库
2. 避免重复写入
3. 必要时只更新文档元数据

这是一种非常典型也很实用的工程策略。

### 16.5 入库链路到这里才算真正结束

只有 `write_to_pgvector(...)` commit 完成后，这批数据才算真正进入正式知识库。

所以一定不要把：

1. 解析完成
2. 切片完成
3. 增强完成

误认为“已经入库完成”。

真正完整的离线入库链路是：

`文档 -> 解析 -> 清洗 -> 切片 -> 增强 -> 基础关系补链 -> chunk_drafts -> 质量门控 -> 向量化 -> 语义/步骤/因果关系补全 -> 实体物化 -> pgvector 正式写库`

## 19. 在线问答第 1 步：先检索，不先回答

### 19.1 检索核心在哪

核心检索逻辑在 [core/rag/retriever.py](/E:/all_project/AI/rag-enhancement/core/rag/retriever.py)。

最值得先抓的是：

`HybridRetriever.retrieve(...)`

这个函数基本就是普通 RAG 召回的主心脏。

### 19.2 检索顺序怎么走

按代码顺序，它大致这么跑：

#### 先看是不是图表编号类快速命中

也就是先尝试 `_media_ref_retrieve(...)`。

如果用户问的是类似：

1. “图 1-2”
2. “表 3-1”

这种显式媒体引用，那么系统会优先走快速路径。

如果命中，就会：

1. 扩展相关块
2. 做粗过滤
3. 提前返回

这意味着它对教材场景下的“图表编号问法”做了专门优化。

#### 如果不是快速命中，就先做 query embedding

调用的是：

`embed_query_cached(query)`

这里说明查询本身也会向量化，而且还有缓存。

#### 然后分情况看是否走 retrieval snapshot

如果快照功能开启，且没有额外 filters，会优先尝试：

1. 拉 snapshot
2. 做 enhanced 折叠
3. 扩展 related
4. 粗过滤

这是一条“更快的召回路径”。

#### 如果 snapshot 不走或失败，再走标准混合召回

标准路径是：

1. dense 检索 `_dense_retrieve(...)`
2. sparse 检索 `_sparse_retrieve(...)`
3. structured 检索 `_structured_retrieve(...)`
4. RRF 融合 `_rrf_merge(...)`
5. enhanced 折回 child `_fold_enhanced_to_children(...)`
6. 扩展相关块 `_expand_related(...)`
7. 粗过滤 `_coarse_filter(...)`

这一步顺序非常值得记住，因为它几乎就是整个普通 RAG 召回逻辑的压缩版。

### 19.3 这里为什么要“enhanced 折回 child”

这是这个项目非常有特点的一点。

系统允许 `enhanced` 参与命中，但很多时候最终证据会折回到来源 `child`。

这么做的好处是：

1. enhanced 负责提高召回命中率
2. child 负责提供更原始、更可信的证据展示

也就是说：

`enhanced 用来帮你找到`

`child 用来给你展示证据`

这是一个很合理的双层设计。

## 20. 在线问答第 2 步：把检索结果交给生成器

### 20.1 生成器入口在哪

核心生成逻辑在 [core/rag/generator.py](/E:/all_project/AI/rag-enhancement/core/rag/generator.py)。

这里最重要的类是：

`RAGGenerator`

核心函数是：

`generate(query, contexts, temperature=0.1)`

### 20.2 它生成答案前先做什么

不是直接把 query 发给模型。

它先做：

1. 构造 system prompt
2. 构造 user prompt
3. 把每个 context 编号
4. 在 prompt 里要求模型用 `[1][2]` 这种格式标引用

所以它的提示词设计目标很明确：

1. 只能基于给定上下文回答
2. 必须带引用编号
3. 信息不足就要拒答

### 20.3 它为什么能返回 citations

因为它不是靠模型返回 JSON citation 对象，而是先要求模型在答案里写 `[1][2]`。

然后生成器再通过 `_parse_answer_with_citations(...)` 去解析这些编号，把它们映射回实际的上下文块。

所以 citations 的来源不是“模型自己吐出完整结构化引用”，而是：

1. 模型写引用编号
2. 代码再把编号翻译成 citation 对象

这比完全依赖模型生成结构化 citation 更稳。

### 20.4 如果模型拒答怎么办

这里还有一个很实用的 fallback：

`_build_extract_answer_from_contexts(...)`

它的思路是：

如果已经有高置信度证据，但模型还是拒答或没给引用，那就直接从强证据里抽一句或几句内容，拼成一个保底回答。

也就是说：

系统不完全把“能不能答出来”押在生成模型的表现上。

它还有一个抽取式兜底层。

这说明项目在问答层也做了工程稳健性设计，不是完全裸奔。

## 24. 读这个项目时最容易混淆的几个点

### 24.1 解析不等于切片

解析是把文档变成结构化块。  
切片是把结构化块变成知识块。

### 24.2 增强不等于正式入库

增强完成时，数据还没真正入正式库。

### 24.3 enhanced 不等于替换 child

enhanced 是附加层，不是覆盖层。

### 24.4 embedding 后不等于立刻写库

embedding 后还会继续补语义、步骤、因果关系，还会做实体物化。

### 24.5 在线问答不是“直接问大模型”

它始终是“先检索，再生成”。

## 27. 补一段你现在最关心的主链路：切片增强做完以后，代码下一步到底干什么

前面我们已经走到 `hierarchical.chunk(...)` 这一段，也就是：

1. `ContentBlock[]` 已经清洗完了
2. `Chunk[]` 已经切出来了
3. `child / parent / enhanced` 也都已经组织好了
4. `link_related_chunks(...)` 这种基础关系也已经补了一轮
5. `save_chunk_drafts(...)` 也已经把草稿存起来了

很多人看到这里，会下意识以为“这不就已经入库完成了吗”。

其实还差一大截。

从真实代码顺序看，切片增强完成以后，主链路并不是直接写 `chunks` 正式表，而是进入 [backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py) 里的：

`confirm_pipeline(task_id)`

这个函数可以理解成一句大白话：

`把已经做好的 chunk 草稿，变成正式知识库数据`

也就是说，前半段 `run_pipeline_real(...)` 更像“先生产草稿”，后半段 `confirm_pipeline(...)` 才是“把草稿转正”。

### 27.1 为什么项目要故意拆成“草稿阶段”和“正式入库阶段”

这不是多此一举，而是这个项目很重要的工程设计。

因为切片增强做完时，系统虽然已经拿到了很多 `Chunk`，但这时候还有几个问题没解决：

1. 这些 chunk 质量到底过不过关，还没筛
2. 这些 chunk 还没有 embedding，检索还用不了
3. chunk 之间更深一层的语义 / 流程 / 因果关系还没补
4. 图谱层要用的实体还没抽出来、没合并、没落库
5. pgvector 正式库也还没写

所以 `chunk_drafts` 的意义就是：

1. 先把“候选结果”存下来
2. 给人工确认、预览、二次处理留空间
3. 等确认后，再走后半段更贵、更正式的流程

从产品角度看，这样做也更安全。

因为如果切完就直接写正式库，一旦切片质量差、增强乱、结构有问题，后面检索、问答、图谱都会一起被污染。

## 28. `confirm_pipeline(...)` 的真实顺序

这一段建议你直接配合 [backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py:1506) 看。

它的顺序其实很硬，很直：

1. 先把这次任务对应的 `chunk_drafts` 读出来
2. 跑质量门控
3. 对通过门控的 chunk 做向量化
4. 基于向量和文本关系补充 chunk 之间的连接
5. 抽实体、做实体合并、给实体做 embedding
6. 最后把文档、chunks、relations、triples 一起写进 pgvector

如果再翻成对象变化链，就是：

`chunk_drafts`

-> `passed chunks`

-> `embeddings`

-> `enriched relations`

-> `entities / mentions / triples`

-> `documents + chunks + chunk_relations + kg_triples`

这才是“正式入库完成”的完整定义。

## 29. 第一步：先把草稿读回来，而不是直接沿用内存里的结果

`confirm_pipeline(...)` 一上来做的是：

`load_confirmable_chunks(task_id)`

这一步很值得注意。

因为它说明后半段并不依赖前半段函数栈里还留着的 Python 对象，而是重新从草稿存储里读一次。

这样做有几个现实好处：

1. 前后两段流程解耦了
2. 即使服务中间切换、重试、恢复，也能继续
3. 人工确认、编辑草稿以后，后半段吃到的是“确认后的草稿”
4. 后半段逻辑不需要假设前半段内存状态一定还在

这就是典型的“中间状态持久化”设计。

也正因为这样，`chunk_drafts` 在这个项目里不是临时小缓存，而是真正的中间产物层。

## 30. 第二步：质量门控，不是所有切好的 chunk 都能进正式库

这一步调用的是 [core/cleaner/quality_gate.py](/E:/all_project/AI/rag-enhancement/core/cleaner/quality_gate.py) 里的：

`apply_quality_gate(...)`

### 30.1 它的输入和输出是什么

输入：

`list[Chunk]`

输出：

`QualityGateResult`

里面最关键的是：

1. `chunks`：通过门控的 chunk
2. `discarded_count`：被丢掉多少
3. `discarded_chunks`：具体丢了哪些、为什么丢
4. `scores`：如果启用了 LLM 打分，每个 chunk 的分数是多少
5. `metrics`：这一步的耗时和 token 等统计

### 30.2 它到底在筛什么

先说结论：

它不是做“漂亮排版”，也不是做“语义增强”，而是做“最后一道坏数据拦截”。

代码里的规则很直接：

1. 表格 chunk、图片 chunk 默认先放行
2. 普通文本 chunk 先看标点/符号占比是不是太离谱
3. 如果配置要求，还会调 LLM 给 chunk 打质量分
4. 分数不够的就不进正式库

所以这一步的目标非常朴素：

`别把明显垃圾块送进正式知识库`

### 30.3 为什么要把质量门控放在切片之后，而不是解析之后

因为它筛的是 `Chunk`，不是 `ContentBlock`。

这点很重要。

项目在这里关心的已经不是“原文块是不是干净”，而是：

`这个切出来、准备拿去检索的知识块，有没有独立价值`

比如有些内容：

1. 解析时看起来正常
2. 清洗时也不该删
3. 但切成 chunk 以后，可能只剩一句残句
4. 或者符号太多、内容太碎、检索价值很低

这类问题只有到了 `Chunk` 层才看得清。

所以质量门控放在这里，逻辑上是合理的。

### 30.4 这一层和前面的普通清洗有什么本质区别

前面的 `clean_blocks(...)` 更像：

`不要把明显脏的原料送进切片机`

这里的 `apply_quality_gate(...)` 更像：

`切完以后，再把成品里不合格的挑出去`

也就是说：

1. 前面的清洗对象是 `ContentBlock`
2. 这里的质检对象是 `Chunk`
3. 前面偏“原料预处理”
4. 这里偏“成品验收”

## 31. 第三步：只给通过门控的 chunk 做 embedding

质量门控通过后，代码才会做：

`texts = [chunk.content for chunk in passed]`

然后调用 [core/embedding/client.py](/E:/all_project/AI/rag-enhancement/core/embedding/client.py) 里的：

`embed_texts_with_metrics(texts)`

### 31.1 为什么这里不是对所有草稿做向量化

很简单，因为 embedding 很贵。

如果一个 chunk 质量本来就不合格，还给它做 embedding，相当于：

1. 白花模型调用成本
2. 白占向量库存储
3. 白增加后续检索噪声

所以这里一定是：

`先筛，再向量化`

### 31.2 `embed_texts_with_metrics(...)` 到底做了什么

别把它理解成“把一组字符串丢给 embedding API”就完了。

从代码看，它至少还做了这些事：

1. 解析实际要用的 model、batch size、max concurrency、retry 次数
2. 把长列表文本切成多个 batch
3. 支持串行，也支持线程池并发
4. 如果配置了 key pool，会在多 key 之间调度
5. 遇到限流、失败会重试
6. 保证最终输出顺序和输入顺序对齐
7. 汇总 wall time、retryCount、token 用量、key pool 统计

所以它不是一个“单薄的 API 包装器”，而是：

`一个带并发、重试、指标统计的 embedding 执行器`

### 31.3 为什么“顺序对齐”这么重要

因为后面 `passed` 和 `embeddings` 是要一一对应一起写库的。

如果第 17 个 chunk 的向量跑到第 16 个位置，整个库就错了。

所以这段代码里有一个非常关键的工程目标：

`不管并发怎么跑，最后结果顺序必须跟输入文本顺序一致`

这也是它要维护 `offset`、分 batch 回填 `results[...]` 的原因。

### 31.4 这里的 embedding 结果后面给谁用

不是只给向量库用。

它至少服务三件事：

1. 后面正式写 `chunks.embedding`
2. 立刻拿来补 semantic link
3. 后面在线 RAG 检索时做 dense retrieval

所以 embedding 在这个项目里不是“附属步骤”，而是整个检索层的底座之一。

## 32. 第四步：embedding 做完还没结束，还要继续补关系

这是很多人特别容易漏掉的一段。

看 `confirm_pipeline(...)` 会发现，embedding 后面不是立刻 `write_to_pgvector(...)`，中间还插了三块：

1. `link_semantic(passed, embeddings)`
2. `detect_procedure_chunks(passed)` + `link_procedure(passed)`
3. `link_causal(passed)`

也就是说：

`向量化完成 != 可以直接写库`

系统还要利用这批已经成型的 chunk，再补一层更深的关系网络。

### 32.1 语义关系：`link_semantic(...)`

这一步是“拿 embedding 做 chunk 之间的语义相似关系”。

你可以把它理解成：

1. 哪些 chunk 说的是同一个知识点
2. 哪些 chunk 意思非常接近
3. 哪些 chunk 可能互相重复

这层关系的价值在于：

1. 后面 Graph RAG 可以用
2. 普通 RAG 的 related 扩展可以用
3. 结果去重和相邻补充也会更稳

### 32.2 流程关系：`detect_procedure_chunks(...)` + `link_procedure(...)`

这一步是识别“步骤型内容”。

大白话说，就是系统会尝试看：

1. 这是不是某个操作流程
2. 这几个 chunk 之间有没有前后步骤关系

然后补出类似：

1. `next_step`
2. `prev_step`

这样的关系。

这说明项目不是只把知识当成“平面文本块”，而是在尝试恢复一部分顺序结构。

### 32.3 因果关系：`link_causal(...)`

这一步进一步补：

1. 哪段内容是原因
2. 哪段内容是结果

也就是类似：

1. `cause_of`
2. `effect_of`

这一类边。

从这里你就能看出来，这个项目后半段正式入库前，做的已经不是“单纯切片处理”了，而是在逐步把 chunk 变成一个小型知识网络。

## 33. 第五步：实体物化，真正开始往图谱层走

再往下，`confirm_pipeline(...)` 会先单独建一个数据库连接，然后调用：

[core/kg/extraction_pipeline.py](/E:/all_project/AI/rag-enhancement/core/kg/extraction_pipeline.py)

里的：

`materialize_entities(conn, passed, kb_id)`

### 33.1 为什么这一步放在写 chunk 正式库之前

因为实体和 mentions 关系也属于正式知识库的一部分。

系统要先把实体层整理好，后面写库时，chunk 层、关系层、图谱层才是相互一致的。

### 33.2 它到底做了哪几件事

从代码顺序看，过程很清楚：

1. 只遍历 `layer="enhanced"` 的 chunk
2. 从这些 enhanced chunk 里取 `extracted_entities`
3. 用 `EntityMerger().merge(...)` 合并同名、同义、来源重复的实体
4. 用 `generate_definition(...)` 给实体生成定义
5. 对实体自己的“名字 + 定义”再做一次 embedding
6. 给相关 chunk 补 `mentions` 关系
7. 最后 `write_entities(conn, entities)` 写实体表

### 33.3 为什么只从 `enhanced` 层抽实体

这是一个很关键的设计判断。

说明项目作者认为：

1. 原始 `child` 更偏证据层
2. `enhanced` 更偏语义整理层
3. 实体抽取更适合吃“增强后的、更完整、更容易识别语义”的文本

这和前面的设计是一套逻辑：

1. `child` 负责保真
2. `enhanced` 负责补语义
3. 图谱抽取主要依赖补过语义的那层

### 33.4 为什么实体还要单独做 embedding

因为实体不只是展示标签，它后面也可能参与：

1. 图谱检索
2. 实体相似性判断
3. 实体聚合、召回、扩展

所以项目里并不是只有 chunk 有向量。

实体本身也是被向量化的对象。

## 34. 第六步：最后才是 `write_to_pgvector(...)`

等前面的质量、embedding、关系补全、实体物化都做完，才会进入真正的正式落库：

[core/output/pgvector_writer.py](/E:/all_project/AI/rag-enhancement/core/output/pgvector_writer.py:435)

`write_to_pgvector(...)`

### 34.1 这一步为什么一定要放最后

因为它是“最终提交”。

只有到了这里，系统才认为：

1. chunk 内容定下来了
2. embedding 定下来了
3. chunk relations 定下来了
4. triples 也准备好了

所以这里本质上是离线入库链路的最后落地动作。

### 34.2 它真正写了哪些东西

从代码顺序看，它至少做：

1. 确保默认知识库存在
2. 计算文件 hash
3. 建库连接、确保 schema 存在
4. 用 `file_hash` 检查是不是重复文档
5. upsert `documents`
6. 批量写 `chunks`
7. 批量写 `chunk_relations`
8. 批量写 `kg_triples`
9. commit

这里要特别注意一点：

`kg_triples` 也是在这里正式写的

而且代码里明确只从 `layer="enhanced"` 的 chunk 里取三元组。

这又一次说明：

增强层不只是“帮助召回”，还是图谱层的重要来源。

### 34.3 `file_hash` 在这里的意义是什么

这不是为了好看，是为了避免重复入库。

如果同一个文档已经进过库，系统可以：

1. 识别这是不是同一份文件
2. 避免重复写一遍 chunks
3. 必要时只更新文档级元数据

这类“幂等”和“去重”设计，在知识库系统里非常重要。

因为一旦重复写库，后面的检索和问答就会出现：

1. 重复命中
2. 证据重复
3. 统计膨胀

### 34.4 到这一步结束后，任务状态才会被真正标成成功

`confirm_pipeline(...)` 在写库成功后，还会：

1. 记录 export 阶段统计
2. 把 `task["chunk_count"]` 更新为正式写入数
3. 生成一份最新的 `chunks_preview`
4. `clear_chunk_drafts(task_id)` 清掉草稿
5. `_set_task_status(task, "success", done=True)`

这几行很能说明问题。

因为它告诉你：

1. 成功的标准不是“增强做完”
2. 也不是“embedding 做完”
3. 而是“正式写库成功，而且草稿已经清理，任务已转正”

## 37. 把 `quality_gate.py` 再拆细一点：它不是“高大上 AI 审稿”，而是很朴素的二段筛选

如果你直接看 [core/cleaner/quality_gate.py](/E:/all_project/AI/rag-enhancement/core/cleaner/quality_gate.py)，会发现这个文件其实一点都不花。

它的核心思路非常直白：

1. 先用便宜规则筛一遍
2. 如果配置要求，再用 LLM 打分筛第二遍

所以它不是“默认全靠大模型判质量”，而是：

`规则优先，LLM 可选补刀`

### 37.1 第一层：表格块、图片块先放行

这段代码一上来就判断：

1. `chunk.is_table_chunk`
2. `chunk.is_image_chunk`

如果是这两类，直接放进 `kept`。

这一步背后的想法很现实：

1. 表格和图片本来就不适合用“标点比例高不高”这种规则判
2. 这两类内容的价值经常不在普通正文句子里
3. 如果一刀切按文本规则筛，容易误杀

所以项目在这里采取的是：

`文本块严一点，表格块和图片块先别乱杀`

### 37.2 第二层：普通文本先看“是不是一堆符号”

对普通文本 chunk，它会：

1. 先 `strip()`
2. 统计有多少字符既不是字母数字，也不是空白
3. 算一个“符号/标点占比”
4. 如果超过 `max_punct_ratio`，直接丢掉

你可以把这一步理解成：

`这段内容是不是像一坨不成句的杂质`

这种规则虽然简单，但非常有用。

因为很多坏 chunk 的问题并不是“语义高深难懂”，而是：

1. 本来就是 OCR 噪声
2. 被切得太碎
3. 只剩符号、括号、列表碎片
4. 肉眼一看就不像正常知识条目

这类垃圾没必要花 LLM 钱再问一遍。

### 37.3 第三层：如果启用 LLM，就做 1 到 5 分的质量评分

只有当：

1. `min_score > 0`
2. 而且文本块不为空

时，才会走 `_llm_score_chunks(...)`。

这一步的输入不是整篇长文，而是每个 chunk 截断后的前 300 字。

为什么只给 300 字？

因为这一步的目标不是深度总结，而是粗判：

`这个 chunk 值不值得进知识库`

所以这里更像“批量评分器”，不是“精细理解器”。

### 37.4 LLM 打分的输出长什么样

代码要求模型返回的是 JSON 数组，每项包含：

1. `index`
2. `score`
3. `reason`

也就是说，这里不是让模型自由发挥长篇点评，而是让它像批改表格一样回结构化结果。

这有两个好处：

1. 代码容易解析
2. 批处理多段 chunk 比较省 token

### 37.5 如果 LLM 出错怎么办

这段代码很工程，不会因为质检模型出错就把整个流程搞崩。

如果单个 batch 调用失败，它会给这一批默认分数 `3`。

也就是说：

1. 不因为一次质检失败让整个确认流程直接挂死
2. 给一个中间分，尽量继续跑

这说明质量门控在这里是“有价值的防线”，但不是“宁可全流程失败也要死守”的硬门。

### 37.6 `score_only` 是干什么的

这个参数很容易被忽略。

它的意思是：

`只打分，不淘汰`

也就是：

1. 依然记录分数
2. 但不按分数删 chunk

这通常适合：

1. 先观察模型评分效果
2. 想做评估，不想立刻影响正式入库
3. 做灰度验证

所以这个文件不只是“质检器”，它也能充当“质量观测器”。

## 38. 把 `embedding/client.py` 再拆细一点：它不只是发请求，而是在做一整套批处理调度

前面我们说过 `embed_texts_with_metrics(...)` 很重要，这里再把它拆实一点。

### 38.1 它先做的不是调用 API，而是先把运行参数全部算出来

一上来它会先 resolve：

1. 用哪个模型
2. 每批多少条
3. 最大并发多少
4. 失败最多重试几次

也就是说，这一步先做的是“施工计划”，不是“马上开工”。

### 38.2 它会先把文本切成 batch

这一步的对象变化是：

`texts`

-> `[(batch_index, offset, batch_texts), ...]`

这里的 `offset` 很关键。

因为后面无论并发怎么跑，最后都要按原顺序把 embedding 放回去。

### 38.3 它支持两种 key 使用模式

#### 第一种：单 key / 单 client

如果没配置 key pool，它就用一个 client 正常发。

这时候逻辑比较朴素：

1. 调 embedding API
2. 失败就按重试次数补几次

#### 第二种：多 key pool

如果配了 `LLM_EMBEDDING_API_KEY_POOL` 之类的配置，它会建一个 `_EmbeddingKeyPool`。

这个池子不只是“存一堆 key”，它还维护：

1. 每个 key 当前 inflight 数
2. 每个 key 是否在 cooldown
3. 每个 key 的调用次数、成功次数、失败次数、限流次数、累计耗时
4. 整个池子的 throttle/retry/cooldown 统计

所以这不是“随便轮询一下 key”，而是带状态的调度器。

### 38.4 它挑 key 的逻辑不是随机，而是“谁空闲用谁”

`_EmbeddingKeyPool.acquire(...)` 的核心目标是：

`尽量选 inflight 最少、而且没在冷却中的 key`

也就是说，它在追求的是比较稳定的负载摊平，而不是随机撞运气。

### 38.5 遇到限流时，它不是只会傻重试

这里有一个很工程化的点。

如果错误像限流错误：

1. 当前 key 会被记一次 throttle
2. 这个 key 可能进入 cooldown
3. 这批任务会尝试换另一个 key 重跑

也就是说，它区分了两类失败：

1. 普通失败：同 key 重试
2. 限流失败：优先换 key

这就比“所有异常都原地重试”成熟得多。

### 38.6 并发执行后，为什么最后还能顺序不乱

因为它不是把结果 append 到列表末尾，而是按：

`results[offset:offset + len(embeddings)] = embeddings`

这种方式回填。

所以就算第 7 批先回来，第 2 批后回来，也不会把顺序打乱。

这个设计非常关键。

因为后面：

1. `passed[0]` 必须对应 `embeddings[0]`
2. `passed[1]` 必须对应 `embeddings[1]`

一旦错位，整库都污染。

### 38.7 它最后返回的不是裸向量，而是“向量 + 运行指标”

`EmbeddingRun` 里返回两块：

1. `embeddings`
2. `metrics`

指标至少包括：

1. `batchSize`
2. `batchCount`
3. `maxConcurrency`
4. `retryCount`
5. `embeddingWallMs`
6. token 统计
7. key pool 统计

所以这一步不仅为了“算出向量”，也为了后面控制台、日志、性能追踪能看见这一步到底花了什么代价。

## 39. 把三个 linker 分开看：它们补的不是同一种关系

很多人会把 `semantic_linker`、`procedure_linker`、`causal_linker` 混成一句“关系补全”。

这样记太粗了。

其实它们分别在补三种完全不同的关系观。

### 39.1 `semantic_linker.py`：补“意思接近”

看 [core/chunker/semantic_linker.py](/E:/all_project/AI/rag-enhancement/core/chunker/semantic_linker.py)，它做的是：

1. 只看 `layer == "child"` 的 chunk
2. embedding 数量必须和 chunks 数量一一对应
3. 两两比较相似度
4. 超过阈值的，补成语义边

#### 为什么只看 `child`

因为 `child` 是最原始、最稳定的知识单元。

如果把 `enhanced` 也混进来，会出现两个问题：

1. 增强文本本来就更像“解释文本”，容易相互看起来更像
2. 会把原始证据层和补充表达层搅在一起

所以这里的语义网络，是建立在原始 child 证据层上的。

#### 它补两种关系

1. `semantic_similar`
2. `duplicate_of`

区别在于：

1. 相似度高到普通阈值以上，是“语义相近”
2. 高到更高阈值以上，是“可能重复”

这就说明语义 linker 不是只有“像不像”，还顺手做了一层去重判断。

#### 为什么有 `skip_same_parent`

默认会跳过同一个 `parent` 下的 chunk 对。

这很合理。

因为同一标题下面紧挨着切出来的 child，本来就天然相近。

如果还全补成 semantic 边，会把图谱搞得特别稠，而且没太大新增信息量。

### 39.2 `procedure_linker.py`：补“先后顺序”

看 [core/chunker/procedure_linker.py](/E:/all_project/AI/rag-enhancement/core/chunker/procedure_linker.py)，它做了两步：

1. `detect_procedure_chunks(...)`
2. `link_procedure(...)`

#### 第一步：先识别谁像步骤块

它主要看两类信号：

1. 开头是不是“第 1 步 / 第一步 / step 1”这类顺序标记
2. 开头是不是“首先、接着、然后、随后、最后”这类时序词

所以这一步不是靠 embedding，也不是靠 LLM，而是纯规则。

#### 第二步：再在同一个 parent 下面串起来

它会按：

1. `procedure_order`
2. `chunk_index`

排序，然后依次补：

1. `next_step`
2. `prev_step`

注意，它是按 `parent_id` 分组的。

也就是说，项目默认认为“流程关系主要在同一个主题块内部成立”，不会轻易跨大标题乱连步骤。

### 39.3 `causal_linker.py`：补“因和果”

看 [core/chunker/causal_linker.py](/E:/all_project/AI/rag-enhancement/core/chunker/causal_linker.py)，它更简单粗暴。

它主要逻辑是：

1. 只看 `child` chunk
2. 如果当前 chunk 前 200 字里出现“因为、由于、导致、使得、所以、因此”这类词
3. 就拿它前一个 child chunk 当候选因果对
4. 补 `cause_of` / `effect_of`

这套规则明显不完美，但它非常便宜，而且对教材、说明文、因果解释类文本会有一定效果。

所以这一步的思路不是“做超强因果推理”，而是：

`先用低成本规则，把明显的因果线索捞出来`

### 39.4 为什么这三个 linker 都只在正式入库前做

因为它们处理的对象必须是：

1. 已经定型的 chunk
2. 已经通过质量门控的 chunk
3. 最好已经有 embedding 的 chunk

这时候再补关系，才不会前面还没定型、后面关系先乱写。

所以它们放在 `confirm_pipeline(...)` 这一段，顺序上是很稳的。

## 40. 把 `materialize_entities(...)` 再拆实：它做的不是“抽几个名词”，而是搭一层实体子图

前面我们已经讲过这个函数的位置，现在把内部动作再拆细一点。

### 40.1 它先收集原料：只吃 enhanced chunk 里的 `extracted_entities`

对象变化是：

`enhanced chunks`

-> `raw entity mentions`

这说明实体抽取结果本身，其实是在更早的增强阶段就已经埋进 chunk 里了。

`materialize_entities(...)` 现在做的是：

`把这些零散 mention 真正整理成实体层`

### 40.2 `EntityMerger().merge(...)` 的意义是什么

原始抽取里，很容易出现这些情况：

1. 同一个实体在多个 chunk 里反复出现
2. 同一个实体有轻微不同写法
3. 一个实体被多个增强块同时提到

如果不 merge，会怎样？

1. 实体表会膨胀
2. mentions 会很碎
3. 图谱会有很多重复节点

所以 merge 本质是在做：

`把 mention 合成实体节点`

### 40.3 `generate_definition(...)` 是在补什么

merge 之后，实体可能只有名字，没有说明。

这时项目会去生成 `definition`。

大白话理解就是：

`给实体补一句“它是什么”`

这样后面：

1. 实体自己更容易读
2. 实体 embedding 质量更好
3. 图谱检索和展示都更完整

### 40.4 为什么实体 embedding 用的是“名字 + 定义”

代码里做的是：

`f"{entity.name} {entity.definition or ''}"`

不是只拿名字去做向量。

这很合理。

因为很多实体光看名字信息太少，比如：

1. 缩写
2. 专有词
3. 章节里的局部简称

加上定义后，实体向量更接近“这个实体代表的含义”，而不只是“这个词长什么样”。

### 40.5 `mentions` 关系是怎么补回 chunk 的

实体整理好以后，它会按 `source_chunks` 倒查：

1. 哪些 chunk 提到了这个实体
2. 给这些 chunk 加 `mentions -> entity.id`

所以实体层不是孤立写进数据库，而是又回头和 chunk 层连起来。

这就是图谱层逐渐成网的关键一步。

## 41. 把 `pgvector_writer.py` 再拆细：它不是只写 `chunks`，而是在组织一次完整事务

很多人第一次看文件名 `pgvector_writer.py`，容易以为它只是“向量入库工具”。

实际上它干的是：

`把一整批正式知识库对象一次性落进数据库`

### 41.1 第一件事：先判断要不要跳过重复文档

`compute_file_hash(...)` 会先算文件 hash。

然后：

`find_document_id_by_hash(conn, kb_id, file_hash)`

去查当前知识库里是不是已经有这份文档。

如果有，代码不是再把所有 chunk 重写一遍，而是：

1. 只 upsert 文档记录
2. commit
3. 返回 `skipped=True`

这说明这里对“重复文档”采用的是：

`文档级去重`

### 41.2 第二件事：先 upsert `documents`

写 chunk 之前，必须先拿到 `document_id`。

因为后面每个 chunk 都要挂到某个 document 下面。

所以这里的顺序必须是：

`document 先落`

然后才是：

`chunks / relations / triples`

### 41.3 第三件事：`write_chunks_batch(...)` 真正写的是检索主表

它给每个 chunk 组织的字段非常多，不只是正文：

1. `content`
2. `source`
3. `page`
4. `chunk_index`
5. `strategy`
6. `title`
7. `char_count`
8. `is_table_chunk`
9. `is_image_chunk`
10. `image_path`
11. `layer`
12. `parent_id`
13. `related_ids`
14. `search_text`
15. `search_vector`
16. `embedding`

这说明 `chunks` 表本身就是整个检索层最核心的总表。

它同时服务：

1. 展示
2. 全文检索
3. 向量检索
4. 结构回溯

### 41.4 `search_text` 和 `search_vector` 是什么意思

`search_text` 不是原文的简单别名。

它是把：

1. `title`
2. `content`
3. `source`

拼起来形成一个更适合全文检索的文本串。

然后 `search_vector` 再对这个文本做：

`to_tsvector('simple', search_text)`

也就是说：

1. `embedding` 负责 dense retrieval
2. `search_vector` 负责 PostgreSQL 文本检索

这就是普通 RAG 里 dense + sparse 混合检索的数据库基础。

### 41.5 为什么 writer 支持 `values` 和 `copy` 两种写法

这是性能优化点。

`_write_mode()` 决定当前写入方式：

1. `values`
2. `copy`

如果是 `copy`，它会：

1. 先建临时表
2. 用 CSV buffer 批量 COPY 进临时表
3. 再从临时表插到正式表

这样做通常是为了大量写入时更快。

所以这个文件不只是“能写进去”，还在关心“怎么写更快”。

### 41.6 第四件事：`write_chunk_relations_batch(...)` 把边落下来

这一层会遍历每个 chunk 的 `relations`，写进 `chunk_relations`。

这里还有一个小但关键的动作：

它先用 `(src, dst, rel_type)` 做 `seen` 去重。

也就是说，哪怕前面多个步骤补关系时重复加了同一条边，落库前也会先尽量去一下重。

### 41.7 第五件事：`write_kg_triples_batch(...)` 只从 enhanced 层写三元组

这一点前面说过，但值得再强调一次。

代码明确：

1. 只遍历 `chunk.layer == "enhanced"`
2. 把其中的 `extracted_triples` 写进 `kg_triples`
3. `source_chunk` 记录的是 `parent_id or chunk.id`

也就是说，三元组层主要吃的是增强语义层，不是原始 child 原文层。

### 41.8 最后 commit，才算这一整批正式落地

`write_to_pgvector(...)` 不是每写一张表就自己算完成。

它是在：

1. `documents`
2. `chunks`
3. `chunk_relations`
4. `kg_triples`

都准备好后，最后一次 `conn.commit()`。

所以从事务角度看，这一整批数据是尽量被当成一个整体提交的。

这就是为什么它更像“正式发布”而不是“随手落点数据”。

## 45. `run_rag_pipeline(...)` 才是普通在线问答的真实主流程骨架

看 [backend/adapters/rag_adapter.py](/E:/all_project/AI/rag-enhancement/backend/adapters/rag_adapter.py)，它一上来会实例化四个角色：

1. `HybridRetriever()`
2. `ParentChildReranker()`
3. `RAGGenerator()`
4. `RAGScorer()`

这四个对象基本就把普通 RAG 的在线主链路完整拆开了。

你可以先把它们粗记成：

1. `Retriever`：先找候选证据
2. `Reranker`：把候选证据重新排顺序
3. `Generator`：基于证据生成答案
4. `Scorer`：给这次问答打分

所以普通在线链路的主顺序其实非常清晰：

`retrieve -> rerank -> generate -> score`

### 45.1 为什么这里要单独再来一个 reranker

因为“找得到”不等于“排得好”。

检索器的目标更偏：

`尽量不要漏`

而重排器的目标更偏：

`把最该进答案上下文的块排到前面`

这两个目标并不完全一样。

所以项目在在线阶段没有偷懒直接：

`retrieve -> generate`

而是明确插了一层：

`retrieve -> rerank -> generate`

这会让最终答案上下文更稳。

### 45.2 这条链路还有一个特殊分支：媒体编号快速直达

如果 retriever 发现用户问的是：

1. “图 1-2”
2. “表 3-1”

这类媒体编号型问题，`run_rag_pipeline(...)` 会走短路分支：

1. 不做正常 rerank
2. 不调生成器
3. 直接构造定位回答
4. relevance / faithfulness 直接给到 `1.0`

这说明项目对教材场景下的“图表编号问法”是做了专门工程优化的。

## 46. 第一步：`HybridRetriever.retrieve(...)` 先做候选召回

核心入口在：

[core/rag/retriever.py](/E:/all_project/AI/rag-enhancement/core/rag/retriever.py:621)

`HybridRetriever.retrieve(...)`

这个函数建议你不要把它理解成“单一检索算法”，它其实是一个分支很多的召回编排器。

### 46.1 第一分支：先看是不是图表编号直达

它第一件事不是做 query embedding，而是：

`_media_ref_retrieve(query, kb_id, top_n=top_k)`

这一步的核心意思是：

`如果用户已经明确点名某个图/表编号，就优先走专门直达路径`

#### 这条路径为什么值得单独开

因为这类问题没必要每次都：

1. 先向量化 query
2. 再 dense 检索
3. 再 BM25
4. 再 RRF

明明用户已经给了很强的结构化线索。

所以这条路径更像：

`结构化定位优先级高于通用语义召回`

#### 命中后它还会做什么

命中媒体块后，它还会：

1. `_expand_related(...)` 扩展相关块
2. `_coarse_filter(...)` 做粗过滤
3. 直接返回

所以它不是只拿一条孤立 chunk，而是尽量把相关证据也顺手带上。

### 46.2 第二分支：普通问题先做 query embedding

如果不是媒体编号直达，它才会做：

`embed_query_cached(query)`

这一步很关键，因为它说明：

1. 用户 query 本身也要向量化
2. 而且还有 TTL 缓存

也就是说，对重复或相近时段的查询，系统不一定每次都重新打 embedding API。

### 46.3 第三分支：如果启用了 retrieval snapshot，先走快路径

如果：

1. snapshot 功能开启
2. 而且这次没有额外 filters

系统会优先尝试：

1. `fetch_retrieval_snapshot(...)`
2. `fold_enhanced_snapshot(...)`
3. `expand_related_snapshot(...)`
4. `_coarse_filter(...)`

这条路径的本质可以理解成：

`提前准备好的召回快照，尽量减少现场临时检索开销`

如果成功，就直接返回。

如果失败，也不会把整个流程打死，而是自动回退到标准混合召回路径。

这又体现了一个很工程化的思路：

`快路径能用就用，失败就回退，不影响主功能`

### 46.4 第四分支：标准混合召回

如果前面两个捷径都没走通，它才进入标准主路径：

1. `_dense_retrieve(...)`
2. `_sparse_retrieve(...)`
3. `_structured_retrieve(...)`
4. `_rrf_merge(...)`
5. `_fold_enhanced_to_children(...)`
6. `_expand_related(...)`
7. `_coarse_filter(...)`

这一串就是普通在线召回最核心的一套组合拳。

## 47. `_dense_retrieve(...)`：向量检索在做什么

这一步会拿 query vector 去打数据库里的：

`c.embedding <=> query_vector`

也就是 pgvector 的向量距离检索。

### 47.1 它检索的不是全部层，而是默认只看 `child + enhanced`

代码里有：

`_DEFAULT_RETRIEVAL_LAYERS = ("child", "enhanced")`

所以它默认不会把 `parent` 也拿来做主召回层。

这很重要。

因为项目对三层 chunk 的分工很明确：

1. `child`：原始证据主层
2. `enhanced`：增强召回层
3. `parent`：结构导航层

所以 parent 不是主召回主力。

### 47.2 为什么 dense 结果里还保留那么多字段

因为后面不是只要一个分数。

还要继续做：

1. 折回 child
2. 扩 related
3. 重排
4. 生成引用

所以候选对象里会保留：

1. `document_id`
2. `document_name`
3. `page`
4. `layer`
5. `parent_id`
6. `related_ids`
7. `chunk_index`
8. `image_path`

也就是说，retriever 在返回的不是“分数数组”，而是“后续整条链路都能继续处理的候选对象”。

---

## 48. `_sparse_retrieve(...)`：BM25 在补什么

这一层就是稀疏检索，也可以理解成关键词检索。

### 48.1 它为什么还有缓存

因为 `_build_bm25_index(kb_id)` 会先把当前知识库里的 chunk 拉出来，建一个 BM25 索引。

这个索引会缓存在：

`_bm25_cache`

里。

原因很简单：

1. 一次查询都重建 BM25 太浪费
2. 同一知识库会被反复查

所以这里做了一个进程内缓存。

### 48.2 稀疏检索和 dense 检索的角色差异

dense 更擅长：

1. 同义表达
2. 语义接近
3. 用户说法和原文不完全一致

sparse 更擅长：

1. 关键词明确命中
2. 专有名词
3. 编号、术语、短语直接匹配

所以两者不是互相替代，而是互补。

这也是为什么后面要做 hybrid merge。

---

## 49. `_structured_retrieve(...)`：不是每次都用，但它说明项目支持结构化过滤

这个函数只有在传了 `filters` 的时候才真正起作用。

它支持按这类条件收窄：

1. `source`
2. `title_like`
3. `is_table`
4. `page_range`

这说明项目的在线检索并不是只能“纯自然语言模糊查”。

在需要的时候，它也可以：

`自然语言 + 结构条件`

一起用。

这对控制台、管理侧、精确定位类场景很有用。

---

## 50. `_rrf_merge(...)`：为什么三路结果不是简单拼接，而是要融合

这里用的是 RRF，也就是 Reciprocal Rank Fusion。

你不需要先背数学公式，先记它的大白话作用：

`把多路召回里都排得靠前的候选，整体往前推`

### 50.1 为什么不用“谁分高就听谁”

因为 dense、sparse、entity/structured 这几路分数根本不在一个量纲上。

如果硬拼原始分数，往往会：

1. 某一路分数范围天生大
2. 直接把别的路全部淹掉

RRF 的好处就是：

1. 少依赖不同算法的原始分数可比性
2. 更依赖“排名是否稳定靠前”

这对混合召回很实用。

### 50.2 融合后候选还会带上“来自哪些通道”

它会在候选对象里记：

`sources = ["embedding", "bm25", ...]`

这意味着后面不仅知道“这个候选分高”，还知道：

`它到底是靠哪几路一起把它推上来的`

这对 trace 和诊断很有帮助。

---

## 51. `_fold_enhanced_to_children(...)`：这一步是整个项目在线链路最有特点的地方之一

这一层你一定要吃透。

因为它直接体现了项目对 `enhanced` 和 `child` 的分工哲学。

### 51.1 它在做什么

大白话说：

`允许 enhanced 帮忙命中，但尽量把最终证据折回它的原始 child`

也就是说：

1. 如果命中的是普通 child，就直接保留
2. 如果命中的是 enhanced，而且它有 `parent_id`
3. 就去把对应 child 找回来
4. 最终返回 child 作为主证据对象

### 51.2 为什么要这样折回

因为增强块常常更适合“找得到”，但原始 child 更适合“拿出来给人看”。

增强块的问题在于：

1. 它是补充表达
2. 可能不是原文
3. 可能更偏总结

而 child 的优势在于：

1. 更贴近原文
2. 更适合证据展示
3. 更适合 citation 回溯

所以项目做的是：

`enhanced 负责提高召回`

`child 负责承担证据`

这正是你前面离线链路里看到的那套双层设计，在在线链路里的落地。

### 51.3 它还会保留哪些“我是被 enhanced 命中的”痕迹

不会把 enhanced 的痕迹彻底抹掉。

它还会保留：

1. `matched_enhanced_id`
2. `matched_enhanced_text`
3. `matched_by`

也就是说，系统既把证据折回 child，又保留“这次是 enhanced 帮你命中的”这条 trace。

这很工程化，也很适合调试。

---

## 52. `_expand_related(...)`：为什么最终返回里常常会多出一些“分不高但相关”的块

这一步会看每个候选的：

`related_ids`

然后把这些关联块补进来，默认给一个较低分数，比如 `0.3`。

### 52.1 这不是为了让它们抢第一，而是为了补上下文

related 扩展的意义不是：

`把它们变成主命中`

而是：

`把和主命中紧密相连的上下文也顺手拉进候选池`

这样做的好处是：

1. 最终生成答案时上下文更完整
2. 图表、相邻步骤、同主题块更容易一起出现
3. 用户看到证据时不容易只看到孤零零一块

### 52.2 为什么 related 扩展放在 fold 之后

因为系统先想确定“最终主证据是谁”，再围绕主证据扩展相关块。

如果先扩 related 再折回 child，链路会更乱，也更容易重复。

---

## 53. `_coarse_filter(...)`：最后再做一次统一粗过滤

前面做了这么多召回和扩展，最终还会统一用：

1. `min_score`
2. `top_n`

再裁一遍。

这一步的目标非常现实：

`别把太弱的候选也一起塞进后面的重排和生成`

所以 retriever 的最后一步，不是“谁来都收”，而是再做一道收口。

---

## 54. 第二步：`ParentChildReranker.rerank(...)` 重新安排“哪些证据最应该进答案上下文”

核心入口在：

[core/rag/reranker.py](/E:/all_project/AI/rag-enhancement/core/rag/reranker.py)

`ParentChildReranker.rerank(...)`

### 54.1 它第一件事是给 child 做 rerank

`_rerank_children(...)` 会：

1. 把 child 候选拿出来
2. 调 rerank API
3. 得到一组新的 `rerank_score`

如果 rerank API 不可用，或者没打出有效分，就回退用原始 `score`。

这说明 rerank 在这里是增强器，不是硬依赖。

### 54.2 为什么它名字叫 `ParentChildReranker`，但最后主要还是保留 child 粒度

因为这个项目最终答案证据还是更偏 child。

文件里甚至有一行注释，意思很明确：

`最终证据保持 child 粒度，parent 更偏结构导航，不拿来替代 child 证据`

所以它虽然叫 parent-child reranker，但不是把 parent 当成答案证据主角。

### 54.3 `_window_reorder(...)` 在做什么

这个函数很实用。

如果候选是 parent，它会去数据库里把这个 parent 下的一组 child 拉出来，然后围绕 `best_child_id` 取一个小窗口。

最后把这个窗口拼成：

`context_window`

大白话说就是：

`不要只给生成器看孤零零一句，要尽量给它看命中点附近的一小段局部上下文`

这一步非常关键，因为生成器通常吃“局部上下文窗口”比吃“单一碎片句子”效果更稳。

### 54.4 `use_llm_check` 是干什么的

这是最后一道可选人工智能复核。

`_llm_final_check(...)` 会逐个问模型：

`这段内容和这个问题到底相不相关？只回答 yes / no`

如果模型判断不相关，就可以把这个候选刷掉。

所以它更像：

`最终上下文前的 LLM 守门员`

不过注意，它是可选的，不是默认每次必须跑。

## 55. 第三步：`RAGGenerator.generate(...)` 才真正开始产出答案

核心入口在：

[core/rag/generator.py](/E:/all_project/AI/rag-enhancement/core/rag/generator.py:168)

`RAGGenerator.generate(...)`

### 55.1 如果一个上下文都没有，它会直接拒答

这很合理。

因为这个项目的目标不是“强行编答案”，而是“基于知识库回答”。

所以没上下文就直接返回：

`根据现有文档无法回答该问题`

### 55.2 它先构 prompt，不是直接裸问模型

`_build_rag_prompt(...)` 会把每个 context 编号成：

`[1] 来源，位置`

再加上一段截断后的内容。

也就是说，生成器不是把一堆原始 JSON 扔给模型，而是先整理成：

`可引用、可定位、带编号的上下文列表`

### 55.3 它为什么强制要求 `[1][2]` 这种引用

因为后面代码要靠这个编号把答案里的引用反解出来。

也就是说：

1. 模型负责在答案里写引用编号
2. 代码负责把编号映射回真实 citation 对象

这比让模型直接输出一整套复杂 citation JSON 稳得多。

### 55.4 `_parse_answer_with_citations(...)` 在做什么

它会从模型答案里找：

`[数字]`

然后：

1. 去 contexts 里找到对应块
2. 补成 citation 对象
3. 给出 `source / page / location / snippet / chunk_id`

所以 citation 不是模型“一次性全生成好”的，而是：

`模型给编号`

`代码做映射`

### 55.5 为什么它还有抽取式 fallback

这是这套生成器很务实的地方。

如果出现这些情况：

1. 模型拒答
2. 模型没给 citations
3. 模型报错

系统不会立刻全盘失败，而是会尝试：

`_build_extract_answer_from_contexts(...)`

它会从高分上下文里抽最有用的句子，拼出一个保底答案。

这说明项目不是把“能不能答出来”完全押在生成模型是否听话上，而是给自己留了一个工程保底层。

### 55.6 这个 fallback 和普通生成有什么本质区别

普通生成是：

`让模型组织答案`

抽取式 fallback 是：

`从已有高置信证据里摘句子拼答案`

所以 fallback 更保守，但通常也更稳。

## 56. 第四步：`RAGScorer.score(...)` 生成后还要再打分

核心入口在：

[core/rag/scorer.py](/E:/all_project/AI/rag-enhancement/core/rag/scorer.py)

### 56.1 它默认先做规则分，不先上 LLM

`_rule_score(...)` 主要产出两类分：

1. `relevance_score`
2. `faithfulness_score`

#### `relevance_score` 怎么来

它本质上是看这批 contexts 的 `rerank_score` / `dense_score` 平均水平。

也就是说，它更像：

`这次找到的证据整体强不强`

#### `faithfulness_score` 怎么来

它不是让模型判断“答案像不像真话”，而是检查：

1. 答案里引用了多少条 citation
2. 这些 citation 是否真能对上当前 contexts

所以这个 fidelity/faithfulness 分更像：

`答案有没有老实引用它当前拿到的上下文`

### 56.2 `use_llm_score=True` 时才会再调一次模型评分

这一层是可选的。

如果启用，模型会被问一个更简单的问题：

`请对这次问答质量打 1-5 分`

再换算成 `0-1`。

所以：

1. 规则分负责稳定、便宜、可解释
2. LLM 分负责补一层更主观的整体质量感知

这两层不是替代关系，而是叠加关系。

## 57. 最后一步：`rag_service.py` 再把内部结果封装成外部返回，并把过程记下来

等 `run_rag_pipeline(...)` 返回后，`run_rag_query(...)` 还要做很多“产品化收口”动作。

### 57.1 它返回给前端的不只是一个 answer

最终返回结构里至少有：

1. `answer`
2. `cannotAnswer`
3. `citations`
4. `candidates`
5. `timings`
6. `recallChannels`
7. `trace`

这说明这个项目的在线问答目标不是：

`只吐一句答案`

而是：

`把答案、证据、耗时、召回渠道、链路痕迹一起交给前端`

### 57.2 `trace` 是干什么的

它会把这次流程拆成几段展示：

1. retrieval
2. rerank
3. generate
4. score

并给每段挂上 `latencyMs`。

这对前端控制台、调试、用户解释都非常有价值。

### 57.3 它还会额外记录哪些东西

除了把结果返回给调用方，它还会继续写三类记录：

1. `append_evaluation(...)`
2. `append_rag_query_log(...)`
3. `append_llm_call_log(...)`

也就是说，一次问答结束时，系统不只关心“用户看到了什么”，还关心：

1. 这次问答质量后来能不能评估
2. 这次请求日志能不能追
3. Token 花在哪些环节上

这就是它明显比 demo 更像产品系统的地方。

## 58. 把普通在线问答主链路压缩成一句最准确的话

如果你想把这整段在线主链路压成一句尽量不容易记错的话，建议记成：

`用户提问后，系统先做访问校验，再用 HybridRetriever 做多路召回，必要时走图表编号直达或 snapshot 快路径；然后用 ParentChildReranker 重排并组织局部上下文窗口；再由 RAGGenerator 基于带编号证据生成答案并解析 citations，必要时走抽取式 fallback；最后由 RAGScorer 评估质量，并由 rag_service 把答案、证据、时延、日志和用量统一收口`

这句话比较长，但它是贴近真实代码的。

## 61. 现在继续分叉支线：Graph RAG 是从哪里开始和普通 RAG 不一样的

Graph RAG 对外入口也还是在：

[backend/services/rag_service.py](/E:/all_project/AI/rag-enhancement/backend/services/rag_service.py)

对应函数：

`run_graph_rag_query(...)`

### 61.1 它和普通 RAG 一样的地方

先别急着只看差异，先把相同点记住。

它和普通 `run_rag_query(...)` 一样，也会先做：

1. 知识库访问校验
2. `request_id` 生成
3. 调下游 pipeline
4. 记录 query log

所以它不是完全独立的一套系统，仍然共享：

1. 同一批知识库
2. 同一套鉴权逻辑
3. 同一套日志和治理外壳

### 61.2 它和普通 RAG 真正开始分叉的地方

普通 RAG 调的是：

`run_rag_pipeline(...)`

Graph RAG 调的是：

`run_graph_rag_pipeline(...)`

也就是 [backend/adapters/rag_adapter.py](/E:/all_project/AI/rag-enhancement/backend/adapters/rag_adapter.py) 里的另一条入口。

而这个入口内部会调用：

[core/rag/graph_retriever.py](/E:/all_project/AI/rag-enhancement/core/rag/graph_retriever.py)

里的：

`GraphRetriever.retrieve(...)`

所以普通 RAG 和 Graph RAG 的真正分叉点，可以记成：

`普通 RAG：HybridRetriever -> Reranker -> Generator -> Scorer`

`Graph RAG：GraphRetriever`

也就是说，Graph RAG 不是“在普通 RAG 后面再加一点图谱功能”，而是从检索编排层就换了一套思路。

## 62. `GraphQueryRequest` 比普通查询多了什么

看 [backend/schemas/requests.py](/E:/all_project/AI/rag-enhancement/backend/schemas/requests.py)，Graph RAG 的请求模型是：

`GraphQueryRequest`

它和普通 `QueryRequest` 相比，多两个很关键的东西：

1. `explain`
2. `intent`

### 62.1 `explain` 是干什么的

它控制要不要把路径解释带出来。

大白话说就是：

`不仅告诉你命中了什么，还告诉你这些结果是沿着什么关系一路扩出来的`

这就是 Graph RAG 和普通 RAG 很不一样的一点。

普通 RAG 更偏结果导向：

`这是答案，这是证据`

Graph RAG 更愿意再多给一层：

`这是我为什么把这些证据串起来的路径`

### 62.2 `intent` 是干什么的

它允许调用方直接指定意图。

如果不指定，系统就自己猜。

这个“意图”会直接影响图扩展时：

1. 哪类关系更重要
2. 哪些边权更高
3. 最终扩图方向偏向哪里

所以它不是无关紧要的说明字段，而是会真影响检索行为。

## 63. `GraphRetriever.retrieve(...)` 的总思路：先找种子，再沿图扩

先给你一句最该记住的话：

普通 RAG 更像：

`先把最相关块找出来，再重排，再回答`

Graph RAG 更像：

`先找一批可信种子块，再沿关系网络往外扩，再把扩出来的子图结果组回来`

这就是两者思维方式的根本差别。

### 63.1 它第一步先决定“这是哪类问题”

一上来它会做：

`classify_intent(query)`

或者如果调用方手动传了 `intent`，就直接用外部指定值。

也就是说，Graph RAG 不是完全不看问题类型的盲扩图，而是：

`先判断问题想问什么，再决定扩图优先方向`

## 64. `intent_router.py`：Graph RAG 先做的是问题意图分类

看 [core/rag/intent_router.py](/E:/all_project/AI/rag-enhancement/core/rag/intent_router.py)，会发现它目前非常朴素，是规则分类，不是大模型分类。

### 64.1 它大致分哪几类

目前主要分：

1. `procedure`
2. `concept`
3. `data`
4. `visual`
5. `general`

### 64.2 它怎么判

就是看 query 里有没有一些关键词。

比如：

1. “如何、步骤、流程、方法”更像 `procedure`
2. “什么是、定义、含义、概念”更像 `concept`
3. “多少、数据、比例、年份”更像 `data`
4. “图、示意图、图示、展示”更像 `visual`

所以这一层非常像：

`低成本意图路由器`

### 64.3 为什么这里不用复杂模型，先用规则就够了

因为它的作用不是给用户做精细语义画像，而是只需要解决一个更实用的问题：

`后面的图扩展，优先沿哪种关系走`

对这个目的来说，规则法已经能先跑起来，而且：

1. 快
2. 便宜
3. 可解释
4. 好改

## 65. Graph RAG 的第一段召回，其实仍然复用了普通 RAG 的几块能力

这点很重要，别误以为 Graph RAG 完全抛弃了普通 RAG 的召回基础。

在 [core/rag/graph_retriever.py](/E:/all_project/AI/rag-enhancement/core/rag/graph_retriever.py) 里，它还是先做：

1. `embed_query_cached(query)`
2. `_dense_retrieve(query_vec, kb_id, 20)`
3. `_sparse_retrieve(query, kb_id, 20)`
4. `_entity_retrieve(query, kb_id, ...)`
5. `_rrf_merge(...)`
6. `_expand_related(...)`
7. `_coarse_filter(...)`

也就是说，Graph RAG 的前半截，并不是“凭空开始扩图”，而是先把一批靠谱的 seed chunks 找出来。

### 65.1 为什么 Graph RAG 也还需要 dense + sparse

因为如果种子都找偏了，后面图扩得再漂亮也没用。

图扩展不是魔法。

它依赖一批“起点靠谱”的 seed。

所以 Graph RAG 的现实思路是：

`先借普通 RAG 的召回基本功找起点`

`再发挥图谱优势往外扩`

### 65.2 Graph RAG 比普通 RAG 多出来的一块：`_entity_retrieve(...)`

普通在线主链路里虽然也有实体层，但主召回核心讲的是 dense/sparse/structured。

Graph RAG 这里显式把：

`_entity_retrieve(...)`

拉进第一段主召回。

这一步会尝试：

1. 找跟 query 名称或别名相近的实体
2. 再通过 `entity_mentions` 找回提到这些实体的 chunk

所以 Graph RAG 比普通 RAG 更重视：

`从实体节点反查文本块`

这已经很像“图入口”了。

### 65.3 为什么 `data` 意图下，实体召回的 embedding 会被特殊处理

代码里有：

`query_vec if resolved_intent != "data" else None`

也就是说，数据类问题时，实体召回不强依赖 query embedding。

这背后的直觉是：

1. 数据类问题更可能依赖精确实体名、数值项、表格项
2. 不一定适合把实体相似度当主导

这说明 Graph RAG 已经开始根据意图微调召回策略了。

## 66. `seed_top` 是什么：Graph RAG 不是对全图乱扩，而是只从少量种子开始

Graph RAG 前半段融合完以后，会做：

`seed_top = _coarse_filter(..., top_n=max(top_k * 2, 10))`

这个 `seed_top` 就是“种子候选集合”。

大白话说：

`先挑出一批最值得沿图继续追的起点`

这一步非常重要，因为它说明 Graph RAG 的图扩展不是：

`把整个知识图谱跑一遍`

而是：

`从少量高质量起点做局部扩张`

这才现实，也更可控。

## 67. 真正体现 Graph RAG 特性的核心步骤：`graph_expand(...)`

这一步在：

[core/rag/graph_expander.py](/E:/all_project/AI/rag-enhancement/core/rag/graph_expander.py)

核心函数：

`graph_expand(seeds, kb_id, intent, max_hops=2, max_neighbors=50)`

这就是 Graph RAG 最像 Graph RAG 的地方。

### 67.1 它不是抽象“推理”，而是沿 `chunk_relations` 实打实地走边

函数里直接查的是：

`chunk_relations`

也就是说，它并不是在空中脑补关系，而是在使用离线入库阶段已经落下来的那些边：

1. `mentions`
2. `semantic_similar`
3. `duplicate_of`
4. `next_step`
5. `prev_step`
6. `cause_of`
7. `effect_of`
7. 以及其他关系

所以 Graph RAG 的基础，不是另起炉灶，而是吃前面离线阶段已经建好的关系网。

### 67.2 它怎么扩：本质上是带权 BFS

你可以把这段逻辑粗理解成：

1. 从 seed 开始
2. 维护一个 `frontier`
3. 每次拿一个点出来
4. 沿关系边往外走
5. 走一步 hop 就加一层 path
6. 分数随着 hop 衰减
7. 到达 `max_hops` 或 `max_neighbors` 就收

所以它本质上很像：

`限制层数、限制规模的图遍历`

而不是无限扩张。

### 67.3 它的分数为什么会随着 hop 变远而衰减

代码里有个核心公式：

`score = 上一步权重 * 边权重 * (0.6 ** hop) * 意图优先级`

你不用背公式，只要记住意思：

1. 越远的点，天然分更低
2. 边本身权重越高，扩过去越可信
3. 不同意图对不同关系类型偏好不同

也就是说，Graph RAG 不是只看“能不能走到”，还在看：

`值不值得走到`

### 67.4 `mentions` 为什么被特殊处理

如果当前关系类型是 `mentions`，它不会只把实体节点当成终点。

而是会继续去查：

`entity_mentions`

把“提到这个实体的其他 chunk”也拉出来。

这一步很关键。

因为它实现的是：

`chunk -> entity -> other chunks`

这种跨文本块跳转。

这就是图检索最有用的一类能力之一：

普通向量相似很难稳定做这种“通过共同实体连接文本”的跳跃。

### 67.5 `path` 在这里是什么意思

每次往外扩时，它都会把路径记下来。

路径里至少会记：

1. 从哪个 chunk 出发
2. 到哪个点
3. 用的什么 `rel_type`
4. 边权是多少

如果开了 `explain=true`，这条 path 最后会被带到结果里。

所以 Graph RAG 不是只有“结果更复杂”，还多了：

`我怎么走到这个结果的解释链`

## 68. Graph RAG 为什么要有 `INTENT_REL_PRIORITY`

这是 [core/rag/graph_expander.py](/E:/all_project/AI/rag-enhancement/core/rag/graph_expander.py) 里特别关键的一张表。

它大致定义了：

1. 概念类问题，哪些关系更值钱
2. 步骤类问题，哪些关系更值钱
3. 数据类问题，哪些关系更值钱
4. 图示类问题，哪些关系更值钱
5. 一般问题，哪些关系更值钱

### 68.1 例子：概念类问题更偏好哪些边

像：

1. `mentions`
2. `sibling`
3. `semantic_similar`

会更重要。

这很合理，因为问概念时，更希望找到：

1. 提到同一实体的块
2. 同主题同层级的块
3. 语义接近的块

### 68.2 例子：步骤类问题更偏好哪些边

像：

1. `next_step`
2. `prev_step`

权重就更高。

也就是说，当问题像“怎么做”时，Graph RAG 会更愿意沿流程边走。

### 68.3 这张表的意义

它说明 Graph RAG 不是“固定同一套扩图逻辑”，而是：

`同一张关系网，针对不同问题类型，用不同的走法`

这就是它和普通 RAG 最大的差别之一。

## 69. 图扩完以后，Graph RAG 还要把结果重新水合成可展示对象

图扩展阶段返回的还不是完整可展示结果，它更多只是：

1. `id`
2. `score`
3. `path`
4. `channel`

所以后面 `GraphRetriever` 还要做：

`_hydrate_results(...)`

### 69.1 什么叫“水合”

就是把一个简化的命中 ID，重新补回完整 chunk 信息。

它会再去拿：

1. chunk 内容
2. 页码
3. 层级
4. source
5. 相关实体

这样最后才能给前端真正可读的结果。

### 69.2 它还会额外查每个 chunk 关联的实体

`_load_entities_for_chunks(...)` 会查：

`entity_mentions JOIN entities`

把每个命中 chunk 对应的实体名、类型、定义补出来。

这说明 Graph RAG 最终给人的不是“只多一堆边”，而是：

`文本块 + 相关实体 + 可选路径解释`

## 70. `_build_context(...)`：Graph RAG 的上下文长什么样

Graph RAG 最终也会拼一份 `context`，但它和普通 RAG 的 context 气质不太一样。

它会把每个命中结果组织成：

1. 来源和页码
2. chunk 正文
3. 相关实体列表
4. 如果要求解释，再补路径

所以它的 context 更像：

`一份带实体注释和关系路径的局部子图说明`

而不是普通 RAG 那种“单纯把证据片段排起来”。

## 71. `run_graph_rag_query(...)` 最后返回的东西，和普通 RAG 有什么明显不同

普通 RAG 返回里重点是：

1. `answer`
2. `citations`
3. `candidates`
4. `timings`
5. `trace`

Graph RAG 返回里重点是：

1. `intent`
2. `intentSource`
3. `results`
4. `stats`

也就是说，当前 Graph RAG 更像：

`面向图扩展检索结果展示和解释`

而不是：

`直接产出自然语言最终答案`

这点非常关键。

### 71.1 为什么会这样

因为从现在代码看，Graph RAG 这一支重点还放在：

1. 图扩检索
2. 结果节点整理
3. 路径解释

还没有像普通 RAG 那样接一个完整的 `generator -> scorer -> citations answer` 闭环。

所以你可以把当前 Graph RAG 理解成：

`更偏图检索与图解释工作台`

而普通 RAG 才是：

`更偏最终问答成品链路`

## 72. 普通 RAG 和 Graph RAG，到底差在哪

如果你只想记最核心的差异，记下面这组对照最稳：

### 72.1 相同点

1. 都先做知识库访问校验
2. 都复用同一份知识库数据
3. 都会用到 dense / sparse 等基础召回能力
4. 都会记 query log

### 72.2 不同点

1. 普通 RAG 的主目标是“给答案”
2. Graph RAG 的主目标是“找子图并解释路径”
3. 普通 RAG 检索后走 `rerank -> generate -> score`
4. Graph RAG 检索后走 `intent classify -> graph expand -> hydrate -> explain`
5. 普通 RAG 更强调 citation 和最终答案
6. Graph RAG 更强调实体、关系、path 和 intent-aware expansion

所以它们不是“一个轻版一个重版”的关系，而是：

`同源数据上的两种不同消费方式`

## 76. `backend/routes`：后端真正对外暴露的不是 service 函数，而是 API 路由

我们前面一直在讲 `ingestion_service.py`、`rag_service.py`，那是因为它们更接近业务主链。

但真实请求进来时，最先经过的不是 service，而是 route。

### 76.1 普通 RAG 和 Graph RAG 的对外入口在 `backend/routes/rag.py`

看 [backend/routes/rag.py](/E:/all_project/AI/rag-enhancement/backend/routes/rag.py)，会发现它非常薄，但非常重要。

它主要做三件事：

1. 接收 `QueryRequest` / `GraphQueryRequest`
2. 通过 `Depends(get_current_identity)` 注入当前身份
3. 把异常翻译成 HTTP 状态码

也就是说，前面我们一直讲的：

1. `run_rag_query(...)`
2. `run_graph_rag_query(...)`

不是直接暴露给外部系统的，而是被这两个路由包着：

1. `POST /api/rag/query`
2. `POST /api/rag/graph-query`

### 76.2 为什么 route 层值得单独讲

因为它决定了“后端主链是怎么变成 HTTP API 的”。

它负责把：

`Python 内部对象 + 异常`

翻成：

`HTTP 请求体 + HTTP 响应 + 状态码`

比如这里就很明确：

1. `ValueError` 转成 `404`
2. 其他异常转成 `503`

这说明对外暴露时，项目在语义上把：

1. “资源不存在/不可访问”
2. “服务内部执行失败”

分开了。

### 76.3 入库链路的对外入口在 `backend/routes/ingestion.py`

这个文件更能体现“主链外壳层”的意义。

它对外暴露的不是一个单一上传接口，而是一组完整入口：

1. 文件上传入库
2. 网页入库
3. 备份 CSV 快速导入
4. 任务详情查询
5. 任务删除
6. 任务 SSE 流
7. 任务重试
8. 草稿预览
9. 草稿编辑
10. 草稿删除
11. 草稿合并
12. 草稿确认

这说明离线入库主链在真实产品里，并不是一个“一发完事”的黑盒，而是被拆成了很多可操作节点。

## 77. `rag_adapter.py`：它不是多余中转层，而是在线主链真正的“拼装器”

前面虽然已经读过 [backend/adapters/rag_adapter.py](/E:/all_project/AI/rag-enhancement/backend/adapters/rag_adapter.py)，但这里要专门强调一下它的定位。

### 77.1 它的作用不是再实现算法，而是把算法零件装成流水线

它自己不做：

1. 向量检索
2. 重排模型
3. 答案生成
4. 评分逻辑

这些分别都在别的模块。

它做的是：

1. 建 retriever
2. 建 reranker
3. 建 generator
4. 建 scorer
5. 规定它们的调用顺序
6. 处理普通分支和媒体编号短路分支
7. 汇总时延和 LLM 用量

所以它更像：

`在线问答业务流水线装配器`

### 77.2 为什么这个中间层值得保留

因为如果没有 adapter，这些逻辑就会散到：

1. `rag_service.py`
2. `retriever.py`
3. `generator.py`

里面互相缠绕。

有了 adapter 以后，结构更清楚：

1. `service` 负责对外服务编排与日志
2. `adapter` 负责把核心模块接成 pipeline
3. `core/rag/*` 负责具体能力实现

这是一种很典型、也很健康的分层。

## 78. `access_control.py`：后端主链里“为什么很多地方会直接 404”要靠它解释

看 [backend/services/access_control.py](/E:/all_project/AI/rag-enhancement/backend/services/access_control.py)，会发现它不是主算法层，但它解释了后端一个很重要的行为特征：

`很多资源无权访问时，不是 403，而是直接 404`

### 78.1 它统一封了哪些访问检查

至少包括：

1. `require_kb_access(...)`
2. `require_document_access(...)`
3. `require_task_access(...)`
4. `require_chunk_draft_access(...)`

也就是说：

1. 知识库
2. 文档
3. 入库任务
4. 切片草稿

这些资源的访问裁剪，后端不是各处手写，而是尽量统一收口。

### 78.2 为什么它常常返回 404 而不是 403

因为它走的是“资源不可见”语义，而不只是“你没权限”语义。

这样做的好处是：

1. 不暴露这个资源是否真实存在
2. 对租户隔离和资源裁剪更安全
3. 更符合很多后台系统对私有资源的处理方式

所以如果你后面看到某些地方“明明像权限问题，却返回 404”，背后基本就是这层在起作用。

### 78.3 它还顺手做了什么

不仅拦，还会：

`append_audit_log(action="access.denied", ...)`

也就是说，权限拒绝不是静默发生，而是会落审计。

这又一次说明这个项目的后端外壳层，不只是让功能能跑，还在考虑治理和追踪。

## 79. `chunk_draft_service.py`：为什么前面说 `chunk_drafts` 不是小缓存，这个文件能直接证明

看 [backend/services/chunk_draft_service.py](/E:/all_project/AI/rag-enhancement/backend/services/chunk_draft_service.py)，它直接把 `chunk_drafts` 这层做成了一个完整服务，而不是顺手写两行 SQL。

### 79.1 它真正保存了什么

`save_chunk_drafts(...)` 保存的不只是正文，还包括：

1. `chunk_id`
2. `chunk_index`
3. `content`
4. `source`
5. `page`
6. `strategy`
7. `layer`
8. `title`
9. `parent_id`
10. `related_ids`
11. `is_table_chunk`
12. `is_image_chunk`
13. `image_path`
14. `enhanced_text`
15. `extracted_entities`
16. `extracted_triples`
17. `relations`

这说明草稿层保留的信息量已经非常接近正式 chunk 层了。

也就是说，它不是只给你存一份“简化预览文本”，而是尽量把确认前所需信息都留住。

### 79.2 为什么这层支持编辑、删除、合并

因为它就是给“确认前人工干预”准备的。

从这个文件本身就能看出来：

1. 可以 `update_chunk_draft(...)`
2. 可以 `delete_chunk_draft(...)`
3. 可以 `merge_chunk_drafts(...)`
4. 可以 `list_chunk_drafts(...)`

这说明项目在产品层面对离线入库的理解不是：

`模型切完就算真理`

而是：

`模型先给候选草稿，人和系统再一起把草稿转正`

### 79.3 它为什么还有过期清理

有：

`cleanup_expired_chunk_drafts()`

这说明草稿层虽然重要，但它仍然是“中间层”，不是正式长期存储层。

所以项目对它的态度是：

1. 功能上很认真
2. 生命周期上仍然把它当草稿

这个边界是清楚的。

## 80. `task_store.py`：为什么任务状态能被查询、SSE 能持续推、任务能重试，靠的是它

看 [backend/services/task_store.py](/E:/all_project/AI/rag-enhancement/backend/services/task_store.py)，它解释了离线任务主链另一个很重要的基础设施问题：

`任务状态到底存在哪`

### 80.1 它不是死绑 Redis，而是 Redis 优先、内存降级

这里的逻辑是：

1. 优先尝试 Redis
2. Redis 不可用就退回内存 `_mem_tasks`

所以任务系统不是“一旦 Redis 挂了，整个后端主链直接全瘫”，而是有降级兜底。

### 80.2 它保存的不是算法数据，而是任务状态数据

这里更关注的是：

1. 任务 ID
2. 状态
3. 创建时间 / 更新时间
4. 分阶段进度
5. 结果摘要

也就是说，它服务的是：

1. `/api/ingestion/tasks/{task_id}`
2. SSE 流
3. 重试
4. 列表台账

而不是直接服务检索或问答本身。

### 80.3 为什么它对主链也很重要

因为离线入库是长任务，不是瞬时请求。

如果没有 task store，就很难稳定支持：

1. 后台异步执行
2. 用户刷新页面后继续看状态
3. SSE 持续推送进度
4. 失败后重试
5. 管理台账查询

所以虽然 `task_store.py` 不是算法链的一环，但它是离线主链从“脚本式流程”升级为“产品级任务系统”的关键支撑。

## 83. 现在做第一个后端专项：数据库表和主链对象变化逐表对照

## 84. 先记一张总图：后端数据库不是一层，而是至少 6 层

如果按主链职责粗分，当前数据库大致可以这样理解：

1. 知识库主数据层
2. 文档与 chunk 检索层
3. 关系与图谱层
4. 草稿与任务中间层
5. 身份与权限治理层
6. 日志、评测、用量、审计观测层

这 6 层不是完全隔离，但这样记最不容易乱。

## 85. 第一层：知识库主数据层

### 85.1 `knowledge_bases`

定义在 [core/db/schema.py](/E:/all_project/AI/rag-enhancement/core/db/schema.py)。

这张表就是全项目最顶层的“库目录”。

你可以把它理解成：

`一个知识库实例的主记录`

里面不只是名字和描述，还包括治理字段：

1. `id`
2. `name`
3. `description`
4. `default_strategy`
5. `tenant_id`
6. `created_by`
7. `owner_user_id`
8. `owner_status`
9. `owner_invalid_reason`
10. `status`
11. `deleted_at`

### 85.2 这张表在主链里的位置

它不是某一步骤生成出来的中间结果，而是所有主链都依附的最上层容器。

也就是：

1. 文档属于某个 `knowledge_base`
2. chunks 属于某个 `knowledge_base`
3. triples 属于某个 `knowledge_base`
4. query logs 也会标记属于哪个 `knowledge_base`

所以它是“主容器层”，不是“处理结果层”。

## 86. 第二层：文档与 chunk 检索层

这是和离线入库主链最紧密的一层。

### 86.1 `documents`

这张表保存的是：

`文档级记录`

它主要对应离线主链里的：

`一份原始输入文档，经过处理后，作为正式知识库文档存在`

关键字段包括：

1. `id`
2. `kb_id`
3. `filename`
4. `file_hash`
5. `chunk_count`
6. `source_storage`
7. `source_path`
8. `source_url`
9. `parser_provider`
10. `created_at`
11. `updated_at`

### 86.2 `documents` 在哪一步写入

是在：

`write_to_pgvector(...)`

里先 upsert `documents`，再写 `chunks`。

所以文档层一定比 chunk 层先落。

### 86.3 `chunks`

这是整个检索层最核心的一张表。

它对应的是离线主链里经过：

1. 切片
2. 增强
3. 质量门控
4. embedding
5. 关系补全

之后，最终进入正式知识库的 chunk。

关键字段包括：

1. `id`
2. `kb_id`
3. `document_id`
4. `content`
5. `source`
6. `page`
7. `chunk_index`
8. `strategy`
9. `title`
10. `char_count`
11. `is_table_chunk`
12. `is_image_chunk`
13. `image_path`
14. `layer`
15. `parent_id`
16. `related_ids`
17. `search_text`
18. `search_vector`
19. `embedding`

### 86.4 这张表为什么特别重要

因为普通 RAG 和 Graph RAG 都离不开它。

它同时服务：

1. 文本展示
2. 稀疏检索
3. 向量检索
4. parent/child/enhanced 结构回溯
5. related 扩展
6. 图谱结果水合

所以可以说：

`chunks` 是整个项目在线消费最核心的底座表。

### 86.5 `search_text` / `search_vector` / `embedding` 三者不要混

它们虽然都在 `chunks` 里，但服务目标不同：

1. `search_text`：供全文检索准备的拼接文本
2. `search_vector`：PostgreSQL 文本检索索引对象
3. `embedding`：pgvector 向量检索对象

也就是说，一个 chunk 在库里并不是只存一份正文，而是同时准备了：

1. 文本检索入口
2. 向量检索入口

这就是 hybrid retrieval 的表层基础。

## 87. 第三层：关系与图谱层

这一层和 Graph RAG 联系最紧。

### 87.1 `chunk_relations`

这张表保存的是：

`chunk 与 chunk / entity 之间的边`

关键字段包括：

1. `kb_id`
2. `src_id`
3. `dst_id`
4. `rel_type`
5. `weight`
6. `source`
7. `evidence`

### 87.2 它对应主链里的哪些动作

至少对应这些步骤产生的边：

1. `link_related_chunks(...)`
2. `link_semantic(...)`
3. `link_procedure(...)`
4. `link_causal(...)`
5. `materialize_entities(...)` 里给 chunk 补 `mentions`

所以它不是只存一种关系，而是一个混合边层。

### 87.3 `kg_triples`

这张表保存的是三元组：

1. `s`
2. `p`
3. `o`
4. `confidence`
5. `source_chunk`

这张表主要对应：

`enhanced chunk` 里的 `extracted_triples`

也就是说，它更偏：

`从增强语义层抽出来的知识图谱表达`

### 87.4 `entities`

这张表保存的是实体节点。

关键字段包括：

1. `id`
2. `kb_id`
3. `name`
4. `aliases`
5. `type`
6. `definition`
7. `embedding`

它主要对应：

`materialize_entities(...)` 合并、定义补全、向量化后的实体对象`

### 87.5 `entity_mentions`

这张表保存的是：

`实体在哪些 chunk 里被提到`

本质上它是在把：

`entity`

和

`chunk`

重新连起来。

这张表对 Graph RAG 非常重要，因为 Graph RAG 里有一个关键跳转就是：

`chunk -> entity -> 其他 mention 这个实体的 chunks`

没有 `entity_mentions`，这条跨文本块跳跃就很难做稳。

这一层是离线入库产品化的关键。

### 88.1 `chunk_drafts`

这是“确认前中间层”的正式数据库表表达。

它保存的几乎是 chunk 的全套草稿信息，包括：

1. `chunk_id`
2. `chunk_index`
3. `content`
4. `source`
5. `page`
6. `strategy`
7. `layer`
8. `title`
9. `parent_id`
10. `related_ids`
11. `is_table_chunk`
12. `is_image_chunk`
13. `image_path`

15. `extracted_entities`
16. `extracted_triples`
17. `relations`
18. `user_edited`
19. `is_deleted`
20. `expires_at`

### 88.2 `chunk_drafts` 在主链里的角色

`run_pipeline_real(...)` 和 `confirm_pipeline(...)` 之间的中间状态层

所以对象变化可以直接记成：

`Chunk[]`

## 89. 第五层：身份与权限治理层

这一层不是主算法产物，但它贯穿所有主链。

### 89.1 `kb_identity_*`

这层对主链的影响主要体现在：

1. 某个用户能看到哪些知识库

3. 查询日志、审计日志要按谁来归属

### 89.2 `kb_auth_sessions`

`后端当前登录态与身份快照承载表`

### 89.3 `kb_sso_used_credentials`

这张表更偏正式 SSO 安全治理，主要是：

`一次性凭证防重放`

## 90. 第六层：日志、评测、用量、审计观测层

这一层是“主链结束以后，系统把过程记下来”的地方。

### 90.1 `kb_rag_query_logs`

这张表保存的是一次问答请求的脱敏摘要。

不是完整 query/answer 原文，而是：

1. `request_id`
2. `pipeline_domain`
3. `kb_id`
4. `query_hash`
5. `query_summary`
6. `answer_summary`
7. `cannot_answer`
8. `relevance_score`
9. `faithfulness_score`
10. `prompt_tokens`
11. `completion_tokens`
12. `total_tokens`
13. `latency_ms`
14. `status`
15. `error_code`

这张表对应：

`run_rag_query(...)` / `run_graph_rag_query(...)` 结束后的查询日志收口

### 90.2 `kb_llm_call_logs`

这张表保存的是：

`主链中每一次模型调用的分阶段记录`

它不仅记在线生成，还记：

1. 入库质检 LLM
2. 入库 embedding
3. 在线 rerank LLM check
4. 在线 generator
5. 在线 scorer 的可选 LLM 评分

所以它是跨离线和在线两边的统一模型调用台账。

### 90.3 `kb_token_usage_hourly`

这是从 LLM call logs 汇总出来的小时级 rollup。

也就是说：

1. `kb_llm_call_logs` 更细
2. `kb_token_usage_hourly` 更适合报表

这两张表是细台账和汇总台账的关系。

### 90.4 `kb_audit_logs`

这是治理和安全动作的审计层。

它对应的不是业务答案，而是：

1. access denied
2. 配置变更
3. API Key 操作
4. owner 迁移
5. 高风险治理动作

所以它和 `kb_rag_query_logs` 不同：

1. query logs 记录“问答发生了什么”
2. audit logs 记录“治理和安全动作发生了什么”

## 92. 哪些表是“主数据”，哪些表是“过程数据”，哪些表是“观测数据”

这是很值得强行分清的一层。

### 92.1 主数据

1. `knowledge_bases`
2. `documents`
3. `chunks`
4. `chunk_relations`
5. `kg_triples`
6. `entities`
7. `entity_mentions`

这些表构成了知识库本体。

### 92.2 过程数据

1. `chunk_drafts`
2. `kb_identity_sync_runs`
3. `kb_auth_sessions`
4. `task_store` 里的运行态任务状态

这些更偏“主链运行过程中暂存、过渡、承载状态”的层。

### 92.3 观测数据

1. `kb_rag_query_logs`
2. `kb_llm_call_logs`
3. `kb_token_usage_hourly`
4. `kb_audit_logs`

这些不直接决定知识内容，但决定你能不能：

1. 追请求
2. 看 token
3. 做报表
4. 做审计

这个区分一旦记住，后面你看任何表就不容易混。

## 94. 现在做第二个后端专项：日志、评测、token 用量、审计是怎么挂在主链上的

前面我们已经知道：

1. 主链怎么跑
2. 主链结果落到哪些主数据表

但在真实产品里，只知道“数据写进去了”还不够。

你还需要知道：

1. 这次请求有没有被记下来
2. 每一步模型调用有没有单独记
3. token 花在哪个环节了
4. 哪些高风险动作会进审计

这就是这一节要补的东西。

先给一句最不容易错的大白话：

`这个项目把“业务数据”和“观测数据”分得很开：知识内容写主数据表，过程痕迹写日志/用量/审计表`

## 95. 先记住 4 类不同的“记录”

后端这里最容易混的，就是“日志”这个词其实指了几种不同东西。

最好直接强行分成 4 类：

1. 查询日志
2. 模型调用日志
3. token 小时汇总
4. 审计日志

它们分别对应：

### 95.1 查询日志：`kb_rag_query_logs`

记录的是：

`一次问答请求的摘要`

### 95.2 模型调用日志：`kb_llm_call_logs`

记录的是：

`主链里每一次 LLM / embedding / 打分等模型调用`

### 95.3 小时汇总：`kb_token_usage_hourly`

记录的是：

`把模型调用日志按小时聚合后的统计报表数据`

### 95.4 审计日志：`kb_audit_logs`

记录的是：

`权限拒绝、配置变更、治理动作、安全动作`

如果你先把这四层分清，后面就不容易乱。

## 96. `core/db/query_logs.py`：这不是只有“query logs”，而是整个观测落库中心

文件名虽然叫 `query_logs.py`，但你读代码会发现它其实同时承载了三大类落库函数：

1. `append_rag_query_log(...)`
2. `append_llm_call_log(...)`
3. `append_audit_log(...)`

也就是说，它更像：

`后端观测数据写库中心`

而不是狭义“查询日志工具”。

这点很重要，因为很多观测链路最后都会汇到它。

---

## 97. 第一类：`append_rag_query_log(...)` 记录“这次问答发生了什么”

先看 `RagQueryLogRecord`。

这类记录关注的是：

1. 哪个请求
2. 哪个知识库
3. 问了什么的摘要
4. 回了什么的摘要
5. 有没有拒答
6. 分数怎样
7. token 总量怎样
8. 总耗时怎样

### 97.1 它故意不保存原始 query 和原始 answer 正文

这一点很关键。

代码里写的是：

1. `query_hash`
2. `query_summary`
3. `answer_summary`

也就是说，它不是把用户完整问题和完整答案原文通通塞进日志表，而是做脱敏摘要。

这说明项目在这里追求的是：

`可观测`

但不是：

`把所有正文一股脑儿都存起来`

### 97.2 这类日志在哪些主链里会写

至少有：

1. 普通 RAG 完成后 `run_rag_query(...)`
2. Graph RAG 完成后 `run_graph_rag_query(...)`

所以它是：

`在线消费链路级日志`

不是离线入库主链级日志。

### 97.3 为什么 Graph RAG 现在的 query log 看起来比普通 RAG 简单

因为当前 Graph RAG 这条线重点还不是“最终答案生成”，所以它写 query log 时没有像普通 RAG 那样塞完整 token 汇总和最终 answer 结果。

这和前面主链分析是一致的：

1. 普通 RAG 更像成品问答链路
2. Graph RAG 更像图检索与解释链路

---

## 98. 第二类：`append_llm_call_log(...)` 记录“主链里每一次模型调用”

这是很多人最容易低估的一层。

因为你可能会想：

`不是已经有 query log 了吗，为什么还要单独记 llm call log`

答案是：

一次 query 里通常不止一次模型调用。

### 98.1 一次普通在线问答里，至少可能有这几种模型调用

1. `rerankLlmCheck`
2. `generate`
3. `score`

也就是：

1. 重排后的可选 LLM 最终检查
2. 生成答案
3. 可选 LLM 问答评分

### 98.2 一次离线入库里，也会写模型调用日志

比如：

1. 质检 LLM
2. embedding
3. 可能还有前面的增强阶段

这说明 `kb_llm_call_logs` 是横跨：

1. 离线入库链
2. 在线普通 RAG 链
3. 部分评测链

的一张统一调用台账表。

### 98.3 这类日志比 query log 细在哪里

`RagQueryLogRecord` 更像“整次请求摘要”。

`LlmCallLogRecord` 更像“请求里的单次模型调用明细”。

它至少细到这些字段：

1. `pipeline_domain`
2. `pipeline_stage`
3. `feature_name`
4. `provider`
5. `model_name`
6. `prompt_tokens`
7. `completion_tokens`
8. `total_tokens`
9. `latency_ms`
10. `status`
11. `error_code`

也就是说，它回答的是：

`到底是哪一步、哪个模型、花了多少 token、耗时多少`

而不是只回答：

`这次问答大概怎么样`

---

## 99. `append_llm_call_log(...)` 写完以后，还会立刻顺手更新小时汇总

这是一个非常关键的设计点。

很多系统会把“调用明细表”和“汇总表”分两个完全独立任务慢慢跑。

这里不是完全这样。

`append_llm_call_log(...)` 在插入 `kb_llm_call_logs` 后，会立刻调用：

`_upsert_token_usage_hourly(...)`

### 99.1 这意味着什么

意味着：

1. 细粒度调用台账先落
2. 同一个事务里就把小时级汇总加上去

所以控制台里的 token 用量报表，不一定非要等离线批处理跑完才更新。

它会尽量跟着主链实时累加。

### 99.2 小时汇总表到底汇总了什么

它按这些维度聚合：

1. `hour_bucket`
2. `tenant_id`
3. `kb_id`
4. `api_key_id`
5. `pipeline_domain`
6. `pipeline_stage`
7. `feature_name`

然后累加：

1. `request_count`
2. `prompt_tokens`
3. `completion_tokens`
4. `total_tokens`
5. `latency_ms_sum`
6. `error_count`
7. `estimated_cost`

也就是说，这张表不是“把总 token 一锅端”，而是支持按链路、阶段、功能点拆开的。

这对产品化报表非常关键。

---

## 100. `refresh_token_usage_hourly(...)` 说明汇总表不是只能增量写，也支持回刷

这个函数非常有工程意味。

它的作用是：

`根据 kb_llm_call_logs 重新回算某个时间窗的小时汇总`

### 100.1 为什么需要回刷

因为现实里可能会出现：

1. 历史日志补录
2. 身份归属修复
3. 汇总逻辑升级
4. 某些记录之前没汇对

这时候如果汇总表只能“往上加”，你就很难纠错。

有了 `refresh_token_usage_hourly(...)`，就说明项目把：

1. 细台账
2. 汇总表

分成了“源数据”和“可重建汇总”的关系。

这是一种很稳的做法。

---

## 101. 第三类：`append_audit_log(...)` 记录的不是模型调用，也不是问答摘要，而是治理动作

这一层你一定要和 query logs 分开记。

### 101.1 审计日志关注的核心问题不是“内容是什么”，而是“动作是什么”

`AuditLogRecord` 关注的是：

1. `action`
2. `resource_type`
3. `resource_id`
4. `kb_id`
5. `outcome`
6. `risk_level`
7. `summary`
8. `metadata`

也就是说，它在记录：

`谁对什么资源做了什么动作，结果如何，有什么风险级别`

### 101.2 哪些地方会写审计

从后端扫描结果看，至少这些地方会写：

1. `access_control.py`
2. `kb_service.py`
3. `identity_service.py`
4. `console_service.py`
5. `routes/identity.py`
6. `routes/openapi_v1.py`

这说明审计日志不是某一条业务链专属，而是跨系统治理层的统一底座。

### 101.3 为什么权限拒绝也要单独进审计

因为“拒绝”本身也是重要安全事件。

比如：

1. 某人频繁访问不该看的资源
2. 某个 API Key 越权试探
3. 某个用户对高风险配置做了失败尝试

这些都不是普通 query log 能说明的。

所以 `access.denied` 单独进审计，是很合理的。

## 102. `evaluation_store.py`：评测记录和数据库日志不是一回事

这个点非常容易被忽略。

看 [backend/services/evaluation_store.py](/E:/all_project/AI/rag-enhancement/backend/services/evaluation_store.py)，会发现它当前不是写数据库，而是写：

`data/results/console_evaluations.json`

### 102.1 这意味着什么

说明当前“控制台评测记录”这块，还不是和主日志表完全统一的数据库持久化模型。

它更像：

`控制台近期开箱即用的轻量评测存储`

### 102.2 它记录什么

主要记录：

1. `kbId`
2. `query`
3. `answer`
4. `relevanceScore`
5. `faithfulnessScore`
6. `llmScore`
7. `cannotAnswer`
8. `failureReason`
9. `createdAt`

### 102.3 为什么它和 `kb_rag_query_logs` 不能混为一谈

因为：

1. `kb_rag_query_logs` 更偏通用请求日志
2. `evaluation_store` 更偏控制台评测视角的结果留存

前者是数据库里的统一日志体系。

后者目前是控制台侧的轻量评测存储。

所以你可以把它理解成：

`一个偏产品工作台的小评测仓`

而不是统一观测主表。

## 103. 离线入库主链是怎么挂上模型调用日志的

前面我们主要讲在线 RAG 的日志，现在回头看离线主链。

在 [backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py) 里，有一个关键中间函数：

`_record_ingestion_llm_usage(...)`

### 103.1 它的作用

就是把离线主链里某一步产出的 metrics，映射成 `LlmCallLogRecord` 再写入 `kb_llm_call_logs`。

也就是说，离线主链很多阶段虽然不是“直接生成答案”，但只要调用了模型能力，也会被纳入统一用量台账。

### 103.2 它在哪些阶段会被调用

从代码看，至少包括：

1. 切片增强相关阶段
2. 质量门控阶段
3. embedding 阶段

也就是说，离线主链不是“只有在线问答才有 token 统计”。

离线主链也是一等公民。

### 103.3 为什么 embedding 也走 LLM call log

因为在这个项目里，“模型调用”不是只指 chat completion。

embedding 也是模型能力调用，也有：

1. provider
2. model_name
3. token/维度/耗时
4. request 归属

所以把 embedding 纳入统一台账是合理的。

## 104. 在线普通 RAG 主链是怎么挂上日志和用量的

看 [backend/services/rag_service.py](/E:/all_project/AI/rag-enhancement/backend/services/rag_service.py)，普通 RAG 在返回结果前，至少做了三件观测相关动作：

1. `append_evaluation(...)`
2. `_record_rag_llm_usage(...)`
3. `append_rag_query_log(...)`

### 104.1 `append_evaluation(...)`

这是控制台评测层。

### 104.2 `_record_rag_llm_usage(...)`

这是把：

1. rerank LLM check
2. generate
3. score

这几类可选模型调用拆出来，逐项记进 `kb_llm_call_logs`。

### 104.3 `append_rag_query_log(...)`

这是整次问答摘要的最后收口。

也就是说，普通 RAG 最终不是只写一条日志，而是：

1. 评测一份
2. 模型调用明细若干条
3. 查询摘要一条

这三层一起构成完整观测。

## 105. Graph RAG 现在为什么日志层比普通 RAG 轻

`run_graph_rag_query(...)` 当前只明确做了：

`append_rag_query_log(...)`

而且写入内容也比较轻。

这和它当前链路定位一致：

1. 还没有完整生成答案闭环
2. 重点还是图检索结果和解释

所以它当前观测重点更偏：

`这次 graph query 发生过`

而不是像普通 RAG 那样细拆：

1. generation token
2. score token
3. evaluation record

这也说明 Graph RAG 当前成熟度路径和普通 RAG 不完全一样。

## 106. 哪些记录是“请求级”，哪些是“调用级”，哪些是“聚合级”

这也是非常值得强行分清的一层。

### 106.1 请求级

代表：

`一次业务请求整体`

对应：

1. `kb_rag_query_logs`
2. 部分控制台评测记录

### 106.2 调用级

代表：

`一次请求内部的某次模型调用`

对应：

1. `kb_llm_call_logs`

### 106.3 聚合级

代表：

`把很多调用结果汇总成报表口径`

对应：

1. `kb_token_usage_hourly`

### 106.4 治理级

代表：

`谁对系统资源做了什么动作`

对应：

1. `kb_audit_logs`

一旦这四层记清，以后就不容易把“一个查询为什么有多条日志”当成异常。

## 107. 为什么这些日志函数都在强调“失败不能影响主链”

你如果看 `append_rag_query_log(...)`、`append_llm_call_log(...)`、`append_audit_log(...)`，会发现它们都有很明显的共同点：

1. 尽量 `try/except`
2. 写失败就回滚
3. 返回 `False`
4. 但不应该把主业务直接打崩

这说明项目在观测层的原则是：

`观测很重要，但观测写失败不应反过来拖垮主业务`

这是一种非常典型的产品级后端取舍。

如果反过来，主业务成功了，只因为日志表一时不可写就把用户请求整体搞失败，用户体验会很差。

所以这里采用的是：

`尽量记，但不要让记日志变成业务硬依赖`
