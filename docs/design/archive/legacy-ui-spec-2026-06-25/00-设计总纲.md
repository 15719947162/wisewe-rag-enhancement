# 00 设计总纲

## 1. 产品定义

`WiseWe RAG Console` 是一套面向内部运营、算法调优、知识库管理与效果验证的全链路控制台，不是普通聊天界面，也不是传统 CRUD 后台。

它服务于三个核心对象：

- `知识库资产`
- `离线建库流水线`
- `在线问答与评测链路`

## 2. 设计目标

本项目的 UI 不追求“视觉炫酷”，而追求以下四点：

1. `高可信`
让用户感知系统稳定、专业、可审计。

2. `强解释`
让用户能够从结果一路追溯到证据、参数、上游阶段与错误原因。

3. `高密度`
在有限屏幕中承载复杂链路信息，但不产生拥挤和疲劳感。

4. `可运维`
适合长期使用、排查问题、比较策略、观察趋势。

## 3. 目标用户

- 产品经理 / 项目负责人
- 算法工程师 / RAG 工程师
- 数据运营 / 知识库运营
- QA / 效果评测人员
- 平台运维 / 集成方

## 4. 风格方向

采用 `Premium Data Console` 方案。

关键词：

- 可信
- 克制
- 高级
- 理性
- 技术感
- 内容可读

风格组合：

- `Data-Dense Dashboard`
- `Trust & Authority`
- `Editorial Evidence View`

## 5. 不采用的方向

- 不做偏消费级对话产品的轻社交风
- 不做泛互联网紫色 SaaS 风
- 不做重玻璃拟态或装饰性粒子背景
- 不做“酷炫但不透明”的黑盒 AI 界面

## 6. 设计原则

### 6.1 可信优先

任何视觉决策，都应优先服务于可信感，而不是新奇感。

### 6.2 链路透明

每个关键结果都要能回答四个问题：

- 从哪来
- 为什么是它
- 经过了哪些处理
- 还能回到哪里看证据

### 6.3 高密不压迫

允许高信息密度，但必须通过层级、排版、容器边界、筛选与折叠控制认知负荷。

### 6.4 内容比容器更重要

对于 chunk、citation、context window、错误原因，文本内容本身比卡片视觉更重要。

### 6.5 工作台优先

所有核心页面都按“操作与诊断工作台”设计，而不是静态汇报页设计。

## 7. 全局信息架构

建议一级导航如下：

- `总览`
- `知识库`
- `离线建库`
- `在线问答`
- `评测分析`
- `系统配置`

## 8. 链路映射

### 离线建库

PDF -> 上传 OSS -> 云解析 -> ContentBlock -> 清洗 -> 切片 -> 质量过滤 -> 向量化 -> CSV / pgvector

### 在线问答

用户问题 -> HybridRetriever -> coarse filter -> ParentChildReranker -> context window -> RAGGenerator -> citation align -> RAGScorer

## 9. 设计成功标准

当 UI 达到目标时，用户应该能做到：

- 在 10 秒内看清链路当前状态
- 在 30 秒内定位一次失败发生在哪个阶段
- 在 1 分钟内解释一次答案为什么成立或为什么不能回答
- 在 2 分钟内比较两种切片或召回策略的差异
