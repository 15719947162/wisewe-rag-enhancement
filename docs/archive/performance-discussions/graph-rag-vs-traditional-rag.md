# Graph RAG 与传统 RAG 的差异和召回边界

## 核心结论

Graph RAG 不应该被理解为传统 RAG 的全面替代。更准确的定位是：

```text
传统 RAG 更稳，Graph RAG 更会扩展。
```

传统 RAG 主要解决“找到相似文本片段”的问题，Graph RAG 进一步解决“找到知识之间的关系”的问题。Graph RAG 在多跳、因果、流程、概念关系类问题上更有优势，但在普通事实型问题上，不一定比传统 RAG 更准。

## 传统 RAG 的典型优势

传统 RAG 的典型链路是：

```text
用户问题
-> 向量召回 / BM25
-> rerank
-> 拼接上下文
-> LLM 生成答案
```

它适合回答答案集中在少数文本片段里的问题，例如：

- 某个概念的定义是什么？
- 某张图或表在哪里？
- 某一节教材讲了什么？
- 某段原文如何解释一个术语？
- 用户 query 和原文在字面或语义上高度相似的问题。

这类问题里，传统混合 RAG 往往更稳。原因是链路短、噪声少、证据直接围绕文本相似度和关键词匹配产生。

## Graph RAG 的典型优势

Graph RAG 多使用：

```text
实体
关系
三元组
路径
子图扩展
```

它更适合回答：

- 多跳推理问题
- 因果链问题
- 流程链问题
- 概念关系问题
- 跨章节综合问题
- “A 和 B 有什么关系”
- “为什么 A 会导致 B”
- “某个过程包含哪些步骤”

例如：

```text
感受器 -> 传入神经 -> 神经中枢 -> 传出神经 -> 效应器
```

对于这类流程型问题，Graph RAG 可以沿着关系路径补全上下文，而传统 RAG 可能只召回其中某一段。

## Graph RAG 为什么可能召回更差

Graph RAG 会沿实体和关系扩展候选内容，这会带来召回完整性，也会带来噪声。

典型风险：

1. 实体抽取错误

   把普通词、标题残片、OCR 噪声错误识别为实体，会让扩展方向偏离问题。

2. 关系类型错误

   把“相关”误判成“因果”，或把“并列”误判成“上下位”，会影响路径解释和候选扩展。

3. 同名概念混淆

   教材中常有重复术语、章节标题、图表编号。如果不结合文档、章节、页码范围消歧，Graph RAG 可能跨章节扩错。

4. 扩展过宽

   一个实体如果连接了太多边，Graph RAG 会召回大量“相关但不回答问题”的内容。

5. 图谱缺边

   如果关系抽取不完整，真正需要的路径断裂，Graph RAG 反而可能找不到关键证据。

6. 图谱状态不完整

   文档刚完成基础入库、图谱仍在后台构建时，Graph RAG 的覆盖率天然低于普通混合 RAG。

因此，Graph RAG 可能召回到“相关知识”，但不一定召回到“最直接答案”。

## 推荐使用策略

项目中更适合采用混合策略：

```text
Hybrid RAG = 主召回
Graph RAG = 扩展召回
Reranker / Scorer = 最终裁判
```

推荐链路：

```text
用户问题
-> intent 判断
-> 普通混合 RAG 召回 top candidates
-> 需要关系扩展时触发 Graph RAG
-> 合并 candidates
-> rerank / scorer
-> 生成答案和引用
```

不要让 Graph RAG 单独决定最终证据。Graph 扩展出的候选内容必须再经过 rerank 或 scorer，避免因为关系扩展过宽而把答案召偏。

## 教材级书籍中的特殊差异

教材级文档和普通短文档不同。它通常具备清晰章节结构、概念层级、定义、图表、例题、流程、因果链和跨章节知识点。因此传统 RAG 与 Graph RAG 的差异会更加明显。

### 传统 RAG 在教材中的优缺点

传统 RAG 更适合教材里的局部事实查询，例如：

- 查定义：某个概念是什么？
- 查原文：某一节如何描述某个知识点？
- 查图表：某张图或表在哪里？
- 查章节内容：某一章讲了哪些内容？
- 查局部证据：某句话附近的解释是什么？

优点：

- 证据直接，通常能命中最贴近原文的 `child` chunk。
- 链路短，噪声少，响应稳定。
- 对定义、术语、图表编号、章节标题等问题更稳。
- 更适合作为默认问答保底链路。

缺点：

- 主要依赖文本相似度和关键词匹配。
- 教材越厚，同名术语、相似标题越多，越容易召回到相似但不属于当前语境的内容。
- 对流程、因果、概念关系和跨章节综合问题支持较弱。
- 容易只召回局部片段，答案缺少知识链路。

### Graph RAG 在教材中的优缺点

Graph RAG 更适合教材里的知识网络型问题，例如：

- A 和 B 有什么关系？
- 为什么 A 会导致 B？
- 某个过程分几步？
- 某个概念属于哪个上位概念？
- 这一章的知识点如何串起来？
- 图表、定义、例题分别支撑哪个知识点？

优点：

- 可以把章节、概念、图表、例题、流程、因果关系显式连接起来。
- 更适合跨 chunk、跨章节的综合问答。
- 能返回关系路径，解释为什么这些证据被召回。
- 适合后续扩展到概念图谱导航、学习路径推荐、知识点网络总结。

缺点：

- 图谱质量决定召回上限。
- 实体抽取错误、关系类型错误、同名概念未消歧时，会带来错误扩展。
- 对简单定义题可能扩展过多，召回“相关但不直接回答”的内容。
- 入库和后台构建成本高于传统 RAG。

## 用户问答召回结果上的差异

传统 RAG 的召回结果通常是：

```text
几个高相似度文本片段
```

Graph RAG 的召回结果更像：

```text
核心文本片段
+ 相关实体
+ 关系路径
+ 邻近知识点
+ 跨 chunk 证据
```

例如用户问：

```text
反射弧为什么能完成反射？
```

传统 RAG 可能召回：

- “反射弧由感受器、传入神经、神经中枢、传出神经、效应器组成”这一段。
- 与“反射弧”语义相似的定义段。

Graph RAG 可能召回一条路径：

```text
感受器 -> 传入神经 -> 神经中枢 -> 传出神经 -> 效应器
```

并补充每个节点对应的教材片段。它更容易形成过程解释。

但如果用户问：

```text
反射的定义是什么？
```

传统 RAG 通常更好。它可以直接召回定义段。Graph RAG 如果扩展过多，可能把反射弧、神经调节、条件反射等相关内容都带入上下文，反而干扰最终答案。

## 教材场景下的推荐分工

教材知识库不建议在传统 RAG 和 Graph RAG 中二选一，而应按问题类型分工：

```text
定义 / 原文 / 图表定位 / 单点知识：
  传统混合 RAG 为主

流程 / 因果 / 概念关系 / 跨章节综合：
  Graph RAG 增强

最终答案：
  统一 rerank + 引用校验
```

产品层可以让系统自动判断问题类型。普通问题默认传统混合 RAG；关系型、流程型、因果型问题再启用 Graph RAG。这样既保留传统 RAG 的准确性，又能发挥教材知识图谱在复杂知识组织上的优势。

## 产品层建议

1. 普通问答默认走传统混合 RAG。

   适合定义、原文、图表定位、章节内容等直接事实型问题。

2. 关系型问题启用 Graph RAG。

   适合因果、流程、对比、概念关系、跨章节综合类问题。

3. 暴露图谱覆盖状态。

   Graph RAG 响应应包含类似：

   ```text
   graphStatus
   graphCoverage
   relationCount
   pathCount
   ```

   如果图谱不完整，应明确提示并降级使用普通混合召回。

4. 路径证据和关系置信度应可追踪。

   对 Graph RAG 的结果，除了返回文本证据，还应返回实体路径、关系类型和置信度，方便用户判断为什么召回这些内容。

5. 图谱扩展要有边界。

   可以按问题类型限制扩展深度：

   - 定义型：不扩展或只扩展 1 跳。
   - 流程型：沿 `next_step / prev_step` 扩展。
   - 因果型：沿 `cause_of / effect_of` 扩展。
   - 概念关系型：沿上下位、组成、相似关系扩展。

## 初步结论

Graph RAG 不一定提高所有召回准确率。它提高的是复杂关系问题的召回完整性、路径解释性和跨 chunk 组织能力。

对简单事实问题，传统混合 RAG 通常更稳；对多跳、因果、流程和概念关系问题，Graph RAG 更有价值。

项目后续更合理的方向不是“用 Graph RAG 替代传统 RAG”，而是让二者组合：

```text
普通混合 RAG 保底准确率
Graph RAG 补复杂关系召回
统一 rerank / scorer 控制最终证据质量
```
