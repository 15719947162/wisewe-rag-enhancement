  • page_no：后续证据引用必须用；
  • block_id：每个知识块的唯一身份；
  • bbox：以后可以在页面图片上高亮原文位置；
  • block_type：区分正文、标题、图片、表格、图注、脚注。

这就是后续“可信证据链”的基础。

四、第三步：不要“先让大模型处理后再切片”，建议改成“先结构化，再增强，再切片优化”

您现在的设想是：

扫描件 → MinerU 获取文本图片 → 大模型处理 → 再切片。

我的建议是稍微调整为：

扫描件 → MinerU 结构化解析 → 原文清洗 → 初步结构切分 → 大模型增强 → 最终知识切片 → 入库。

原因是：大模型处理文本时，有可能会改写、合并、遗漏、补充。医学教材做 RAG，最怕“增强层污染原文层”。

正确做法

大模型应该做这些事：

  • 修复标题层级；
  • 判断某一段属于哪个知识点；
  • 提取关键词；
  • 抽取医学术语；
  • 生成摘要；
  • 生成教学问题；
  • 生成图片说明；
  • 判断这段适合关联哪些题库、病例、实训项目。

但大模型不应该做这些事：

  • 直接改写教材原文后替代原文；
  • 把多个不同知识点混在一起；
  • 自行补充教材没有写的内容；
  • 删除页码、图号、表号等证据线索；
  • 把图表内容随意总结成不准确文本。

一句话：

原文层必须忠实，增强层可以智能。

五、第四步：切片怎么切，这是核心竞争力之一

切片不是简单按 500 字、800 字切。医学教材切片要遵循“教学语义单元”。

1. 切片的核心原则

我建议用一句话定义：

一个 chunk 最好只解决一个明确的教学问题。

例如在《基础护理学》的“静脉输液”章节中，不要把“概念、目的、操作流程、注意事项、常见并发症、处理方法”全部放在一个 chunk 里。

应该拆成：

  • 静脉输液的概念；
  • 静脉输液的目的；
  • 静脉输液前评估；
  • 静脉输液用物准备；
  • 静脉输液操作流程；
  • 输液速度调节；
  • 输液过程观察；
  • 常见故障处理；
  • 输液反应识别；
  • 输液外渗处理；
  • 静脉炎预防与护理；
  • 操作后的记录与健康教育。

这才适合 Agent 精准检索。

2. 推荐三层切片法

第一层：父级切片

父级切片通常对应一个小节，例如：

第X章 静脉输液与输血
第X节 静脉输液
小节：输液过程中的观察与护理

父级切片不一定直接 embedding，而是作为上下文容器。

第二层：子级切片

子级切片是真正用于 embedding 的内容。

例如：

知识点：输液过程中的滴速观察
来源：基础护理学 / 第X章 / 静脉输液 / 第128页
内容：教材原文……

子级切片长度建议：

  • 普通概念类：300–600 个中文字符；
  • 操作流程类：500–1000 个中文字符；
  • 表格类：按表格主题切，不按字数切；
  • 图片类：按图片说明、图注、关联正文一起切；
  • 复杂知识点：允许一个父级下挂多个子级。

第三层：增强切片

增强切片不是原文，而是为检索服务的补充文本，例如：

本段主要讲静脉输液过程中护士应观察患者局部穿刺部位、输液速度、患者主诉和不良反应。相关问题包括：输液局部肿胀怎么办？输液速度过快有什么风险？如何判断输液外渗？

这个增强切片可以提升召回率，但回答时必须回到原文证据。

六、医学教材切片的具体技巧

技巧一：标题路径必须进入 chunk

不要只存正文，要把标题路径一起放进去。

例如：

教材：基础护理学
章节：第X章 静脉输液与输血
小节：静脉输液过程中的观察与护理
知识点：输液外渗的识别与处理
正文：……

这样学生问“输液局部肿了怎么办”，系统更容易定位到“静脉输液”而不是泛泛地找到“水肿”“疼痛”“炎症”。

技巧二：操作流程类知识不能机械按字数切

护理教材里大量内容是流程型知识，例如：

  • 操作前评估；
  • 操作前准备；
  • 操作中核对；
  • 操作中观察；
  • 操作后记录；
  • 注意事项；
  • 异常情况处理。

这类内容应该按流程阶段切，而不是按字符数切。

建议切成：

chunk_1：操作前评估与准备用物
chunk_2：操作中关键步骤
chunk_3：操作后观察与记录
chunk_4：异常情况与处理

技巧三：表格要单独处理

医学教材中的表格非常关键。比如：

  • 常见输液故障及处理；
  • 常见输液反应及护理措施；
  • 常用溶液种类和作用；
  • 护理诊断与护理措施对照表。

不要把表格转成一大段自然语言。应该保留表格结构。

建议每个表格保存三份内容：

  1. 原始表格 HTML；
  2. 表格转 Markdown；
  3. 表格摘要和行级知识点。

例如：

{
  "block_type": "table",
  "table_title": "常见输液故障及处理",
  "html": "<table>...</table>",
  "markdown": "|故障|表现|处理|...",
  "summary": "本表说明静脉输液过程中常见故障、表现及处理方法。",
  "related_knowledge_points": ["输液故障", "输液不滴", "穿刺部位肿胀"]
}

技巧四：图片不能只存 OSS 地址

您现在的做法是：

图片存阿里云 OSS → 保存地址 → 检索到 ID 后再获取图片。

这个方向是对的，但还不够。

图片至少要保存 6 类信息：

字段 作用
image_id 图片唯一 ID
oss_key OSS 存储路径
page_no 来自教材哪一页
bbox 图片在页面中的位置
caption 教材原始图注
visual_summary 大模型生成的图片说明
related_text_block_ids 图片关联的正文段落
knowledge_points 图片对应知识点

重点是：

向量检索时通常不是检索 OSS 地址，而是检索图片的文字说明、图注、关联正文和知识点标签。

当用户问到相关问题时，系统先通过文字索引找到图片，再通过 image_id 获取 OSS 图片。

建议 OSS 地址不要直接暴露公网固定 URL，而是通过后端生成临时访问地址，避免教材图片泄露。

技巧五：建立“原文 chunk”和“检索增强 chunk”的双轨机制

一个 chunk 可以有两个文本版本：

1. 用于展示和引用的原文

教材原文：……

2. 用于 embedding 的检索文本

教材：基础护理学
章节：静脉输液
知识点：输液外渗的识别与处理
同义表达：输液局部肿胀、针眼周围疼痛、液体渗出、穿刺部位异常
正文摘要：本段讲述……
原文：……

这样做的好处是：向量检索更容易命中，但最终回答仍然引用原文。

七、术语标准化是什么意思

您问得非常关键。

术语标准化就是把教材中不同说法、学生口语表达、题库表达、临床表达统一映射到同一个标准知识点。

例如学生可能问：

  • “输液的地方肿了怎么办？”
  • “针眼附近鼓起来了怎么办？”
  • “是不是输液外渗？”
  • “静脉输液局部疼痛怎么处理？”
  • “液体没有进血管怎么办？”

这些问题在语义上可能都指向同一类知识点：

输液外渗 / 穿刺部位异常 / 静脉输液并发问题

如果不做术语标准化，embedding 也能部分召回，但不够稳定。尤其是医学教育场景里，学生问法、教师问法、教材说法、考试说法往往不一样。

术语标准化要做什么

建议建立一个内部医学教育术语表：

{
  "standard_term": "输液外渗",
  "aliases": [
    "液体外渗",
    "局部肿胀",
    "穿刺部位肿胀",
    "输液部位疼痛",
    "液体渗出"
  ],
  "english": "extravasation",
  "category": "护理操作并发症",
  "subject": "基础护理学",
  "related_terms": ["静脉输液", "静脉炎", "输液反应"],
  "related_chapters": ["静脉输液与输血"]
}

术语标准化在哪里发挥作用

主要有 4 个地方：

  1. 教材入库时
    给 chunk 打标准知识点标签。
  2. 题库关联时
    判断一道题考的是哪个知识点。
  3. 用户提问时
    把口语问题扩展成标准医学术语。
  4. 学习分析时
    把学生错题、问答、练习表现归到统一知识点。

例如学生问：

“输液的时候局部鼓包，是不是正常？”

系统应该先识别：

局部鼓包 → 穿刺部位肿胀 → 可能关联输液外渗 / 静脉炎 / 穿刺失败

然后再检索对应教材证据。

这比单纯 embedding 更稳定。

八、向量化到底是什么

向量化就是把文本变成一组数字，让系统能计算“语义相似度”。

但在医学教材 RAG 里，真正要注意的是：

不是把整页教材扔进去向量化，而是把经过结构化、切片、术语标注后的知识块向量化。

推荐向量化对象

不要向量化这些：

  • 整本书；
  • 整章；
  • 整页扫描 OCR 文本；
  • 没有标题路径的孤立段落；
  • 大模型自由改写后的总结文本。

应该向量化这些：

  • 一个明确知识点；
  • 一个操作流程阶段；
  • 一个表格主题；
  • 一张图片的图注 + 图片说明 + 关联正文；
  • 一个题目解析对应的知识点；
  • 一个病例决策点对应的教材依据。

推荐向量化模板

例如一个用于 embedding 的文本可以这样构造：

教材：基础护理学
版本：第X版
章节：第X章 静脉输液与输血
小节：静脉输液过程中的观察与护理
知识点：输液外渗的识别与处理
同义词：局部肿胀、穿刺部位疼痛、液体渗出、输液部位异常
教学场景：护理操作、基础护理实训、护士资格证备考
正文：……

这个模板本身就是竞争力。

因为大多数公司只是“把文本切块后 embedding”，而您们可以做到：

带教材结构、带医学术语、带教学场景、带题库关联、带病例关联的向量化。

这就是医学教育垂直领域的差异化。

九、题库关联具体做什么

题库关联不是简单把题库也 embedding 进去。

它的目标是：

每一道题都能找到它考查的教材知识点、证据段落、能力层级和常见错误原因。

例如一道题：

开展静脉输液操作前，护士首先应进行哪项工作？
A. 调节滴速
B. 选择穿刺部位
C. 核对医嘱和患者信息
D. 固定针柄

这道题不能只是作为一个独立题目存在，而应该关联到：

{
  "question_id": "q_000123",
  "subject": "基础护理学",
  "chapter": "静脉输液与输血",
  "knowledge_points": ["静脉输液操作流程", "三查七对", "患者身份核对"],
  "evidence_chunk_ids": [
    "basic_nursing_p126_b04",
    "basic_nursing_p127_b02"
  ],
  "difficulty": "基础",
  "cognitive_level": "记忆/理解",
  "exam_scene": "护士资格证/校内实训考核",
  "common_errors": ["忽略操作前核对", "直接进入穿刺步骤"]
}

题库关联有什么价值

它可以支撑 5 个核心产品能力：

  1. 学生做错题后，自动回到教材原文；
  2. 智能辅导智能体可以解释“为什么选这个，为什么不选其他”；
  3. 教师可以看到学生薄弱知识点；
  4. 系统可以自动生成同知识点变式题；
  5. 学校可以看到课程、章节、知识点、考试结果之间的关系。

这就是从“知识库”升级到“教学数据资产”。

十、病例关联具体做什么

病例关联是把教材知识点和临床情境连接起来。

基础护理学中很多知识不是孤立概念，而是在真实场景中使用的。例如：

患者输液 20 分钟后，主诉穿刺部位疼痛，局部皮肤肿胀，输液滴速变慢。

这个病例应关联到：

  • 静脉输液；
  • 穿刺部位观察；
  • 输液外渗；
  • 静脉炎鉴别；
  • 输液故障处理；
  • 护理记录；
  • 健康教育。

病例数据结构可以这样设计：

{
  "case_id": "case_iv_001",
  "case_title": "静脉输液过程中局部肿胀案例",
  "scene": "基础护理实训",
  "patient_info": "模拟患者，基本信息略",
  "symptoms": ["穿刺部位疼痛", "局部肿胀", "滴速变慢"],
  "decision_points": [
    {
      "question": "护士应首先观察和判断什么？",
      "related_knowledge_points": ["输液外渗", "输液过程观察"],
      "evidence_chunk_ids": ["basic_nursing_p128_b03"]
    }
  ],
  "teaching_objectives": ["识别输液异常", "进行初步处理", "完成护理记录"]
}

病例关联的价值

它可以支撑：

  • AISP 虚拟病人问诊；
  • 临床思维训练；
  • 护理思维训练；
  • OSCE 考站设计；
  • 实训任务设计；
  • 智能陪练智能体；
  • 教师备课智能体；
  • 学生个性化辅导智能体。

这正是天堰的优势。别人做知识库，您们可以做：

教材知识点 + 题库 + 病例 + 实训 + 评价数据的一体化闭环。

十一、证据引用是什么

证据引用就是 Agent 回答问题时，不能只给答案，还要说明：

  • 依据哪本教材；
  • 哪一版；
  • 哪一章；
  • 哪一节；
  • 第几页；
  • 哪个段落；
  • 哪张图；
  • 哪个表格。

例如学生问：

“输液时局部肿胀应该怎么办？”

系统回答不能只是：

“可能是输液外渗，应停止输液并进行处理。”

更好的回答应该是：

根据《基础护理学》第X版第X章“静脉输液与输血”中关于“输液过程观察与异常处理”的内容，局部肿胀、疼痛、滴速异常等情况提示需要重点观察穿刺部位并判断是否存在输液外渗或相关异常。建议在教学训练中按照教材要求进行观察、停止相关操作、报告教师或带教老师，并完成护理记录。

证据来源：
1. 《基础护理学》第X章 静脉输液与输血，第128页，段落 basic_nursing_p128_b03
2. 图X-X 静脉输液穿刺部位观察示意图，image_id: img_00045

前端还可以支持：

  • 点击引用，打开教材原页图片；
  • 高亮对应段落；
  • 显示原图；
  • 显示表格；
  • 显示关联题目；
  • 显示关联病例。

这就是“可信 RAG”。

医学教育系统没有证据引用，就很难进入学校和医院的正式教学场景。

十二、模型评测是什么

模型评测不是评测大模型“聪不聪明”，而是评测整个 RAG 系统是否可靠。

建议建立一个 医学教材 RAG 标准评测集。

1. 评测问题类型

以《基础护理学》为例，至少设计 7 类问题：

类型 示例
直接知识问答 静脉输液的目的是什么？
操作流程问答 静脉输液操作前应做哪些评估？
异常处理问答 输液过程中局部肿胀怎么办？
易混淆概念 输液外渗和静脉炎有什么区别？
表格检索 常见输液故障及处理有哪些？
图片理解 这张穿刺示意图说明了什么操作要点？
题目解析 这道护士资格证题为什么选 C？

2. 评测指标

至少看 6 个指标：

指标 含义
Recall@5 前 5 个检索结果是否包含正确教材证据
Recall@10 前 10 个结果是否召回正确证据
Citation Accuracy 引用页码、章节、段落是否正确
Answer Faithfulness 回答是否忠实于教材
Hallucination Rate 是否编造教材没有的内容
Teaching Usefulness 对学生学习是否有帮助

3. 为什么评测是核心壁垒

因为 embedding 模型换来换去，谁都可以换。

但如果您们沉淀了：

  • 1000 个基础护理学标准问题；
  • 500 个护士资格证题目解析评测样本；
  • 300 个护理操作流程评测样本；
  • 200 个病例推理评测样本；
  • 每个问题都有标准证据 chunk；
  • 每次更新模型、切片策略、reranker 都能自动回归测试。

这就是别人很难短期复制的资产。

一、知识库和知识图谱的先后关系

1. 不建议一开始就完整建设知识图谱

如果一开始就说“我要先把《基础护理学》做成完整知识图谱”，很容易陷入几个问题：

第一，工程量巨大。
一本教材里有章节、概念、操作流程、注意事项、并发症、护理措施、图表、题目、病例、能力目标，全部人工或半自动整理成图谱，周期会很长。

第二，图谱容易脱离真实检索场景。
图谱做得很漂亮，但学生真实提问时，系统还是要回到教材原文、页码、段落、图片证据。如果没有 RAG 知识库做底座，图谱会变成“概念关系展示系统”，而不是“可回答问题的教学智能体底座”。

第三，医学教材存在大量表述性知识。
例如“输液过程中应观察穿刺部位有无肿胀、疼痛、渗液”等内容，既是文本知识，也是操作要点。它不一定一开始就适合抽成非常规范的图谱三元组。

所以，完整知识图谱不能放在第一步。

2. 也不建议先完全不考虑图谱

反过来，如果第一期只做：

OCR → 切片 → embedding → 向量库

后面再想加知识图谱，会遇到大返工：

  • chunk 没有标准知识点 ID；
  • 章节结构没有稳定编码；
  • 图片和正文没有关联；
  • 题库无法挂接到知识点；
  • 病例无法挂接到操作流程；
  • 学生错题无法汇总到能力维度；
  • 后期无法做知识点掌握度分析。

所以第一期即使不建设完整知识图谱，也必须让每个知识 chunk 天然具备图谱化能力。

第三种：正式知识图谱

当候选关系经过以下环节后，再进入正式图谱：

  • 规则校验；
  • AI 二次复核；
  • 抽样人工审核；
  • 真实问答评测验证；
  • 题库和病例使用验证。

正式图谱才用于：

  • 学生学习路径推荐；
  • 知识点掌握度分析；
  • 题目推荐；
  • 病例推荐；
  • 教师备课；
  • 课程诊改；
  • 能力画像。

所以先后顺序应该是：

图谱骨架设计 → 教材知识库建设 → AI抽取图谱候选 → 评测验证 → 正式图谱沉淀 → 题库病例深度挂接。

三、第一期做教材 RAG 时，知识点切分能不能由 AI 来做？

可以，而且应该主要由 AI 来做。

但不能简单理解为：

“把教材丢给大模型，让它帮我切。”

正确做法是：

人定义规则和标准，AI 执行批量加工，系统做自动校验，人只做抽检和疑难复核。

这才符合您“能让机器做的不要让人做”的方向。

人主要做什么？

人不应该逐段切教材。人应该做四件事：

  1. 定义切片规则；
  2. 定义知识点 schema；
  3. 做第一章或第一节的标杆样本；
  4. 抽检 AI 输出质量。

AI 主要做什么？

AI 可以承担：

  • 章节结构识别；
  • 知识点切分；
  • chunk 标题生成；
  • chunk 摘要生成；
  • 术语抽取；
  • 同义词生成；
  • 图文关联；
  • 图片说明生成；
  • 表格结构化；
  • 能力点标注；
  • 题库关联候选；
  • 病例关联候选；
  • 图谱关系候选；
  • 质量自检。

我的建议是：AI 做 90%，人做 10% 的规则制定和质量验收。

四、不要用一个大提示词做完，要拆成多个 AI 工序

五、全部用 AI 做是否可行？

可以，但必须是“AI流水线”，不是“AI自由发挥”

可行的前提有 5 个：

第一，必须有结构化输入。
MinerU 输出后，最好使用 JSON，而不是只给 AI 一大段 Markdown。

第二，必须有严格 JSON Schema。
AI 每一步都必须输出固定结构，方便程序校验。

第三，必须有置信度字段。
例如每个知识点、关系、术语都给 confidence，低置信度进入复核队列。

第四，必须有自动质检规则。
例如 chunk 没有页码、没有原文证据、没有知识点 ID，就不能入库。

第五，必须有抽样人工验收。
不是让人做生产，而是让人做质量控制。

所以结论是：

AI 可以做主生产力，但系统规则、数据结构、评测集和质量门禁必须由我们设计。

六、用最强模型是否值得？

我的观点很明确：

不要用最强模型处理所有内容。应该用最强模型做标准制定、疑难样本、质量审查和评测裁判；批量生产用次一级高性价比模型。

原因很简单：教材知识工程是批处理任务，不是每一步都需要最强推理模型。

推荐模型分工

环节 推荐模型策略
制定切片规则 GPT-5.5 / Claude Opus 级别
第一章标杆样本 GPT-5.5 / Claude Opus 级别
章节结构识别 Sonnet / GPT-5.4 / 国内强模型
大批量知识点切分 Sonnet / GPT-5.4 / GPT-5 mini 级别
术语抽取 Sonnet / GPT-5.4 / 国内模型
图片解释 强多模态模型，疑难图用 GPT-5.5
图谱关系抽取 Sonnet 级别即可，疑难关系用 GPT-5.5 复核
质量抽检 GPT-5.5 / Claude Opus 级别
最终评测裁判 GPT-5.5 / Claude Opus 级别

  • 如果每一步都用 Opus 级模型，可能更高。

八、我建议的最优成本质量组合

第一阶段：做“标杆章节”

先拿《基础护理学》“静脉输液”章节做标杆。

使用最强模型：

  • GPT-5.5；
  • Claude Opus 级别；
  • 再加一个国产强模型做对照。

目标不是省钱，而是把标准打出来：

  • 切片规则；
  • 术语规范；
  • 图文关联规则；
  • 图谱关系规则；
  • 输出 JSON Schema；
  • 评测题集；
  • 质量标准。

这一步花 几百元人民币 都值得。

第二阶段：批量加工全书

全书批量加工时，不要全部用最强模型。

建议：

  • 80% 批量任务：Sonnet 4.5 / GPT-5.4 / 国产强模型；
  • 15% 质量复核：GPT-5.5；
  • 5% 疑难问题：GPT-5.5 + 人工专家确认。

这样整体质量和成本最平衡。

第三阶段：建立自动回归评测

每处理完一个章节，就自动生成：

  • 20 个知识点问答；
  • 10 个操作流程题；
  • 10 个异常处理题；
  • 5 个图表理解题；
  • 5 个病例应用题；
  • 5 个易混淆概念题。

这些题反过来测试知识库。

如果检索不到正确 chunk，说明：

  • 切片有问题；
  • 术语标准化有问题；
  • embedding 文本模板有问题；
  • 图谱关系有问题；
  • rerank 有问题。

这才是持续优化机制。

十、真正的竞争力在哪里？

不是 embedding 提示词本身，而是以下 8 个资产。

1. 医学教材知识切分规则

别人按字数切。
我们按医学教育语义切。

例如：

  • 概念；
  • 目的；
  • 适应证；
  • 禁忌证；
  • 操作步骤；
  • 注意事项；
  • 异常处理；
  • 护理评价；
  • 健康教育；
  • 考点；
  • 病例决策点。

这是行业 Know-how。

2. 教材原文与增强文本双轨机制

原文用于证据引用。
增强文本用于高召回检索。

这样既能准，又能追溯。

3. 医学术语标准化体系

把教材语言、学生语言、考试语言、临床语言统一起来。

例如：

针眼鼓包 → 局部肿胀 → 液体外渗 → 输液外渗

这比单纯 embedding 更重要。

4. 图文表一体化处理能力

医学教材大量知识在图表里。

必须做到：

  • 图片有说明；
  • 表格有结构；
  • 图表有关联正文；
  • 图表可以被检索；
  • 图表可以被引用；
  • 图表可以进入题库和病例。

5. 知识图谱骨架

先定义节点和关系，后续所有教材、题库、病例都按同一套结构进入。

这会形成长期数据资产。

6. 题库与教材证据关联

每一道题都知道：

  • 考哪个知识点；
  • 对应教材哪一页；
  • 为什么选这个；
  • 错选项对应什么误区；
  • 学生错了以后该补哪个知识点。

这就是智能辅导智能体的底层能力。

7. 病例与能力图谱关联

每个病例不只是一个故事，而是挂接：

  • 知识点；
  • 操作步骤；
  • 临床判断；
  • 护理措施；
  • 能力点；
  • 评分点。

这会直接支撑 AISP、护理思维、OSCE、实训评价。

8. 自动评测体系

谁有评测集，谁就能持续优化。

最终天堰要形成自己的：

医学教材 RAG 评测基准库

例如：

  • 基础护理学 1000 个标准问题；
  • 护士资格证 1000 道题目映射；
  • 护理操作 300 个流程问题；
  • 病例推理 300 个场景；
  • 图表理解 200 个问题。

这是比模型更重要的壁垒。

教材知识点 → 题目 → 病例 → 实训 → 评价 → 学情分析

这就是产品价值展示。

十二、最终建议

我的最终判断是：

第一期不要重资产建设完整知识图谱，但必须图谱化设计。

具体路线：

  1. 先设计图谱骨架和数据 Schema；
  2. 用 AI 建设教材 RAG 知识库；
  3. 每个 chunk 同步生成知识点、术语、能力点和候选关系；
  4. 用评测集筛选和修正这些候选关系；
  5. 再沉淀正式知识图谱、能力图谱、题库图谱、病例图谱；
  6. 最终形成医学教育知识工程平台，而不是单纯知识库。

关于模型，我建议：

GPT-5.5 / Claude Opus 级模型用于标准制定、样板章节、疑难复核和最终评测；批量生产不要全部使用最贵模型。

真正的护城河不是“用了最强模型”，而是：

用最强模型帮助我们建立一套可复制、可评测、可追溯、可持续进化的医学教材知识工程流水线。
