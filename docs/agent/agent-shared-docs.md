# Agent Shared Docs Contract

> 适用对象：人类、Codex、Claude、GSD  
> 目标：让不同运行时共用少量高价值文档，而不是各写各的状态。

---

## 共享文档清单

| 路径 | 用途 | 是否共享写入 | 说明 |
|------|------|-------------|------|
| `todo.md` | 任务账本 | 是 | 活动任务、待办、阻塞、完成记录 |
| `bug.md` | 问题账本 | 是 | 缺陷、阻塞、设计性结论 |
| `AGENTS.md` | 项目级运行说明 | 低频 | 只在工作流规则变化时修改 |
| `.planning/STATE.md` | GSD 项目状态 | 谨慎 | 优先由 GSD 维护 |
| `.planning/ROADMAP.md` | GSD 路线图 | 谨慎 | 仅 phase / milestone 变化时修改 |
| `.planning/REQUIREMENTS.md` | GSD 需求清单 | 谨慎 | 仅需求边界变化时修改 |

---

## `todo.md` 规则

`todo.md` 是跨代理共享的“活任务账本”，适合记录：

- 当前进行中的任务
- 下一步要做的任务
- 阻塞任务
- 最近刚完成、但仍需可追踪的任务

### 字段约定

| 字段 | 说明 |
|------|------|
| `ID` | 稳定任务编号，如 `T-001` |
| `状态` | `TODO` / `DOING` / `BLOCKED` / `DONE` / `CANCELLED` |
| `Owner` | `human` / `codex` / `claude` / `gsd` / `shared` |
| `类型` | `feature` / `bugfix` / `docs` / `research` / `ops` |
| `范围` | 模块或目录 |
| `任务` | 一句话描述 |
| `依赖` | 依赖的任务 ID 或 `-` |
| `更新时间` | `YYYY-MM-DD` |
| `备注` | 可选说明 |

### 更新原则

1. 新任务新增条目，不抢占别人的条目含义。
2. 状态变化优先原位更新，不要删除历史任务。
3. 任务完成后，优先改成 `DONE`，而不是直接移除。
4. 如果任务只在当前会话内临时存在、不值得跨会话追踪，可以不写入。

---

## `bug.md` 规则

`bug.md` 是跨代理共享的问题账本，适合记录：

- 真实缺陷
- 复现明确的回归
- 环境阻塞
- 经过确认的设计性结论

### 状态约定

- `🔴` 阻塞
- `🟡` 待改进 / 待处理
- `🟢` 已修复
- `🔵` 设计决策 / 非 bug 结论

### 更新原则

1. 保持现有分类结构，不要随意改成另一种格式。
2. 新问题尽量追加到对应分类表格。
3. 已有问题修复时，更新状态与修复说明。
4. 不删除旧记录，除非是明确重复且已人工确认合并。

---

## `.planning/**` 规则

`.planning/**` 是 GSD 的规划与状态源文件，不是普通备忘录。

### 默认策略

- 日常实现任务：优先只读
- phase / milestone 管理任务：允许写
- 仅当用户明确要求、或 GSD 工作流本身需要时，才直接改写

### 典型读取顺序

1. `.planning/STATE.md`
2. `.planning/ROADMAP.md`
3. 目标 phase 的 `CONTEXT.md`
4. 目标 phase 的 `PLAN.md`
5. 目标 phase 的 `SUMMARY.md`

---

## 冲突处理

出现多人 / 多代理同时修改时，按以下规则处理：

1. 事实优先于表述。
2. 最新且可验证的状态优先。
3. 保留历史，不做大段覆盖式重写。
4. 不确定时，在备注中标注冲突，而不是擅自清洗历史。

---

## 推荐最小流程

开始任务前：

1. 看 `todo.md`
2. 看 `bug.md`
3. 看 `AGENTS.md`
4. 若任务跨阶段，再看 `.planning/STATE.md`

结束任务后：

1. 更新 `todo.md`
2. 如发现问题或修复问题，更新 `bug.md`
3. 如影响 phase 状态，再补 `.planning/**`
