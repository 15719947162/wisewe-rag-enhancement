## 1. 项目定位

### 这个项目到底是干什么的？

它不是单纯的 PDF 查看器，也不是“直接把问题扔给大模型”的壳。

它的目标是：

1. 把教材、文档、资料处理成结构化知识库。
2. 让知识库支持检索、问答、证据回溯。
3. 在普通 RAG 之外，同时保留 Graph RAG 所需的关系和三元组。

一句话说：

`文档处理系统 + 知识库系统 + RAG 问答系统`

### 它和 Dify 是什么关系？

当前主链路不依赖 Dify。

更接近的真实路径是：

`文档 -> 本项目处理 -> 本项目数据库 -> 本项目检索 -> OpenAI-compatible 模型回答`

所以：

1. 不是“处理完自动同步到 Dify”。
2. 也不是“最终回答由 Dify 工作流完成”。

## 2. 关键对象

### `ContentBlock` 是什么？

`ContentBlock` 是解析器输出的结构化内容块。

它还不是最终给检索用的知识块，而是文档被解析后的中间结构。  
常见类型有：

1. 标题
2. 正文
3. 表格
4. 图片

对应代码：

- [core/models/content_block.py](/E:/all_project/AI/rag-enhancement/core/models/content_block.py)

### `Chunk` 是什么？

`Chunk` 是真正进入知识库、参与检索和问答的基本单元。

可以把它理解成：

“从多个结构化内容块中整理出来的、适合被检索和引用的知识块”

### `enhanced chunk` 是什么？

`enhanced chunk` 不是替换原始 chunk，而是附加的一层增强块。

它的作用通常是：

1. 补背景
2. 补术语解释
3. 补图表说明
4. 生成更适合召回的表达

所以关系通常是：

`child chunk` 是原始知识单元  
`enhanced chunk` 是挂在它下面的增强版本

## 3. 解析器与 provider

### 这个项目是自动判断该用哪个解析器吗？

不是。

当前是“配置驱动选择 provider”，不是“系统先分析 PDF 再自动决定 provider”。

更准确地说：

1. 先读取配置里的 provider。
2. 再分发到对应解析器模块。

### 目前主要有哪些 provider？

当前主要可以关注三个：

1. `mineru`
2. `mineru_official`
3. `ali_document_mind`

### 三者的核心区别是什么？

可以粗略理解成：

1. `mineru`：默认通道，走 302AI 代理的 MinerU。
2. `mineru_official`：官方 MinerU Precision API 直连。
3. `ali_document_mind`：阿里云另一套文档解析服务，不属于 MinerU 体系。

共同点是：

1. 最终都要产出 `ContentBlock[]`。
2. 后面都会进入同一套清洗、切片、向量化、写库链路。

区别主要在：

1. 上游服务不同
2. 鉴权不同
3. 任务提交方式不同
4. 输出映射细节不同

## 4. 清洗、切片、增强

### 清洗是在做什么？

清洗不是“美化文本”，而是“减少垃圾进入知识库”。

典型目标：

1. 去掉空块
2. 去掉噪声块
3. 去掉重复或明显无价值内容

### 切片是在切什么？

切的不是原始 PDF 字节流，而是：

`解析并清洗后的 ContentBlock[]`

也就是说，项目不是直接对 PDF 文件做粗暴切片，而是先解析成结构化块，再重组为知识块。

### `ContentBlock -> Chunk` 是复制吗？

不是复制，是重组。

常见情况：

1. 多个正文块合并成一个或多个 `child chunk`
2. 一个表格块直接生成一个 table chunk
3. 一个图片块直接生成一个 image chunk

### 为什么规则切片之后还要增强？

因为“规则能切开”不等于“切得适合召回”。

增强的价值主要在：

1. 让碎片内容更完整
2. 让图表更容易被命中
3. 让术语和上下文关系更容易被检索
4. 为知识图谱补实体和三元组

### 增强有哪几类？

在 `hierarchical` 策略里，增强大致分四类：

1. `text`
2. `fragment`
3. `table`
4. `image`

不同类型走不同增强逻辑。

## 5. 检索与问答

### 问答是直接问模型吗？

不是。

真实顺序更接近：

1. 用户提问
2. 系统先检索证据
3. 再用证据组织上下文
4. 最后调用模型生成答案

### retriever 在做什么？

retriever 负责把“问题”变成“候选证据”。

通常会包含：

1. dense 检索
2. BM25 / sparse 检索
3. graph 扩展
4. 特殊引用类召回
5. 候选融合与折叠

对应代码：

- [core/rag/retriever.py](/E:/all_project/AI/rag-enhancement/core/rag/retriever.py)

### generator 在做什么？

generator 不是负责找证据，而是负责：

1. 消化已有上下文
2. 调模型生成答案
3. 解析引用
4. 必要时做 fallback

对应代码：

- [core/rag/generator.py](/E:/all_project/AI/rag-enhancement/core/rag/generator.py)

## 6. 数据库关系

最核心的一条关系链是：

`knowledge_bases -> documents -> chunks`

围绕 `chunks` 再扩展出：

1. `chunk_relations`
2. `entities`
3. `entity_mentions`
4. `kg_triples`
5. `chunk_drafts`

可以这样记：

1. `documents` 是文档层。
2. `chunks` 是检索层。
3. `chunk_relations` 是关系层。
4. `kg_triples` 是图谱层。
5. `chunk_drafts` 是入库前中间层。

## 7. 常见问答

### 问：增强完成是不是就已经入库完成了？

不是。

增强完成只是 chunk 生产结束。  
后面通常还有：

1. 质量门控
2. 向量化
3. 语义/流程/因果关系补全
4. 实体物化
5. 正式写库

### 问：enhanced chunk 是不是单独放一张表？

不是。

它通常仍然落在 `chunks` 表里，只是 `layer="enhanced"`。

### 问：为什么项目里既有普通 RAG，又有 Graph RAG？

因为它不只存向量，还存：

1. chunk 之间的关系边
2. 实体
3. 实体提及
4. 三元组

所以它天然可以同时支撑：

1. 普通基于 chunk 的召回
2. 基于关系和图谱的扩展召回

### 问：如果一个知识点被规则切散了怎么办？

项目不是强求“绝不切散”，而是通过后续机制补回来，比如：

1. `parent_id`
2. `chunk_index`
3. `related_ids`
4. 相邻关系
5. enhanced 补上下文
6. 检索阶段的扩展召回

### 问：为什么 embedding 也算模型调用？

因为 embedding 本身就是模型能力，不是简单字符串处理。

项目里会对：

1. 文档 chunk
2. 用户 query

分别做向量化，然后再进入相似度检索。

## 9. 补充问答：切片增强做完以后，下一步是什么

### 问：我们前面已经走到“切片完成并做了增强”，那下一步到底是什么？

下一步不是直接开始在线问答，也不是立刻就算“正式入库完成”。

真实代码顺序是：

1. 先把 `chunk_drafts` 草稿读出来
2. 跑质量门控
3. 对通过门控的 chunk 做 embedding
4. 补语义关系、步骤关系、因果关系
5. 物化实体，给实体生成定义和向量
6. 最后才正式写入 pgvector

也就是：

`增强完成`

并不等于

`正式入库完成`

中间还隔着一整段 `confirm_pipeline(...)`。

### 问：为什么增强完了还不能直接算完成？

因为增强只是把 chunk 生产出来了，但正式知识库还没闭环。

这时候至少还有几件大事没做：

1. 还没筛掉低质量 chunk
2. 还没生成向量
3. 还没补 chunk 之间更深层关系
4. 还没抽实体、写实体
5. 还没把 `documents / chunks / chunk_relations / kg_triples` 正式写库

所以增强更像“半成品已经做好”，不是“商品已经入仓”。

### 问：这段后半程的入口代码在哪？

在：

[backend/services/ingestion_service.py](/E:/all_project/AI/rag-enhancement/backend/services/ingestion_service.py:1506)

对应函数：

`confirm_pipeline(task_id)`

你可以直接把它理解成：

`把 chunk 草稿转成正式知识库数据`

### 问：那前半段和后半段怎么区分最容易记？

最容易记的方式是：

1. `run_pipeline_real(...)`：先做解析、清洗、切片、增强、基础关系、草稿保存
2. `confirm_pipeline(...)`：再做质量门控、向量化、关系补全、实体物化、正式写库

如果你想用更白的话记：

1. 前半段：先把文档做成可确认草稿
2. 后半段：再把草稿转成正式知识库

### 问：后半段里最紧挨着增强的第一步是什么？

是质量门控。

也就是：

[core/cleaner/quality_gate.py](/E:/all_project/AI/rag-enhancement/core/cleaner/quality_gate.py)

里的：

`apply_quality_gate(...)`

这一步的作用不是再做一次普通清洗，而是站在“正式入库前最后验收”的角度，看哪些 chunk 值得进正式库，哪些不值得。

### 问：质量门控后为什么马上就是 embedding？

因为 embedding 很贵，所以一定要先筛再做。

如果一个 chunk 质量本来就不合格，还给它生成向量，就是：

1. 浪费模型调用成本
2. 浪费向量库存储
3. 给后续检索增加噪声

所以这条顺序不能反。

### 问：embedding 做完是不是就可以写库了？

还不行。

embedding 后面还会补三类关系：

1. 语义关系
2. 步骤关系
3. 因果关系

再后面还要做实体物化。

所以更准确的理解应该是：

`embedding 只是正式入库前半段中的一个关键步骤，不是最后一步`

### 问：那真正的最后一步是什么？

是：

[core/output/pgvector_writer.py](/E:/all_project/AI/rag-enhancement/core/output/pgvector_writer.py)

里的：

`write_to_pgvector(...)`

只有到这里 commit 成功，这次文档才算真正进入正式知识库。

### 问：质量门控和前面的 `clean_blocks(...)` 有什么区别？

最容易记的区分方式是：

1. `clean_blocks(...)` 处理的是 `ContentBlock`
2. `apply_quality_gate(...)` 处理的是 `Chunk`

前者更像原料预处理，目标是别让明显垃圾块进入切片阶段。

后者更像成品质检，目标是别让“虽然切出来了、但其实不值得检索”的 chunk 进入正式知识库。

### 问：embedding 这一步为什么代码看起来这么复杂？

因为它不是简单调一次 API。

它还要同时解决这些工程问题：

1. 大批量文本怎么分 batch
2. 怎么并发跑
3. 出错怎么重试
4. 限流时要不要换 key
5. 多 key 怎么调度
6. 并发跑完以后怎么保证向量顺序不乱
7. 怎么把耗时、token、retry 统计出来

所以 `embedding/client.py` 本质上更像“向量化执行器”，不只是“向量化客户端”。

### 问：为什么语义关系、步骤关系、因果关系要拆成 3 个 linker？

因为它们补的是 3 种不同类型的知识连接：

1. `semantic_linker` 补“意思接近”
2. `procedure_linker` 补“前后步骤”
3. `causal_linker` 补“原因和结果”

这 3 类边的判断依据完全不一样：

1. 语义关系主要靠 embedding 相似度
2. 步骤关系主要靠顺序词和编号规则
3. 因果关系主要靠因果触发词规则

所以拆开写比混成一个“大关系补全器”更清楚，也更容易单独调。

### 问：为什么实体物化主要吃 `enhanced` 层，而不是直接吃 `child` 层？

因为 `child` 更偏原始证据，`enhanced` 更偏语义整理后的表达。

实体抽取通常更适合从“语义更完整、表述更清楚”的文本里拿结果。

所以项目这里的思路是：

1. `child` 负责保真和展示证据
2. `enhanced` 负责补语义和支撑图谱抽取

### 问：`pgvector_writer.py` 为什么说不是只写向量？

因为它真正落的不是一列 embedding，而是一整套正式知识库数据：

1. `documents`
2. `chunks`
3. `chunk_relations`
4. `kg_triples`

而且 `chunks` 里也不只是正文和向量，还包括：

1. 标题
2. 页码
3. 分层信息
4. 图片路径
5. parent / related 结构信息
6. 供全文检索用的 `search_vector`

所以这个文件本质是在做“正式事务写库”。

### 问：在线问答是不是就是“把 query 丢给模型”？

不是。

真实顺序更接近：

1. 先做知识库访问校验
2. 先检索候选证据
3. 再重排候选证据
4. 再把证据组织成上下文窗口
5. 再让生成器回答
6. 最后再评分、记日志、记 token 用量

所以它始终是：

`先找证据，再回答`

不是：

`直接问模型`

### 问：为什么 retriever 后面还要再来一个 reranker？

因为 retriever 更偏“别漏掉”，reranker 更偏“谁最值得进最终答案上下文”。

检索器通常会多拿一些候选，宁可宽一点。

重排器再把这些候选重新排顺序，尽量把最该给生成器看的块放到前面。

所以它们不是重复劳动，而是前后分工不同。

### 问：命中了 `enhanced`，为什么最后给用户看的常常还是 `child`？

因为这个项目的设计是：

1. `enhanced` 更适合提高召回
2. `child` 更适合做原始证据展示

所以系统常常会：

1. 先靠 `enhanced` 命中
2. 再折回它来源的 `child`
3. 最终用 `child` 做 citation 和证据展示

最容易记的一句话就是：

`enhanced 帮你找到，child 帮你举证`

### 问：为什么生成器还要强制模型写 `[1][2]` 这种编号？

因为后面的代码要靠这个编号把答案里的引用解析回真实上下文块。

也就是说：

1. 模型负责在答案里写引用编号
2. 代码负责把编号映射回 citation 对象

这比让模型直接输出复杂 JSON 引用结构更稳。

### 问：如果模型拒答或者不老实不给引用怎么办？

项目没有把全部希望押在模型是否听话上。

如果：

1. 模型拒答
2. 模型没给 citations
3. 模型调用报错

系统还会尝试抽取式 fallback，也就是直接从高分上下文里挑句子，拼一个保底答案。

所以它不是“模型一失手，整次回答就彻底废掉”。

### 问：`relevance_score` 和 `faithfulness_score` 是一回事吗？

不是。

你可以先这样记：

1. `relevance_score`：这批证据跟问题整体像不像、强不强
2. `faithfulness_score`：答案里的引用是不是老老实实对得上当前上下文

前者更偏“找得对不对”。

后者更偏“答得有没有老实引用证据”。

### 问：Graph RAG 和普通 RAG 是不是只是“多查一点关系”？

不能只这么理解。

更准确地说：

1. 普通 RAG 的主目标是“给答案”
2. Graph RAG 的主目标是“沿关系网扩结果并解释路径”

普通 RAG 的主链路更像：

`retrieve -> rerank -> generate -> score`

Graph RAG 的主链路更像：

`seed recall -> intent classify -> graph expand -> hydrate -> explain`

所以它们不是简单的“查得多一点”关系，而是在线消费方式都不一样。

### 问：Graph RAG 为什么还要先做普通召回，不能直接从图里查吗？

因为图扩展也得有起点。

如果一开始的 seed chunk 找偏了，后面沿图走得再远也没意义。

所以 Graph RAG 的现实做法是：

1. 先借普通 RAG 的 dense / sparse / entity 召回找到一批靠谱种子
2. 再从这些种子沿 `chunk_relations` 和实体关系继续往外扩

所以它不是抛弃普通召回，而是在普通召回上再叠一层图扩展。

### 问：Graph RAG 里的 `intent` 为什么重要？

因为它会影响“扩图时优先走哪种边”。

比如：

1. 步骤类问题更偏好 `next_step / prev_step`
2. 概念类问题更偏好 `mentions / semantic_similar / sibling`
3. 图示类问题更偏好 `refers_to`

也就是说，同一张关系网，不同问题类型的走法不一样。

这就是 `intent` 的意义。

### 问：Graph RAG 现在为什么不像普通 RAG 那样直接给最终自然语言答案？

从当前代码看，Graph RAG 这一支的重点还在：

1. 图扩检索
2. 实体补充
3. 路径解释
4. 结果整理

它当前更像“图检索与解释工作台”，而不是已经完全闭合成“图问答生成器”。

所以你现在看到的返回重点是：

1. `intent`
2. `results`
3. `stats`
4. 可选 `path`

而不是像普通 RAG 那样重点返回：

1. `answer`
2. `citations`
3. `scores`

### 问：Graph RAG 里的 `path` 是什么？

它就是系统从 seed 往外扩图时走过的关系路径记录。

里面会记类似：

1. 从哪个 chunk 出发
2. 走到了哪个点
3. 经过了什么 `rel_type`
4. 边权是多少

如果开了 `explain=true`，这条路径就会被带回结果里，方便解释“为什么会扩到这里”。

### 问：后端主链不是已经讲完了吗，为什么还要补 `routes`、`access_control`、`task_store` 这些层？

因为如果只讲核心算法文件，你只能理解：

`功能逻辑怎么跑`

但还不够理解：

`真实后端系统怎么对外提供服务`

这几层分别负责：

1. `routes`：把内部函数变成 HTTP API
2. `access_control`：统一做资源裁剪和拒绝审计
3. `chunk_draft_service`：支撑“先草稿、后确认、再转正”
4. `task_store`：支撑长任务状态保存、SSE、重试和任务台账

所以它们不是算法核心，但是真实后端外壳层里不可缺的部分。

### 问：为什么很多无权访问不是 403，而是 404？

因为这里走的是“资源不可见”语义，不只是“你没权限”语义。

这样做的好处是：

1. 不暴露资源是否存在
2. 更适合租户隔离
3. 更符合私有资源裁剪场景

所以很多地方看起来像权限问题，后端却故意返回 404，这是设计行为，不是随便写的。

### 问：`chunk_drafts` 到底是临时缓存，还是正式中间层？

更准确地说，它是“认真设计过的中间草稿层”。

因为它不仅存文本，还存：

1. 层级信息
2. related_ids
3. enhanced_text
4. extracted_entities
5. extracted_triples
6. relations

而且还支持：

1. 预览
2. 编辑
3. 删除
4. 合并
5. 过期清理

所以它不是随手缓存，而是确认前的重要中间状态层。

### 问：`task_store` 为什么重要，它不是只是存状态吗？

它虽然“只是存状态”，但离线入库本来就是长任务。

如果没有它，就很难稳定支持：

1. 异步后台执行
2. 任务详情查询
3. SSE 实时推送
4. 页面刷新后继续看进度
5. 失败后重试
6. 后台任务台账

所以它不是主算法层，但它是把离线入库做成产品级任务系统的关键基础设施。

### 问：`documents`、`chunks`、`chunk_drafts` 这三个最容易混，怎么区分？

最容易记的方式是：

1. `documents`：文档级正式记录
2. `chunk_drafts`：确认前的中间草稿层
3. `chunks`：确认后的正式检索层

也就是说：

`一份文档`

会先变成很多草稿 chunk，

草稿确认后，

才变成正式 `chunks`。

### 问：`chunk_relations`、`kg_triples`、`entity_mentions` 这三张关系表有什么区别？

它们不是一回事。

可以这样记：

1. `chunk_relations`：chunk 和 chunk / entity 之间的通用边
2. `kg_triples`：更像图谱语义表达的三元组
3. `entity_mentions`：实体在哪些 chunk 里被提到

最直接的理解是：

1. `chunk_relations` 更偏“检索层和导航层关系”
2. `kg_triples` 更偏“知识图谱表达”
3. `entity_mentions` 更偏“实体和文本块的连接器”

### 问：为什么 Graph RAG 现在没有单独一张“Graph RAG 结果表”？

因为它当前重点是“消费已有关系层”，不是“把图检索结果长期持久化成新主数据”。

它主要复用的是：

1. `chunks`
2. `chunk_relations`
3. `entities`
4. `entity_mentions`
5. `kg_triples`

也就是说，Graph RAG 现在更像一种在线查询与扩展方式，而不是一套额外主数据存储。

### 问：怎么快速区分主数据、过程数据、观测数据？

可以先这样记：

1. 主数据：知识库本体本身，像 `documents / chunks / chunk_relations / entities / kg_triples`
2. 过程数据：主链运行中的中间层，像 `chunk_drafts`、任务状态、身份同步运行记录
3. 观测数据：为了追踪、报表、审计而记的，像 `kb_rag_query_logs / kb_llm_call_logs / kb_token_usage_hourly / kb_audit_logs`

这个区分一旦记住，后面看表就不容易乱。

### 问：`kb_rag_query_logs` 和 `kb_llm_call_logs` 有什么本质区别？

最简单的记法是：

1. `kb_rag_query_logs`：一次请求整体摘要
2. `kb_llm_call_logs`：这次请求内部每一次模型调用明细

也就是说：

1. query log 更偏“整次问答”
2. llm call log 更偏“问答里每一步模型花了什么”

所以一个请求通常只对应一条 query log，但可能对应多条 llm call log。

### 问：为什么有了 `kb_llm_call_logs` 还要 `kb_token_usage_hourly`？

因为两者用途不同：

1. `kb_llm_call_logs` 是细台账
2. `kb_token_usage_hourly` 是报表汇总层

前者适合查具体请求、具体阶段、具体模型调用。

后者适合按小时、按知识库、按链路、按阶段看总量趋势。

可以把它们理解成：

`明细表`

和

`汇总表`

的关系。

### 问：审计日志和查询日志是不是一回事？

不是。

查询日志更偏：

`一次问答请求发生了什么`

审计日志更偏：

`谁对系统资源做了什么治理或安全动作`

比如：

1. 权限拒绝
2. 配置修改
3. API Key 操作
4. owner 迁移

这些都应该进审计，不应该混进普通问答查询日志。

### 问：为什么很多日志函数都写成“失败也不能影响主链”？

因为项目在这里的原则是：

`观测很重要，但不能反过来拖垮主业务`

也就是说，如果一次问答本来已经能成功回答，不应该因为日志表瞬时写失败，就把整次问答也判成失败。

所以这里采用的是：

1. 尽量写
2. 写失败就回滚自己的日志事务
3. 但不要反过来把主业务打崩

这是一种很典型的产品级后端取舍。

### 问：Graph RAG 为什么当前观测层看起来比普通 RAG 轻？

因为当前 Graph RAG 的定位还更偏：

1. 图检索
2. 路径解释
3. 结果浏览

还没有像普通 RAG 那样完全闭合成：

`rerank -> generate -> score -> answer`

的最终问答成品链。

所以它当前日志重点更偏“这次 graph query 发生过”，而不是像普通 RAG 那样细拆多段模型调用和最终评分。
