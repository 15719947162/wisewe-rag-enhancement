# 页面覆盖：入库管理

> 本文件中的规则仅覆盖 `docs/design/system/MASTER.md` 在入库管理页面中的部分。

## 页面目标

这是文档入库和索引构建的操作工作台。它必须让流水线中的每一个处理阶段都可见、可查、可解释。

## 语言覆盖

- 本页面所有可见 UI 文案必须使用 `简体中文`。
- `PDF`、`LLM`、`CSV`、`pgvector` 等必要技术标识可保留英文。
- 面向用户的标签、状态描述、错误原因、操作提示、导出说明必须为中文。
- 队列状态、Diff 描述、筛选条件、导出结果不得默认输出英文。

## 核心用户任务

- 上传并排队文档
- 查看解析结果
- 比较清洗前后结果
- 比较不同分块策略
- 检查质量筛选决策
- 导出结果或写入 pgvector

## 布局覆盖

使用 `按阶段优先的生产线工作台`：

- 顶部：任务头部、阶段步进器和主操作
- 左侧：任务列表或文档队列
- 中间：阶段结果工作区
- 右侧：包含证据、参数和错误信息的详情抽屉
- 主视图优先展示任务状态、阶段耗时、处理结果和最新日志摘要

## 必备视图

1. `Parse 视图`
- PDF 元数据
- 解析状态
- 各类内容块数量
- 示例内容块预览

2. `Clean 视图`
- 清洗前后对比
- 被移除内容块的原因
- 开启时展示 LLM 清洗决策

3. `Chunk 视图`
- 策略选择器
- 输出数量对比
- parent / child / enhanced 层级树预览
- 表格 / 图片 / 文本 chunk 标签

4. `Quality Gate 视图`
- 被丢弃的 chunk 数量
- 丢弃原因分布
- 每个 chunk 的评分预览

5. `Export 视图`
- CSV 结果
- pgvector 写入状态
- `kb_id` 与文档关联摘要

## 关键设计规则

- 每个阶段都必须包含 `状态`、`输入数量`、`输出数量`、`耗时`、`原因`
- 对比视图必须左右并排，而不是只藏在标签页里
- chunk 层级结构必须足够直观
- 按 block 类型和 chunk 类型筛选必须足够快
- SSE 实时日志不能抢占主视图，只能作为辅助信息
