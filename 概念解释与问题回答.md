## 1. 项目定位

### 这个项目到底是干什么的？

它不是单纯的 PDF 查看器，也不是“把问题直接丢给大模型”的薄壳。

它的目标是：

1. 把教材、文档、资料处理成结构化知识库
2. 让知识库支持检索、问答、证据回溯
3. 在普通 RAG 之外，同时保留 Graph RAG 需要的关系和三元组

一句话理解：

`文档处理系统 + 知识库系统 + RAG 问答系统`

### 它和 Dify 是什么关系？

当前主链路不依赖 Dify。

更接近的真实路径是：

`文档 -> 本项目处理 -> 本项目数据库 -> 本项目检索 -> OpenAI-compatible 模型回答`

所以：

1. 不是“处理完自动同步到 Dify”
2. 也不是“最终回答由 Dify 工作流完成”

## 2. 关键对象

### `ContentBlock` 是什么？

`ContentBlock` 是解析器输出的结构化内容块。
它还不是最终参与检索的知识块，而是文档被解析后的中间结构。

常见类型有：

1. 标题
2. 正文
3. 表格
4. 图片

对应代码：

- [core/models/content_block.py](/abs/path/E:/study/wisewe-rag-enhancement/core/models/content_block.py:1)

### `Chunk` 是什么？

`Chunk` 是真正进入知识库、参与检索和问答的基本单元。
可以把它理解成：

“从多个结构化内容块中整理出来的、适合被检索和引用的知识块”

`Chunk` 模型定义见：

- [core/models/content_block.py](/abs/path/E:/study/wisewe-rag-enhancement/core/models/content_block.py:226)

### `enhanced chunk` 是什么？

`enhanced chunk` 不是替换原始 chunk，而是在原始 `child chunk` 基础上再挂一层增强表达。

它的作用通常是：

1. 补背景
2. 补术语解释
3. 补图表说明
4. 生成更适合召回的表达

所以通常关系是：

`child chunk` 是原始证据块  
`enhanced chunk` 是挂在它下面的增强层

## 3. 解析器与 provider

### 这个项目是自动判断该用哪个解析器吗？

不是。

当前是“配置驱动选择 provider”，不是“系统先分析 PDF 再自动决定 provider”。

更准确地说：

1. 先读取配置里的 `provider`
2. 再分发到对应解析器模块

### 目前主要有哪些 provider？

当前主要可关注三个：

1. `mineru`
2. `mineru_official`
3. `ali_document_mind`

### 三者的核心区别是什么？

可以粗略理解成：

1. `mineru`：默认通道，走 302AI 代理的 MinerU
2. `mineru_official`：官方 MinerU Precision API 直连
3. `ali_document_mind`：阿里云另一套文档解析服务，不属于 MinerU 体系

共同点是：

1. 最终都要产出 `ContentBlock[]`
2. 后面都会进入同一套清洗、切片、向量化、写库链路

区别主要在：

1. 上游服务不同
2. 鉴权不同
3. 任务提交方式不同
4. 输出映射细节不同

## 4. 清洗、切片、增强

### 清洗是在做什么？

清洗不是“美化文本”，而是“减少垃圾进入知识库”。

典型目标：

1. 去掉空块
2. 去掉噪声块
3. 去掉重复或明显无价值内容

### 切片是在切什么？

切的不是原始 PDF 字节流，而是：

`解析并清洗后的 ContentBlock[]`

也就是说，项目不是直接对 PDF 做粗暴切片，而是先解析成结构化块，再重组成知识块。

### `ContentBlock -> Chunk` 是复制吗？

不是复制，是重组。

常见情况：

1. 多个正文块合并成一个或多个 `child chunk`
2. 一个表格块直接生成一个 table chunk
3. 一个图片块直接生成一个 image chunk

### 为什么规则切片之后还要增强？

因为“规则能切开”不等于“切得适合召回”。

增强的价值主要在：

1. 让碎片内容更完整
2. 让图表更容易被命中
3. 让术语和上下文关系更容易被检索
4. 为知识图谱补实体和三元组

### 增强有哪几类？

在 `hierarchical` 策略里，增强大致分四类：

1. `text`
2. `fragment`
3. `table`
4. `image`

不同类型走不同增强逻辑。

增强链路主文件：

- [core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:1)

## 5. 检索与问答

### 问答是直接问模型吗？

不是。

真实顺序更接近：

1. 用户提问
2. 系统先检索证据
3. 再用证据组织上下文
4. 最后调用模型生成答案

### retriever 在做什么？

retriever 负责把“问题”变成“候选证据”。

通常会包括：

1. dense 检索
2. BM25 / sparse 检索
3. graph 扩展
4. 特殊引用类召回
5. 候选融合与折叠

对应代码：

- [core/rag/retriever.py](/abs/path/E:/study/wisewe-rag-enhancement/core/rag/retriever.py:1)

### generator 在做什么？

generator 不是负责找证据，而是负责：

1. 消化已有上下文
2. 调模型生成答案
3. 解析引用
4. 必要时做 fallback

对应代码：

- [core/rag/generator.py](/abs/path/E:/study/wisewe-rag-enhancement/core/rag/generator.py:1)

## 6. 数据库关系

最核心的一条关系链是：

`knowledge_bases -> documents -> chunks`

围绕 `chunks` 再扩展出：

1. `chunk_relations`
2. `entities`
3. `entity_mentions`
4. `kg_triples`
5. `chunk_drafts`

可以这样记：

1. `documents` 是文档层
2. `chunks` 是检索层
3. `chunk_relations` 是关系层
4. `kg_triples` 是图谱层
5. `chunk_drafts` 是入库前中间层

## 7. 常见问答

### 问：增强完成是不是就已经入库完成了？

不是。

增强完成只代表 chunk 生产结束。
后面通常还有：

1. 质量门控
2. 向量化
3. 语义/流程/因果关系补全
4. 实体物化
5. 正式写库

### 问：enhanced chunk 是不是单独放一张表？

不是。

它通常仍然落在 `chunks` 表里，只是 `layer="enhanced"`。

### 问：为什么项目里既有普通 RAG，又有 Graph RAG？

因为它不只存向量，还存：

1. chunk 之间的关系边
2. 实体
3. 实体提及
4. 三元组

所以它天然可以同时支持：

1. 基于 chunk 的普通召回
2. 基于关系和图谱的扩展召回

### 问：如果一个知识点被规则切散了怎么办？

项目不是强求“绝不切散”，而是通过后续机制补回来，比如：

1. `parent_id`
2. `chunk_index`
3. `related_ids`
4. 相邻关系
5. enhanced 补上下文
6. 检索阶段的扩展召回

### 问：为什么 embedding 也算模型调用？

因为 embedding 本身就是模型能力，不是简单字符串处理。

项目里会对：

1. 文档 chunk
2. 用户 query

分别做向量化，然后再进入相似度检索。

## 8. 补充说明：hierarchical 增强输入、返回结构与 chunk 结构

### 问：`hierarchical` 增强到底发给 LLM 什么？

增强不是对 `parent chunk` 做的，而是围绕 `child chunk` 做的。
也就是说，先有基础切片，再决定某个 `child` 要不要补一层 `enhanced chunk`。

按类型看，大致是这样：

1. `text` 增强
   - 输入：`章节标题 title + child.content`
   - 目标：生成一句摘要，外加几个更适合检索命中的问题表达
   - 对应实现：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:720)

2. `fragment` 增强
   - 输入：`章节标题 title + 缺少独立上下文的片段内容`
   - 目标：补背景、补术语解释、补检索问题
   - 对应实现：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:947)

3. `table` 增强
   - 输入：`章节标题 title + table_content`
   - 目标：概括表格主题、解释术语、提炼规律或结论
   - 对应实现：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:904)

4. `image` 增强
   - 输入分两种：
   - 有 VL 模型时：发图片本身的 base64 加提示词
   - 没有 VL 模型时：发 `alt_text/图片说明文字`
   - 目标：描述图片内容、说明教学作用、补关键概念或问题
   - 对应实现：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:822)

### 问：LLM 返回的增强结果期望长什么样？

代码设计上的理想返回是一个可解析的 JSON，对应结构大致是：

```json
{
  "summary": "该知识点的摘要",
  "questions": ["问题1", "问题2"],
  "entities": [
    {
      "name": "机器学习",
      "type": "概念",
      "aliases": ["ML"]
    }
  ],
  "triples": [
    {
      "s": "机器学习",
      "p": "属于",
      "o": "人工智能",
      "confidence": 0.7
    }
  ]
}
```

解释：

1. `summary`
   - 供增强切片主内容使用

2. `questions`
   - 供检索召回扩展使用

3. `entities`
   - 供实体抽取与图谱物化使用

4. `triples`
   - 供 `kg_triples` 写库使用

运行时对应的解析结果模型是 `EnhancedOutput`：

- [core/chunker/enhanced_parser.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/enhanced_parser.py:57)

解析入口函数是：

- [parse_enhanced_response(...)](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/enhanced_parser.py:119)

但运行时不是强制要求模型一定返回标准 JSON。
项目这里做了容错：

1. 如果能从返回文本里提取出 JSON，就按结构化结果解析
2. 如果提取不到 JSON，就把整段返回文本当作 `summary`
3. 这时 `questions/entities/triples` 可能为空

所以你可以把它理解成两种情况：

1. 理想情况：结构化增强结果
2. 兜底情况：只有一段增强摘要文本

### 问：`parent`、`child`、`enhanced` 三种 chunk 的数据结构有什么区别？

三者底层其实都是同一个 `Chunk` 模型，不是三张不同的表，也不是三种不同的类。
差异主要靠字段值区分，特别是：

1. `layer`
2. `parent_id`
3. `enhanced_text`
4. `extracted_entities`
5. `extracted_triples`
6. `token_cost`

`Chunk` 关键字段定义见：

- [core/models/content_block.py](/abs/path/E:/study/wisewe-rag-enhancement/core/models/content_block.py:226)

公共字段大致包括：

```json
{
  "id": "uuid",
  "content": "切片内容",
  "source": "xxx.pdf",
  "page": 12,
  "chunk_index": 5,
  "strategy": "hierarchical",
  "title": "章节标题",
  "char_count": 123,
  "is_table_chunk": false,
  "is_image_chunk": false,
  "image_path": null,
  "layer": "child",
  "parent_id": null,
  "enhanced_text": null,
  "extracted_entities": [],
  "extracted_triples": [],
  "relations": [],
  "token_cost": 0
}
```

### 问：`parent chunk` 是什么样？

`parent` 更像章节容器，通常来自标题块。

对应构建位置：

- [core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:1304)

最小理解可以写成：

```json
{
  "content": "第三章 数据预处理",
  "strategy": "hierarchical",
  "title": "第三章 数据预处理",
  "layer": "parent",
  "parent_id": null,
  "enhanced_text": null,
  "extracted_entities": [],
  "extracted_triples": [],
  "token_cost": 0
}
```

特点：

1. `layer="parent"`
2. 通常没有上级 `parent_id`
3. 主要承载章节结构，不是增强结果

### 问：`child chunk` 是什么样？

`child` 是基础知识块，是真正从正文、表格、图片块整理出来的可检索单元。

对应构建位置：

1. 文本 child：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:2045)
2. 表格 child：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:1325)
3. 图片 child：[core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:1351)

最小理解可以写成：

```json
{
  "content": "这里是一段正文知识点内容",
  "strategy": "hierarchical",
  "title": "第三章 数据预处理",
  "layer": "child",
  "parent_id": "对应parent的id",
  "enhanced_text": null,
  "extracted_entities": [],
  "extracted_triples": [],
  "token_cost": 0
}
```

如果它是表格或图片切片，还会有：

1. `is_table_chunk=true`
2. 或 `is_image_chunk=true`
3. 图片场景下还会有 `image_path`

特点：

1. `layer="child"`
2. `parent_id` 指向章节级 `parent`
3. 它是原始证据块，不是增强块

### 问：`enhanced chunk` 是什么样？

`enhanced` 是在某个 `child` 基础上额外生成的一层增强块，不替代原始 `child`。

对应构建位置：

- [core/chunker/hierarchical.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/hierarchical.py:2234)

最小理解可以写成：

```json
{
  "content": "[LLM增强] 这里是摘要后的增强内容",
  "strategy": "hierarchical",
  "title": "第三章 数据预处理",
  "layer": "enhanced",
  "parent_id": "对应child的id",
  "enhanced_text": "这里是去掉前缀后的增强摘要",
  "extracted_entities": [
    {
      "name": "机器学习",
      "type": "概念",
      "aliases": []
    }
  ],
  "extracted_triples": [
    {
      "s": "机器学习",
      "p": "属于",
      "o": "人工智能",
      "confidence": 0.7
    }
  ],
  "token_cost": 128
}
```

这里最容易记错的一点是：

1. `enhanced.parent_id` 指向的不是章节 `parent`
2. 而是它来源的那个原始 `child.id`

也就是说层次关系通常是：

`parent -> child -> enhanced`

而不是：

`parent -> enhanced`

### 问：怎么一句话区分三者？

最容易记的方式是：

1. `parent`：章节容器
2. `child`：原始证据块
3. `enhanced`：挂在 `child` 下的增强表达层

再压缩成一句：

`parent 负责结构，child 负责证据，enhanced 负责补语义和提升召回`

## 9. 向量与语义关系

### 问：生成向量到底有什么用？

最核心的作用是：

`把文本变成机器可比较的语义坐标`

如果没有向量，系统通常只能做：

1. 看关键词是否一致
2. 看字面字符串是否相似
3. 看有没有显式规则关系

这会漏掉很多“字面不同但意思接近”的内容。比如：

1. “机器学习是人工智能的一个分支”
2. “机器学习属于人工智能领域”

这两句字面不完全相同，但语义很接近。向量的价值，就是让系统能比较这种“意思上的接近”。

### 问：为什么是向量，不是别的格式？

因为向量天然适合做相似度计算。

你可以把一段文本理解成高维空间里的一个点。这个点不是 2 维、3 维，而是几百维甚至上千维。embedding 模型会把文本编码成这样一个坐标，让：

1. 语义接近的文本在空间里更接近
2. 语义差异大的文本在空间里更远

所以向量不是“文本的可读摘要”，而是“文本在语义空间里的位置”。

### 问：向量长什么样？

本质上就是一串浮点数。

例如：

```text
[0.018, -0.274, 0.091, ..., 0.442]
```

在项目代码里，语义链接器接收的就是：

`list[list[float]]`

也就是：

1. 外层列表：很多个 chunk 的向量
2. 内层列表：单个 chunk 的浮点数数组

对应代码：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:77)

### 问：单个数字有意义吗？

通常没有直接的人类可读意义。

不要把某一维理解成“主题A”或“主题B”。真正有意义的是整组数的整体分布。模型训练的目标不是让某一维可解释，而是让整组向量之间的相对位置有语义规律。

所以：

1. 单独看第 137 维没什么意义
2. 整个向量和其他向量比较时才有意义

### 问：项目里生成向量后具体拿来干什么？

至少有两类直接用途：

1. 检索
2. 建关系

检索时：

1. 用户问题会先被转成 query 向量
2. 再拿 query 向量去比库里的 chunk 向量
3. 找最接近的 chunk 作为候选证据

建关系时：

1. 已通过质量门控的 chunk 会批量生成向量
2. 系统会比较 chunk 之间的向量相似度
3. 给语义接近的 chunk 补关系边

所以向量既服务于“问问题找证据”，也服务于“入库时建立语义网络”。

### 问：怎么通过向量做关联？

不是直接看“数字差多少”，而是看“两个向量方向像不像”。

这个项目里用的是：

`余弦相似度（cosine similarity）`

对应实现：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:59)

可以把它粗略理解成：

1. 两个向量方向越接近
2. 余弦值越接近 1
3. 语义越相似

所以关联流程大致是：

1. 先给每个 chunk 生成 embedding
2. 两两计算相似度
3. 超过阈值的进入候选
4. 再从候选里挑最有价值的关系保留

### 问：项目里向量间关系是怎么建立的？

主入口在：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:77)

逻辑可以概括成：

1. 只处理 `layer == "child"` 的 chunk
2. 确保 `embeddings` 和 `chunks` 一一对应
3. 计算 child chunk 两两之间的相似度
4. 相似度超过阈值才进入候选
5. 每个 chunk 只保留 topK 个相近对象
6. 根据相似度高低给关系类型
7. 最后写成双向关系边

默认参数是：

1. `threshold = 0.85`
2. `dup_threshold = 0.95`
3. `topk = 10`

对应代码：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:50)

### 问：怎么知道两个向量之间“有关系”还是“没关系”？

严格说，不是向量自己告诉系统“有关系”，而是系统根据相似度阈值做工程判断。

当前项目里的判断规则是：

1. 相似度 `>= 0.95`：`duplicate_of`
2. 相似度 `>= 0.85` 且 `< 0.95`：`semantic_similar`
3. 相似度 `< 0.85`：不建语义关系

也就是说：

1. 数学层先给出一个相似度分数
2. 业务层再决定这个分数算不算“关系”

对应逻辑：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:356)

### 问：`duplicate_of` 和 `semantic_similar` 有什么区别？

区别在于相似度高到什么程度。

1. `duplicate_of`
   - 分数极高
   - 更像“可能是重复内容”或“表达几乎等价”

2. `semantic_similar`
   - 分数较高
   - 说明主题接近、意思相关
   - 但不一定是同一句话、同一事实的重复表述

所以这两类关系都来自向量相似度，但语义强度不同。

### 问：为什么不是所有相似的向量都连边？

因为会关系爆炸。

如果库里有很多 chunk，两两比较后会出现大量“有点像”的结果。全部连起来会导致：

1. 图结构过密
2. 检索扩展时噪声变多
3. 关系价值下降
4. 后续处理成本上升

所以项目做了两层收口：

1. 只保留超过阈值的
2. 每个 chunk 最多只保留 topK 个相似关系

另外默认还会跳过同一个 `parent` 下的 child chunk，避免同章节相邻内容把语义边刷满。

对应代码：

- [core/chunker/semantic_linker.py](/abs/path/E:/study/wisewe-rag-enhancement/core/chunker/semantic_linker.py:197)

### 问：向量关系是不是就等于事实关系？

不是。

向量关系能说明的主要是：

1. 内容语义接近
2. 主题相关
3. 可能重复
4. 检索时可以互相补充

但它不能直接证明：

1. 因果关系
2. 步骤先后
3. 引用关系
4. 逻辑蕴含
5. 事实真伪

所以项目里才会把关系拆成不同类型的 linker：

1. `semantic_linker`
2. `procedure_linker`
3. `causal_linker`

也就是说，向量只负责“语义接近”这一层，不负责包打天下。

### 问：一句话怎么理解 embedding 和向量关系？

可以这样记：

`embedding` 是把文本压成高维语义坐标，`semantic_linker` 则是在这些坐标之间按相似度阈值补“语义相关”或“可能重复”的边。

1. 有过期时间
2. 会被清理
3. 还没经过最终质量门控和向量化写库

所以最贴切的理解是：

`它不是性能缓存，而是正式入库前的可操作中间工作层。`

### 问：代码里说“一份 chunks、一份 embeddings、通过索引位置绑定”是什么意思？

意思是：运行时并不是把向量直接塞回每个 `Chunk` 对象里，而是同时维护两份并行列表：

1. `chunks`
2. `embeddings`

然后默认约定：

`chunks[i]` 对应的向量就是 `embeddings[i]`

也就是说，它们不是靠对象引用绑定，也不是靠字典 key 绑定，而是靠“相同下标位置”绑定。

### 问：这种位置绑定在主链路里是怎么形成的？

在正式入库链路里，先有通过质量门控后的：

`passed = quality_result.chunks`

然后代码会做：

1. `texts = [chunk.content for chunk in passed]`
2. `embedding_run = embed_texts_with_metrics(texts)`
3. `embeddings = embedding_run.embeddings`

对应位置：

- [backend/services/ingestion_service.py](/abs/path/E:/study/wisewe-rag-enhancement/backend/services/ingestion_service.py:1029)
- [backend/services/ingestion_service.py](/abs/path/E:/study/wisewe-rag-enhancement/backend/services/ingestion_service.py:1056)

这里其实发生了三次形态变化：

1. `passed` 是 `list[Chunk]`
2. `texts` 是 `list[str]`
3. `embeddings` 是 `list[list[float]]`

但顺序一直没变，所以自然得到：

1. `texts[0]` 来自 `passed[0].content`
2. `embeddings[0]` 对应 `texts[0]`
3. 所以 `embeddings[0]` 也就对应 `passed[0]`

### 问：为什么项目要用这种方式，而不是 `chunk_id -> embedding`？

因为批量向量化最自然的数据结构就是数组。

这样做有几个好处：

1. 批量请求更简单
2. 返回结果更容易按顺序回填
3. 后续 NumPy / 矩阵计算更方便
4. 不需要额外维护大字典映射

所以工程上选的是：

`chunks[i] <-> embeddings[i]`

而不是：

`embedding_map[chunks[i].id]`

### 问：这种设计最重要的前提是什么？

最重要的前提就是：

`顺序绝不能乱`

如果 `passed[0]` 是 A，而 `embeddings[0]` 实际上是 B 的向量，那后果会非常严重：

1. chunk 和向量错配
2. 检索语义坐标错误
3. 语义关系建立错误
4. 最终写库结果被污染

所以这个设计的纪律是：

1. 只要改动 `chunks` 的顺序
2. 就必须同步改动 `embeddings`
3. 不能只改一边

### 问：后面的代码是怎么依赖这种位置绑定的？

最典型的是语义关系补全：

`link_semantic(passed, embeddings)`

- [backend/services/ingestion_service.py](/abs/path/E:/study/wisewe-rag-enhancement/backend/services/ingestion_service.py:1061)

`semantic_linker` 一上来就要求：

后面它会用 chunk 的位置索引，直接去取同位置的 embedding 做相似度计算。

所以这个模块完全依赖一个约定：

`第 i 个 chunk 的向量，就是第 i 个 embedding`

### 问：它是不是等于“给 chunk 加了个 embedding 字段”？

从最终落库结果看，可以近似这样理解，因为正式 `chunks` 表里确实会有 `embedding` 字段。

但从运行时逻辑看，不是“立刻给每个 `Chunk` 对象加字段”，而是：

2. 再单独生成一份 `embeddings`
3. 后续函数同时拿这两份列表工作

所以更准确的说法是：

`运行时是两条平行数组，落库后才变成 chunk 记录带 embedding 字段。`

### 问：所以模型评分看重的本质是什么？

2. 文笔好不好
3. 是否像人类写作文一样完整

而是在判断：

1. 有没有知识点
2. 有没有独立检索价值
3. 信息是不是太碎
4. 是不是明显噪声
5. 是否值得为它付出 embedding 和入库成本
