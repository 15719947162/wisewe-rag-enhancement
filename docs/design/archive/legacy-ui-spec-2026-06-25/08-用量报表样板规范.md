# 08 用量报表样板规范

本规范以 `Token 用量统计` 页面作为控制台 UI 升级的第一块验证样板。目标不是做一个孤立漂亮页面，而是把后续日志、审计、配额、成本、性能类页面都能复用的标准先跑通。

## 产品定位

用量报表属于 `治理与审计型工作台`，用户打开页面时首先要判断三件事：

- 当前总消耗是多少，是否有异常放大
- 消耗主要来自哪个链路、知识库或 API Key
- 当前统计口径覆盖了什么，没有覆盖什么

因此页面必须优先表达 `可信数据 + 可比较维度 + 清晰口径`，不要做营销化大横幅、装饰性渐变或无意义图形。

## 视觉标准

### 气质

- 高端：通过克制色彩、清晰对齐、稳定边框和留白体现，不通过夸张阴影体现
- 简洁：一屏内只保留关键指标、主要维度和口径说明
- 互联网化：交互反馈明确，数据层级扫描效率高，控件有明确状态
- 企业级：所有统计必须带来源、口径、时间或边界说明

### 色彩

- 主操作使用 `--color-brand-primary`
- 数据强调采用多角色色板，不使用单一蓝灰或大面积紫蓝渐变
- 深靛蓝用于主操作、导航锚点和总体秩序
- 青绿用于输入、健康、已验证和知识流动
- 琥珀用于成本、警戒、边界说明和待补能力
- 紫罗兰用于总量、结构关系和图谱类信息
- 玫红仅用于失败、风险或异常消耗提示，避免大面积使用
- 表格和排行区域使用白底、浅灰蓝边框、极轻 hover 背景
- 警告口径说明使用琥珀色点位或弱背景，不使用整块高饱和黄

### 字体与数字

- 指标数字使用等宽字体，便于比较位数
- 页面标题使用中文业务标题，不以英文口号为主
- 技术字段如 `kb_rag_query_logs`、`API Key` 可保留英文，但必须搭配中文解释

## 信息架构

页面推荐四段式：

1. `报表头部`
   - 页面标题
   - 统计口径
   - 数据来源标签
   - 刷新与 Top N 控件

2. `核心指标`
   - 请求数
   - Total Token
   - Prompt Token
   - Completion Token
   - 平均延迟

3. `维度排行`
   - 按链路域
   - 按知识库
   - 按 API Key
   - 每一行必须同时展示请求数、Token、平均延迟和相对占比条

4. `口径边界`
   - 已覆盖数据
   - 未覆盖能力
   - 后续升级方向

## 交互标准

- `Top N` 选择后不自动刷新，必须通过明确的 `刷新` 按钮提交，避免误触发请求
- 刷新按钮 loading 时保持尺寸稳定
- 维度排行的 hover 态只改变背景和边框，不做位移
- 空状态要解释“为什么为空”和“产生什么数据后会出现”
- 错误状态必须使用 `role="alert"`，不能只靠颜色提示

## 图表标准

Token 用量验证样板优先使用 `排行条 + 表格`，不引入复杂图表库。

原因：

- 当前 API 只提供聚合桶，不提供时间序列
- 排行条可以表达占比和相对大小
- 表格保留精确值，适合审计和排查

后续当 API 增加小时级 rollup 后，再补：

- 折线图：小时 / 天级 Token 趋势
- 堆叠柱状图：Prompt / Completion 构成
- 异常点标注：高延迟、高消耗请求

## 可复用组件规则

后续日志与用量类页面可复用以下模式：

- `ReportHero`：标题、口径、来源、控件
- `MetricCard`：指标名称、数值、解释、可选强调色
- `RankedUsagePanel`：维度标题、行数、排行条、精确值表格
- `ScopeNotice`：覆盖范围、边界、下一步能力

## 验收标准

- 一屏内能看出总请求、总 Token 和主要消耗来源
- 视觉上不再呈现通用紫色 SaaS 风格
- 所有中文文案清晰、无中英混杂主导
- 表格在移动端可横向滚动，不撑破页面
- 错误、空状态、loading 状态齐全
- `frontend npm run typecheck` 通过
