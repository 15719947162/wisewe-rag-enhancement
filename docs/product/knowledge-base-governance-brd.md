# 知识库治理与开放能力业务需求文档

版本：v0.3  
日期：2026-06-18  
状态：草案  
适用系统：WiseWe RAG 教材 / 文档知识库系统  

## 1. 背景

当前 WiseWe RAG 系统已经具备知识库创建、文档入库、解析、清洗、分块、向量化、pgvector 存储、在线问答、Graph RAG、控制台配置和任务观测能力。随着系统从技术验证走向正式业务交付，需要补齐面向租户、组织、应用和外部调用方的治理能力。

本需求结合 AI 基座租户端已有 MySQL 权限表进行设计。知识库系统只消费 AI 基座同步的租户、用户、角色和用户角色关系等只读身份数据，用于展示、审计和判断管理员身份；菜单权限、会话记录、学生、教职工、分组、标签等复杂组织画像不纳入知识库系统同步范围。知识库系统只负责知识库业务能力，不维护可写的权限、用户和角色主数据。

## 2. 硬性边界

1. 知识库系统不维护可写用户主数据。
2. 知识库系统不维护可写角色主数据。
3. 知识库系统不维护权限策略表。
4. 知识库系统不维护可写用户与角色授权关系。
5. 知识库系统可以保存 AI 基座同步过来的租户、用户、角色、用户角色关系只读快照，但这些快照只用于展示、审计、租户隔离和管理员身份判断，不得成为身份或权限事实源。
6. 第一期租户管理员 / 知识库管理员身份优先按 AI 基座角色快照中的 `role_code=superManager` 判断；命中则视为本租户知识库管理员，未命中则按具备知识库自有归属权限的普通用户处理。
7. 知识库系统不同步菜单权限、会话记录、学生、教职工、分组、标签等复杂组织画像。
8. 知识库系统不提供独立的知识库授权规则配置。
9. 个人创建的知识库默认当前业务归属人为创建人，默认仅当前业务归属人可见、可管理、可操作相关文档和入库链路。
10. 管理员可查看和管理所在租户下所有知识库；平台管理员可按 AI 基座约束查看和管理全局知识库。
11. 知识库系统可以保存业务资源、API Key、配置、任务、日志和审计记录，但这些数据不得成为用户、角色和权限的事实源。
12. 知识库系统颁发的 API Key 仅在知识库系统自身 OpenAPI 内有效，只能消费知识库系统自身能力。
13. 知识库 API Key 不得作为 AI 基座访问凭证、用户身份、角色身份、服务账号或其他业务系统通用凭证使用。
14. 如 AI 基座身份数据或鉴权能力不可用，默认拒绝需要鉴权的新增、修改、删除、导出等敏感操作；可读取的降级能力必须由业务方另行确认。
15. AI 基座账号禁用、离职或移出租户后，该用户不得继续访问知识库后台或操作知识库资源；其历史创建的知识库不自动删除，由租户管理员接管、转移当前业务归属人或删除，平台管理员可按全局范围处理。

## 3. 目标

1. 知识库系统接入 AI 基座租户端身份与鉴权能力，以 AI 基座同步身份数据和知识库业务可见性规则共同约束后台访问。
2. 知识库具备独立对外 API 能力，支持自身 API Key、HTTP API、调用统计、阈值告警和审计。
3. 知识库具备前端管理后台，覆盖知识库、文档、入库、问答、API Key、配置和日志，但不在后台维护用户、角色和权限规则。
4. 知识库具备配置中心，支持全局、租户、知识库、API Key 多层业务配置。
5. 知识库具备日志管理能力，支持任务日志、API 调用日志、RAG 查询日志、配置变更日志和审计日志。
6. 不破坏现有离线入库、在线 RAG、Graph RAG 和控制台链路。

## 4. 非目标

1. 不替换 AI 基座租户端权限系统。
2. 不在知识库系统内建设独立 IAM。
3. 不在知识库系统内复制 AI 基座菜单、会话、学生、教职工、分组、标签和复杂权限规则。
4. 不提供知识库系统侧的用户授权、角色授权、学生范围授权、教职工范围授权配置页面。
5. 不要求第一期实现完整计费结算，也不要求第一期按额度、QPS、并发或 token 预算强制拦截调用，只要求保留调用量、额度、QPS、并发和 token 消耗统计与告警。
6. 不要求第一期替换现有解析、分块、向量化、RAG 生成链路。

## 5. 业务范围

### 5.1 范围内

- AI 基座登录态 / SSO / 身份摘要接入。
- AI 基座租户、用户、角色、用户角色关系只读同步数据消费。
- 租户管理员 / 知识库管理员识别默认使用 `role_code=superManager`；平台管理员角色映射另行确认，只配置角色编码 / 名称映射，不分配角色。
- 知识库当前业务归属人可见、管理员全量可见的业务访问规则。
- 知识库业务资源管理。
- 知识库对外 API Key 管理。
- 对外 HTTP API。
- 管理后台页面。
- 多层业务配置中心。
- 结构化日志与审计。
- API 调用统计、额度统计与阈值告警。
- 身份校验、用户及权限同步和业务访问判断日志与异常告警。

### 5.2 范围外

- AI 基座权限表结构改造。
- AI 基座授权规则配置。
- 菜单权限、会话记录、学生、教职工、分组、标签等组织画像同步落库。
- 知识库本地权限策略维护。
- 统一登录系统重建。
- 外部模型供应商计费。
- 复杂工单审批流。
- 跨区域多活部署。

## 6. AI 基座身份数据与权限边界

知识库系统仅消费 AI 基座同步的必要身份数据。同步数据用于展示、审计、租户隔离和管理员身份判断，不用于在知识库系统内重新建设一套 IAM。

### 6.1 允许同步的只读数据

| 数据 | AI 基座来源表 | 知识库系统使用方式 |
| --- | --- | --- |
| 租户 | `system_tenant` | 展示租户名称、租户状态；业务数据按 `tenant_id` 隔离 |
| 用户 | `sys_user` | 展示创建人、上传人、操作者等基础信息；不保存密码、密码 hash、盐值 |
| 角色 | `sys_role` | 判断是否为租户管理员 / 知识库管理员、平台管理员等管理身份；不在知识库内维护角色 |
| 用户角色关系 | `sys_user_role` | 只读判断用户关联角色；不在知识库内新增、修改、删除关系 |

可同步字段建议：

- `tenant_id`、租户名称、租户状态。
- `user_id`、账号名、显示名、手机号或邮箱脱敏值、账号状态、所属租户。
- `role_id`、角色编码 `role_code`、角色名称、角色状态。
- `user_id` 与 `role_id` 的只读关联。

### 6.2 不纳入同步范围的数据

| 数据 | AI 基座来源表 | 处理方式 |
| --- | --- | --- |
| 菜单权限 | `system_menu`、`sys_role_menu` | 不同步；知识库前端页面按自身功能域展示，后端按当前业务归属人 / 管理员规则兜底 |
| 会话记录 | `b_user_session_record` | 不同步；知识库只维护自身短期 session 和 RAG / API 调用日志 |
| 学生画像 | `b_student_info` | 不同步；当前知识库不做学生范围授权 |
| 教职工画像 | `b_staff_basic` | 不同步；当前知识库不做教职工范围授权 |
| 教职工分组 | `b_staff_group`、`b_staff_group_member` | 不同步 |
| 教职工标签 | `b_staff_tag`、`b_staff_basic.tags` | 不同步 |
| Agent 权限 | `agent_permission` | 不作为知识库权限来源 |
| 访问地址记录 | `agent_access_address`、`digit_use_record` | 不同步；知识库记录自身 OpenAPI 访问日志 |
| AI 基座密码与登录凭证 | `sys_user` 等 | 严禁同步、保存、展示或写入日志 |

### 6.3 知识库业务访问规则

知识库系统第一期只采用简洁业务规则：

1. 个人创建的知识库，默认仅创建人本人可见。
2. 当前业务归属人可对归属给自己的知识库执行查看、修改、删除、启用、禁用等管理操作。
3. 当前业务归属人可操作自己归属知识库下的文档、入库任务、草稿、确认入库、问答、图谱和评测链路。
4. 租户管理员可查看和管理本租户下所有知识库及其文档、入库、问答、图谱、评测、API Key、日志和配置。
5. 租户管理员 / 知识库管理员判断优先使用 `role_code=superManager`；命中则具备本租户知识库管理员能力，未命中则按普通用户的知识库自有归属权限判断。
6. 平台管理员可在 AI 基座平台管理员角色映射范围内查看和管理全局知识库；平台管理员编码不从 `superManager` 推断，需独立确认。
7. 非当前业务归属人且非管理员，默认不可见、不可操作他人知识库。
8. 所有规则必须受 `tenant_id` 限制，跨租户访问直接拒绝。
9. 普通用户可删除自己创建的知识库，以及该知识库下的文档、入库任务、草稿、图谱和评测数据。
10. 配置中心按配置层级授权：知识库当前业务归属人可管理本人归属知识库的运行配置；租户管理员可管理本租户知识库配置和 API Key 配置；平台管理员可管理全局配置；普通用户不可管理 API Key、全局配置或租户主数据。
11. AI 基座账号被禁用、离职或移出租户后，该用户不得继续登录知识库后台，也不得继续访问其曾创建的知识库。
12. 失效用户历史创建的知识库默认保留，不自动删除；文档、入库任务、图谱、评测、日志和 token 统计继续归属原知识库与租户。
13. 租户管理员可对本租户内失效用户创建的知识库执行查看、删除、接管或转移当前业务归属人；平台管理员可按全局管理范围处理。
14. 接管或转移当前业务归属人后，普通用户可见性按当前业务归属人判断；`created_by` 保留原创建人用于审计，不因接管或归属转移被覆盖。
15. API Key 是租户级技术凭证，不因某个知识库原创建人失效而自动失效；是否可用由租户状态、API Key 状态、绑定知识库状态、能力范围和安全策略共同决定。
16. 后续如需要共享给指定用户、角色、班级、组织等，应作为后续迭代新增能力，不在当前边界内实现。

### 6.4 身份数据同步方式

第一期身份数据同步采用“受控增量同步（仅 AI 基座 SSO 的 `superManager` 可触发）+ 登录时兜底刷新”。

1. 受控增量同步负责刷新 AI 基座租户、用户、角色、用户角色关系只读快照，作为后台访问判断和 OpenAPI 租户状态校验的本地依据；只有 AI 基座 SSO session 且当前身份命中 `role_code=superManager` 时才允许触发，平台管理员、普通用户和旧请求头身份不能触发五类 delta 入库。
2. 同步成功后，日志管理必须新增 / 展示“用户及权限同步”日志，记录本轮租户、用户、角色、用户角色、删除事件数量、`last_sync_at`、`max_updated_at`、快照版本、状态和失败原因，便于复核是否真的产生租户侧同步数据。
3. 用户通过 AI 基座免登录 / SSO 进入知识库后台时，知识库系统必须基于身份摘要兜底刷新当前用户、租户、角色和用户角色关系快照，避免仅依赖同步任务导致登录态使用过旧数据。
4. 首次上线或历史数据迁移前，仍需先完成一次身份快照初始化；初始化失败时普通用户不得进入业务后台。
5. 受控同步失败时按旧快照降级策略处理：低风险读取可在有效窗口内使用旧快照，新增、修改、删除、导出、配置变更、API Key 管理、接管 / 转交等敏感操作拒绝。
6. webhook / 事件推送不是第一期硬依赖；后续如 AI 基座提供账号禁用、角色变更、租户停用等高敏事件，可作为加速刷新通道接入，但不能替代本地访问判断。

## 7. 总体业务原则

1. AI 基座租户端是租户、用户、角色和用户角色关系的身份事实源。
2. 知识库系统是业务能力系统，不是权限系统。
3. 知识库系统可保存 AI 基座同步的租户、用户、角色、用户角色关系只读快照，但只用于展示、审计、租户隔离和管理员身份判断。
4. 知识库系统不消费 AI 基座菜单权限、按钮权限点、会话记录、学生画像、教职工画像、分组或标签。
5. 后台页面、按钮和接口是否可访问，由知识库业务规则判断：同租户、当前业务归属人可管理本人归属知识库、`role_code=superManager` 的租户管理员可管理本租户全部知识库，平台管理员按独立映射管理全局知识库。
6. 知识库系统只记录 `tenant_id`、`actor_id`、`resource_id`、`api_key_id` 等业务关联和日志字段，不维护这些 ID 对应的可写用户、角色和权限定义。
7. 对外 API Key 只代表知识库系统颁发的技术访问凭证，不代表组织权限来源。
8. 对外 API Key 仅在知识库系统自身 OpenAPI 域内有效，不得用于 AI 基座或其他系统鉴权。
9. API Key 可控制调用能力、知识库范围、IP、统计阈值和过期时间；API Key 的创建、绑定、禁用和轮换仅允许租户管理员或平台管理员操作。
10. 配置中心管理业务配置，不管理权限配置；全局配置仅平台管理员可改，知识库配置可由当前业务归属人或管理员修改，API Key 配置仅管理员可改，租户主数据只读同步。
11. 日志中心记录访问判断结果和拒绝原因，但不存储完整权限规则。
12. AI 基座身份校验失败、身份快照不可用或账号 / 租户状态不明确时，默认拒绝敏感操作。
13. 所有跨租户访问必须被拒绝。
14. 用户状态变化只影响该用户后续登录和操作权限，不自动删除其历史知识库业务数据；历史资源处置由管理员按租户边界接管、转移当前业务归属人或删除。

## 8. 用户角色

下列角色仅作为业务使用者描述，不由知识库系统维护。实际身份和角色来自 AI 基座同步数据；知识库只按当前业务归属人、租户管理员、平台管理员三类业务身份执行访问判断。

| 使用者 | 描述 | 主要诉求 |
| --- | --- | --- |
| 平台超级管理员 | AI 基座角色快照中命中平台管理员角色映射的用户 | 全局可见、全局配置、故障排查 |
| 租户管理员 / 知识库管理员 | AI 基座角色快照中命中 `role_code=superManager` 的用户 | 管理本租户知识库、API Key、配置和日志 |
| 知识库当前业务归属人 | 当前负责某个知识库的 AI 基座用户，默认等于原创建人，可由管理员接管或归属转移后变化 | 管理本人归属的知识库、文档、入库、问答、图谱和评测 |
| 知识库使用者 | 可进入知识库后台且访问本人知识库的用户 | 安全访问、准确问答、可见证据 |
| 外部应用 | 通过知识库 API Key 调用能力的系统 | 稳定接口、API Key、调用统计、阈值告警 |
| 运维人员 | 在 AI 基座具备运维权限的用户 | 日志、告警、错误定位、审计 |
| 审计人员 | 在 AI 基座具备审计权限的用户 | 可追溯、可导出、不可篡改记录 |

## 9. 功能域划分

知识库系统的功能按功能域划分，现有控制台的所有入口都应能映射到下列分类。某些页面会同时覆盖多个功能域，但统计和责任归属以主域为准。

### 9.1 功能域总览

| 功能域 | 业务能力 | 典型控制台入口 | 优先级 |
| --- | --- | --- | --- |
| 身份与访问域 | AI 基座登录、SSO、只读身份快照、当前业务归属人 / 管理员访问判断 | AI 基座登录、SSO callback、后台页面、后台 API | P0 |
| 知识库业务域 | 知识库创建、编辑、启停、删除、总览 | 知识库列表、单库总览 | P0 |
| 文档与入库域 | 文档上传、解析、清洗、切片、向量化、草稿、确认入库 | 全局入库看板、离线入库页、单库文档管理、单库入库任务 | P0 |
| 在线问答域 | RAG 问答、Graph RAG 问答、引用回溯、答案调试 | 全局问答调试、在线 RAG 页、单库问答调试 | P0 |
| 图谱与评测域 | 图谱预览、关系预览、评测样本、质量分析 | 单库知识图谱、全局评测记录、单库评测记录 | P1 |

## 10. AI 基座身份接入与访问判断需求

### 10.1 身份与访问判断入口

知识库系统需要对以下入口完成 AI 基座身份校验、租户隔离和知识库业务访问判断：

| 入口 | 判断要求 |
| --- | --- |
| 管理后台页面 | 校验 AI 基座登录态、租户状态和账号状态；支持 AI 基座免登录跳转 |
| 后台 API | 校验知识库短期 session、租户隔离、当前业务归属人 / 管理员规则 |
| 文件上传 | 仅知识库当前业务归属人、本租户管理员或平台管理员可操作 |
| 草稿确认 | 仅知识库当前业务归属人、本租户管理员或平台管理员可操作 |
| 文档查看 / 下载 | 仅知识库当前业务归属人、本租户管理员或平台管理员可操作 |
| RAG 问答 | 仅知识库当前业务归属人、本租户管理员或平台管理员可操作 |
| Graph RAG 问答 | 仅知识库当前业务归属人、本租户管理员或平台管理员可操作 |
| 配置中心 | 全局配置仅平台管理员可操作；租户主数据只读同步；知识库配置由当前业务归属人、本租户管理员或平台管理员操作；API Key 配置仅本租户管理员或平台管理员操作 |
| 日志中心 | 当前业务归属人可看本人归属知识库日志；租户管理员可看本租户日志；平台管理员可看全局日志 |
| API Key 管理 | 仅本租户管理员或平台管理员可创建、禁用、删除和轮换 |

### 10.2 AI 基座免登录 / SSO

知识库后台不提供独立账号密码登录能力。用户登录、密码校验、账号状态和会话真实性均由 AI 基座租户端完成。知识库只接收 AI 基座确认后的身份摘要，并在知识库系统内建立短期后台 session。

推荐免登录流程：

```text
用户已登录 AI 基座租户端
-> 用户点击 AI 基座中的知识库入口
-> AI 基座生成一次性 SSO code 和 state
-> 浏览器跳转到知识库 /sso/callback?code=xxx&state=yyy
-> 知识库后端校验 state
-> 知识库后端使用 code 调用 AI 基座换取身份摘要
-> AI 基座返回 tenant_id、actor_id、display_name、tenant_status、expires_in
-> 知识库创建自身后台 session
-> 前端进入知识库后台
-> 后续后台接口按 tenant_id、owner_user_id 和管理员角色快照做业务访问判断
```

知识库 session 只允许保存：

```text
session_id
tenant_id
actor_id
display_name 可选
login_time
expire_time
last_active_time
```

约束要求：

1. 知识库不得同步、保存或校验用户密码、密码 hash、盐值和登录凭证。
2. URL 中只允许携带一次性 SSO code，不允许携带用户密码、AI 基座 token、角色列表或完整权限数据。
3. SSO code 必须短有效期，建议 30-60 秒，且只能使用一次。
4. `state` / `nonce` 必须校验，防止 CSRF、伪造跳转和重放。
5. 知识库后端换取身份时，必须使用服务端到服务端调用，不得由前端直接换取。
6. 知识库 session cookie 必须设置 `HttpOnly`、`Secure`、`SameSite`。
7. 知识库 session 仅代表用户已通过 AI 基座身份认证，不代表用户拥有任何知识库操作权限。
8. 每次后台敏感操作必须在服务端重新执行租户隔离和当前业务归属人 / 管理员规则判断，不得只依赖前端页面隐藏。
9. AI 基座退出登录、账号禁用或租户停用时，知识库应通过短 session、身份快照同步、登录态校验或事件通知失效 session。
10. 第一期身份数据同步方式采用“受控增量同步（仅 AI 基座 SSO 的 `superManager` 可触发）+ 登录时兜底刷新”；webhook / 事件通知仅作为后续可选加速通道，不作为第一期唯一依赖。

AI 基座返回身份摘要建议：

```json
{
  "tenantId": "10001",
  "actorId": "20001",
  "username": "zhangsan",
  "displayName": "张三",
  "tenantStatus": "active",
  "expiresInSeconds": 7200
}
```

### 10.3 业务访问上下文

每次执行后台访问判断时，知识库系统至少使用：

```text
tenant_id
actor_id
resource_type
resource_id
action
kb_id
document_id 可选
api_key_id 可选
resource_created_by
resource_owner_user_id
actor_role_codes
is_tenant_admin
is_platform_admin
client_ip
user_agent
request_id
```

说明：

- `actor_id` 只来自 AI 基座登录态、SSO 换取后的知识库 session 或可信调用上下文。
- 知识库系统可根据 `actor_id` 查询只读用户快照和只读用户角色快照，用于展示、审计和管理员身份判断。
- `resource_id` 是知识库系统自己的业务资源 ID。
- `resource_created_by` 来自知识库业务表，例如 `knowledge_bases.created_by`、`documents.uploaded_by`。
- `resource_owner_user_id` 表示知识库当前业务归属人，默认等于 `created_by`；管理员接管或转交后可变化，普通用户访问判断优先按该字段执行。
- `actor_role_codes` 来自 AI 基座角色快照；知识库不在本地分配角色，也不从菜单权限或按钮权限点推断管理员身份。
- `is_tenant_admin` 第一期按 `actor_role_codes` 是否包含 `superManager` 判断；命中 `superManager` 即视为本租户租户管理员 / 知识库管理员。
- 未命中 `superManager` 的用户，默认按普通用户处理，只能访问和操作本人作为当前业务归属人的知识库及其子资源。
- `is_platform_admin` 只来自 AI 基座独立的平台管理员角色映射；平台管理员身份不从 `superManager` 自动推断。
- 所有访问判断必须先校验 `tenant_id`，跨租户直接拒绝。

### 10.4 访问判断结果

知识库后台访问判断建议输出统一结果：

```json
{
  "allow": true,
  "tenantId": "10001",
  "actorId": "20001",
  "decisionId": "kb_authz_xxx",
  "reasonCode": "ALLOW_BY_OWNER",
  "message": "允许访问",
  "expiresInSeconds": 60
}
```

拒绝时返回：

```json
{
  "allow": false,
  "tenantId": "10001",
  "actorId": "20001",
  "decisionId": "kb_authz_xxx",
  "reasonCode": "DENY_NOT_OWNER_OR_ADMIN",
  "message": "无权访问该知识库"
}
```

知识库系统只保存访问判断结果日志，不保存 AI 基座菜单、会话、学生、教职工、分组、标签或复杂权限规则。

### 10.5 缓存规则

1. 允许对访问判断结果做短 TTL 缓存，默认不超过 60 秒。
2. 缓存 key 必须包含 `tenant_id`、`actor_id`、`resource_type`、`resource_id`、`action`。
3. 涉及删除、配置修改、API Key 管理、日志导出等敏感操作不建议缓存。
4. 身份摘要、账号状态、租户状态或角色快照有明确有效期时，以较小值为准。
5. 缓存只保存 allow / deny 结果和 reasonCode，不保存用户、角色、权限规则。
6. 身份快照同步失败或延迟时，低风险读取类操作可临时使用旧快照，默认允许旧快照最大年龄为 10 分钟，配置范围建议限制在 5-15 分钟。
7. 旧快照降级只允许用于知识库列表、详情、普通业务日志查看等低风险读取；新增、修改、删除、导出、配置变更、API Key 管理、接管 / 转交等敏感操作必须拒绝。
8. 使用旧快照完成访问判断时，必须在访问判断日志中标记 `identity_snapshot_stale=true`、记录快照版本、快照时间和降级原因，并触发身份同步可用性告警。
9. 普通用户知识库共享不在第一期边界内；第一期仅支持“个人创建、个人可见”和“管理员全量可见”，不提供按用户、角色、组织或外部应用的显式共享授权。

### 10.6 异常规则

| 场景 | 处理 |
| --- | --- |
| AI 基座身份校验成功且业务规则 allow | 继续执行业务操作 |
| AI 基座身份校验失败 | 拒绝操作，记录拒绝日志 |
| 租户、账号或角色快照不可用 | 默认拒绝敏感操作，记录告警 |
| 身份快照同步失败或延迟但存在未超过旧快照窗口的快照 | 仅允许低风险读取类操作；敏感操作拒绝；记录降级日志和告警 |
| 身份快照超过旧快照窗口 | 后台访问和敏感操作拒绝，提示重新登录或联系管理员 |
| 创建人账号禁用、离职或移出租户 | 该用户后续登录和操作拒绝；历史知识库保留，由租户管理员接管、转移当前业务归属人或删除；`created_by` 不覆盖 |
| 业务规则 deny | 拒绝操作，记录拒绝日志 |
| AI 基座登录态校验超时或 5xx | 默认拒绝敏感操作，记录告警 |
| 本地缓存命中 allow | 在 TTL 内允许低风险读取类操作 |
| 本地缓存命中 deny | 在 TTL 内拒绝 |

## 11. 知识库业务资源需求

### 11.1 知识库管理

知识库系统负责维护知识库业务资源：

- 知识库 ID。
- 租户 ID。
- 名称。
- 描述。
- 默认切片策略。
- 状态。
- 创建时间。
- 更新时间。
- 创建人 ID。
- 更新人 ID。
- 当前业务归属人 ID。
- 接管 / 转交时间。
- 接管 / 转交操作者 ID。

注意：

- 创建人、更新人只作为审计字段，不代表可写用户表。
- 当前业务归属人用于普通用户可见性和可操作性判断，默认等于创建人；管理员接管或转交后可变更。
- 创建人字段不得因接管或转交被覆盖，必须保留原创建人用于审计。
- 创建知识库：通过 AI 基座身份校验且账号、租户状态正常的用户可以创建。
- 查看、修改、删除、启用、禁用知识库：仅同租户当前业务归属人、本租户管理员或平台管理员可以操作。
- 知识库是否对某人可见，由 `tenant_id`、当前业务归属人和管理员角色映射共同决定；历史创建人仅用于审计展示。
- AI 基座账号被禁用、离职或移出租户后，其历史知识库不自动删除；本租户管理员可接管、转交或删除。

### 11.2 文档管理

知识库系统负责维护文档业务资源：

- 文档 ID。
- 租户 ID。
- 知识库 ID。
- 文件名。
- 文件 hash。
- 源文件路径或 URL。
- 解析 provider。
- 切片数量。
- 创建时间。
- 更新时间。
- 上传人 ID。

文档查看、下载、删除必须先校验文档所属知识库，再按同租户当前业务归属人 / 管理员规则判断。

### 11.3 入库任务

知识库系统负责维护入库任务：

- 任务 ID。
- 租户 ID。
- 知识库 ID。
- 文件名。
- 阶段状态。
- 阶段进度。
- 阶段日志。
- 错误信息。
- 创建人 ID。
- 创建时间。

入库任务操作包括上传、重试、删除、草稿查看、草稿编辑、草稿确认。每个操作都必须校验任务所属知识库，并按同租户当前业务归属人 / 管理员规则判断。

### 11.4 删除与恢复策略

删除策略按知识库归属执行。文档、入库任务、草稿切片、向量、图谱、评测等子资源的删除权限，不按各自上传人、任务创建人或操作人单独拆分，而是统一回到所属知识库的 `owner_user_id`、租户管理员和平台管理员判断。

删除原则：

1. 知识库、文档、入库任务默认采用软删除，记录 `deleted_at`、`deleted_by`、`delete_reason` 和操作审计日志。
2. 当前业务归属人可删除本人归属知识库及其子资源；租户管理员可删除本租户全部知识库及其子资源；平台管理员可按全局范围删除。
3. 删除知识库时，该知识库下的文档、入库任务、草稿、切片、向量、图谱、评测结果应立即对后台列表、OpenAPI、RAG、Graph RAG 和评测查询不可见。
4. 删除文档时，该文档对应的切片、向量、图谱关系、评测引用和问答证据应立即不可被检索、召回或引用。
5. 删除入库任务时，应删除或隐藏该任务的草稿、临时文件和任务态数据；已确认入库的数据仍按所属知识库 / 文档删除规则处理。
6. 日志、操作审计、API 调用记录、访问判断记录、RAG 查询摘要、token 消耗统计不随业务资源硬删，默认保留脱敏记录用于审计、追溯和成本统计。
7. API Key 优先提供禁用、过期和轮换；删除 API Key 时不得删除历史调用日志和审计记录，只保留 key 指纹、前缀 / 后四位、删除人、删除时间和删除原因。
8. 恢复仅面向软删除保留期内的知识库、文档和入库任务；恢复操作只允许当前业务归属人、租户管理员或平台管理员执行，并必须写审计日志。
9. 物理删除只作为管理员高危操作或保留期到期后的后台清理任务，执行前必须确认租户、知识库归属、资源范围和审计留痕。
10. 物理删除不得破坏审计闭环；如需满足合规删除，应保留不可逆脱敏后的最小审计记录。

## 12. 对外 API 需求

### 12.1 API 入口

后台 API 继续使用：

```text
/api/*
```

对外 API 使用独立前缀：

```text
/openapi/v1/*
```

### 12.2 API Key 定位

API Key 是知识库系统颁发给外部应用的技术访问凭证，仅用于消费知识库系统自身 OpenAPI。API Key 可控制：

- 可调用接口范围。
- 可访问知识库范围。
- 过期时间。
- IP 白名单。
- 日调用量统计阈值。
- 总调用量统计阈值。
- 并发或 QPS 统计阈值。

API Key 是租户级技术凭证，创建、绑定、禁用、轮换等管理操作仅允许本租户管理员或平台管理员执行。API Key 不代表用户、角色、组织权限、AI 基座身份或其他系统身份。

API Key 禁止用于：

- 登录 AI 基座。
- 调用 AI 基座接口。
- 作为 AI 基座服务账号。
- 代表 AI 基座用户、角色、学生、教职工或其他组织画像身份。
- 调用知识库系统以外的其他业务系统。
- 作为跨系统单点登录或通用开放平台凭证。

### 12.3 API Key 管理

API Key 需支持：

1. 创建。
2. 命名。
3. 绑定租户。
4. 绑定一个或多个知识库。
5. 绑定接口能力范围。
6. 设置过期时间。
7. 设置 IP 白名单。
8. 设置每日调用量统计阈值。
9. 设置总调用量统计阈值。
10. 设置 QPS 或并发统计阈值。
11. 禁用。
12. 删除。
13. 轮换。
14. 查看最近使用时间。
15. 查看调用统计。

API Key 绑定和调用规则：

1. API Key 归属当前租户。
2. API Key 可绑定一个或多个知识库，单个 API Key 默认最多绑定 20 个知识库；超过 20 个知识库时建议拆分多个 API Key，避免外部系统滥配和后续治理困难。
3. 每个知识库创建后，应在后台显式展示知识库 ID，方便外部系统配置调用参数。
4. 外部系统调用问答、文档、图谱等知识库资源接口时，默认必须指定 `kb_id`。
5. `kb_id` 必须属于 API Key 绑定范围，并且必须属于同一租户。
6. 第一版不支持“未传 `kb_id` 时默认查询全部知识库”。
7. 未传 `kb_id` 时返回 `KB_ID_REQUIRED`。
8. 跨知识库查询不作为默认能力；如后续支持，必须使用独立高级接口显式传 `kb_ids`，并限制知识库数量、候选数、rerank 数、token 上限和是否允许跨库。
9. 文档上传、入库和其它写操作不是默认查询能力的一部分；如需通过 OpenAPI 执行，必须显式授予 `kb.document.upload`、`kb.task.write` 等能力，并强制执行签名、IP 白名单、nonce 防重放和审计。

API Key 安全要求：

- 创建时只展示一次明文。
- 数据库只保存 hash。
- 前端只显示 key 前缀和后四位。
- 删除和禁用必须写审计日志。
- 轮换后旧 key 可设置宽限期。
- API Key 不允许放在 URL query、前端页面、移动端包体或公开配置文件中。
- API Key 推荐仅保存在调用方服务端密钥管理系统中。
- 生产环境对外调用推荐使用签名请求，不推荐只使用长期 Bearer Token。
- IP 白名单第一期必须支持，但可按 API Key 配置为空；生产对外调用建议配置 IP 白名单。
- Bearer Token 模式仅建议用于内网、低风险系统间联调或过渡场景。
- 生产对外调用默认推荐启用签名模式；高风险接口必须强制签名。
- 生产环境默认采用强签名 + IP 白名单 + nonce 防重放；其他系统不得把真实 API Key 放在前端、客户端、URL 参数、日志或普通业务请求参数中。
- 如外部系统确需在业务参数中携带“凭证标识”或“鉴权引用”，只能使用 `api_key_id`、`app_id`、别名、配置引用或一次性代理 token，不得传输真实 API Key 明文。

### 12.4 系统级强签名与防注入

为降低 API Key 暴露、重放、伪造和恶意请求注入风险，知识库 OpenAPI 应支持两种认证强度：

| 模式 | 适用场景 | 要求 |
| --- | --- | --- |
| 基础模式 | 内网、低风险系统间调用 | `Authorization: Bearer <api_key>`、IP 白名单、用量统计、阈值告警、审计日志 |
| 签名模式 | 生产对外调用、跨系统集成 | `app_id + api_key_id + timestamp + nonce + body_hash + signature` |

签名模式请求头：

```text
X-KB-App-Id: app_xxx
X-KB-Key-Id: key_xxx
X-KB-Timestamp: 2026-06-17T10:00:00Z
X-KB-Nonce: random-128-bit
X-KB-Body-SHA256: hex_sha256(request_body)
X-KB-Signature: HMAC-SHA256(method + path + timestamp + nonce + body_hash)
X-KB-Signature-Version: v1
```

签名串规范：

```text
METHOD + "\n" +
PATH + "\n" +
QUERY_STRING + "\n" +
TIMESTAMP + "\n" +
NONCE + "\n" +
BODY_SHA256
```

签名模式校验规则：

1. `timestamp` 与服务端时间偏差默认不得超过 5 分钟。
2. `nonce` 在有效窗口内不得重复使用。
3. `body_hash` 必须与请求体一致。
4. `signature` 必须使用服务端保存的密钥 hash 派生校验，原始密钥不得明文存储。
5. `signature_version` 必须是服务端支持的版本，第一期建议固定为 `v1`。
6. 签名失败、时间过期、nonce 重放、body 被篡改时必须拒绝。
7. 同一 API Key 可配置是否强制签名模式。
8. 高风险接口，如文档上传、批量查询、日志导出、配置修改，默认要求签名模式。

防泄露要求：

1. API Key 明文只在创建时返回一次。
2. API Key 泄露后可立即禁用。
3. API Key 支持轮换，新旧 key 宽限期默认不超过 24 小时。
4. API Key 支持按知识库、接口能力、IP、QPS、每日调用量统计爆炸半径；第一期 QPS、调用量、并发和 token 预算只做统计、报表和告警，不作为强制拦截条件。
5. API Key 异常调用需触发告警，例如异常 IP、异常时间段、调用量突增、连续签名失败。
6. API Key 不允许被第三方页面、浏览器脚本、客户端 App 直接持有。

防注入要求：

1. 所有 OpenAPI 请求必须做 schema 校验，拒绝未知字段和超长字段。
2. 问答 query、metadata、filter 等字段必须限制长度、类型和字符集。
3. 文档上传接口必须限制文件类型、大小、页数和 MIME。
4. 外部请求中的任何自然语言内容都只能作为用户输入，不得覆盖系统提示词、开发者提示词、检索策略或权限上下文。
5. RAG 生成阶段必须保持系统提示词和用户 query 隔离，外部系统不得通过 query 修改系统规则。
6. 对疑似 prompt injection、越权指令、批量探测、数据外传诱导的请求，应记录风险标签并可配置拒绝或降级。
7. 返回结果不得包含敏感配置、API Key、内部路径、AI 基座 token 或完整鉴权上下文。

### 12.5 API Key 与 AI 基座约束

1. 创建 API Key 前，知识库系统必须校验当前操作者是否为本租户管理员或平台管理员。
2. 绑定知识库前，必须校验 API Key、目标知识库和操作者处于同一租户，且操作者为本租户管理员或平台管理员。
3. 外部应用调用 API 时，知识库系统先校验 API Key，再校验请求中的 `kb_id` 是否存在且是否绑定到该 key。
4. API Key 调用不得绕过 `tenant_id` 边界。
5. API Key 不得被映射为 AI 基座应用身份或服务账号。
6. API Key 不得被转发给其他系统作为鉴权凭证。
7. API Key 绑定多个知识库只代表授权范围，不代表单次请求跨库召回。
8. OpenAPI 调用不要求每次实时请求 AI 基座校验租户状态，默认以知识库本地 AI 基座租户只读快照为准。
9. 租户快照状态为 `active` 且快照更新时间在 5 分钟内时，OpenAPI 可继续执行 API Key、`kb_id`、能力范围、IP、签名等安全校验，并记录用量统计和超阈状态。
10. 租户快照更新时间超过 5 分钟但不超过 10 分钟时，可降级允许 OpenAPI 调用，但必须记录 `tenant_snapshot_stale=true`、快照时间、快照年龄和降级原因，并触发身份同步 / 租户状态告警。
11. 租户快照更新时间超过 10 分钟时，OpenAPI 调用必须拒绝，返回 `TENANT_STATUS_STALE`。
12. 租户快照状态为停用、冻结、删除、不明确或不可用时，OpenAPI 调用必须拒绝，返回 `TENANT_DISABLED` 或 `TENANT_UNAVAILABLE`。
13. API Key 自身禁用、过期、未绑定目标 `kb_id`、能力不足、IP 不允许或签名失败时，应优先按 API Key / 请求自身规则拒绝，不因租户快照仍在新鲜窗口内放行；第一期额度、QPS、并发和 token 预算超阈只记录和告警，不拒绝调用。

## 13. 前端管理后台需求

### 13.1 导航结构

管理后台应包含：

1. 总览
2. 知识库
3. 入库任务
4. 在线问答
5. 评测
6. API Key
7. 配置中心
8. 日志中心
9. 身份与访问监控
10. Token 消耗统计

不提供知识库本地用户管理、角色管理和权限策略管理页面。

### 13.2 页面与操作可见性

1. 前端不消费 AI 基座菜单权限或按钮权限点。
2. 前端导航按知识库自身功能域展示；具体列表数据由后端按租户、当前业务归属人和管理员规则过滤。
3. 操作按钮可按后端返回的 `canEdit`、`canDelete`、`canUpload`、`canManageApiKey` 等业务能力标记展示。
4. 前端隐藏按钮只作为体验优化，后端接口必须再次执行租户隔离和当前业务归属人 / 管理员规则判断。
5. 身份快照不可用、账号停用、租户停用或业务访问判断失败时，默认显示无权限或不可用状态。
6. 知识库列表和详情可展示原创建人、当前业务归属人和账号状态；普通用户只看到本人当前归属的知识库，管理员可看到本租户内全部知识库。

### 13.3 知识库管理

功能要求：

- 创建知识库。
- 编辑知识库名称、描述、默认切片策略。
- 查看知识库状态。
- 查看文档数量、切片数量、最近入库时间。
- 删除知识库。
- 恢复软删除保留期内的知识库。
- 禁用 / 启用知识库。
- 查看原创建人、当前业务归属人和创建人账号状态。
- 管理员接管或转交失效用户创建的知识库。

验收标准：

- 非当前业务归属人且非管理员不可见或不可操作他人知识库。
- 创建人账号禁用、离职或移出租户后，该账号不能继续访问知识库；管理员仍可查看、接管、转交或删除其历史知识库。
- 删除知识库前必须二次确认。
- 删除操作必须写入审计日志。
- 删除知识库默认软删除，并立即使该知识库下文档、入库任务、切片、向量、图谱、评测在后台、OpenAPI、RAG 和 Graph RAG 中不可见。
- 软删除保留期内的恢复操作必须再次校验当前业务归属人 / 管理员规则，并写入审计日志。
- 知识库禁用后不可通过后台或 API 查询。

- 删除任务。
- 恢复软删除保留期内的文档或入库任务。
- 下载原文。
- 导出 CSV。

每个操作对应的允许 / 拒绝都由后端按同租户、知识库当前业务归属人和管理员规则决定。
删除文档、入库任务、草稿、切片、向量、图谱和评测数据时，删除权限统一按所属知识库的 `owner_user_id`、租户管理员和平台管理员判断，不按文档上传人或任务创建人单独判断。

1. 第一期不建设审批流。
2. 普通用户不得导出全量日志；如提供导出，只允许导出本人归属知识库的业务日志摘要。
3. 租户管理员可导出本租户范围日志，平台管理员可导出全局日志。
4. 导出必须限制时间范围和最大条数，默认单次时间范围不超过 31 天，默认单次导出不超过 10 万条。
5. 导出内容必须按日志脱敏规则处理，不得包含完整 prompt、API Key 明文、模型密钥、AI 基座 token、用户密码或完整身份 / 访问判断上下文。
6. 每次导出必须写审计日志，记录操作者、租户、知识库范围、时间范围、日志类型、导出条数、requestId 和导出结果。

日志可见性：

| 日志类型 | 租户管理员 / 平台管理员 | 普通用户 |
| --- | --- | --- |
| 入库任务日志 | 本租户全部 / 全局 | 仅本人归属知识库 |
| 文档操作日志 | 本租户全部 / 全局 | 仅本人归属知识库 |
| RAG 查询日志 | 本租户全部 / 全局 | 仅本人归属知识库 |
| Graph RAG 查询日志 | 本租户全部 / 全局 | 仅本人归属知识库 |
| 评测日志 | 本租户全部 / 全局 | 仅本人归属知识库 |
| Token 消耗统计 | 本租户全部 / 全局 | 仅本人归属知识库 |
| API 调用日志 | 本租户全部 / 全局 | 不可见，或仅展示本人归属知识库统计摘要 |
| API Key 审计日志 | 本租户全部 / 全局 | 不可见 |
| 配置变更日志 | 本租户全部 / 全局 | 不可见 |
| 用户及权限同步日志 | 本租户全部 / 全局 | 不可见 |
| 身份与访问判断日志 | 本租户全部 / 全局 | 不可见 |

### 13.8 身份与访问监控页面

页面应支持：

- 查看 AI 基座 SSO / 身份校验可用性。
- 查看身份快照同步状态和最近同步时间。
- 查看当前旧快照降级窗口配置、旧快照命中次数和最大快照年龄。
- 查看访问判断平均耗时和 p95 耗时。
- 查看 allow / deny 数量。
- 查看 deny reasonCode 分布。
- 查看身份校验超时次数。
- 查看访问判断失败告警。
- 按租户、知识库、接口动作筛选。

### 13.9 Token 消耗统计页面

页面应支持按链路管道归属统计大模型 token 消耗，不允许只提供一个无法拆账的全局总数。

页面应支持：

- 按租户、知识库、文档、入库任务、问答请求、API Key、操作者 ID 筛选。
- 按链路管道筛选，例如入库链路、向量化链路、在线问答链路、Graph RAG 链路、评测链路。
- 按链路阶段筛选，例如解析增强、清洗、切片增强、embedding、rerank、答案生成、评测打分。
- 按 provider、模型、调用状态、时间范围筛选。
- 查看 input tokens、output tokens、total tokens、cached tokens、reasoning tokens。
- 查看调用次数、成功次数、失败次数、平均耗时、p95 耗时。
- 查看按日、按小时、按知识库、按链路阶段聚合的趋势。
- 从统计记录下钻到对应 taskId、requestId 或 RAG 查询日志。
- 导出统计明细和聚合报表。

统计页面只展示 token 和调用指标，不展示完整 prompt、用户密码、AI 基座 token、API Key 明文或完整身份 / 访问判断上下文。

## 14. 配置中心需求

### 14.1 配置层级

配置按以下层级生效：

```text
API Key 配置 > 知识库配置 > 全局配置 > 环境变量默认值
```

说明：

- 这里的配置是业务运行配置，不是权限配置。
- 租户主数据只读同步，不在知识库配置中心修改租户名称、状态、归属、角色或用户关系。
- API Key 配置只记录 key 的技术限制，不记录组织权限。

### 14.2 配置类型

| 类型 | 示例 |
| --- | --- |
| 字符串 | 模型名称 |
| 数字 | 并发数、batch size |
| 布尔 | 是否开启 Graph RAG |
| JSON | 权重配置、复杂 provider 配置 |
| 密钥 | 模型 API Key、AK/SK、Token |

### 14.3 敏感配置规则

1. 密钥类配置不允许明文读取。
2. 前端只展示是否已配置和脱敏值。
3. 保存空值不应误清除原值，除非用户明确点击清除。
4. 每次变更记录操作人 ID、时间、旧值摘要、新值摘要。
5. 密钥字段审计日志不得保存明文。

### 14.4 配置生效规则

第一期配置中心只承诺热生效和灰度生效，不承诺所有配置都能后台热改。底层基础设施、敏感密钥和全局运行时容量类配置可以在配置中心展示、脱敏、审计，但实际变更走管理员运维流程。

配置按生效方式分为三类：

| 生效方式 | 配置范围 | 生效要求 |
| --- | --- | --- |
| 热生效 | Prompt Profile；API Key 策略，包括过期时间、能力范围、绑定知识库、签名策略、IP 白名单、统计阈值；知识库问答参数，包括 `top_k`、`rerank_top_n`、`temperature`、`max_tokens`、引用策略；日志保留天数、脱敏策略、告警阈值 | 保存后立即进入配置解析链路，无需重启；必须记录配置版本、操作者和审计日志 |
| 灰度生效 | 模型选择；rerank provider；embedding provider / embedding 模型；切片策略、清洗策略、质量过滤策略；Graph RAG 扩展深度、关系类型白名单；系统提示词大版本 | 按知识库或 API Key 范围逐级生效；必须支持版本化、预览、回滚和效果追踪 |
| 重启或运维变更 | 数据库连接；pgvector / Redis / OSS 配置；模型密钥、AK/SK 等敏感密钥；解析 provider 大切换；文件存储路径；队列 worker 数；后台任务并发全局上限 | 配置中心可展示脱敏值、记录变更意图和审计，但不承诺后台直接热改；实际变更需走管理员运维流程，必要时重启或重新部署 |

通用规则：

1. 知识库任务创建时应记录当时生效配置快照。
2. RAG 查询日志应记录关键配置版本。
3. 灰度配置必须能按知识库、API Key 或请求维度定位生效来源。
4. 对质量、成本或召回结果影响较大的配置，默认进入灰度生效类别，不直接全局热切。
5. 修改配置前必须执行服务端访问判断；全局配置仅平台管理员可修改，知识库配置可由当前业务归属人、本租户管理员或平台管理员修改，API Key 配置仅本租户管理员或平台管理员可修改；租户主数据不在知识库侧修改，只读同步。

### 14.5 Prompt Profile 与 API 配置覆盖

系统提示词、回答风格、引用格式等生成策略可以通过配置中心管理，也可以通过受控 API 配置方式生效覆盖，但不能允许普通问答接口直接传入原始 `system_prompt`。

推荐使用 Prompt Profile：

1. 当前业务归属人、本租户管理员或平台管理员可在配置中心创建归属当前知识库的 Prompt Profile；本租户管理员或平台管理员可创建归属 API Key 范围的 Prompt Profile。
2. Prompt Profile 归属知识库或 API Key 可用范围，不承接租户主数据编辑能力。
3. 外部问答调用时只允许传 `prompt_profile_id`，不允许传 raw `system_prompt`。
4. 知识库系统校验 `prompt_profile_id` 绑定当前 `kb_id` 或当前 API Key。
5. Prompt Profile 必须版本化，支持启用、禁用、回滚和审计。

普通问答接口允许覆盖的参数应采用白名单：

| 参数 | 是否允许 API 覆盖 | 规则 |
| --- | --- | --- |
| `top_k` | 允许 | 限定最小值、最大值 |
| `rerank_top_n` | 允许 | 不得超过系统配置上限 |
| `temperature` | 允许 | 限定范围 |
| `max_tokens` | 允许 | 不得超过知识库或 API Key 配置的单次请求安全上限；第一期 token 总预算超阈只统计和告警，不拒绝业务调用 |
| `answer_format` | 允许 | 枚举值 |
| `citation_mode` | 允许 | 枚举值 |
| `prompt_profile_id` | 允许 | 必须属于当前知识库或 API Key 范围 |
| `system_prompt` | 不允许 | 不得在普通问答接口直接传入 |
| 模型 API Key、AK/SK、Token | 不允许 | 只能由管理员在配置中心维护 |
| IP 白名单、签名策略、安全策略 | 不允许 | 只能由管理员维护 |

如需通过 API 修改系统提示词，应使用独立配置管理接口，例如：

```text
PUT /openapi/v1/knowledge-bases/{kb_id}/config/prompt-profiles/{profile_id}
```

配置管理 API 要求：

1. 仅租户管理员或平台管理员创建的管理员级 API Key 可调用。
2. API Key 必须具备 `kb.config.write` 能力。
3. 必须强制签名模式。
4. 必须启用 IP 白名单。
5. 每次变更必须写配置审计日志。
6. 必须保留版本、变更摘要、操作者、requestId 和回滚能力。
7. 不允许通过普通问答接口顺带修改配置。

## 15. 日志与审计需求

### 15.1 日志对象

| 日志 | 说明 | 保留建议 |
| --- | --- | --- |
| 入库任务日志 | upload、parse、clean、chunk、quality、embedding、export 阶段日志 | 180 天 |
| API 调用日志 | 对外 API 请求、响应状态、耗时、错误码 | 180 天 |
| RAG 查询日志 | 问题、答案摘要、引用、耗时、评分 | 180 天 |
| 大模型调用日志 | provider、模型、链路管道、阶段、token 消耗、耗时、状态 | 180 天 |
| 用户及权限同步日志 | sync_mode、last_sync_at、max_updated_at、租户 / 用户 / 角色 / 用户角色 / 删除事件数量、状态和失败原因 | 180 天 |
| 身份与访问判断日志 | allow / deny、reasonCode、decisionId、耗时 | 180 天 |
| 配置审计日志 | 配置新增、修改、删除、回滚 | 365 天 |
| API Key 审计日志 | API Key 创建、禁用、删除、轮换 | 365 天 |
| 错误日志 | 系统错误、provider 错误、统计超阈告警和安全拒绝错误 | 180 天 |
| 小时 / 日聚合统计 | API 调用量、超阈次数、token 消耗聚合、成本趋势 | 2 年 |

保留周期原则：

1. 明细类日志短保留，默认 180 天，包括入库任务、API 调用、RAG / Graph RAG 查询、大模型调用明细、身份与访问判断、错误日志。
2. 审计类日志中保留，默认 365 天，包括操作审计、API Key 创建 / 禁用 / 删除 / 轮换、配置变更、日志导出。
3. 聚合类统计长保留，默认 2 年，包括小时 / 日维度 token 聚合、调用量聚合、超阈次数和成本趋势。
4. 删除知识库、文档、入库任务或 API Key 时，不得删除保留期内的日志、审计和聚合统计；只允许保留脱敏后的资源标识、key 指纹、租户、知识库、时间和统计字段。
5. 到达保留期后的清理必须由后台任务执行，清理任务自身必须写审计日志。

敏感信息保留原则：

1. 日志保留证据链，不保留可复现敏感内容；默认只记录元数据、摘要、hash、fingerprint、引用 ID、风险标签和统计字段。
2. API Key、模型 Key、AK/SK、SSO code、session token、签名 secret、AI 基座 token、用户密码、密码 hash 和盐值永不明文入库、入日志或进入导出文件。
3. OpenAPI 请求日志默认记录 `request_id`、`tenant_id`、`kb_id`、`api_key_id` 或 key fingerprint、接口名、状态码、耗时、错误码、IP、user agent、签名校验结果和 token 用量，不默认保存完整请求体、完整响应体或真实 API Key。
4. RAG / Graph RAG 查询日志默认保存脱敏后的 query / answer 摘要、query hash、长度、引用 chunk id、命中文档 id、风险标签、耗时和 token 用量，不默认长期保存完整用户问题、完整模型回答、完整 prompt 或完整上下文。
5. 入库任务日志只记录文件名、文件 hash、页数、大小、任务阶段、错误码、错误摘要、文档 id、chunk id 和耗时；解析出的文档正文、表格全文、图片 OCR 文本不得写入运行日志。
6. Prompt Profile、系统提示词和模型参数变更只记录 profile id、版本号、配置 hash、变更摘要、变更人、时间和审计结果；不得在普通日志中记录完整 system prompt 或供应商密钥。
7. 如确需排障保留原始 query、answer、prompt 片段或 provider 原始响应，必须由管理员显式开启调试留痕模式，限定租户 / 知识库 / requestId 范围，默认保留 7-14 天，到期自动清理，并记录开启、查看、导出和关闭审计日志。
8. nonce 防重放记录只保留签名校验所需窗口，建议 10-30 分钟；如需安全排查最长不超过 24 小时，且只保存 nonce hash、时间窗、api_key_id 和校验结果。
9. 日志导出文件必须短期有效，建议 24 小时自动清理；导出文件不得包含未脱敏密钥、完整 prompt、完整文档正文、完整用户输入或完整模型输出。

### 15.2 日志字段

所有结构化日志至少包含：

```text
id
tenant_id
kb_id
request_id
actor_id
api_key_id
auth_decision_id
auth_reason_code
action
resource_type
resource_id
status
latency_ms
ip
user_agent
error_code
error_message
created_at
```

说明：

- `actor_id` 是 AI 基座返回的操作者 ID，只作日志关联。
- `auth_decision_id` 是知识库访问判断结果 ID。
- `auth_reason_code` 是知识库访问判断原因码，例如 `ALLOW_BY_OWNER`、`ALLOW_BY_TENANT_ADMIN`、`DENY_NOT_OWNER_OR_ADMIN`。
- 日志中不保存用户详情、角色详情或权限规则。
- `api_key_id` 只记录知识库 API Key 元数据 ID 或 fingerprint，不记录 API Key 明文。
- `error_message` 必须做敏感信息过滤，不得透出密钥、token、完整请求体、完整 prompt、完整文档正文或 provider 原始敏感响应。

大模型调用日志额外包含：

```text
pipeline_domain
pipeline_stage
provider
model_name
operation_name
input_tokens
output_tokens
total_tokens
cached_tokens
reasoning_tokens
token_source
```

说明：

- `pipeline_domain` 和 `pipeline_stage` 用于标记 token 消耗归属，在大模型调用日志中不得为空。
- token 字段只记录用量，不记录完整 prompt、模型密钥或供应商原始响应中的敏感内容。

### 15.3 审计要求

以下操作必须写审计日志：

- 创建知识库。
- 修改知识库。
- 删除知识库。
- 创建 API Key。
- 禁用 API Key。
- 删除 API Key。
- 轮换 API Key。
- 修改配置。
- 回滚配置。
- 删除文档。
- 确认入库。
- 导出日志。

## 16. 额度、阈值与用量统计需求

### 16.1 统计与阈值维度

系统应支持以下统计和阈值维度：

- API Key 每日调用次数。
- API Key 总调用次数。
- API Key QPS。
- API Key 并发数。
- API Key 触发的大模型 token 消耗。
- 单文件上传大小。
- 单知识库文档数量。
- 单租户存储容量。

第一期仅支持租户、知识库、API Key 和系统级用量统计。用户级、角色级、学生 / 教职工范围级额度不纳入知识库系统范围；如后续需要展示 AI 基座已有额度结果，只能只读展示，不落为知识库本地规则。

第一期硬边界：

1. QPS、每日调用量、总调用量、并发数和 token 预算只做统计、报表、趋势、超阈标记和告警。
2. 超过统计阈值时，不拒绝 OpenAPI 问答、Graph RAG、文档读取等业务调用。
3. 统计记录必须同时保存实际值、配置阈值、是否超阈、超阈时间和触发来源。
4. `RATE_LIMITED` 和 `QUOTA_EXCEEDED` 仅作为后续启用强限制时的预留错误码，第一期默认不返回。
5. 安全鉴权、请求结构和资源边界错误不属于“只统计”范围，仍必须拒绝。

### 16.2 超阈与拒绝处理

| 场景 | 处理 |
| --- | --- |
| AI 基座判定租户停用或不可用 | 拒绝调用，返回 `BASE_IDENTITY_DENIED` 或 AI 基座原因码 |
| API Key 达到每日调用量阈值 | 记录实际调用量、阈值和 `usage_threshold_exceeded=true`，触发告警，第一期不拒绝调用 |
| API Key 达到 QPS 阈值 | 记录实际 QPS、阈值和 `qps_threshold_exceeded=true`，触发告警，第一期不拒绝调用 |
| API Key 达到并发阈值 | 记录实际并发、阈值和 `concurrency_threshold_exceeded=true`，触发告警，第一期不排队、不拒绝 |
| API Key 触发 token 预算阈值 | 按链路管道记录 token 实际值、阈值和 `token_threshold_exceeded=true`，触发告警，第一期不拒绝调用 |
| 上传文件过大 | 拒绝上传 |
| AI 基座身份快照显示租户停用 | 拒绝全部新增调用 |

仍需拒绝的安全和结构性场景：

- API Key 无效、禁用、过期。
- API Key 未绑定目标 `kb_id`。
- API Key 能力不足。
- IP 不在白名单。
- 签名缺失、签名失败、请求过期、nonce 重放、body hash 不一致。
- `kb_id` 缺失、跨租户、知识库不存在或不可访问。
- 请求 schema 非法、payload 过大、文件类型或大小不符合要求。
- 租户状态不可用、停用、冻结、删除或快照超过允许新鲜度窗口。

### 16.3 大模型 Token 消耗统计

系统需要统计大模型 token 消耗，并按链路管道归属拆分。统计结果不能只提供全局总数，必须能回答“哪个链路、哪个阶段、哪个知识库、哪次任务或请求消耗了多少 token”。

统计归属原则：

1. 每一次大模型调用必须归属到一个明确的 `pipeline_domain` 和 `pipeline_stage`。
2. `pipeline_domain` 表示链路管道，`pipeline_stage` 表示该链路下的具体阶段。
3. OpenAPI、后台页面、API Key 只是调用入口，不作为 token 消耗的最终归属；实际消耗应归到被触发的业务链路。
4. 同一次模型调用只能计入一个链路阶段，避免全局统计和链路统计重复计数。
5. 链路汇总等于该链路下各阶段消耗之和；全局汇总只能作为各链路汇总之和展示。
6. provider 未返回 token 用量时，可以按 tokenizer 估算，但必须标记为估算值。

链路管道建议：

| pipeline_domain | 归属范围 | 典型阶段 |
| --- | --- | --- |
| `ingestion` | 离线入库链路 | 文档解析增强、清洗、切片增强、表格理解、图片理解、质量检查 |
| `embedding` | 向量化链路 | query embedding、document embedding、batch embedding |
| `online_rag` | 在线普通 RAG 链路 | query 改写、rerank、上下文压缩、答案生成、引用校验 |
| `graph_rag` | Graph RAG 链路 | 实体识别、图谱扩展、关系筛选、图谱问答生成 |
| `evaluation` | 评测链路 | 相关性评分、忠实度评分、拒答判断、评测总结 |
| `openapi_invocation` | OpenAPI 协议层自身模型调用 | 仅当 OpenAPI 层存在独立模型调用时使用；一般业务问答仍归入 `online_rag` 或 `graph_rag` |

统计维度：

- 租户。
- 知识库。
- 文档。
- 入库任务。
- RAG 请求。
- API Key。
- 操作者 ID。
- provider。
- 模型名称。
- 链路管道。
- 链路阶段。
- 调用状态。
- 时间粒度，至少支持小时和天。

统计指标：

- input tokens。
- output tokens。
- total tokens。
- cached tokens。
- reasoning tokens。
- 调用次数。
- 成功次数。
- 失败次数。
- 平均耗时。
- p95 耗时。
- 估算值数量和占比。

展示要求：

- 系统总览可展示全局 token 趋势，但必须可下钻到链路管道。
- 单库总览可展示当前知识库 token 消耗趋势和阶段分布。
- 入库任务页展示当前任务各阶段 token 消耗。
- 问答调试页展示当前 requestId 的 RAG / Graph RAG token 消耗。
- 评测记录页展示每次评测的 token 消耗。
- API Key 管理页展示该 key 触发的 token 消耗，但归属仍按实际链路管道拆分。

## 17. 数据模型建议

### 17.1 数据库设计总览

数据库设计必须围绕知识库业务资源、开放接口凭证、配置、日志、统计和只读身份快照展开，不建设本地 IAM。AI 基座租户端仍是租户、用户、角色和用户角色关系的事实源；菜单、会话、学生、教职工、分组、标签等数据不进入知识库数据库边界。

硬性数据库边界：

1. 知识库系统不得新增、修改、删除 AI 基座的权限、用户和角色主数据。
2. 知识库系统允许保存租户、用户、角色、用户角色关系只读快照，但只能由同步任务写入，不允许后台页面或 OpenAPI 修改。
3. 知识库系统不得新增可写本地用户表、角色表、用户角色关系表、权限策略表、权限主体表、密码表或密码 hash 表。
4. 知识库系统不得通过后台页面、OpenAPI、任务脚本或配置中心维护 AI 基座的用户、角色和权限规则。
5. 知识库系统不得把 AI 基座同步数据转换成可编辑的本地权限事实源。
6. `actor_id`、`tenant_id`、`auth_decision_id`、`auth_reason_code`、`identity_snapshot_version` 只能作为业务关联和审计字段，不代表本地用户、角色或权限。
7. API Key 表只能表达知识库 OpenAPI 的技术访问范围，不表达组织权限、用户身份或角色身份。
8. 所有包含 `tenant_id` 的表都用于租户隔离、业务查询和审计，不得反向推导或维护权限规则。

推荐按以下数据域组织：

| 数据域 | 表类型 | 说明 |
| --- | --- | --- |
| AI 基座身份快照 | 租户、用户、角色、用户角色关系只读快照 | 仅用于展示、审计、租户隔离和管理员身份判断 |
| 知识库业务资源 | 知识库、文档、切片、图谱、入库任务 | 保存知识库自身业务数据 |
| OpenAPI 凭证 | API Key、能力范围、知识库绑定、签名配置、nonce 记录 | 仅用于知识库 OpenAPI 技术鉴权 |
| 配置治理 | 配置档案、配置值、配置变更日志 | 管理解析、切片、向量、问答、安全和日志配置 |
| 日志审计 | 操作审计、身份同步日志、用户及权限同步日志、访问判断日志、API 调用日志、RAG 查询日志、任务日志 | 记录行为、结果、错误码和追溯字段 |
| 用量统计 | 日维度用量、小时 token 聚合、API Key 用量、超阈标记 | 支撑统计报表、告警和成本分析；第一期不作为强制拦截依据 |
| 后台会话 | 短期 session、SSO state / nonce 可选记录 | 仅保存 AI 基座认证后的短期登录态，不保存密码和权限 |

敏感字段处理：

- API Key 原文只在创建时返回一次，数据库只保存 hash、key 前缀和后四位。
- 签名 nonce 只保存有效窗口内的去重记录，到期后可清理。
- 密钥类配置应加密存储，前端只展示脱敏值或是否已配置。
- 日志不得记录用户密码、AI 基座 token 明文、API Key 明文、模型密钥明文或完整权限规则。
- 日志和导出不得默认记录完整用户 query、完整模型 answer、完整 prompt、完整 provider 响应、完整文档正文、表格全文或图片 OCR 文本。
- 调试留痕数据必须独立标记 `debug_trace=true`，限定范围、短期保留、自动清理，并纳入审计和导出脱敏规则。
- 删除、禁用、轮换 API Key 和修改敏感配置必须写审计日志。

### 17.4 历史数据初始化与迁移

首次上线或从现有知识库系统迁移时，必须先完成 AI 基座身份快照初始化和历史业务数据归属补齐，再向普通用户开放知识库后台。

历史知识库补 `tenant_id`：

1. 如果现有系统只有单租户数据，统一补默认租户 ID。
2. 如果已有 AI 基座租户映射，按历史创建人或当前业务归属人映射租户。
3. 映射不出来的知识库进入“待认领 / 待管理员处理”状态，不对普通用户可见。

历史知识库补 `owner_user_id`：

1. 优先使用历史 `created_by` 作为 `owner_user_id`。
2. 如果 `created_by` 在 AI 基座快照中不存在、禁用、离职或移出租户，则知识库保留，但进入“失效归属人”状态。
3. 处于“失效归属人”状态的知识库仅租户管理员或平台管理员可见，可接管、转交或删除。

`created_by` 迁移规则：

1. `created_by` 只作为不可变审计字段。
2. 不因迁移、接管或转交覆盖 `created_by`。
3. 历史创建人无法映射时，保留原始值或迁移占位值，并记录迁移原因。

身份快照初始化：

1. 首次上线前先同步 AI 基座租户、用户、角色、用户角色关系只读快照。
2. 同步成功后再开放普通用户后台入口。
3. 如果快照为空或同步失败，只允许平台管理员进入初始化 / 诊断页面，普通用户不可进入业务后台。

历史 API Key：

1. 如果当前系统没有正式 API Key，上线后全部新建，不迁移旧 key。
2. 如果已有测试 key，默认全部置为禁用或过期，由管理员重新颁发。
3. 不允许把旧明文 key 回写到新系统。

历史日志与 token 统计：

1. 能补 `tenant_id`、`kb_id`、`owner_user_id` 的记录按新规则补齐。
2. 补不出来的记录保留为只读历史归档，不参与普通用户查询和成本统计，仅允许管理员查询。
3. 历史日志和统计不得补写完整 prompt、API Key 明文、模型密钥、AI 基座 token 或完整权限上下文。

迁移脚本要求：

1. 必须可重复执行。
2. 必须支持失败回滚或至少支持失败后重新执行。
3. 必须输出迁移报告，包括总数、成功数、待认领数、失效归属人数、失败数和失败原因。
4. 必须写迁移审计日志。

## 18. 核心业务流程

### 18.1 用户访问后台

```text
用户已登录 AI 基座
-> 点击 AI 基座中的知识库入口
-> AI 基座生成一次性 SSO code 和 state
-> 跳转知识库 /sso/callback
-> 知识库后端校验 state 并使用 code 向 AI 基座换取身份摘要
-> 知识库创建短期后台 session
-> 知识库后台读取只读身份快照并判断管理员身份
-> 前端渲染知识库自身功能页面
-> 后端每次接口调用执行租户隔离、当前业务归属人 / 管理员规则判断
```

### 18.2 创建知识库

```text
用户点击创建知识库
-> 后端校验 AI 基座身份摘要、账号状态和租户状态
-> 后端确认当前用户处于有效租户
-> 写入 knowledge_bases 业务数据，created_by 和 owner_user_id 默认均为当前用户
-> 写审计日志，记录 actor_id 和 auth_decision_id
-> 返回知识库详情
```

### 18.3 上传文档入库

```text
用户上传 PDF
-> 后端校验知识库 tenant_id 与当前用户 tenant_id 一致
-> 后端校验当前用户是知识库当前业务归属人、本租户管理员或平台管理员
-> 创建入库任务
-> 执行解析、清洗、切片、向量化、写库
-> 写任务日志和审计日志
```

### 18.4 失效用户知识库接管 / 转交

```text
AI 基座同步到用户禁用、离职或移出租户状态
-> 知识库停止该用户新建 session，并拒绝其后续敏感操作
-> 该用户历史创建的知识库继续保留，不自动删除
-> 租户管理员进入知识库列表筛选失效创建人或失效归属人
-> 管理员选择接管、转交给本租户有效用户，或删除知识库
-> 后端校验管理员身份、租户边界和目标用户有效状态
-> 接管 / 转交时更新 owner_user_id、transferred_by、transferred_at
-> created_by 保持原创建人不变
-> 写操作审计日志和访问判断日志
```

### 18.5 外部应用调用问答

```text
外部应用携带 API Key 请求 /openapi/v1/knowledge-bases/{kb_id}/query
-> 校验 API Key
-> 校验本地租户状态快照和快照新鲜度，不每次实时请求 AI 基座
-> 如 API Key 要求签名模式，校验 timestamp、nonce、body_hash 和 signature
-> 校验 IP 白名单
-> 校验 API Key 能力范围
-> 校验 API Key 绑定的租户和知识库
-> 记录 API Key 用量，判断 QPS、并发、调用量和 token 预算是否超阈并触发告警，第一期不因超阈拦截
-> 校验请求 schema、字段长度和风险标签
-> 解析生效配置
-> 调用 RAG
-> 写 API 调用日志和 RAG 查询日志
-> 返回答案、引用、requestId
```

### 18.6 配置变更

```text
用户进入配置中心
-> 修改配置
-> 后端按配置层级执行访问判断
-> 全局配置仅平台管理员可改
-> 租户同步信息仅只读展示，不允许知识库侧修改
-> 知识库配置可由当前业务归属人、本租户管理员或平台管理员修改
-> API Key 配置仅本租户管理员或平台管理员修改
-> 校验配置格式
-> 保存配置值
-> 刷新运行时配置缓存
-> 写配置变更日志
```

### 18.7 查看日志

```text
用户进入日志中心
-> 后端按租户、知识库、当前业务归属人 / 管理员规则过滤可见日志范围
-> 按租户、知识库、requestId、taskId 等条件查询日志
-> 返回日志列表
```

## 19. 非功能需求

### 19.1 安全

- 所有管理后台接口必须校验 AI 基座身份态，并执行服务端访问判断。
- 所有 OpenAPI 接口必须鉴权。
- API Key 不允许明文存储。
- 生产对外调用推荐启用签名模式。
- API Key 不允许出现在 URL query、前端代码、移动端包体或公开配置文件中。
- 敏感配置不允许明文返回。
- 操作审计不可被普通管理员删除。
- 跨租户访问必须被拒绝。
- 文件下载必须校验同租户、知识库当前业务归属人 / 管理员规则。
- 日志导出必须按租户、知识库、当前业务归属人 / 管理员规则过滤。
- 知识库系统不得保存可写用户、角色、权限规则副本；只允许保存 AI 基座只读身份快照。
- OpenAPI 请求必须做 schema 校验、长度限制和风险日志记录。
- 普通问答接口不得接受 raw `system_prompt`，系统提示词只能通过 Prompt Profile 或管理员级配置 API 变更。

### 19.2 性能

- AI 基座身份校验 p95 建议小于 100ms。
- 知识库访问判断 p95 建议小于 30ms。
- API Key 校验 p95 应小于 30ms。
- 知识库列表接口 p95 应小于 500ms，不含 AI 基座异常等待时间。
- 对外 RAG 查询不应因日志写入明显增加主链路耗时，日志可异步落库。
- 访问判断缓存命中时，低风险读取类操作不应明显增加延迟。

### 19.3 可用性

- AI 基座身份校验或身份快照不可用时，敏感新增、修改、删除、导出操作默认拒绝。
- 身份快照同步失败或延迟时，低风险读取类操作可使用未超过旧快照窗口的快照，默认窗口 10 分钟，可配置范围建议为 5-15 分钟；超过窗口后访问拒绝。
- 旧快照降级期间不得执行新增、修改、删除、导出、配置变更、API Key 管理、接管 / 转交等敏感操作。
- AI 基座用户禁用、离职或移出租户后，该用户后续登录和敏感操作默认拒绝，其历史知识库不自动删除。
- 日志写入失败时，不应影响核心问答成功返回，但必须记录降级告警。
- 配置中心保存失败时必须返回明确错误，不允许静默成功。
- API Key 用量统计组件异常时，不应影响已通过安全鉴权的核心问答成功返回，但必须记录降级告警；如后续启用强限制，再单独定义保守拒绝策略。

### 19.4 可观测性

- 每个对外请求必须有 `requestId`。
- 每个入库任务必须有 `taskId`。
- 每次服务端访问判断应记录耗时、结果和 reasonCode。
- 每次访问拒绝应记录拒绝原因。
- 每次配置生效应能查到来源层级。
- 每次 RAG 查询应记录主要阶段耗时。
- 每次大模型调用应记录 `pipeline_domain`、`pipeline_stage`、provider、模型、token 用量、耗时和状态。
- Token 消耗统计应能按链路管道、链路阶段、租户、知识库、任务、请求和 API Key 下钻。

## 20. 验收标准

### 20.1 身份接入与访问判断验收

- 未携带 AI 基座登录态的后台请求被拒绝。
- 未完成 SSO code 换取身份摘要的后台请求被拒绝。
- SSO code 只能使用一次，过期后不可再次登录。
- 知识库不得保存用户密码、密码 hash、盐值或登录凭证。
- AI 基座账号停用、租户停用或身份快照不可用时，敏感操作被拒绝。
- 身份快照同步失败或延迟时，低风险读取类操作可在 10 分钟默认旧快照窗口内继续；窗口可配置但必须限制在 5-15 分钟。
- 使用旧快照时，新增、修改、删除、导出、配置变更、API Key 管理、接管 / 转交等敏感操作必须被拒绝。
- 旧快照降级命中必须能在身份与访问监控页面和访问判断日志中查到。
- AI 基座账号禁用、离职或移出租户后，该用户不能继续登录知识库后台或操作知识库资源。
- 知识库访问判断返回 deny 时，后台接口被拒绝。
- 知识库访问判断返回 allow 时，后台接口继续执行业务。
- 知识库系统没有新增可写用户、角色、权限策略维护表。
- 知识库系统没有用户管理、角色管理、授权策略配置页面。
- 每次访问判断结果可在日志中心按 requestId 查到。
- 首次上线前 AI 基座租户、用户、角色、用户角色关系只读快照必须同步成功；快照为空或同步失败时，普通用户不得进入业务后台。

### 20.2 知识库访问验收

- 不同租户用户不能互相看到知识库。
- 当前业务归属人可以看到并管理归属给自己的知识库。
- 当前业务归属人可以上传、查看、确认入库和删除自己归属知识库下的文档。
- 非当前业务归属人且非管理员不能看到或操作他人知识库。
- 租户管理员可以查看和管理本租户全部知识库。
- 租户管理员可以接管、转交或删除本租户失效用户历史创建的知识库。
- 平台管理员可以查看和管理全局知识库。
- 接管或转交知识库时，`created_by` 不得被覆盖，必须保留原创建人用于审计。
- 所有迁移后可见知识库必须具备 `tenant_id`。
- 所有普通用户可见知识库必须具备有效 `owner_user_id`；无法映射租户或归属人的历史知识库必须进入“待认领 / 待管理员处理”或“失效归属人”状态。
- 普通用户不能看到未归属、跨租户或失效归属人的历史知识库。
- 租户管理员和平台管理员能看到待处理列表，并能执行接管、转交或删除。
- 知识库、文档和入库任务删除默认软删除；软删除后业务查询、OpenAPI、RAG、Graph RAG 和评测查询不可见。
- 文档、入库任务、草稿、切片、向量、图谱和评测数据的删除权限必须按所属知识库当前业务归属判断，不按上传人或任务创建人单独判断。
- 日志、审计和 token 统计不随知识库或文档删除而硬删，必须保留脱敏记录用于审计和成本追溯。
- 日志查看和导出必须按租户、知识库、当前业务归属人 / 管理员规则过滤。
- 历史迁移脚本必须可重复执行，可失败后重新执行或回滚，并输出迁移报告和审计日志。

### 20.3 OpenAPI 与系统级强签名验收

- 创建 API Key 前必须校验当前操作者为本租户管理员或平台管理员。
- 创建 API Key 时仅展示一次明文。
- 禁用 API Key 后调用失败。
- 过期 API Key 调用失败。
- 不具备对应能力范围的 API Key 不能调用对应接口。
- API Key 只能调用知识库系统自身 OpenAPI，不能调用后台管理 API。
- API Key 不能作为 AI 基座 Token、服务账号或其他业务系统凭证使用。
- API Key 是租户级凭证，可绑定多个知识库，单个 API Key 默认最多绑定 20 个知识库，但单次知识库资源调用必须指定 `kb_id`。
- 未指定 `kb_id` 的知识库资源调用必须返回 `KB_ID_REQUIRED`。
- API Key 绑定多个知识库时，单次请求只在指定 `kb_id` 内召回，不得默认跨全部绑定知识库召回。
- 生产对外调用默认支持或强制签名模式。
- 高风险接口默认要求签名模式。
- Bearer Token 模式仅建议用于内网、低风险联调或过渡场景。
- IP 白名单第一期必须支持，生产对外调用建议配置 IP 白名单。
- 配置为签名模式的 API Key，缺少签名时调用失败。
- 请求时间戳过期时调用失败。
- nonce 重放时调用失败。
- body hash 与请求体不一致时调用失败。
- 不在 IP 白名单的请求被拒绝。
- 超过每日调用量、QPS、并发或 token 预算统计阈值时，第一期调用仍继续执行，但必须记录超阈标记并触发告警；`QUOTA_EXCEEDED` 和 `RATE_LIMITED` 仅作为后续强限制预留错误码。
- 签名无效、请求过期、schema 非法、payload 过大、IP 不允许等场景必须返回对应错误码。
- OpenAPI 调用不应每次实时请求 AI 基座校验租户状态，应默认使用本地租户快照完成校验。
- 本地租户快照更新时间在 5 分钟内且状态为 `active` 时，OpenAPI 可继续执行后续校验。
- 本地租户快照更新时间超过 5 分钟但不超过 10 分钟时，OpenAPI 可降级放行，但必须写入 `tenant_snapshot_stale=true` 日志并触发告警。
- 本地租户快照更新时间超过 10 分钟时，OpenAPI 必须返回 `TENANT_STATUS_STALE`。
- 租户停用、冻结、删除、不明确或不可用时，OpenAPI 必须返回 `TENANT_DISABLED` 或 `TENANT_UNAVAILABLE`。
- 每次调用可以在日志中心按 requestId 查到。

### 20.4 配置中心验收

- 修改配置前必须通过对应配置层级的访问判断。
- 全局配置仅平台管理员可修改。
- 租户同步信息仅只读展示，不允许在知识库配置中心修改。
- 知识库配置可覆盖全局默认值。
- API Key 配置可覆盖知识库配置中允许被 key 级别覆盖的安全和调用参数。
- 敏感配置不明文回显。
- 配置变更可查看操作人 ID、时间和变更摘要。
- 配置保存失败时前端显示失败。
- 普通问答接口不得直接传入 raw `system_prompt`。
- 外部调用如需切换提示词，只能传 `prompt_profile_id`，且必须通过租户、知识库或 API Key 范围校验。
- 配置管理 API 修改 Prompt Profile 时必须强制签名、写审计、版本化并支持回滚。

### 20.5 日志验收

- 入库任务能查看阶段日志。
- API 调用能查看状态码、耗时、错误码。
- RAG 查询能查看问题、答案摘要、引用数量和耗时。
- 大模型调用日志能查看 provider、模型、链路管道、链路阶段、token 用量和耗时。
- 用户及权限同步日志能查看 sync_mode、`last_sync_at`、`max_updated_at`、五类变更计数、状态和失败原因。
- 身份与访问判断日志能查看 allow / deny、reasonCode、decisionId 和耗时。
- 日志支持按租户、知识库、操作者 ID、API Key、requestId、时间范围筛选。
- 日志导出操作必须先通过当前业务归属人 / 管理员访问判断。
- 第一期日志导出不建设审批流，但必须限制时间范围、最大条数、导出字段脱敏并写审计日志。
- 日志保留周期符合明细 180 天、审计 365 天、聚合统计 2 年。
- 日志默认只保存证据链和脱敏摘要，不保存 API Key、模型密钥、AI 基座 token、完整 prompt、完整文档正文、完整用户输入或完整模型输出。
- 调试留痕模式必须管理员显式开启、限定范围、默认 7-14 天自动清理，并对开启、查看、导出和关闭写审计日志。
- nonce 防重放记录只在签名窗口内保留，常规建议 10-30 分钟，安全排查最长不超过 24 小时。
- 导出文件必须短期有效，建议 24 小时自动清理。
- 保留期内删除知识库、文档、入库任务或 API Key 不得删除日志、审计和聚合统计，只能保留脱敏关联。
- 普通用户只能查看本人归属知识库的业务日志摘要；租户管理员可查看本租户全量日志；平台管理员可查看全局日志。

### 20.6 Token 消耗统计验收

- 每次大模型调用必须归属到唯一 `pipeline_domain` 和 `pipeline_stage`。
- 入库链路 token 消耗可按解析增强、清洗、切片增强、embedding 等阶段拆分查看。
- 在线问答链路 token 消耗可按 query 改写、rerank、上下文压缩、答案生成、引用校验等阶段拆分查看。
- Graph RAG 链路 token 消耗可按实体识别、图谱扩展、关系筛选、图谱问答生成等阶段拆分查看。
- 评测链路 token 消耗可按相关性评分、忠实度评分、拒答判断、评测总结等阶段拆分查看。
- 系统总览展示的全局 token 总数必须等于各链路管道聚合之和。
- 单库总览展示的 token 统计只包含当前知识库。
- API Key 维度可查看该 key 触发的 token 消耗，但仍需按实际业务链路拆分。
- provider 未返回 token 时，估算值必须带 `estimated` 标记。
- Token 统计页面不得展示完整 prompt、API Key 明文、模型密钥、AI 基座 token 或完整身份 / 访问判断上下文。

## 21. 迭代规划

整体建议拆成 6 个迭代推进。迭代顺序遵循：先立身份边界，再形成业务闭环，再开放外部能力，再补治理和运维统计。每个迭代都必须延续硬边界：知识库系统不维护可写用户、角色和权限规则；租户、用户、角色、用户角色关系事实以 AI 基座租户端为准；知识库只执行租户隔离、当前业务归属人和管理员业务访问规则。

### 21.1 迭代 1：身份与访问底座

目标：完成知识库后台接入 AI 基座的身份认证、只读身份快照和访问判断底座，先把安全边界立住。

关联功能域：

- 身份与访问域。
- 监控报表域的基础身份与访问可观测能力。

交付内容：

- AI 基座登录态校验。
- AI 基座免登录 / SSO callback。
- 一次性 SSO code 换取身份摘要。
- AI 基座租户、用户、角色、用户角色关系只读快照同步。
- 租户管理员 / 知识库管理员识别规则，第一期固定按 `role_code=superManager` 判断。
- 平台管理员角色映射配置，平台管理员编码需由 AI 基座侧单独确认。
- 知识库短期后台 session。
- 后台 API 访问判断中间件。
- 前端页面和操作业务能力标记。
- 租户隔离基础字段。
- 历史知识库 `tenant_id`、`owner_user_id` 初始化和待处理状态标记。
- 历史 API Key、日志和 token 统计迁移策略。
- 迁移报告和迁移审计日志。
- 最小操作审计日志。
- 身份同步日志、用户及权限同步日志和访问判断日志。

验收重点：

- 未通过 AI 基座登录或 SSO 的用户不能进入知识库后台。
- 知识库不保存用户密码、密码 hash、盐值或登录凭证。
- 知识库不新增可写用户、角色、权限策略维护表。
- 后台接口不能只依赖前端页面或按钮隐藏，必须服务端执行租户隔离和当前业务归属人 / 管理员规则判断。
- AI 基座身份校验失败、身份快照不可用或账号 / 租户状态不明确时，敏感操作默认拒绝。
- 历史知识库迁移后必须完成 `tenant_id` 补齐；普通用户可见知识库必须有有效 `owner_user_id`，否则进入管理员待处理状态。

前置依赖：

- AI 基座提供登录态校验或 SSO code 换身份摘要接口。
- AI 基座提供租户、用户、角色、用户角色关系同步来源或查询接口。
- 租户管理员 / 知识库管理员默认按 `role_code=superManager` 判断；仅需确认 AI 基座是否存在独立平台管理员角色编码，以及该编码是否具备跨租户全局范围。
- AI 基座同步或查询结果需要能区分用户有效、禁用、离职、移出租户等状态，用于拒绝失效用户访问。

### 21.2 迭代 2：知识库业务闭环

目标：完成知识库、文档和入库的后台主链路，让用户可以从创建知识库到文档入库形成闭环。

关联功能域：

- 知识库业务域。
- 文档与入库域。

交付内容：

- 知识库列表。
- 知识库创建、编辑、删除、启用、禁用。
- 知识库当前业务归属人字段。
- 失效用户创建知识库的管理员接管、转交和删除。
- 知识库、文档、入库任务的软删除、恢复和保留期后物理清理策略。
- 子资源删除权限按所属知识库归属判断。
- 单库总览。
- 文档管理。
- 文档上传。
- 入库任务创建。
- 解析、清洗、切片、向量化、写库。
- 草稿切片查看、编辑、删除、合并。
- 确认入库。
- 失败任务重试。
- 全局入库看板。
- 单库入库任务。
- 入库阶段日志和 SSE 实时日志。
- 文档下载和 CSV 导出。

验收重点：

- 用户可见、创建、修改、删除知识库的规则为：同租户、当前业务归属人可管理本人归属知识库、租户管理员可管理本租户全部知识库、平台管理员可管理全局知识库。
- 用户上传、查看、确认入库和删除文档前，必须先校验文档所属知识库，再执行当前业务归属人 / 管理员规则。
- 失效用户历史创建的知识库不自动删除；租户管理员可接管、转交或删除，且接管 / 转交必须写审计日志。
- 删除知识库、文档、入库任务后，相关数据立即从后台列表、OpenAPI、RAG、Graph RAG 和评测查询中不可见。
- 恢复软删除资源必须重新校验所属知识库当前业务归属人 / 管理员规则。
- 不同租户的知识库、文档、任务数据互不可见。
- 从创建知识库到上传文档、确认入库、生成可检索数据可完整跑通。
- 入库失败能查看错误原因并支持重试。

前置依赖：

- 迭代 1 的身份、租户和访问判断底座完成。
- 现有解析、切片、向量化、pgvector 写入链路可被后台任务调用。

### 21.3 迭代 3：问答、图谱与评测

目标：让知识库可被实际使用，并提供证据回溯、图谱预览和质量评测能力。

关联功能域：

- 在线问答域。
- 图谱与评测域。
- 文档与入库域的查询后验证能力。

交付内容：

- 全局问答调试。
- 单库问答调试。
- 普通 RAG 查询。
- Graph RAG 查询。
- 引用和证据回溯。
- 查询耗时、召回结果和答案摘要记录。
- 单库知识图谱预览。
- 切片关系、实体关系和三元组预览。
- 全局评测记录。
- 单库评测记录。
- 评测样本、相关性、忠实度、拒答情况展示。

验收重点：

- 用户问答、查看证据、查看图谱和查看评测前，必须先通过同租户、当前业务归属人 / 管理员访问判断。
- 普通 RAG 和 Graph RAG 均能返回答案、引用和 requestId。
- 单库工作台内能围绕当前知识库完成问答、图谱和评测查看。
- 全局评测记录可跨知识库巡检，但仍受租户、当前业务归属人和管理员访问边界限制。
- 不伪造没有后端支撑的历史问答数据。

前置依赖：

- 迭代 2 已完成知识库与文档入库闭环。
- 普通 RAG、Graph RAG、图谱预览和评测记录接口具备可用数据来源。

### 21.4 迭代 4：OpenAPI 与 API Key

目标：支持外部系统以知识库 API Key 安全调用知识库，并建立系统级凭证安全闭环。

关联功能域：

- 开放接口域。
- 额度与用量域的 API Key 调用统计和超阈告警。
- 日志审计域的 API 调用审计。

交付内容：

- OpenAPI v1 基础协议：`/openapi/v1/*` 路由、请求 / 响应结构、`requestId`、分页、错误格式。
- API Key 生命周期：创建、查看、禁用、删除、轮换、过期、明文一次性展示、hash 存储。
- API Key 授权边界：租户级归属、绑定一个或多个知识库、默认最多绑定 20 个知识库、绑定接口能力范围，禁止调用后台管理 API。
- API Key 调用租户状态校验：不实时请求 AI 基座，使用本地租户快照；5 分钟内正常校验，5-10 分钟降级放行并告警，超过 10 分钟拒绝。
- OpenAPI 知识库资源调用必须指定 `kb_id`，未指定时返回 `KB_ID_REQUIRED`。
- API Key 绑定多个知识库时，单次请求仍只在指定 `kb_id` 内召回；跨库查询后续作为显式高级接口设计。
- 系统级强签名认证：签名请求头、签名串、HMAC 算法版本、timestamp 时间窗、nonce 防重放、body hash 防篡改。
- 强签名适用策略：Bearer Token 仅建议用于内网或低风险联调；生产对外调用推荐强签名，高风险接口默认强制签名。
- IP 白名单、QPS、并发数、每日调用量、总调用量统计阈值。
- 系统级错误码规范：认证类、签名类、绑定类、统计阈值预留类、请求结构类、身份状态类、访问判断类错误码。
- 安全与注入防护：schema 校验、未知字段拒绝、字段长度限制、prompt injection 风险标签、敏感信息禁止返回。
- 审计与告警：API 调用日志、签名失败日志、nonce 重放日志、异常 IP 告警、调用量突增告警。
- API Key 管理页面。
- 后台内置 OpenAPI 调用文档页，仅管理员可见，不匿名公开；展示接口、签名规则、错误码和示例，不展示真实 API Key。
- API Key 管理操作仅允许本租户管理员或平台管理员。

验收重点：

- API Key 只能调用知识库系统自身 OpenAPI，不能调用后台管理 API。
- API Key 不能作为 AI 基座 Token、服务账号或其他业务系统凭证使用。
- 创建、绑定、禁用、轮换 API Key 前必须校验当前操作者为本租户管理员或平台管理员。
- 未传 `kb_id` 的知识库资源调用必须被拒绝，不得默认跨租户或跨全部绑定知识库召回。
- 文档上传、入库和其它写操作必须显式授予对应 OpenAPI 能力，不能默认继承查询能力。
- 生产对外调用支持或强制签名模式。
- timestamp 过期、nonce 重放、body hash 不一致、签名无效、IP 不允许等场景必须拒绝并返回明确错误码。
- OpenAPI 调用日志可按 requestId、API Key、租户、知识库追踪。

前置依赖：

- 迭代 1 的身份与访问底座完成。
- 迭代 2 和迭代 3 的核心业务能力已具备可开放接口。

### 21.5 迭代 5：配置中心

目标：完成多层业务配置治理，让解析、切片、向量、问答、安全和日志参数可管理、可审计、可回滚。

关联功能域：

- 配置治理域。
- 日志审计域的配置审计能力。

交付内容：

- 全局配置。
- 租户同步信息只读查看。
- 知识库配置。
- API Key 配置。
- Prompt Profile 管理。
- 配置管理 API，可受控修改 Prompt Profile。
- 配置优先级解析。
- 配置搜索和分组。
- 解析、清洗、切片、向量、问答、Graph RAG、安全、日志配置。
- 配置生效方式分类：热生效、灰度生效、重启或运维变更。
- 灰度配置版本化、预览、回滚和效果追踪。
- 敏感配置脱敏。
- 配置变更预览。
- 配置保存。
- 配置回滚。
- 配置变更日志。
- 配置操作接入配置层级访问判断。
- 普通问答接口支持 `prompt_profile_id`，但不允许 raw `system_prompt` 覆盖。
- 配置管理 API 修改 Prompt Profile 时强制签名、IP 白名单、审计、版本化和回滚。

验收重点：

- 配置中心只管理业务运行配置，不管理用户、角色和权限规则。
- 配置优先级符合 `API Key 配置 > 知识库配置 > 全局配置 > 环境变量默认值`。
- 租户名称、状态、租户归属、用户、角色和用户角色关系只能从 AI 基座租户端同步，配置中心不得修改。
- 敏感配置不明文回显，审计日志不得保存密钥明文。
- 配置保存失败时必须返回明确错误，不允许静默成功。
- 修改配置、回滚配置前必须通过配置层级访问判断。
- 第一期开通热生效和灰度生效能力，不承诺所有配置都能后台热改。
- Prompt Profile、API Key 策略、知识库问答参数、日志保留天数、脱敏策略和告警阈值应支持热生效。
- 模型选择、rerank provider、embedding provider / 模型、切片策略、清洗策略、质量过滤策略、Graph RAG 扩展参数和系统提示词大版本应支持按知识库或 API Key 灰度生效并可回滚。
- 数据库连接、pgvector / Redis / OSS 配置、模型密钥、AK/SK、解析 provider 大切换、文件存储路径、队列 worker 数和后台任务并发全局上限属于重启或运维变更，不承诺后台直接热改。
- 系统提示词可通过 Prompt Profile 或管理员级配置 API 变更，不允许普通 query 请求直接覆盖。

前置依赖：

- 迭代 1 的身份与访问底座完成。
- 已按热生效、灰度生效、重启或运维变更三类明确配置生效边界。

### 21.6 迭代 6：日志、监控、用量与 Token 统计

目标：形成可运维、可审计、可追踪、可拆账的运营治理闭环。

关联功能域：

- 日志审计域。
- 额度与用量域。
- 监控报表域。

交付内容：

- 日志中心页面。
- 入库任务日志结构化。
- API 调用日志。
- RAG 查询日志。
- Graph RAG 查询日志。
- 大模型调用日志。
- 用户及权限同步日志。
- 身份与访问判断日志。
- 配置变更日志。
- 错误聚合。
- 身份与访问监控。
- 系统总览指标增强。
- 用量统计。
- Token 消耗统计页面。
- `kb_llm_call_logs`。
- `kb_token_usage_hourly`。
- 按链路管道和阶段聚合 token 消耗。
- 日志导出。

验收重点：

- 每次大模型调用必须归属到唯一 `pipeline_domain` 和 `pipeline_stage`。
- Token 消耗能按 `ingestion`、`embedding`、`online_rag`、`graph_rag`、`evaluation` 拆分。

- 日志支持按租户、知识库、操作者 ID、API Key、requestId、taskId、时间范围筛选。
- 日志导出操作必须先通过当前业务归属人 / 管理员访问判断。

前置依赖：

- 迭代 2、迭代 3、迭代 4 的核心链路已经产生可统计日志。
- 大模型 provider 能返回 token 用量；无法返回时需具备估算和 `token_source=estimated` 标记能力。

## 22. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 基座身份校验或身份快照同步不可用 | 知识库后台和敏感操作不可用 | 默认拒绝敏感操作，监控告警，明确降级策略 |
| AI 基座身份校验耗时高 | 登录、后台进入和身份刷新变慢 | 短 session、超时告警，低风险读取可按已缓存访问判断降级 |
| AI 基座 `superManager` 语义变化 | 租户管理员 / 知识库管理员可见范围变化 | 将 `role_code=superManager` 作为第一期明确合同，若 AI 基座调整语义或编码，需走配置变更、审计日志和身份快照刷新 |
| AI 基座平台管理员角色映射变化 | 平台管理员全局可见范围变化 | 平台管理员角色独立配置化，变更写审计日志并刷新身份快照 |
| SSO code 泄露或重放 | 非法免登录 | 一次性 code、短有效期、state 校验、服务端换取身份摘要 |
| API Key 泄露 | 外部滥用 | hash 存储、IP 白名单、调用统计、超阈告警、禁用、轮换 |
| API Key 被放到前端或客户端 | 易被抓包或反编译 | 只允许服务端保存，生产建议签名模式和 IP 白名单 |
| API Key 被重放 | 被截获请求重复调用 | timestamp + nonce + body hash + HMAC 签名 |
| 合法调用方注入恶意 query | 诱导越权、数据外传或提示词污染 | schema 校验、长度限制、系统提示词隔离、风险标签和审计 |
| 日志保留完整 query、answer、prompt 或文档正文 | 敏感信息泄露，日志系统变成数据外泄面 | 默认只存脱敏摘要、hash、引用 ID、风险标签和统计字段；原文仅允许短期调试留痕 |
| 调试留痕长期保存 | 排障数据累积成高风险敏感数据 | 管理员显式开启、限定范围、默认 7-14 天自动清理，开启、查看、导出和关闭均写审计 |
| 错误日志透出密钥或 provider 原始响应 | 密钥泄露或供应商敏感信息泄露 | error_message 入库前做敏感信息过滤，禁止记录 key、token、完整请求体和原始敏感响应 |
| API Key 绕过租户边界 | 安全事故 | API Key 必须绑定租户和知识库，跨租户直接拒绝 |
| API Key 被误当作跨系统凭证 | 越权访问或边界混乱 | 明确仅知识库 OpenAPI 有效，其他系统不得接受该凭证 |
| Token 统计归属不清 | 成本无法拆账，无法定位高消耗链路 | 每次模型调用强制记录 pipeline_domain 和 pipeline_stage |
| Token 统计重复计数 | 总量失真，影响报表和成本判断 | 单次模型调用只归属一个链路阶段，全局汇总由链路汇总相加 |
| provider 不返回 token 用量 | 统计缺口 | 使用估算并标记 token_source=estimated |
| 日志量过大 | DB 压力增加 | 异步写入、冷热分层、保留策略 |
| 配置误操作 | 影响解析或问答质量 | 变更预览、审计、回滚、分层生效 |
| 跨租户数据泄露 | 高风险安全事故 | 所有核心表补租户字段，所有查询强制 tenant filter |
| 删除权限按子资源创建人判断 | 同一知识库内文档和任务权限割裂，管理员接管后仍无法治理 | 删除权限统一回到所属知识库 `owner_user_id`、租户管理员和平台管理员判断 |
| 误删知识库或文档 | 业务数据不可恢复，影响问答服务 | 默认软删除、二次确认、审计日志、保留期内恢复，物理删除仅管理员高危操作或后台清理任务 |
| 删除后仍被召回 | 已删除内容泄露到问答和图谱结果 | 软删除后立即从后台、OpenAPI、RAG、Graph RAG、评测查询中过滤 |
| 删除日志和统计导致审计断链 | 成本和安全追溯缺失 | 日志、审计、API 调用和 token 统计保留脱敏记录，不随业务资源硬删 |
| 用户离职或禁用后资源无人维护 | 知识库长期无人负责，文档和 API 调用难以治理 | 保留历史知识库，由租户管理员接管、转交或删除；`created_by` 保留审计，`owner_user_id` 表示当前归属 |
| 接管时覆盖原创建人 | 审计链路丢失 | 接管 / 转交只能更新当前业务归属人和转交字段，不得覆盖 `created_by` |
| 身份快照旧数据误放大权限 | 账号或角色变化后仍短时间按旧状态读取 | 旧快照只允许低风险读取，默认 10 分钟、可配置 5-15 分钟；敏感操作拒绝并记录 `identity_snapshot_stale=true` |
| API Key 调用实时依赖 AI 基座 | OpenAPI SLA 被 AI 基座抖动拖垮 | 使用本地租户状态快照和新鲜度窗口；超过 10 分钟拒绝，5-10 分钟降级告警 |

## 23. 待确认问题

1. AI 基座是否提供 SSO code 换身份摘要接口，以及身份摘要字段、有效期和错误码。
2. AI 基座是否提供支持增量同步所需的更新时间、版本号或变更游标字段。
3. AI 基座是否存在独立平台管理员角色编码或角色名称；`role_code=superManager` 是否仅代表租户范围管理员，是否存在跨租户全局范围例外。
4. AI 基座身份校验接口的超时时间和 SLA 是多少。
5. SSO 退出登录、账号禁用、租户停用是否支持 webhook 或事件通知；该能力仅作为后续加速刷新通道，不影响第一期受控增量同步（仅 AI 基座 SSO 的 `superManager` 可触发）+ 登录时兜底刷新方案。
6. 平台管理员角色映射变更是否需要审批或双人确认；`superManager` 语义或编码变更是否需要走同等审批。
