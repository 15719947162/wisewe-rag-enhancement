# 当前召回机制与传统 RAG 召回机制的差异

## 核心结论

传统 RAG 的召回核心是“相似文本检索”。当前项目的召回核心更接近：

```text
教材证据定位 + 多通道融合 + 证据回溯
```

也就是说，当前系统不是简单找“哪几段文字和问题相似”，而是在判断：

- 用户是不是在找某张图或某张表？
- 用户是不是在问精确术语、编号、章节或页码？
- 是否需要语义相似内容？
- 命中的 enhanced 是否能折回真实 child 证据？
- 是否需要补相邻片段、引用片段或关联片段？
- 最终证据能否回到教材页码、文件名和 chunk 位置？

## 传统 RAG 召回链路

传统 RAG 常见链路是：

```text
query
-> query embedding
-> vector search top_k
-> rerank
-> context
-> answer
```

稍微增强的传统 RAG 会加入 BM25：

```text
query
-> vector recall
-> BM25 recall
-> merge / rerank
-> answer
```

它的优势是简单、快、稳定，尤其适合普通文档中的事实问答。用户问“什么是 X”，系统找到最相似的几段文本，通常就能回答。

但教材场景里存在大量图表编号、页码位置、章节层级、定义、例题、相似术语、流程链、图文对应和表格对应关系。只靠相似文本召回往往不够。

## 当前项目召回链路

当前普通问答召回大致是：

```text
query
-> media_ref 图表编号直达判断
-> dense 向量召回
-> sparse BM25 召回
-> structured filters
-> RRF 融合
-> enhanced 折回 child
-> related_ids 扩展
-> coarse filter
-> rerank
-> answer
```

整体上，当前系统可以理解为三类召回能力组合：

```text
media_ref:
  图表定位短路

Hybrid RAG:
  默认主召回

Graph RAG:
  关系型 / 多跳 / 流程因果增强召回
```

## 关键差异

### 1. media_ref 图表编号直达

教材中常见这类问题：

```text
图 1-3-3 在哪里？
表 2-1 说明了什么？
```

传统 RAG 可能仍会走 embedding、BM25、rerank 和 LLM 生成，慢且不稳定。当前机制会先识别图表编号，命中后直接查询数据库里的图片或表格 chunk，跳过完整 RAG 链路。

优点：

- 响应快。
- 定位准确。
- 避免 LLM 编造。
- 适合教材图表查询。

风险：

- 依赖入库阶段图表编号提取准确。
- OCR 把编号识别错会影响直达。
- 标题或内容中没有编号时可能命不中。

### 2. dense + sparse 双通道

当前不是单纯向量召回，而是同时使用：

```text
dense:
  语义相似，依赖 embedding / pgvector

sparse:
  关键词、编号、术语匹配，依赖 BM25
```

dense 适合语义改写问题，例如：

```text
为什么植物需要光合作用？
```

BM25 更适合精确术语、符号、编号、章节标题，例如：

```text
ATP
Na+ 通道
图 1-3-3
神经中枢
第 3 节
```

传统纯向量 RAG 在精确符号上容易漂移；纯 BM25 又不理解语义。当前混合方式更适合教材。

### 3. RRF 融合

dense 和 sparse 的分数不是同一种东西，不能直接相加。当前用 RRF 做排序融合。

直观理解：

```text
一个 chunk 如果在多个通道里都排得靠前，它更值得保留。
```

优点：

- 降低单通道失误。
- 不强行比较向量分和 BM25 分。
- 对混合召回更稳。

风险：

- 如果某个通道召回噪声很多，仍可能挤进候选。
- RRF 后的分数不是语义相似度，前端展示时不能误解为准确率。
- 仍需要 rerank 或 scorer 做最终裁判。

### 4. child + enhanced 检索，引用折回 child

当前三层切片的在线召回口径是：

```text
检索：child + enhanced
引用：child
导航：parent
```

`enhanced` 的作用是提高召回可见性。例如原文是：

```text
感受器接受刺激后产生兴奋。
```

enhanced 可能补充为：

```text
该段说明反射弧中感受器受到外界刺激后如何启动神经冲动传导过程。
```

用户问“反射弧为什么能完成反射”时，enhanced 更容易被召回。

但最终引用不能直接引用 enhanced，因为 enhanced 是模型生成或加工内容，不是教材原文。因此命中 enhanced 后需要折回对应 `child`，把真实教材片段作为证据。

优点：

- 提升召回率。
- 保持引用可信。
- 兼顾语义扩展和证据回溯。

风险：

- enhanced 质量差会召回偏。
- enhanced 和 child 绑定必须稳定。
- 折回 child 后，如果原文片段太短，可能还需要上下文窗口补充。

### 5. related_ids 扩展

传统 RAG 通常只拿 top_k 片段。当前会根据 `related_ids` 补充一些关联 chunk。

这对教材有用，因为一个知识点可能分散在：

- 当前段。
- 上一段定义。
- 下一段解释。
- 相邻图表。
- 表格说明。
- 被引用 chunk。

related 扩展可以补完整上下文。

风险：

- 扩展过宽会带来噪声。
- 关联边质量不高时会召偏。
- related 不应该替代 rerank，只能作为候选补充。

### 6. structured filters

当前链路具备结构化过滤能力，例如：

- 按来源过滤。
- 按标题过滤。
- 按页码范围过滤。
- 按是否表格过滤。

教材问答里这很重要：

```text
只查第 2 章
只看表格
只看某份教材
只看第 30-40 页
```

当前常规控制台问答不一定总是传 filters，所以这部分能力仍有继续产品化空间。

### 7. Graph RAG 是增强路径，不是替代路径

普通 `/api/rag/query` 是混合 RAG。Graph RAG 不是替代它，而是另一条关系型增强路径。

推荐理解：

```text
普通混合 RAG:
  默认主链路，负责事实型、定义型、图表定位、单点知识问答。

Graph RAG:
  增强链路，负责流程、因果、概念关系、多跳和跨章节综合问题。
```

## 用户问答结果上的差异

传统 RAG 返回的候选通常是：

```text
几个相似文本片段
```

当前机制返回的候选更像：

```text
直接定位证据
+ 语义相似片段
+ 关键词命中片段
+ enhanced 折回原文片段
+ 关联上下文片段
+ 图表 / 表格候选
```

因此用户会感受到：

- 查图表编号更快。
- 查教材术语更稳。
- 答案引用更能回到页码和文档位置。
- 复杂问题会召回更多上下文。
- 简单问题如果扩展过多，可能出现噪声。

## 当前机制的优势

1. 更适合教材

   图表、页码、章节、术语都被纳入召回设计。

2. 更适合证据回溯

   最终证据落在 `child`，可以对应文档、页码和 chunk。

3. 更兼顾语义和精确匹配

   dense 负责语义，BM25 负责关键词、编号和术语。

4. 更容易扩展到 Graph RAG

   related、entity、chunk_relations 这些设计为后续图谱召回打基础。

## 当前机制的风险

1. 链路复杂，问题定位更难

   召回不准时，需要判断问题来自 dense、BM25、enhanced、related、rerank 还是 Graph 扩展。

2. 通道计数容易误导

   RRF 是融合算法，不是独立召回源。前端如果展示“RRF 数量”，用户可能误解。

3. enhanced 会引入噪声

   它提高可召回性，但也可能把问题带偏。

4. related 扩展需要控制边界

   不能把所有相关内容都塞进上下文。

5. 还需要意图路由

   定义题、图表定位题、流程题、关系题不应该完全走同一套参数。

## 后续优化方向

建议把召回策略做成按问题类型路由：

```text
图表定位题:
  media_ref -> 直接返回

定义 / 原文题:
  dense + BM25 -> rerank
  少扩展或不扩展 graph

术语 / 编号题:
  BM25 权重更高
  dense 辅助

流程 / 因果题:
  Hybrid RAG + Graph RAG
  允许关系路径扩展

跨章节综合题:
  Hybrid RAG 初召回
  Graph RAG 补链路
  rerank / scorer 严格裁判
```

目标是让当前机制从“所有问题都走一条复杂链路”，进化成：

```text
按意图选择召回策略
```

## 初步结论

传统 RAG 是文本相似召回。当前项目是面向教材的证据召回系统。

它不仅要找相似内容，还要定位图表、保护原文引用、融合关键词和语义、补上下文，并为 Graph RAG 留出关系扩展空间。
