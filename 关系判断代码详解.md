# 关系判断代码详解

这份文档只讲一件事：

这个项目里，`chunk` 和 `chunk`、`chunk` 和 `entity` 之间的各种“关系”，到底是怎么判断出来的，什么时候判断，写到哪里，后面检索时又是怎么拿来用的。

我尽量按真实代码顺序讲，而且用大白话，不省略关键细节。

---

## 一、先把全局顺序讲清楚

很多人一看到“关系判断”，容易以为这是一个单独模块，输入一堆 chunk，输出一堆边，就完了。

但在这个项目里，不是这样。

关系判断分成两大批：

1. 第一批是基础规则关系
2. 第二批是增强后的高级关系

真实顺序可以先记成这样：

```text
切片完成
  -> link_related_chunks()
     先做基础规则关系
     - refers_to
     - adjacent
     - sibling
     - enhanced 继承

确认入库 confirm_pipeline()
  -> embedding
  -> link_semantic()
     做语义相似关系
     - semantic_similar
     - duplicate_of
  -> detect_procedure_chunks()
     先识别“这是不是步骤块”
  -> link_procedure()
     做步骤关系
     - next_step
     - prev_step
  -> link_causal()
     做因果关系
     - cause_of
     - effect_of
  -> materialize_entities()
     做实体物化
     - mentions

最终写库
  -> chunk_relations
  -> entities
  -> entity_mentions

在线 Graph RAG
  -> graph_expand()
     根据关系类型继续扩图
```

所以要理解“关系判断”，你不能只看一个文件，至少要把这些地方一起看：

- `core/chunker/linker.py`
- `core/chunker/relation_utils.py`
- `core/chunker/semantic_linker.py`
- `core/chunker/procedure_linker.py`
- `core/chunker/causal_linker.py`
- `core/kg/extraction_pipeline.py`
- `core/rag/graph_expander.py`

---

## 二、先认识最底层的关系工具：`relation_utils.py`

文件：

- `core/chunker/relation_utils.py`

这个文件不负责“判断业务逻辑”，它负责的是：

1. 有没有这条关系
2. 怎么安全地加一条关系
3. 怎么同时加双向关系
4. 怎么做一点小规范化

也就是说，很多具体判断模块最后都要落到这里。

---

## 三、`has_relation()`：先查重，避免重复加边

代码入口：

- `has_relation(chunk, target_id, rel_type=None)`

它的逻辑非常直接：

```python
return any(
    relation.target_id == target_id and (rel_type is None or relation.rel_type == rel_type)
    for relation in chunk.relations
)
```

大白话解释：

- 它会遍历当前 `chunk.relations`
- 看里面是不是已经存在一条边，目标是不是这个 `target_id`
- 如果你还指定了 `rel_type`，那就连关系类型也必须一样
- 只要找到一条，就返回 `True`

这意味着什么？

这意味着项目里判断“重复关系”的标准，不是看 `evidence`，也不是看 `weight`，而是看：

- `target_id`
- `rel_type`

只要这两个一样，就认为已经有了，不再重复加。

比如：

```text
child-001 已经有
  -> target_id = child-002
  -> rel_type = sibling

后来又有另一个规则也想加：
  child-001 -> child-002 -> sibling

那就不会再加第二次
```

但是如果关系类型不同，就可以共存：

```text
child-001 -> child-002 -> refers_to
child-001 -> child-002 -> sibling
```

这两条是允许同时存在的，因为类型不一样。

## 四、`add_relation()`：真正把关系塞进 `chunk.relations`

代码入口：

- `add_relation(chunk, target_id, rel_type, weight=1.0, source="rule", evidence="")`

它内部做了几件事。

### 4.1 第一件事：先查重

```python
if has_relation(chunk, target_id, rel_type):
    return
```

也就是说，重复关系直接丢弃。

### 4.2 第二件事：构造 `Relation`

真正 append 的结构大概是这样：

```python
Relation(
    target_id=target_id,
    rel_type=rel_type,
    weight=_normalize_weight(weight),
    source=source,
    evidence=evidence[:20],
)
```

这里有三个容易漏掉的细节。

### 4.3 细节一：`weight` 会被轻微归一化

内部调用了：

- `_normalize_weight(weight)`

规则非常克制，不是大范围归一化，只修两种边角情况：

1. 如果 `weight` 比 `1.0` 稍微大一点点，但只是浮点误差
2. 如果 `weight` 比 `0.0` 稍微小一点点，但只是浮点误差

对应逻辑：

```python
if 1.0 < weight <= 1.0 + _WEIGHT_EPSILON:
    return 1.0
if -_WEIGHT_EPSILON <= weight < 0.0:
    return 0.0
```

大白话解释：

- 有些相似度理论上应该是 `1.0`
- 但浮点数算出来可能是 `1.0000000001`
- 这种会被压回 `1.0`

反过来：

- 理论上不该出现负值
- 但如果因为数值误差出来一个 `-0.0000000001`
- 也会被拉回 `0.0`

### 4.4 细节二：`evidence` 只保留前 20 个字符

注意这里不是整个 evidence 都存：

```python
evidence=evidence[:20]
```

这说明设计意图很明确：

- `evidence` 只是一个轻量证据片段
- 不是完整原文副本
- 主要用于说明“这条边是怎么来的”

比如：

- `如图3-1所示`
- `因而`
- `cos=0.912`
- `同parent_id`

这些短证据够解释关系来源，但不会把整段正文复制进去。

### 4.5 细节三：`source` 会记录关系来源

常见来源有：

- `rule`
- `embedding`
- `entity`

大白话解释：

- `rule`：规则判断出来的
- `embedding`：向量相似度判断出来的
- `entity`：实体物化时补进去的

后面排查问题时，这个字段很重要。

---

## 五、`add_bidirectional_relation()`：一次加两条反向边

很多关系天然是双向的。

比如：

- `semantic_similar`
- `duplicate_of`
- `refers_to` 在这个项目的基础 linker 里也是双向补的
- `sibling`
- `adjacent`

这个工具函数做的事很简单：

```python
add_relation(src, dst.id, rel_type, ...)
add_relation(dst, src.id, rel_type, ...)
```

大白话解释：

- 从 A 指向 B 加一条
- 再从 B 指向 A 加一条

但要注意：

这个函数不是万能正确。

有些关系不能双向对称，比如：

- `next_step`
- `prev_step`
- `cause_of`
- `effect_of`
- `mentions`

这些就不能直接用这个函数，而是要手工一前一后分别加不同类型。

---

## 六、`filter_by_type()`：后续按类型过滤关系

这个函数很简单：

```python
return [relation for relation in relations if relation.rel_type in types]
```

它本身没有复杂逻辑，但体现出设计思路：

- 所有关系最终都先落在 `chunk.relations`
- 之后想拿哪一类，就再过滤

也就是说，这个项目不是“每种关系单独存一个字段”，而是统一放在一个关系列表里。

---

## 七、第一批关系：`link_related_chunks()` 做基础规则关系

文件：

- `core/chunker/linker.py`

这是最早执行的一批关系判断。

这个文件开头的注释其实已经把规则说得很清楚：

1. 引用匹配
2. 邻近匹配
3. 同父块匹配
4. enhanced 继承父 child 的关系

注意这一步发生得很早。

它的定位不是“高智能”，而是：

- 局部
- 可审计
- 尽量稳定

也就是宁愿简单一点，也要可控。

---

## 八、`link_related_chunks()` 先怎么分块

函数一开始先把 chunks 分成三类：

```python
img_chunks = [(i, c) for i, c in enumerate(chunks) if c.is_image_chunk]
table_chunks = [(i, c) for i, c in enumerate(chunks) if c.is_table_chunk]
text_chunks = [
    (i, c)
    for i, c in enumerate(chunks)
    if not c.is_image_chunk and not c.is_table_chunk and c.layer not in ("parent", "enhanced")
]
```

大白话解释：

### 8.1 图片块

- `is_image_chunk == True`

### 8.2 表格块

- `is_table_chunk == True`

### 8.3 文本块

这里不是“所有不是图表的都算文本”，它还排除了：

- `parent`
- `enhanced`

所以这里真正做基础关系时，文本块主要是：

- 普通 `child` 文本块

这很关键。

因为它说明基础规则关系的主战场是：

- 正文 child
- 图片 child
- 表格 child

不是 parent，也不是 enhanced。

enhanced 关系稍后统一“继承”，而不是跟 child 一起重新跑一遍四条规则。

---

## 九、`relation_keys`：这个函数自己也维护了一层去重

除了 `relation_utils.py` 的通用查重，这个文件内部还自己建了一套键集合：

```python
relation_keys: dict[str, set[tuple[str, str]]] = {
    chunk.id: {(relation.target_id, relation.rel_type) for relation in chunk.relations}
    for chunk in chunks
}
```

为什么要这样做？

因为这个文件内部没有直接调用 `add_relation()`，而是自己写了一个局部 `_add_relation()`。

原因很简单：

- 它处理的是当前函数里的大批量关系添加
- 直接在本地用 set 去重，效率更直接

这也再次说明它的去重标准仍然是：

- `target_id`
- `rel_type`

## 十、Rule 1：`refers_to`，明确引用图表

这是最重要的基础规则之一。

场景很常见：

- 正文里写“如图3-1所示”
- 另一个 chunk 里真有“图3-1”

那系统就把这两个 chunk 连起来。

---

## 十一、它怎么识别“图3-1”和“表3-1”

相关正则在文件开头：

- `_FIG_REF`
- `_TABLE_REF`
- `_FIG_LABEL`
- `_TABLE_LABEL`

还有一个辅助：

- `_FULLWIDTH_DIGITS`

### 11.1 `_FIG_REF` / `_TABLE_REF`

这两个是拿来从正文里找“引用”的。

比如正文里有：

- `如图3-1所示`
- `见图 3 - 1`
- `表3-1`

它就会把编号那部分提出来。

### 11.2 `_FIG_LABEL` / `_TABLE_LABEL`

这两个是拿来从图块、表块自己的内容里找“标签”的。

比如图片块里可能写：

- `图3-1 神经元结构示意图`

表格块里可能写：

- `表3-1 神经元分类`

### 11.3 `_FULLWIDTH_DIGITS`

这个东西是把全角数字翻译成半角数字。

大白话解释：

有些解析结果里，数字不一定是标准半角 `123`，可能混进全角版本。

所以系统会先统一一下，避免：

- 正文里是一个写法
- 图表标签里是另一个写法
- 结果明明是同一个编号，却对不上

---

## 十二、`_normalize_ref()`：把编号统一成可比较格式

代码大意：

1. 先做全角转半角
2. 再把空格和各种横线删掉

比如：

```text
图 3 - 1
图3-1
图3－1
图3—1
```

最后都可能变成同一个标准 key，比如：

```text
31
```

或者某种统一去掉分隔符后的形式。

重点不是它最终长什么样，而是：

- 同一个编号，不同写法，最后尽量映射成同一个 key

---

## 十三、`_extract_refs()`：提取标准 key 和原始证据

这个函数返回的是：

```python
list[tuple[str, str]]
```

含义是：

1. 标准化后的 key
2. 原始匹配到的文本

比如：

```text
正文："如图3-1所示，..."

提取结果可能是：
[
  ("31", "如图3-1")
]
```

为什么两个都要？

因为：

- key 用来做匹配
- 原始文本用来当 `evidence`

---

## 十四、Rule 1 的完整流程

### 14.1 先扫图片块，建立图标签表

```python
for idx, chunk in img_chunks:
    for key, _label in _extract_refs(_FIG_LABEL, chunk.content[:300]):
        fig_label_map.setdefault(key, idx)
```

说明：

- 只看图片块前 300 个字符
- 从里面找类似“图3-1”
- 建立 `编号 -> chunk索引` 的映射

### 14.2 再扫表格块，建立表标签表

逻辑一样：

```python
table_label_map[key] = idx
```

### 14.3 最后扫正文块，查正文里有没有提图提表

```python
for idx, chunk in text_chunks:
    for key, evidence in _extract_refs(_FIG_REF, chunk.content):
        if key in fig_label_map:
            _link(idx, fig_label_map[key], "refers_to", evidence)
```

表格同理。

### 14.4 `_link()` 会补双向关系

它不是只加：

- 文本 -> 图片

它还会反过来再加：

- 图片 -> 文本

所以最终这两边都知道彼此有关联。

---

## 十五、Rule 1 的真实意义

大白话讲，`refers_to` 解决的是这种问题：

```text
正文说：
  “如图3-1所示……”

图在另一个 chunk 里
表也在另一个 chunk 里
```

如果没有这个关系：

- 用户问到正文
- 系统只召回正文
- 却带不出真正的图表

加了这个关系之后：

- 命中正文时，可以顺着 `refers_to` 找到图表
- 命中图表时，也能反向回到引用它的正文

所以它非常适合：

- 图文联动
- 表文联动
- 证据回溯

---

## 十六、Rule 2：`adjacent`，没有明确引用也给你一个附近关系

代码段大意：

```python
for idx, chunk in img_chunks + table_chunks:
    if chunk.related_ids:
        continue
    for back in range(idx - 1, max(idx - 5, -1), -1):
        prev = chunks[back]
        if not prev.is_image_chunk and not prev.is_table_chunk and prev.layer not in ("parent", "enhanced"):
            _link(idx, back, "adjacent", "邻近5块内")
            break
```

这里有几个关键点。

### 16.1 只处理图片块和表格块

因为这个规则是“媒体块去找它附近的文本块”。

### 16.2 `if chunk.related_ids: continue`

这行很关键。

它的意思是：

- 如果这个媒体块已经有相关关系了
- 就不再额外补 `adjacent`

大白话理解：

项目作者的意思不是“关系越多越好”，而是：

- 如果已经明确知道它引用了谁，或者已经有现成相关块
- 那就不要再用“离得近”这种更弱的规则添乱

### 16.3 只向前找，不向后找

搜索范围是：

- 从当前块往前看
- 最多回看 5 个块

为什么？

因为很多教材、文档里，图表通常是跟在说明文字后面出现的。

所以默认假设是：

- 图表前面的文字更可能是它的说明文字

### 16.4 找到第一个合格文本块就停

不是找所有附近文本，而是找最近的那一个。

这说明 `adjacent` 的设计非常保守：

- 它不是做“同区域全连接”
- 它只是给无明确引用的媒体块找一个最近落点

---

## 十七、Rule 2 的真实意义

这个关系主要解决的是：

- 有图
- 有表
- 但正文没写“如图X-X”
- 或者解析出来后编号丢了
- 或者 OCR 不稳定没识别出来

这时至少别让图表完全孤立。

系统会说：

- 好，既然我不知道你明确引用谁
- 那我至少把你挂到前面最近的一段正文上

所以 `adjacent` 本质上是一个兜底关系。

它比 `refers_to` 弱，但比完全没关系强。

---

## 十八、Rule 3：`sibling`，同一个 `parent_id` 下的局部共检索关系

这条规则的逻辑是：

1. 先按 `parent_id` 分组
2. 只拿 `layer == "child"` 的块
3. 同组内，把媒体块和文本块两两连起来

核心代码：

```python
if any(chunk.parent_id for chunk in chunks):
    parent_groups: dict[str, list[int]] = {}
    for idx, chunk in enumerate(chunks):
        if chunk.parent_id and chunk.layer == "child":
            parent_groups.setdefault(chunk.parent_id, []).append(idx)
```

然后：

```python
media_in_group = [idx for idx in group if chunks[idx].is_image_chunk or chunks[idx].is_table_chunk]
text_in_group = [idx for idx in group if not chunks[idx].is_image_chunk and not chunks[idx].is_table_chunk]
```

最后：

```python
for media_idx in media_in_group:
    for text_idx in text_in_group:
        _link(media_idx, text_idx, "sibling", "同parent_id")
```

---

## 十九、`sibling` 不是“所有兄弟互连”，而是“媒体对文本互连”

这个细节很重要。

很多人会误以为：

- 同一个 `parent_id` 下的所有 child 全部互连

但实际不是。

它只做：

- 媒体块 <-> 文本块

不做：

- 文本块 <-> 文本块
- 图片块 <-> 图片块
- 表格块 <-> 表格块

为什么这么设计？

因为它的目的不是“把一整章搅成全连接图”。

它的目的更具体：

- 让本章里的图片、表格，不至于和正文脱节

所以它是一个服务于“图文共检索”的局部关系。

---

## 二十、Rule 3 的真实意义

哪怕正文没有明确写：

- `如图3-1`
- `见表3-1`

只要它们在同一个父章节下面，系统也认为：

- 这张图
- 这个表
- 和这段正文

大概率属于同一个局部语义区。

所以 `sibling` 也是一种兜底，但比 `adjacent` 更有结构依据。

`adjacent` 依据是：

- 物理位置近

`sibling` 依据是：

- 结构归属近

---

## 二十一、Rule 4：enhanced 继承父 child 的关系

这是整个基础关系里特别关键的一条。

代码大意：

```python
child_id_to_idx = {chunk.id: idx for idx, chunk in enumerate(chunks) if chunk.layer == "child"}

for chunk in chunks:
    if chunk.layer != "enhanced" or not chunk.parent_id:
        continue
    parent_idx = child_id_to_idx.get(chunk.parent_id)
    if parent_idx is None:
        continue
    for relation in chunks[parent_idx].relations:
        _add_relation(...)
```

大白话解释：

### 21.1 enhanced 自己不重新跑四条基础规则

它不是把 enhanced 当成正文再重新做一遍：

- 图表引用识别
- 邻近关系
- sibling

而是直接说：

- 你的父 child 已经判断好了
- 你把父亲的关系照抄过来

### 21.2 enhanced 的 `parent_id` 指向的是 child

这里一定要注意：

- 对 `child` 来说，`parent_id` 通常指向章节 parent
- 对 `enhanced` 来说，`parent_id` 指向的是它对应的原始 child

也就是说，`parent_id` 在不同 layer 上含义不完全一样。

### 21.3 继承的是整条关系

继承时复制的不是只有 `target_id`，而是把这些都带过去：

- `target_id`
- `rel_type`
- `weight`
- `source`
- `evidence`

所以 enhanced 和它的 child，在关系网络上会非常相似。

---

## 二十二、为什么 enhanced 一定要继承关系

这个问题很重要。

因为在线检索里，经常会先命中 enhanced。

原因是：

- enhanced 更短
- 更像问句
- 更适合召回

但真正给用户展示时，又常常要回到 child。

如果 enhanced 不继承关系，会出现什么问题？

比如：

- enhanced 命中了“神经元组成部分”
- 但它自己不知道“如图3-1”对应哪张图
- 也不知道同章节表格是谁

那图文联动就断了。

继承之后：

- enhanced 命中也能带出跟 child 一样的关系网络

所以这是在线召回很重要的一环。

---

## 二十三、基础关系阶段的小结

到 `link_related_chunks()` 结束时，系统已经拿到一批不依赖向量的关系：

- `refers_to`
- `adjacent`
- `sibling`
- `enhanced` 继承来的这些关系

这些关系的特点是：

1. 很快
2. 可解释
3. 不依赖 embedding
4. 很适合图文表联动

但它们还不够。

因为它们解决的是：

- 明确引用
- 邻近
- 同结构归属

还没解决：

- 跨章节语义相近
- 步骤链
- 因果链
- 实体链

这些要到后面继续补。

## 二十四、第二批关系的入口：`confirm_pipeline()` 之后

虽然这份文档不展开整条入库链，但为了理解关系判断，顺序必须知道：

1. 先有 chunk
2. 先跑基础关系 `link_related_chunks()`
3. 再做 embedding
4. 再做语义关系
5. 再做流程关系
6. 再做因果关系
7. 再做实体物化和 mentions

为什么语义关系不能放前面？

因为它需要 embedding。

为什么 mentions 又更后面？

因为它依赖 enhanced 抽出的实体，再经过合并、定义生成、实体向量化后才能落库。

## 二十五、`link_semantic()`：用向量相似度补语义边

文件：

- `core/chunker/semantic_linker.py`

这一步的目标是解决：

- 两段话没写“如图”
- 也不在同一父章节
- 但意思很像

那系统也想把它们连起来。

---

## 二十六、`link_semantic()` 的默认参数

默认值在文件开头：

```python
DEFAULT_THRESHOLD = 0.85
DEFAULT_TOPK = 10
DEFAULT_DUP_THRESHOLD = 0.95
DEFAULT_SKIP_SAME_PARENT = True
DEFAULT_ENABLED = True
DEFAULT_BLOCK_SIZE = 256
```

这些参数非常值得逐个解释。

### 26.1 `threshold = 0.85`

低于这个相似度，不建边。

也就是说，项目作者认为：

- 0.85 以下不够像
- 不值得建立语义关系

### 26.2 `topk = 10`

每个 chunk 最多只保留前 10 个相似对象。

这很重要，因为不然关系会爆炸。

### 26.3 `dup_threshold = 0.95`

如果相似度已经高到 0.95 以上，就不只是“像”，而是近似重复，于是关系类型升级成：

- `duplicate_of`

否则就是：

- `semantic_similar`

### 26.4 `skip_same_parent = True`

默认跳过同一个 `parent_id` 下的 child。

大白话解释：

- 同一章里相邻段落本来就很像
- 如果也给它们全加语义边，图会非常乱
- 而且同章节局部关系前面已经有 `sibling` 这类规则兜底

所以语义关系主要想抓的是：

- 跨 parent 的相似内容

### 26.5 `block_size = 256`

这是 numpy 快路径下的分块计算大小，后面细讲。

---

## 二十七、`link_semantic()` 只处理 `child`

这行很关键：

```python
child_pairs = [(idx, chunk) for idx, chunk in enumerate(chunks) if chunk.layer == "child"]
```

它只拿：

- `layer == "child"`

不拿：

- `parent`
- `enhanced`

为什么？

原因很现实：

1. `child` 是真实正文载体
2. `enhanced` 本来就依附 child
3. 如果 child 和 enhanced 都参与语义建边，重复量会非常大

所以设计思路是：

- 语义边先在 child 层建立
- enhanced 需要基础关系时靠继承
- 检索时再通过 parent-child/hydrate 做展示映射

这和前面基础关系只主要在 child 上跑，是一致的。

## 三十、两条执行路径：numpy 快路径和纯 Python 回退

`link_semantic()` 不是只有一种实现，它会先尝试：

- `_link_semantic_numpy()`

如果拿不到，再回退到：

- `_link_semantic_python()`

这说明它的设计目标是：

1. 能快就快
2. 没装 numpy 也能跑

## 三十一、纯 Python 路径 `_link_semantic_python()`

它的逻辑可以拆成四步。

### 31.1 先给每个 child embedding 算范数

```python
norms = {
    global_idx: math.sqrt(sum(value * value for value in embeddings[global_idx])) or 1.0
    for global_idx, _chunk in child_pairs
}
```

也就是先把向量长度算出来，后面算余弦相似度要用。

### 31.2 初始化候选表

```python
candidates: list[list[tuple[float, int]]] = [[] for _global_idx, _chunk in child_pairs]
```

大白话解释：

- 给每个 child 准备一个候选列表
- 列表里存“某个候选有多像，它的位置是多少”

### 31.3 两两计算相似度

双层循环：

```python
for pos_i, (global_i, chunk_i) in enumerate(child_pairs[:-1]):
    for pos_j in range(pos_i + 1, len(child_pairs)):
```

注意它不是全矩阵重复算，而是只算上三角。

这比较省。

### 31.4 同 parent 默认跳过

```python
if skip_same_parent and chunk_i.parent_id and chunk_i.parent_id == chunk_j.parent_id:
    continue
```

这就是前面说的默认策略。

### 31.5 相似度过阈值才互相记进候选

```python
if score >= threshold:
    candidates[pos_i].append((score, pos_j))
    candidates[pos_j].append((score, pos_i))
```

这里也能看到，它是双向记候选。

不是先决定边类型，而是先收集“谁像谁”。

真正加边在下一步。

这个路径更适合大批量数据。

## 三十三、先看 numpy 路径的几个开关和退路

### 33.1 可以通过环境变量强制关掉

```python
if os.getenv("LINKER_SEMANTIC_NUMPY_ENABLED", "true").lower() in {"0", "false", "off"}:
    return None
```

### 33.2 import numpy 失败就回退

```python
try:
    import numpy as np
except Exception:
    return None
```

### 33.3 embedding 转矩阵失败也回退

```python
try:
    matrix = np.asarray(...)
except (TypeError, ValueError):
    return None
```

### 33.4 维度不对也回退

```python
if matrix.ndim != 2:
    return None
```

所以这里不是“失败就报错”，而是：

- 尽量优雅回退到 Python 实现

---

## 三十四、numpy 路径怎么做归一化

```python
norms = np.linalg.norm(matrix, axis=1)
norms[norms == 0.0] = 1.0
matrix = matrix / norms[:, None]
```

大白话解释：

- 每一行是一个 chunk 的 embedding
- 先算每行长度
- 长度为 0 的，强行当 1，避免除零
- 然后每个向量都除以自己的长度

归一化以后：

- 点积就等于余弦相似度

后面就能用矩阵乘法一把算很多相似度。

---

## 三十五、为什么还有 `parent_positions`

如果启用了 `skip_same_parent`，代码会先把：

- 每个 `parent_id`
- 对应哪些 child 位置

做成映射：

```python
parent_positions: dict[str, list[int]] = {}
```

这样后面处理某个 child 时，如果它有父章节，就能快速把“同 parent 的那些位置”一起屏蔽掉。

这比一对一再去比较 parent_id 更高效。

---

## 三十六、为什么还要分块 `block_size`

代码：

```python
for start in range(0, len(child_pairs), block_size):
    stop = min(start + block_size, len(child_pairs))
    similarities = matrix[start:stop] @ matrix.T
```

大白话解释：

- 它没有一次把整个 `N x N` 相似度矩阵全部持久保留
- 而是按块来算

为什么？

因为 child 很多时，整矩阵非常大。

分块的好处：

1. 内存更稳
2. 还能保留矩阵乘法的速度优势

默认块大小是：

- `256`

也可以通过环境变量：

- `LINKER_SEMANTIC_BLOCK_SIZE`

调整。

---

## 三十七、numpy 路径里每个 child 怎么筛候选

对于块里的每一行分数：

```python
mask = scores >= threshold
mask[pos_i] = False
```

先做两件事：

1. 只保留大于等于阈值的
2. 把自己排掉

然后如果要跳同 parent：

```python
if skip_same_parent and chunk_i.parent_id:
    mask[parent_positions.get(chunk_i.parent_id, [])] = False
```

再做一轮屏蔽。

最后：

```python
candidate_positions = np.flatnonzero(mask)
candidate_scores = scores[candidate_positions]
order = np.argsort(-candidate_scores, kind="stable")[:topk]
```

这一步的意思就是：

- 找出所有合格候选
- 按分数从高到低排序
- 只保留前 `topk`

---

## 三十八、真正落边：`_add_semantic_relations_for_positions()`

不管是 Python 路径还是 numpy 路径，最后都会走到这里。

逻辑非常关键：

```python
rel_type = "duplicate_of" if score >= dup_threshold else "semantic_similar"
```

也就是：

- `score >= 0.95` -> `duplicate_of`
- `0.85 <= score < 0.95` -> `semantic_similar`

然后：

```python
add_bidirectional_relation(
    chunk_i,
    chunk_j,
    rel_type=rel_type,
    weight=score,
    source="embedding",
    evidence=f"cos={score:.3f}",
)
```

这里又有几个关键细节。

### 38.1 是双向边

语义相似在这里被视为对称关系。

### 38.2 `source = "embedding"`

明确说明这不是规则来的，是向量来的。

### 38.3 `evidence = "cos=0.xxx"`

证据不是正文片段，而是相似度数值本身。

这是很合理的，因为语义边的证据本来就不是一句显式文本，而是：

- “向量上很像”

### 38.4 计数方式只看一边有没有新增

代码里：

```python
before = len(chunk_i.relations)
...
if len(chunk_i.relations) > before:
    added += 1
```

大白话解释：

- 虽然实际加的是双向边
- 但计数时是按“新增一对关系”记一次

所以返回值 `added` 不是“关系总条数”，更接近“新增关系对数”。

## 三十九、语义关系的小结

到这一步，系统新增了两类语义边：

- `semantic_similar`
- `duplicate_of`

它们的核心特点是：

1. 只在 `child` 层建立
2. 默认跳过同 parent
3. 用 embedding 判断
4. 双向
5. evidence 记录的是相似度

这一步解决的是“跨局部结构的语义连通性”。

## 四十、`detect_procedure_chunks()`：先识别哪些块像步骤

文件：

- `core/chunker/procedure_linker.py`

这里要注意：

流程关系不是直接上来就连边，而是分两步：

1. 先识别“这是不是流程块”
2. 再在同 parent 下排序连边

这比直接硬连更稳。

## 四十一、流程识别只看什么块

代码第一行就限制了范围：

```python
if chunk.layer != "child" or chunk.is_table_chunk or chunk.is_image_chunk:
    continue
```

也就是说，只看：

- `child`
- 纯文本

不看：

- parent
- enhanced
- 图块
- 表块

原因很简单：

- 步骤通常写在正文里
- 图表本身一般不是“步骤块”

---

## 四十二、只取前 80 个字符判断

```python
text = chunk.content[:80].strip().lower()
```

这说明设计假设是：

- 步骤标志通常在开头

比如：

- `第一步：`
- `第2步：`
- `1.`
- `首先`
- `然后`

所以没必要扫全文。

---

## 四十三、`_ORDER_RE`：识别带顺序号的步骤

这个正则虽然终端显示可能乱码，但它的意图很清楚：

- 匹配开头的步骤编号

大致想覆盖这类形式：

- `第一步`
- `第1步`
- `1.`
- `1)`

配合 `_parse_order()` 和 `_CN_NUM`，它可以把：

- 阿拉伯数字
- 中文数字

统一转成整数顺序号。

比如：

- `一` -> `1`
- `二` -> `2`
- `10` -> `10`

## 四十四、`_TEMPORAL_HEAD`：识别没有编号但带时间顺序词的块

这个元组里放的是一批“顺序提示词”。

从代码语义上看，主要是这类词：

- 首先
- 接着
- 然后
- 随后
- 最后
- 之后
- step

大白话解释：

有些教材不会严谨写：

- 第一步
- 第二步

而是写：

- 首先……
- 然后……
- 最后……

那系统也把它当成步骤块。

## 四十五、识别流程块时，标记了什么字段

如果命中编号规则：

```python
chunk.is_procedure_chunk = True
chunk.procedure_order = _parse_order(match.group(1))
```

如果命中时间词规则：

```python
chunk.is_procedure_chunk = True
```

但这时通常没有明确的 `procedure_order`。

也就是说，流程块分两种：

1. 有明确顺序号的
2. 只有顺序语气、没有顺序号的

后面排序时会统一处理。

---

## 四十六、`link_procedure()`：在同 parent 下把步骤串起来

真正连边的函数是：

- `link_procedure(chunks)`

它也分几步。

### 46.1 先按 `parent_id` 分组

```python
for chunk in chunks:
    if chunk.layer == "child" and chunk.parent_id:
        groups.setdefault(chunk.parent_id, []).append(chunk)
```

这说明：

- 步骤关系不跨父章节
- 每个父章节内部单独串自己的步骤链

### 46.2 只取被识别成流程块的 child

```python
proc = [chunk for chunk in group if chunk.is_procedure_chunk]
```

### 46.3 排序规则

```python
proc.sort(key=lambda item: (item.procedure_order or 999, item.chunk_index))
```

这行非常值得细讲。

#### 第一排序键：`procedure_order`

有顺序号的排前面。

#### 没顺序号的当 `999`

也就是：

- `首先`
- `然后`
- `最后`

这类没有明确编号的，默认排到“所有已编号步骤后面”

#### 第二排序键：`chunk_index`

如果两块都没有顺序号，或者顺序号一样，就用原始 chunk 顺序兜底。

所以作者的策略不是“强行理解复杂步骤结构”，而是：

- 能按编号就按编号
- 不能按编号就按原文顺序

---

## 四十七、真正加的是 `next_step` 和 `prev_step`

相邻两步之间，会加两条定向边：

```python
add_relation(prev, curr.id, "next_step", ...)
add_relation(curr, prev.id, "prev_step", ...)
```

这不是双向同类型边，而是：

- A 指向 B：`next_step`
- B 指向 A：`prev_step`

这很合理，因为“下一步”和“上一步”本来就是不同语义。

### 47.1 evidence 存的是对方前 20 个字符

```python
evidence=curr.content[:20]
evidence=prev.content[:20]
```

也就是：

- 这一步的下一步是谁
- 用对方开头内容做简短提示

---

## 四十八、流程关系的小结

这一套最终解决的是：

- 用户命中某一步
- 系统可以顺着关系给出前一步、后一步、整条步骤链

它的特点是：

1. 只作用于 `child` 文本块
2. 不跨 `parent_id`
3. 先识别再连边
4. 有编号优先按编号
5. 没编号再按原始顺序

## 四十九、`link_causal()`：因果关系是最保守的一类规则

文件：

- `core/chunker/causal_linker.py`

如果说流程关系已经算比较简单明确，那因果关系更保守。

它并不会做复杂语义理解。

它做的是一种非常克制的启发式：

- 看到因果触发词
- 就尝试把“前一个 child”当原因

仅此而已。

---

## 五十、因果词表 `_CAUSAL_MARKERS`

从代码语义上看，覆盖的是这类词：

- 因为
- 由于
- 鉴于
- 导致
- 使得
- 引起
- 造成
- 所以
- 因此

大白话解释：

只要当前块前 200 个字符里出现这些词，系统就怀疑：

- 这块可能在表达因果

但只是“怀疑”，不是说它真的理解了复杂因果句法。

---

## 五十一、为什么只看前 200 个字符

代码：

```python
if not any(marker in current.content[:200] for marker in _CAUSAL_MARKERS):
    continue
```

理由和步骤识别类似：

- 因果提示词通常也在句子前半段很容易出现
- 没必要扫太长
- 这样速度也更稳

---

## 五十二、它只在 `child_chunks` 线性序列里看“前一个”

代码第一步先收集：

```python
child_chunks = [chunk for chunk in chunks if chunk.layer == "child"]
```

然后在这个列表里遍历：

```python
previous = child_chunks[idx - 1]
```

这意味着什么？

意味着它不是：

- 找最相关前文
- 找同句主语宾语
- 找整段因果图

它就是最保守的：

- 当前块如果像“结果”
- 那前一个 child 就候选为“原因”

---

## 五十三、为什么它还要检查同 `parent_id`

代码：

```python
if current.parent_id and previous.parent_id and current.parent_id != previous.parent_id:
    continue
```

这说明作者很怕误连。

大白话解释：

- 如果前一个 child 已经属于另一个章节
- 那就算当前块里有“因此”“由于”
- 也不跨章节去硬连

所以因果边只在局部章节内部做。

---

## 五十四、真正加的是 `cause_of` / `effect_of`

加边逻辑：

```python
add_relation(previous, current.id, "cause_of", weight=0.7, source="rule", evidence=current.content[:20])
add_relation(current, previous.id, "effect_of", weight=0.7, source="rule", evidence=previous.content[:20])
```

这又有几个细节。

### 54.1 权重固定是 `0.7`

不是 `1.0`。

这非常能说明作者态度：

- 他知道这条规则只是启发式
- 没有非常强的确定性

所以故意给了一个中等偏保守的权重。

### 54.2 一前一后两条不同类型边

- 前块 -> 当前块：`cause_of`
- 当前块 -> 前块：`effect_of`

### 54.3 evidence 仍然只是短片段

也是各取对方前 20 个字符。

---

## 五十五、为什么说因果规则“保守但粗糙”

它保守在：

1. 只看 child
2. 只看前 200 字
3. 只看前一个 child
4. 不跨 parent
5. 权重只有 0.7

它粗糙在：

- 真实因果关系不一定就在前一块
- 有因果词不等于一定真的是一条高质量因果边

所以你要明白：

这个模块不是在做“论文级因果抽取”。

它是在做：

- 一个低成本、可解释、够用的局部因果提示网络

---

## 五十六、`materialize_entities()`：这里补的是 `mentions`

文件：

- `core/kg/extraction_pipeline.py`

很多人一讲关系判断，只盯着 chunk 对 chunk。

但这个项目里还有一类很重要的关系：

- `mentions`

它是：

- `chunk -> entity`

这个关系特别关键，因为 Graph RAG 扩图时会专门处理它。

---

## 五十七、实体物化不是从所有 chunk 提，而是从 `enhanced` 提

代码：

```python
for chunk in chunks:
    if chunk.layer != "enhanced":
        continue
    for entity in chunk.extracted_entities:
        raw.append((entity, chunk.parent_id or chunk.id))
```

这说明：

### 57.1 只从 enhanced 上取 `extracted_entities`

不是直接从 child 取。

### 57.2 记入 raw 时，source chunk 用的是：

- `chunk.parent_id`
- 或者 `chunk.id`

大白话解释：

通常 enhanced 的 `parent_id` 指向它对应的 child。

所以这里的意思基本就是：

- enhanced 抽出了实体
- 但最终把这个实体归属到它对应的 child 上

这很合理，因为真正正文载体是 child。

---

## 五十八、实体合并之后，怎么回挂到 chunk

实体会先经过：

- `EntityMerger().merge(kb_id, raw)`

然后生成定义：

- `generate_definition(entity, chunk_map)`

再做实体 embedding：

- `embed_texts([f"{entity.name} {entity.definition or ''}" for entity in entities])`

之后按 source chunk 反向归组：

```python
by_chunk: dict[str, list[Entity]] = {}
for entity in entities:
    for chunk_id in entity.source_chunks:
        by_chunk.setdefault(chunk_id, []).append(entity)
```

最后真正回挂关系：

```python
for chunk in chunks:
    for entity in by_chunk.get(chunk.id, []):
        add_relation(chunk, entity.id, rel_type="mentions", source="entity", evidence=entity.name[:20])
```

这里非常关键。

---

## 五十九、`mentions` 的本质是什么

它不是：

- chunk 指向另一个 chunk

而是：

- chunk 指向 entity

所以你要把它理解成一条“跨表关系”：

- chunk 侧写在 `chunk.relations`
- 实体侧最终会写到 `entities`
- 还有一张专门的 `entity_mentions`

也就是说，`mentions` 实际上把：

- chunk 图
- entity 图

接起来了。

---

## 六十、为什么 `mentions` 很重要

因为如果没有它，Graph RAG 很难做“概念跳转”。

比如：

- 命中一个讲“神经元”的 chunk
- 你想扩到所有提到“突触”“动作电位”的相关块

那就需要先有：

- chunk -> entity (`mentions`)

然后实体表里再知道：

- 哪些 chunk 也提到了这个 entity

这就是后面 Graph RAG 的关键桥梁。

---

## 六十一、关系最终怎么被 Graph RAG 用掉：`graph_expand()`

文件：

- `core/rag/graph_expander.py`

这个文件不是“判断关系”，而是“消费关系”。

但不讲它，你前面那些关系为什么有价值就说不透。

---

## 六十二、Graph RAG 有一套关系优先级：`INTENT_REL_PRIORITY`

代码里定义了不同意图下，关系类型的重要性。

比如：

### 62.1 `concept`

```python
{"mentions": 1.0, "sibling": 0.8, "semantic_similar": 0.7}
```

说明问概念时，最看重：

1. `mentions`
2. `sibling`
3. `semantic_similar`

### 62.2 `procedure`

```python
{"next_step": 1.0, "prev_step": 0.9, "sibling": 0.6}
```

说明问流程时，最看重：

1. `next_step`
2. `prev_step`

### 62.3 `data`

更偏好：

- `refers_to`
- `sibling`

### 62.4 `visual`

最看重：

- `refers_to`
- `adjacent`

这说明前面每种关系不是“随便建着玩”，而是后面真的按意图差异化消费。

---

## 六十三、`graph_expand()` 的输入是什么

函数签名：

```python
graph_expand(
    seeds: list[str],
    kb_id: str,
    intent: str,
    *,
    max_hops: int = 2,
    max_neighbors: int = 50,
)
```

大白话解释：

- `seeds`：种子 chunk id
- `kb_id`：在哪个知识库里扩
- `intent`：这是概念问题、流程问题、图表问题还是通用问题
- `max_hops`：最多扩几跳
- `max_neighbors`：最多扩多少邻居

---

## 六十四、Graph RAG 扩图时怎么读关系库

核心 SQL：

```sql
SELECT dst_id::text, rel_type, weight
FROM chunk_relations
WHERE kb_id = %s AND src_id::text = %s
ORDER BY weight DESC
LIMIT %s
```

这说明：

前面那些落在 `chunk.relations` 里的关系，最后会被正式写到：

- `chunk_relations`

在线扩图就是直接查这张表。

所以“关系判断”不是内存里临时玩一下，而是实打实入库、实打实参与查询。

---

## 六十五、Graph RAG 的扩图分数怎么算

代码：

```python
score = float(weight) * float(rel_weight) * (0.6**hop) * rel_priority.get(rel_type, 0.5)
```

拆开看：

1. 当前路径累计权重 `weight`
2. 当前边自己的权重 `rel_weight`
3. 跳数衰减 `0.6**hop`
4. 关系类型优先级 `rel_priority`

大白话解释：

- 边本身越强，得分越高
- 跳得越远，得分越低
- 关系类型越符合当前 intent，得分越高

这就把“关系判断阶段给的 `weight`”和“Graph RAG 扩图策略”接上了。

---

## 六十六、`mentions` 在 Graph RAG 里是特殊分支

这段最值得看：

```python
if rel_type == "mentions":
    SELECT chunk_id::text
    FROM entity_mentions
    WHERE kb_id = %s AND entity_id::text = %s
```

什么意思？

如果当前边是：

- `chunk -> entity`

那 Graph RAG 不会把 entity 当最终文本结果直接停住。

它会继续去 `entity_mentions` 里查：

- 还有哪些 chunk 也提到了这个 entity

然后把这些 chunk 再推进 frontier。

这就形成了：

```text
chunk A
  -> mentions
entity X
  -> mentioned_in
chunk B / chunk C / chunk D
```

也就是说，`mentions` 不是终点，而是“实体桥”。

---

## 六十七、为什么 `mentions` 是关系网络里最像“图谱入口”的边

因为其他大部分边，本质还是：

- chunk 到 chunk

只有 `mentions` 明显把系统引向了：

- entity 维度

所以如果你要理解 Graph RAG 和普通 RAG 的差别，`mentions` 是必须搞明白的一条边。

普通 RAG 更像：

- 直接找相似 chunk

Graph RAG 更像：

- 命中 chunk
- 顺着关系走
- 走到 entity
- 再从 entity 回到别的 chunk

这就是“图扩展”的味道。

## 六十八、把所有关系类型放在一起再看一遍

下面按“来源”和“作用”整理一遍。

### 68.1 基础规则关系

#### `refers_to`

- 来源：`link_related_chunks()`
- 判断方式：正文显式引用图表编号
- 常见用途：图文联动、表文联动、visual/data intent

#### `adjacent`

- 来源：`link_related_chunks()`
- 判断方式：媒体块往前 5 块找最近正文
- 常见用途：图表兜底挂靠、visual intent

#### `sibling`

- 来源：`link_related_chunks()`
- 判断方式：同 `parent_id` 下媒体块和文本块互连
- 常见用途：同章节局部共检索、concept/data/procedure 辅助扩图

### 68.2 语义关系

#### `semantic_similar`

- 来源：`link_semantic()`
- 判断方式：embedding 相似度 `0.85 <= score < 0.95`
- 常见用途：跨章节相似内容扩展

#### `duplicate_of`

- 来源：`link_semantic()`
- 判断方式：embedding 相似度 `score >= 0.95`
- 常见用途：近重复识别

### 68.3 流程关系

#### `next_step`

- 来源：`link_procedure()`
- 判断方式：同父章节下，步骤排序后指向下一步
- 常见用途：procedure intent

#### `prev_step`

- 来源：`link_procedure()`
- 判断方式：同父章节下，步骤排序后反向指向上一步
- 常见用途：procedure intent

### 68.4 因果关系

#### `cause_of`

- 来源：`link_causal()`
- 判断方式：当前块出现因果词，前一个 child 当原因

#### `effect_of`

- 来源：`link_causal()`
- 判断方式：当前块反向指回前一个 child

### 68.5 实体关系

#### `mentions`

- 来源：`materialize_entities()`
- 判断方式：enhanced 抽实体，合并后挂回对应 child
- 常见用途：concept intent、Graph RAG 实体桥扩展

## 六十九、所有关系都写在哪里

要分两层看。

### 69.1 内存态

先写在：

- `chunk.relations`

这是离线处理阶段在 Python 对象上的表现。

### 69.2 落库态

后面会写入关系表：

- `chunk_relations`

实体相关还会落到：

- `entities`
- `entity_mentions`

所以最终数据库里你可以这样理解：

### 69.3 `chunk_relations`

存：

- chunk 到 chunk
- 以及某些实现上表现为 chunk 到 entity id 的关系视图

### 69.4 `entity_mentions`

专门存：

- 实体被哪些 chunk 提到

这个表是 Graph RAG 处理 `mentions` 时的重要补表。

## 七十、真实调用链再按代码意义串一次

你可以把关系判断想成五层。

### 第 1 层：局部可解释关系

- `link_related_chunks()`

解决：

- 图表明确引用
- 图表邻近挂靠
- 同章节媒体与正文联动
- enhanced 继承

### 第 2 层：全局语义关系

- `link_semantic()`

解决：

- 跨章节语义相近
- 近重复内容

### 第 3 层：流程链

- `detect_procedure_chunks()`
- `link_procedure()`

解决：

- 步骤前后顺序

### 第 4 层：因果链

- `link_causal()`

解决：

- 局部因果提示

### 第 5 层：实体桥

- `materialize_entities()`

解决：

- chunk 到 entity
- entity 再回到别的 chunk

这五层叠起来，才构成这个项目完整的关系网络。

## 七十一、几个最容易误解的点

### 71.1 不是所有关系都在一个地方判断

不是只有 `linker.py`。

关系判断是分散在多个模块里的。

### 71.2 不是所有层都参与每一种关系

最核心的判断层是：

- `child`

`enhanced` 更多是：

- 继承
- 抽实体来源
- 在线召回入口

### 71.3 不是所有关系都是双向同类型

双向同类型的典型有：

- `semantic_similar`
- `duplicate_of`
- `refers_to`
- `sibling`
- `adjacent`

定向成对但类型不同的有：

- `next_step` / `prev_step`
- `cause_of` / `effect_of`

### 71.4 `mentions` 不是 chunk 对 chunk

这是最容易漏掉的点之一。

它是：

- chunk 指向 entity

后面再由 `entity_mentions` 把 entity 接回 chunk。

### 71.5 因果关系不是高智能语义推理

它只是保守启发式。

不要把它脑补成完整因果抽取系统。

## 七十二、如果只用一句大白话总结整个设计

这个项目的“关系判断”本质上是在做三件事：

1. 先用最稳的规则，把图、表、同章节内容、enhanced 和 child 连起来
2. 再用 embedding 把跨章节但语义相近的内容连起来
3. 再用步骤、因果、实体，把“怎么做”“为什么会这样”“这个概念还在哪出现过”这些横向线索补出来

最终目的不是为了画图好看，而是为了让在线检索别只会“找到一段像的文字”，而是能顺着关系把更完整的上下文一起带出来。
