# 教材知识库与 RAG 产品技术招标参数

## 1. 建设目标

本项目拟建设面向教材、教辅、讲义及扫描型 PDF 文档的知识库与检索增强问答系统。系统应覆盖教材文档上传、云端/本地协同解析、教材分片、内容清洗、三层切片、向量化、知识入库、混合召回、重排、证据整合、问答生成、来源回溯和运行监控等完整链路。

系统不得仅提供简单的 PDF 转文本或普通向量检索能力，应突出教材类文档的章节结构、页码、图表、表格、图片、知识点、流程关系、因果关系和证据回溯能力。

## 2. 总体技术链路要求

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 2.1 | 完整入库链路 | 应支持从原始教材 PDF 到知识库数据的完整处理，包括文件上传、PDF 体检、分片解析、解析结果合并、清洗、三层切片、质量门控、向量化、关系构建、实体物化和向量数据库写入。 |
| 2.2 | 完整问答链路 | 应支持用户问题输入后的在线检索增强问答，包括查询理解、混合召回、候选融合、增强切片折回、关系扩展、重排、上下文构造、答案生成、引用对齐和评分。 |
| 2.3 | 教材证据闭环 | 系统输出答案时，应能够回溯至原教材文件、页码、切片编号、图片/表格资源、图表编号及相关上下文。 |
| 2.4 | 多知识库隔离 | 应支持按知识库隔离文档、切片、向量、关系、实体和问答记录，支持知识库级管理和检索过滤。 |
| 2.5 | 可观测运行 | 应提供解析、切片、向量化、入库、召回、重排、生成等阶段的耗时、数量、失败、重试、并发和 provider 指标。 |

## 3. 教材分片与解析参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 3.1 | 大教材自动分片 | 系统应支持百页级、数百 MB 级教材 PDF 自动分片解析。对于大体量教材，应按页码范围拆分为多个 shard 并发提交解析，解析完成后按原始全局页码合并。 |
| 3.2 | 分片触发条件 | 应支持按文件大小和页数触发分片。当前验证链路中，Document Mind provider 默认在 `file_size > 150 MB` 或 `page_count > 50` 时启用分片。 |
| 3.3 | 分片粒度 | 应支持配置每片页数上限。当前验证链路采用 `33 页/片` 作为稳定分片配置，并保留最小页数、目标波次等可调参数。 |
| 3.4 | 分片并发 | 应支持并发解析多个分片。当前验证链路采用 `4 worker` 并发解析，并受分片数量、解析 provider 并发上限和凭证池容量共同约束。 |
| 3.5 | 页码一致性 | 分片解析完成后，所有内容块应恢复为原 PDF 的全局页码，用户侧页码展示应与原教材一致。 |
| 3.6 | 文件名一致性 | 分片过程不得把 shard 文件名写入最终证据。最终 `source_file` 应始终保留原始上传文件名。 |
| 3.7 | 失败处理 | 任一分片解析失败时，应能够明确返回失败 shard 编号、原始页码范围和失败原因，便于定位问题。 |
| 3.8 | 分片保存优化 | 应支持分片 PDF 轻量保存策略，以降低分片生成耗时。当前验证链路中 Document Mind 分片使用轻量 cleanup 与 deflate 组合。 |
| 3.9 | 分片调度优化 | 应支持基于页数、文件大小、采样文本和 provider 能力进行分片调度；可保留复杂度感知分片、重片优先等实验开关，但默认策略应以稳定验证结果为准。 |

## 4. 解析多样化能力参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 4.1 | 多 provider 解析 | 系统应支持多解析 provider 接入，包括云端 OCR/版面解析服务、Document Mind 类结构化解析服务和兼容扩展 provider。 |
| 4.2 | 扫描教材解析 | 对扫描版教材，应具备 OCR、页面版面分析、图片识别和图文区域提取能力。 |
| 4.3 | 图文混排解析 | 对图文混排教材，应能同时解析正文、标题、图片、表格、页码和版面结构，不得只保留纯文本。 |
| 4.4 | Markdown 输出 | 应支持结构化 Markdown 解析输出，保留标题层级、段落、列表、表格和图片标记。 |
| 4.5 | 视觉版面输出 | 应支持获取视觉版面信息，包括页面图片、图片区域、表格区域、版面位置和页面级视觉证据。 |
| 4.6 | 图片块生成 | 解析结果中应将教材图片、图示、流程图等转换为独立图片内容块，并保留本地路径、远程 URL 或 data URL 等资源引用。 |
| 4.7 | 表格块生成 | 解析结果中应将教材表格转换为独立表格内容块，保留表格正文、表号、页码和上下文。 |
| 4.8 | 图表编号识别 | 应识别并保留教材中的图号、表号等编号信息，如“图 1-3-3”“表 2-1”等。 |
| 4.9 | 解析增强可控 | 解析 provider 内置 LLM/VLM 增强能力应可配置启停。当前验证链路中已验证关闭 provider 托管 LLM/VLM 后仍可完成完整入库，并保留 `markdown,visualLayoutInfo` 输出。 |
| 4.10 | 解析结果指标 | 解析阶段应输出 provider、分片数、完成分片数、轮询次数、输出块数量、图片块数量、表格块数量、解析耗时等指标。 |

## 5. 当前解析验证链路参考参数

以下参数可作为投标产品能力验证或验收基线参考，具体数值可按部署环境调整。

| 参数类别 | 当前验证值/行为 | 说明 |
| --- | --- | --- |
| 大文件分片触发 | `>150 MB` 或 `>50 页` | Document Mind provider 默认分片触发条件。 |
| 稳定分片粒度 | `33 页/片` | 当前大教材解析验证中的稳定配置。 |
| 分片并发 | `4 worker` | 当前验证链路中解析并发稳定档。 |
| 单 key 并发 | `1 inflight/key` | 控制 provider 限流风险。 |
| 输出格式 | `markdown,visualLayoutInfo` | 同时保留结构化正文和视觉版面证据。 |
| provider 托管增强 | 默认可关闭 | 关闭后仍保留完整入库能力，减少解析长尾耗时。 |
| 真实教材输出示例 | 约 `18500` blocks、`475` image blocks、`81` table blocks | 代表性教材验证中，No-LLM + 视觉版面输出未压垮后续清洗和切片链路。 |
| 完整入库耗时示例 | 最优 warm run 约 `212885 ms` | 当前验证链路包含 parse、chunk、embedding、export 等完整阶段。 |

## 6. 教材三层切片参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 6.1 | 三层切片结构 | 系统应支持 `parent`、`child`、`enhanced` 三层切片结构，分别承担章节导航、检索证据和语义增强召回职责。 |
| 6.2 | Parent 结构层 | `parent` 应来自标题、章节和教材结构，用于上下文容器、章节导航和结果分组，不应作为最终答案的唯一事实证据。 |
| 6.3 | Child 证据层 | `child` 应作为最终证据入口，承载教材正文、表格、图片说明、页码、来源文件、图表编号和切片编号。 |
| 6.4 | Enhanced 增强层 | `enhanced` 应由文本摘要、问题化表达、片段补全、表格摘要、图片描述等增强内容构成，用于提高召回率。 |
| 6.5 | Enhanced 折回 | `enhanced` 命中后，应折回对应 `child`，最终答案引用应落在原始教材证据切片上。 |
| 6.6 | 文本切片 | 应支持按教材章节、段落、知识点和上下文依赖进行文本切片，避免单纯固定长度切片造成知识点割裂。 |
| 6.7 | 表格切片 | 应支持表格独立切片，并为表格生成摘要增强切片，提升表格问答召回能力。 |
| 6.8 | 图片切片 | 应支持图片独立切片，并可结合 VL 模型或文本 fallback 生成图片描述增强切片。 |
| 6.9 | 片段补全 | 对“如上所述”“见图”等上下文依赖片段，应支持片段增强，补充必要上下文以提高召回质量。 |
| 6.10 | 顺序稳定 | 增强任务可并发执行，但最终切片合并顺序应保持与预留 slot 一致，确保入库结果稳定。 |
| 6.11 | 切片关系 | 应自动构建父子关系、相邻关系、语义相似关系、引用关系、流程关系、因果关系等关系边。 |
| 6.12 | 质量门控 | 切片后应执行质量门控，过滤明显无效、过短、异常或不适合作为知识单元的内容，同时保留图片和表格证据。 |

## 7. 当前三层切片验证链路参考参数

| 参数类别 | 当前验证值/行为 | 说明 |
| --- | --- | --- |
| 增强模式 | `parallel_ordered` | 并发增强，但按原 slot 顺序合并。 |
| 文本增强 worker | `16` | 普通文本与片段增强软上限。 |
| 表格增强 worker | `3` | 表格摘要增强软上限。 |
| 图片增强 worker | `4` | 图片/VL 描述增强软上限。 |
| 全局增强并发 | `70` | 当前真实教材验证中的稳定并发档。 |
| LLM client 复用 | `true` | 降低重复 client/连接池初始化开销。 |
| key 池支持 | `LLM_API_KEY_POOL`、`VL_API_KEY_POOL` | 支持多 key 轮转、限流冷却和匿名指标。 |
| 最新验证指标示例 | `enhancePeakConcurrency=70`、`enhanceFailures=0`、`throttle/retry=0` | 真实教材任务中验证三层增强并发稳定。 |

## 8. 向量化与入库参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 8.1 | 批量向量化 | 应支持对教材切片进行批量向量化，保持结果顺序与原切片顺序一致。 |
| 8.2 | 向量化对象 | 应支持对 `child`、`enhanced`、表格摘要、图片描述等内容向量化。 |
| 8.3 | 参数可配置 | 向量模型、API 地址、批大小、并发数、重试次数、超时时间和 key 池应支持配置。 |
| 8.4 | 并发控制 | 应支持有界并发，避免对 embedding provider 产生不可控请求洪峰。 |
| 8.5 | 失败重试 | 单批向量化失败时，应支持批级重试，不应直接重启整篇教材入库任务。 |
| 8.6 | 向量数据库 | 应支持 pgvector 或同等级向量数据库，提供向量相似度检索、元数据过滤、知识库过滤和批量写入。 |
| 8.7 | 关系写入 | 除切片向量外，应同步写入 `chunk_relations`、`kg_triples`、`entities`、`entity_mentions` 等关系与图谱数据。 |
| 8.8 | 入库一致性 | 文档、切片、向量、关系、实体和三元组应保持一致写入，失败时应记录任务状态和失败原因。 |
| 8.9 | 批量写入优化 | 应支持批量写入策略，如 COPY、批量 insert、临时表等方式，提高大教材入库效率。 |

## 10. 混合召回参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 10.1 | 多路召回 | 应支持向量语义召回、关键词/BM25 召回、结构化过滤召回、图表编号直达召回、关系扩展召回和 Graph RAG 召回。 |
| 10.2 | 向量召回 | 应基于 pgvector 等向量数据库执行 dense recall，默认检索层级应覆盖 `child + enhanced`。 |
| 10.3 | 关键词召回 | 应支持 BM25 或同等级关键词召回，用于教材术语、编号、标题和原文短语匹配。 |
| 10.4 | RRF 融合 | 应支持 Reciprocal Rank Fusion 或同等级融合算法，将多路召回结果合并为统一候选集。 |
| 10.5 | Enhanced 折回 | 召回命中 `enhanced` 后，应折回对应 `child`，并保留 matched enhanced 信息用于解释和调试。 |
| 10.6 | 关联扩展 | 应支持基于 `related_ids`、相邻关系、父子关系和引用关系扩展候选切片。 |
| 10.7 | 图表编号直达 | 对“图 3-3”“表 2-1”等查询，应优先执行 media_ref 精确召回，直接定位到对应图片或表格切片。 |
| 10.8 | 图谱召回 | 应支持基于实体、entity_mentions、chunk_relations 和 kg_triples 的 Graph RAG 召回与扩展。 |
| 10.9 | 意图路由 | Graph RAG 应支持概念、流程、数据、视觉、通用等意图识别，并按意图调整关系扩展权重。 |
| 10.10 | 召回过滤 | 应支持按知识库、文档、来源、页码范围、标题、是否表格、是否图片、切片层级等条件过滤。 |

## 11. 重排与证据整合参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 11.1 | 候选重排 | 系统应支持对混合召回候选进行统一重排，提升最终上下文质量。 |
| 11.2 | 重排因素 | 重排应综合向量相似度、关键词命中、图表编号命中、切片类型、关系强度、页码邻近度、章节上下文和召回来源。 |
| 11.3 | Parent/Child 对齐 | 重排后应保持 parent 结构上下文和 child 证据切片之间的关联。 |
| 11.4 | 去重机制 | 对同一 `child` 的原生命中和 enhanced 命中，应进行去重与分数合并，避免重复证据污染上下文。 |
| 11.5 | 证据包生成 | 应将重排结果组织为证据包，至少包含原文片段、文件名、页码、切片编号、切片类型、图表编号、图片/表格资源和召回来源。 |
| 11.6 | 上下文构造 | 应基于证据包构造生成上下文，并保留引用编号，便于答案生成后对齐引用。 |
| 11.7 | 引用对齐 | 答案中的引用应能对齐到具体教材证据；若模型未生成引用但存在高置信证据，应支持基于证据的引用补齐或抽取式 fallback。 |

## 12. 问答生成与可信输出参数

| 序号 | 参数项 | 技术要求 |
| --- | --- | --- |
| 12.1 | 基于证据生成 | 答案生成必须基于召回证据，不得脱离知识库编造。 |
| 12.2 | 资料不足处理 | 当召回证据不足、相关度不足或无法支撑回答时，应明确输出资料不足类提示。 |
| 12.3 | 来源引用 | 答案应提供来源引用，包括文档名、页码、切片编号和必要的图表/表格信息。 |
| 12.4 | 图片/表格展示 | 候选证据中如包含图片或表格，应支持在前端或接口中返回对应资源信息，便于用户查看原始证据。 |
| 12.5 | 评分与评测 | 系统应支持对回答质量、证据忠实度、引用覆盖度等进行评分或记录，便于知识库效果评估。 |

## 14. 验收建议指标

| 验收项 | 建议验收方式 |
| --- | --- |
| 大教材解析 | 使用百页级或数百 MB 扫描教材上传，验证系统自动分片、并发解析、全局页码合并和失败定位能力。 |
| 解析多样化 | 使用纯文本 PDF、扫描 PDF、图文混排 PDF、含表格教材、含图片教材分别验证文本、图片、表格和版面信息输出。 |
| 三层切片 | 抽查入库结果中是否存在 parent、child、enhanced 三层结构，并验证 enhanced 是否能折回 child。 |
| 图片/表格保留 | 查询教材图片和表格，验证图片路径、表格内容、图表编号、页码和来源文件是否完整。 |
| 向量化与入库 | 检查 chunks、embedding、chunk_relations、kg_triples、entities、entity_mentions 是否一致写入。 |
| 图表编号直达 | 使用“图 x-x”“表 x-x”类问题验证是否能直接定位到对应图片或表格切片。 |
| 混合召回 | 使用概念解释、章节问题、图表问题、流程问题、因果问题分别验证 dense、BM25、media_ref、related、Graph RAG 能力。 |
| 重排与证据 | 检查最终答案证据是否来自高相关切片，是否包含文件名、页码和切片编号。 |
| 答案可信度 | 构造知识库无答案问题，验证系统是否明确提示资料不足，避免无依据编造。 |
| 性能观测 | 检查各阶段是否输出 parse/chunk/embedding/export/retrieval/rerank/generation 等耗时和指标。 |

## 15. 产品亮点表述

本产品技术路线面向教材知识库场景设计，区别于传统“PDF 转文本 + 向量库检索”的简单方案。系统通过大教材分片解析、多 provider 解析、多模态内容保留和三层切片，将教材内容组织为可检索、可引用、可图谱扩展的知识单元；通过向量召回、关键词召回、图表编号直达召回、关系扩展召回和 Graph RAG 召回提升命中质量；通过重排、证据整合和引用回溯保障问答输出可信可查。

当前验证链路已覆盖大教材 Document Mind 分片解析、`markdown + visualLayoutInfo` 结构化输出、parent/child/enhanced 三层切片、文本/表格/图片增强、embedding 批量向量化、pgvector 写入、普通 RAG 与 Graph RAG 召回、图表编号直达、证据回溯和控制台进度观测，具备作为产品技术招标参数的完整链路基础。
