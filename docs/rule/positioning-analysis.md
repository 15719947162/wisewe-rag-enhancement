# 项目定位分析：调整后 vs 常见知识库

> 配套文档：`docs/rule/hierarchical-chunking.md`、`docs/rule/hierarchical-chunking-extensions.md`
> 评估对象：在落地 6 个扩展方向（关系类型化 / 跨章节语义边 / 实体层 / 三元组 / 流程链 / Graph RAG）后的项目

---

## 对手分类

先把对手分清楚，否则比较没意义：

| 阵营 | 代表 | 特征 |
|---|---|---|
| RAG 框架库 | LangChain / LlamaIndex | 是积木，不是产品 |
| 企业知识库产品 | Dify / FastGPT / RagFlow / AnythingLLM | 应用层完整（用户 / 权限 / Workflow / Agent） |
| GraphRAG 类 | 微软 GraphRAG / HippoRAG / Neo4j LLM-Graph | 重在图谱构建与社区检测 |
| 垂直高质量 RAG | Vectara / Cohere RAG | 闭源 API，质量高但黑盒 |

**调整后这个项目的真实定位**：PDF 教材场景的「高质量切片 + 关系图谱 + 多路检索」实验型知识库底座，介于 RagFlow（专注解析 + 切片）和 GraphRAG（专注图谱）之间。

---

## 优势（赢在哪）

| 维度 | 这个项目 | 常见知识库 |
|---|---|---|
| **解析质量** | MinerU 云端，表格 / 公式 / 版面 / 多栏全保留 | 多数用 PyPDF / PDFPlumber / Unstructured，表格和公式经常烂 |
| **切片策略多样性** | 6 种策略可横向对比，每种带评测 | Dify / FastGPT 通常只给「按字符 + 重叠」一种 |
| **三层结构** | parent 提供章节上下文，child 检索，enhanced 辅助召回 | 大多只有 flat chunk，靠 metadata 拼接父级 |
| **教材垂直优化** | 操作类放宽切片、章节继承、图表引用号识别 | 通用方案，不分文档类型 |
| **关系类型化** | 边带 `rel_type / weight / evidence`，可审计 | RagFlow 有 raptor，但关系无类型；多数无关系 |
| **LLM 性价比** | enhanced 一次调用产出 4 类产物（摘要 + 问题 + 实体 + 三元组） | 大多分多次调，token 浪费 |
| **混合检索** | embedding + BM25 + entity + 图扩展 + 意图路由 | 多数只做 hybrid（embedding + BM25），无图、无路由 |
| **可解释性** | 命中链路清晰：seed chunk → 关系类型 → 邻居 → entity 定义 | 黑盒向量召回，调试靠玄学 |
| **质量门控** | 规则 + LLM 双层过滤，脏数据进不了库 | 基本无质量门控，垃圾入库 |

**核心差异化**：「把 PDF 拆成图，而不是拆成块」—— 这是多数知识库还没做到的事。

---

## 短板（差在哪 / 真正的硬伤）

### 一、产品化能力（与 Dify / FastGPT 比）

| 短板 | 影响 |
|---|---|
| 无用户体系 / 权限 / 多租户 | 不能直接给团队 / 客户用 |
| 无 Agent / Workflow 层 | 只能「建库 + 查」，不能编排业务流程 |
| 多知识库管理弱 | 项目级是单文档管线，`knowledge_base` 表只是基础 CRUD |
| 无前端运营后台 | 数据维护、问答日志、用户反馈闭环缺失 |

### 二、规模与运维（与生产级比）

| 短板 | 影响 |
|---|---|
| pgvector 适合 ≤ 百万向量；千万级要换 Milvus / Qdrant | 上量后性能塌陷 |
| **无增量更新流程** | 改一页 PDF → 全文重建 |
| 无版本管理 / 回滚 | 一次错误增强污染全库 |
| 解析 + 增强一本书可能数小时 + 数十美元 | 冷启动慢、费用高 |
| 链路长：302.ai + OSS + LLM + Embedding + pgvector | 任一节点挂全链路降级 |
| 无缓存层（query embedding、增强结果） | 重复成本 |

### 三、检索能力的隐性短板

| 短板 | 影响 |
|---|---|
| **没有 reranker**（规划但未落地） | 召回 top20 → top5 这一步全靠原始相似度 |
| 多路召回融合方案缺位（RRF / weighted） | 几路结果怎么合，没设计 |
| 图扩展拖慢延迟 | 1~2 跳扩展在大库下 P95 容易爆 |
| 实体消歧只靠字面 + embedding | 没有 Wikidata / 领域 KB 链接，同名异义难处理 |
| 三元组只用于 RAG 增强，**不能独立查询** | 没有 NL2Cypher / SPARQL，知识图谱「读不出来」 |
| 无 ontology（本体） | Entity 只有类型枚举，无层级关系，推理能力有限 |

### 四、多模态与多语言

| 短板 | 影响 |
|---|---|
| 仅 PDF 一种入口 | Word / PPT / Excel / 网页 / 视频 / 音频不支持 |
| 图片靠 VL 描述 → 文本召回 | 真正的「以图搜图」无法做 |
| 中文场景为主，多语言切片 / embedding 未优化 | 国际化项目用不了 |

### 五、评测与持续迭代

| 短板 | 影响 |
|---|---|
| 无 RAGAS / TruLens / DeepEval 接入 | 策略对比靠人工肉眼 |
| 无问答日志回流 → 难例挖掘 | 无法持续优化召回 |
| 无 A/B 框架 | 新策略上线靠拍脑袋 |

---

## 一句话定位

| 场景 | 适合 / 不适合 |
|---|---|
| ✅ 教材 / 手册 / 标准类 PDF 的高质量知识库底座 | 强烈推荐 |
| ✅ 研究「切片策略 / 关系建模 / Graph RAG」对比的实验平台 | 行业内少有的开放方案 |
| ✅ 给上层 Agent / Dify / 业务系统当数据源 | 适合做「管道」而非「产品」 |
| ❌ 直接交付企业终端用户用的对话产品 | 缺产品层 |
| ❌ 千万级文档、数十万用户的生产部署 | 架构需大改 |
| ❌ 多模态 / 多语言 / 多格式综合知识库 | 不在覆盖范围 |

## 战略层面的建议

如果继续演进，**有三条不冲突的路**，但选错路会浪费精力：

### 路线 1（推荐）：走深 —— 教材 / 培训垂直 + Graph RAG 极致化

- 当行业最好用的「教材知识库引擎」
- 不要去和 Dify 拼产品广度
- 差异化壁垒最高

### 路线 2：走宽 —— 补齐多格式 + 用户体系 + Workflow

- 重做成 Dify 替代品
- 工作量极大且红海
- 几乎肯定打不过头部产品

### 路线 3：走外接 —— Pipeline as a Service

- 把这个项目定位为「知识库构建管道」
- 输出标准接口给 Dify / FastGPT / Agent 用
- 与现有产品互补，不竞争
- 作为商业模式的备选很务实

**按当前代码的气质和 6 个扩展方向的设计，路线 1 最有机会做成差异化产品。**

---

## 与扩展方向的对应关系

下表说明每个扩展方向消除了哪些短板：

| 扩展方向 | 消除的短板 | 强化的优势 |
|---|---|---|
| 方向 1 关系类型化 | 关系无类型、可解释性差 | 可审计的知识图 |
| 方向 2 跨章节语义边 | 跨章节关联缺失 | 主题链贯穿 |
| 方向 3 实体层 | 术语 / 型号召回弱、跨章节贯穿弱 | 精确召回 + 跨文档收敛 |
| 方向 4 三元组抽取 | LLM 性价比低 | 知识图谱原料零成本 |
| 方向 5 流程 / 因果链 | 操作类问题召回不完整 | 教材垂直深化 |
| 方向 6 Graph RAG 检索 | 无图扩展、无意图路由 | 多路召回 + 推理能力 |

未消除（仍是短板）的有：

- 产品化层（用户 / 权限 / Workflow）
- 增量更新 / 版本管理
- reranker / 多路融合算法
- 多模态 / 多语言 / 多格式
- 评测体系（RAGAS / 难例回流）

→ 这些是**下一阶段**（Phase 7+）需要规划的方向。
